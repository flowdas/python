자유 스레딩에 대한 파이썬 지원¶

3.13 버전부터 CPython은 무료 스레딩 (GIL)을 가진 Python 빌드를 지원합니다. 자유 스레딩 실행은 사용 가능한 CPU 코어에서 스레드를 병렬로 실행하여 가용 처리 능력을 온전히 활용할 수 있게 합니다. 모든 소프트웨어가 자동으로 이점을 갖는 것은 아니지만, 스레딩을 염두에 두고 설계된 프로그램은 다중 코어 하드웨어에서 더 빠르게 실행됩니다.

일부 서드파티 패키지, 특히 :term:`확장 모듈 <extension module>`을 가진 패키지는 자유 스레딩 빌드에서 사용하기 위해 준비되지 않았을 수 있으며, :term:`GIL`을 다시 활성화할 수 있습니다.

이 문서는 Python 코드에서 자유 스레딩이 갖는 영향을 설명합니다. 자유 스레딩 빌드를 지원하는 C 확장을 작성하는 방법에 대한 내용은 :ref:`freethreading-extensions-howto`를 참조하십시오.

더 보기

PEP 703 – CPython에서 글로벌 인터프리터 록을 선택 사항으로 만드는 방법에 대한 자유 스레딩 Python 전반의 설명.

설치 방법¶

Python 3.13 부터, 공식 macOS 및 Windows 설치 프로그램은 선택적으로 자유 스레딩 Python 바이너리를 설치하는 것을 지원합니다. 설치 프로그램은 https://www.python.org/downloads/ 에서 사용할 수 있습니다.

다른 플랫폼에 대한 정보는 자유 스레딩 Python 설치를 위한 커뮤니티 유지보수 설치 가이드인 Installing a Free-Threaded Python <https://py-free-threading.github.io/installing-cpython/>\를 참조하십시오.

CPython을 소스에서 빌드할 때, 자유 스레딩 Python 인터프리터 빌드를 위해 --disable-gil 구성 옵션을 사용해야 합니다.

자유 스레딩 Python 식별하기¶

현재 인터프리터가 자유 스레딩을 지원하는지 확인하려면 python -VV 및 sys.version`에 "free-threading build"가 포함되어 있는지 확인하십시오. 새 :func:`sys._is_gil_enabled 함수를 사용하면 실행 중인 프로세스에서 GIL이 실제로 비활성화되었는지 확인할 수 있습니다.

The sysconfig.get_config_var("Py_GIL_DISABLED") configuration variable can be used to determine whether the build supports free threading. If the variable is set to 1, then the build supports free threading. This is the recommended mechanism for decisions related to the build configuration.

자유 스레딩 Python의 글로벌 인터프리터 록¶

CPython의 자유 스레딩 빌드는 환경 변수 PYTHON_GIL 또는 명령줄 옵션 :option:`-X gil`을 사용하여 런타임에 GIL 활성화를 선택적으로 지원합니다.

또한, 명시적으로 자유 스레딩을 지원한다고 표시되지 않은 C-API 확장 모듈을 임포트할 때 GIL이 자동으로 활성화될 수도 있습니다. 이 경우 경고가 출력됩니다.

개별 패키지 문서 외에도, 다음 웹사이트들이 인기 패키지가 자유 스레딩을 지원하는 상태를 추적합니다:

  • https://py-free-threading.github.io/tracking/

  • https://hugovk.github.io/free-threaded-wheels/

스레드 안전성¶

CPython의 자유 스레딩 빌드는 기본 GIL 활성화 빌드와 유사한 스레드 안전성을 Python 수준에서 제공하는 것을 목표로 합니다. dict, list, :class:`set`과 같은 내장형 데이터 타입은 내부 잠금(internal locks)을 사용하여 동시 수정에 대한 보호를 수행하며, 이는 GIL과 유사하게 동작합니다. 그러나 Python은 역사적으로 이러한 내장형 타입의 동시 수정에 대한 특정 동작을 보장하지 않았으므로, 이는 현재 또는 미래의 동작을 보장하는 것이 아니라 현재 구현에 대한 설명으로 간주해야 합니다.

참고

가능하다면, 기본형 타입의 내부 락에 의존하기보다는 threading.Lock 또는 다른 동기화 프리미티브를 사용하는 것이 좋습니다.

알려진 제한 사항¶

이 섹션에서는 자유 스레딩 CPython 빌드의 알려진 제한 사항을 설명합니다.

불멸화¶

자유 스레딩 빌드에서는 일부 객체가 :term:`immortal <immortal>``입니다. 불멸 객체는 해제되지 않으며 참조 횟수가 변경되지 않는 참조 횟수를 갖습니다. 이는 효율적인 멀티스레딩 확장을 막을 수 있는 참조 횟수 경합을 피하기 위해 수행됩니다.

3.14 릴리스 기준, 불멸화는 다음으로 제한됩니다:

  • 코드 상수: 숫자 리터럴, 문자열 리터럴 및 다른 상수들로 구성된 튜플 리터럴.

  • :func:`sys.intern`에 의해 인턴화된 문자열.

프레임 객체¶

프레임이 다른 스레드에서 실행 중인 경우, frame 객체에서 :attr:`frame.f_locals`에 접근하는 것은 안전하지 않으며, 그렇게 하면 인터프리터를 충돌시킬 수 있습니다.

이터레이터¶

여러 스레드에서 동일한 이터레이터 객체에 동시에 접근하는 것은 일반적으로 스레드 안전하지 않으며, 스레드는 중복되거나 누락된 요소를 볼 수 있습니다.

단일 스레드 성능¶

자유 스레딩 빌드는 기본 GIL 활성화 빌드와 비교하여 파이썬 코드를 실행할 때 추가적인 오버헤드를 가집니다. 오버헤드의 양은 워크로드와 하드웨어에 따라 다릅니다. pyperformance 벤치마크 스위트에서 평균 오버헤드는 macOS aarch64의 경우 약 1%에서 x86-64 Linux 시스템의 8%까지 변합니다.

동작 변경¶

이 섹션에서는 자유 스레딩 빌드와 관련된 CPython 동작 변경 사항을 설명합니다.

컨텍스트 변수¶

자유 스레딩 빌드에서는 플래그 Context() 사본으로 시작되게 합니다. 기본 GIL 활성화 빌드에서는 이 플래그가 기본적으로 false이므로 스레드는 빈 :class:`~contextvars.Context()`로 시작합니다.

경고 필터¶

자유 스레딩 빌드에서는 플래그 catch_warnings 컨텍스트 관리자는 경고 필터에 컨텍스트 변수를 사용합니다. 플래그가 false이면 자세한 내용은 :mod:`warnings 모듈을 참조하세요.

메모리 사용량 증가¶

자유 스레딩 빌드는 일반적으로 기본 빌드보다 더 많은 메모리를 사용할 것입니다. 여기에는 여러 가지 이유가 있으며, 주로 설계 결정에서 비롯된 것입니다.

모든 인턴화된 문자열은 불멸합니다¶

최신 파이썬 버전(버전 2.3 이후)에서 문자열을 인턴화(예: :func:`sys.intern`를 사용)하는 것이 그것을 불멸로 만들지는 않습니다. 대신, 해당 문자열에 대한 마지막 참조가 사라지면, 그것은 인턴화된 문자열 테이블에서 제거됩니다. 이는 자유 스레딩 빌드와 달리, 모든 인턴화된 문자열은 인터프리터 종료 시까지 남아 불멸화됩니다.

비 GC 객체는 더 큰 객체 헤더를 가집니다¶

자유 스레딩 빌드는 다른 PyObject 구조를 사용합니다. 기본 빌드와 달리 GC 관련 정보가 PyObject 구조 앞에 할당되는 대신, GC 관련 정보는 일반 객체 헤더의 일부를 이룹니다. 예를 들어, AMD64 플랫폼에서 None 은 자유 스레딩 빌드에서 32바이트를, 기본 빌드에서 16바이트를 사용합니다. GC 객체(딕셔너리 및 리스트와 같은)는 자유 스레딩 빌드가 GC 정보에 추가 공간을 사용하지 않기 때문에 두 빌드에서 동일한 크기입니다.

QSBR이 메모리 해제를 지연시킬 수 있습니다¶

충돌 없는 자료 구조를 안전하게 구현하기 위해서는 SMR(Safe Memory Reclamation) 기법, 즉 quiescent state 기반 재할당(QSBR) 방식이 사용됩니다. 이는 데이터에 접근할 때 충돌이 없도록 하는 메모리 영역이 즉시 메모리를 해제하는 대신 QSBR을 사용하여 free 연산을 지연시키기 때문입니다. 이러한 자료 구조의 예시로 리스트 객체와 딕셔너리 키 객체가 있습니다. QSBR이 어떻게 구현되는지에 대한 더 자세한 내용은 CPython 소스 트리 내의 InternalDocs/qsbr.md 를 참조하십시오. gc.collect() 를 실행하면 QSBR에 의해 보유된 모든 메모리가 실제로 해제되어야 합니다. QSBR이 메모리를 해제할 때에도, 밑단 메모리 할당자가 즉시 해당 메모리를 OS에 반환하지 않을 수 있으므로, 프로세스의 상주 집합 크기(RSS)가 감소하지 않을 수도 있다는 점에 유의하십시오.

mimalloc 할당자 대 pymalloc 할당자¶

기본 빌드는 일반적으로 작은 할당(512바이트 이하)에는 “pymalloc” 메모리 할당자를 사용합니다. free-threaded 빌드는 pymalloc을 사용하지 않으며 모든 Python 객체는 “mimalloc” 할당자를 사용하여 할당됩니다. pymalloc 할당자는 메모리 사용량을 낮게 유지하는 데 도움이 되는 다음 속성들을 가지고 있습니다: 할당 블록당 오버헤드가 작음, 효과적인 메모리 단편화 방지, 그리고 해제된 메모리를 운영 체제에 빠르게 반환합니다. mimalloc 할당자 또한 이 측면에서 훌륭하게 동작하지만, 약간 더 많은 오버헤드를 가질 수 있습니다.

free-threaded 빌드에서는 mimalloc이 여러 개의 분리된 힙(현재 네 개)에 걸쳐 메모리를 관리합니다. 예를 들어, 모든 GC 지원 객체는 자체 힙에서 할당됩니다. 별도의 힙을 사용한다는 것은 한 힙에서 해제된 메모리가 다른 힙을 사용하는 할당에 사용될 수 없다는 것을 의미합니다. 또한, 일부 힙들은 힙의 메모리를 해제할 때(mimalloc 용어로는 “페이지”라고 함) QSBR(quiescent-state based reclamation)을 사용하도록 설정되어 있습니다. QSBR의 사용은 페이지가 해제되는 모든 메모리 블록과 해당 메모리 페이지가 해제되는 시점(새로운 할당이나 OS로 반납되는 경우) 사이에 지연을 만듭니다.

mimalloc 할당자 또한 해제된 메모리를 OS에 반환하는 것을 지연시킵니다. 환경 변수 MIMALLOC_PURGE_DELAY 를 0 으로 설정하여 이 지연을 줄일 수 있습니다. 하지만 이 경우 할당자의 성능이 저하될 가능성이 높다는 점에 유의하십시오.

free-threaded 참조 횟수는 객체가 더 오래 생존하게 할 수 있습니다¶

기본 빌드에서는 객체의 참조 횟수가 0에 도달하면 일반적으로 할당이 해제됩니다. free-threaded 빌드는 현재 스레드가 “소유”하는 객체를 위한 빠른 경로와 다른 객체들을 위한 느린 경로를 갖는 “편향 참조 횟수(biased reference counting)”를 사용합니다. 자세한 내용은 :pep:`703`을 참조하십시오. 객체의 참조 횟수가 “큐에 대기(queued)” 상태에 놓이는 순간마다 할당 해제가 지연될 수 있습니다. 큐에 대기 상태는 바이트코드 평가기(bytecode evaluator)의 “평가 중단기(eval breaker)” 섹션에서 초기화됩니다.

free-threaded 빌드는 또한 “지연 참조 횟수(deferred reference counting)”라는 다른 방식의 참조 횟수 계산을 허용합니다. 이 모드는 객체별로 플래그를 설정하여 활성화할 수 있습니다. 지연 참조 횟수는 다음 유형에 대해 활성화됩니다:

  • 모듈 객체

  • 모듈 최상위 함수

  • 클래스 범위에 정의된 클래스 메서드

  • 디스크립터 객체

  • :class:`threading.local`에 의해 생성된 스레드별 객체

지연 참조 횟수가 활성화되면, Python 함수 스택의 참조는 참조 횟수에 추가되지 않습니다. 이 방식은 특히 여러 스레드에서 사용되는 객체의 참조 횟수 오버헤드를 줄여줍니다. 스택 참조가 계산되지 않기 때문에, 지연 참조 횟수를 가진 객체들은 내부 참조 횟수가 0이 되더라도 즉시 해제되지 않습니다. 대신, 다음 GC 주기에서 검사되며, 해당 객체에 스택 참조가 없으면 해제됩니다. 이는 일반적인 경우와 달리, 이 객체들이 참조 횟수가 0이 될 때가 아니라 GC에 의해 해제됨을 의미합니다.

스레드별 참조 횟수는 객체의 해제를 지연시킬 수 있습니다¶

자주 공유되는 객체의 참조 카운트 필드에 대한 경쟁을 방지하기 위해, 자유 스레드 빌드도 선택된 일부 객체 유형에 대해 “스레드별 참조 카운팅”을 사용합니다. 단일 공유 참조 카운트를 업데이트하는 대신, 각 스레드는 자신만의 로컬 참조 카운트 배열을 유지하며, 이 배열은 객체에 할당된 고유한 id로 인덱싱됩니다. 실제 참조 카운트는 객체의 로컬 카운트가 0으로 떨어질 때 스레드별 카운트를 합산하여 계산됩니다. 스레드별 참조 카운팅은 현재 다음을 위해 사용됩니다:

  • 힙 할당 타입 객체 (Python에서 생성된 클래스)

  • 코드 객체

  • 모듈 객체의 __dict__

스레드별 카운트를 객체로 병합해야만 해제할 수 있으므로, 스레드별 참조 카운팅을 사용하는 객체는 일반적으로 기본 빌드보다 더 늦게 해제됩니다. 특히, 그러한 객체는 참조한 스레드가 안전 지점에 도달하거나 종료될 때까지 일반적으로 해제되지 않습니다. :func:`gc.collect`를 실행하면 스레드별 카운트가 병합되어 이러한 객체를 해제할 수 있게 됩니다.
