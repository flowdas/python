원격 디버깅 첨부 프로토콜¶

이 프로토콜은 외부 도구가 실행 중인 CPython 프로세스에 연결하여 Python 코드를 원격으로 실행할 수 있게 합니다.

대부분의 플랫폼은 다른 파이썬 프로세스에 연결하려면 높은 권한을 필요로 합니다.

원격 디버깅 비활성화¶

원격 디버깅 지원을 비활성화하려면 다음 중 하나를 사용하십시오:

  • 인터프리터를 시작하기 전에 PYTHON_DISABLE_REMOTE_DEBUG 환경 변수를 1 로 설정합니다.

  • -X disable_remote_debug 명령줄 옵션을 사용하십시오.

  • --without-remote-debug 빌드 플래그를 사용하여 Python을 컴파일하십시오.

권한 요구 사항¶

실행 중인 파이썬 프로세스에 원격 디버깅을 위해 연결하려면 대부분의 플랫폼에서 높은 권한이 필요합니다. 특정 요구 사항 및 문제 해결 단계는 사용자의 운영 체제에 따라 다릅니다:

리눅스

트레이서 프로세스는 CAP_SYS_PTRACE 권한 또는 동등한 권한을 가지고 있어야 합니다. 사용자가 소유하고 신호를 보낼 수 있는 프로세스만 트레이스할 수 있습니다. 프로세스가 이미 트레이싱 중이거나, set-user-ID 또는 set-group-ID를 사용하여 실행되는 경우 트레이싱이 실패할 수 있습니다. Yama와 같은 보안 모듈은 트레이싱을 추가로 제한할 수 있습니다.

임시로 ptrace 제한(재부팅까지)을 완화하려면 다음을 실행하십시오:

echo 0 | sudo tee /proc/sys/kernel/yama/ptrace_scope

참고

ptrace_scope 비활성화는 시스템 강화 수준을 낮추므로 신뢰할 수 있는 환경에서만 수행해야 합니다.

컨테이너 내부에서 실행하는 경우, --cap-add=SYS_PTRACE 또는 --privileged 를 사용하고 필요한 경우 root로 실행하십시오.

권한을 높여 명령을 다시 실행해 보십시오:

sudo -E !!

macOS

다른 프로세스에 연결(attach)하려면 일반적으로 디버깅 도구를 높은 권한(elevated privileges)으로 실행해야 합니다. 이는 sudo 를 사용하거나 root로 실행하여 수행할 수 있습니다.

소유한 프로세스에 연결할 때도, 시스템 보안 제한 때문에 디버거가 root 권한으로 실행되지 않으면 macOS에서 디버깅을 차단할 수 있습니다.

Windows

다른 프로세스에 연결하려면, 일반적으로 디버깅 도구를 관리자 권한으로 실행해야 합니다. 명령 프롬프트나 터미널을 관리자로 시작하십시오.

일부 프로세스는 : SeDebugPrivilege 권한이 활성화되어 있지 않으면 관리자 권한으로도 액세스가 불가능할 수 있습니다.

파일이나 폴더 액세스 문제를 해결하려면, 보안 권한을 조정하십시오:

  1. 파일이나 폴더를 마우스 오른쪽 버튼으로 클릭하고 속성 을 선택합니다.

  2. 사용자 및 액세스 그룹을 보려면 보안 탭으로 이동합니다.

  3. 권한을 수정하려면 편집 을 클릭합니다.

  4. 사용자 계정을 선택합니다.

  5. 권한 에서 필요한 대로 읽기 또는 모든 권한 을 확인합니다.

  6. 적용 을 클릭한 다음, 확인하려면 확인 을 클릭합니다.

참고

진행하기 전에 모든 :ref:`permission-requirements`를 충족했는지 확인하십시오.

이 섹션은 외부 도구가 실행 중인 CPython 프로세스 내에 Python 스크립트를 주입하고 실행할 수 있게 하는 낮은 수준의 프로토콜을 설명합니다.

이 메커니즘은 원격 Python 프로세스에게 .py 파일을 실행하도록 지시하는 sys.remote_exec() 함수의 기초를 이룹니다. 하지만 이 섹션은 그 함수의 사용법을 문서화하지 않습니다. 대신, 이 섹션은 대상 Python 프로세스의 pid 와 실행할 Python 소스 파일의 경로를 입력으로 받는 기본 프로토콜에 대한 자세한 설명을 제공합니다. 이 정보는 프로그래밍 언어에 관계없이 프로토콜의 독립적인 재구현을 지원합니다.

경고

주입된 스크립트의 실행은 인터프리터가 안전한 평가 지점에 도달하는지에 달려 있습니다. 그 결과, 실행은 대상 프로세스의 런타임 상태에 따라 지연될 수 있습니다.

주입된 후, 스크립트는 안전한 평가 지점에 도달할 때마다 인터프리터에 의해 대상 프로세스 내에서 실행됩니다. 이 접근 방식은 실행 중인 Python 애플리케이션의 동작이나 구조를 수정하지 않고 원격 실행 기능을 가능하게 합니다.

뒤따르는 절에서는 메모리에서 인터프리터 구조를 찾는 기술, 내부 필드를 안전하게 접근하는 방법, 코드 실행을 트리거하는 방법 등을 포함하여 프로토콜에 대한 단계별 설명을 제공합니다. 해당되는 경우에는 플랫폼별 변형 사항이 명시되며, 각 작업을 명확히 하기 위해 예제 구현이 포함됩니다.

PyRuntime 구조 찾기¶

CPython은 외부 도구가 런타임에 해당 구조를 찾을 수 있도록 PyRuntime 구조를 전용 바이너리 섹션에 배치합니다. 이 섹션의 이름과 형식은 플랫폼마다 다릅니다. 예를 들어, ELF 시스템에서는 .PyRuntime 이 사용되고, macOS에서는 __DATA,__PyRuntime 이 사용됩니다. 도구는 디스크의 바이너리를 검사하여 이 구조의 오프셋을 찾을 수 있습니다.

PyRuntime 구조는 CPython의 전역 인터프리터 상태를 포함하고 있으며, 인터프리터 목록, 스레드 상태, 디버거 지원 필드를 포함한 다른 내부 데이터에 대한 접근을 제공합니다.

원격 Python 프로세스를 사용하려면 디버거가 먼저 대상 프로세스 내에서 PyRuntime 구조의 메모리 주소를 찾아야 합니다. 이 주소는 운영체제가 바이너리를 로드한 위치에 따라 달라지기 때문에 하드코딩하거나 심볼 이름으로부터 계산할 수 없습니다.

PyRuntime 을 찾는 방법은 플랫폼에 따라 다르지만, 단계는 일반적으로 동일합니다:

  1. 대상 프로세스에서 Python 바이너리 또는 공유 라이브러리가 로드된 기본 주소를 찾습니다.

  2. 디스크에 있는 바이너리를 사용하여 .PyRuntime 섹션의 오프셋을 찾습니다.

  3. 식별된 섹션 오프셋을 기본 주소에 더하여 메모리 주소를 계산합니다.

아래 섹션들은 지원되는 각 플랫폼별로 이 작업을 수행하는 방법과 예제 코드를 설명합니다.

Linux (ELF)

Linux에서 PyRuntime 구조를 찾으려면:

  1. 프로세스의 메모리 맵(예: /proc/<pid>/maps)을 읽어서 Python 실행 파일 또는 libpython 이 로드된 주소를 찾습니다.

  2. 바이너리 내의 ELF 섹션 헤더를 구문 분석하여 .PyRuntime 섹션의 오프셋을 얻습니다.

  3. 이 오프셋을 단계 1의 기본 주소에 더하여 PyRuntime 의 메모리 주소를 얻습니다.

다음은 예제 구현입니다:

def find_py_runtime_linux(pid: int) -> int:
    # 단계 1: 메모리에서 Python 실행 파일을 찾으려고 시도합니다
    binary_path, base_address = find_mapped_binary(
        pid, name_contains="python"
    )

    # 단계 2: 실행 파일이 없으면 공유 라이브러리로 대체합니다
    if binary_path is None:
        binary_path, base_address = find_mapped_binary(
            pid, name_contains="libpython"
        )

    # 단계 3: ELF 헤더를 구문 분석하여 .PyRuntime 섹션 오프셋을 얻습니다
    section_offset = parse_elf_section_offset(
        binary_path, ".PyRuntime"
    )

    # 단계 4: 메모리에서 PyRuntime 주소 계산
    return base_address + section_offset

리눅스 시스템에서는 다른 프로세스로부터 메모리를 읽는 두 가지 주요 접근 방식이 있습니다. 첫 번째는 /proc 파일 시스템을 통해, 구체적으로 /proc/[pid]/mem``에서 읽어 프로세스의 메모리에 직접 접근하는 것입니다. 이를 위해서는 적절한 권한(대상 프로세스와 동일한 사용자가거나 root 접근이 필요함)이 필요합니다. 두 번째 접근 방식은 프로세스 간에 메모리를 복사하는 보다 효율적인 방법인 ``process_vm_readv() 시스템 호출을 사용하는 것입니다. ptrace`의 ``PTRACE_PEEKTEXT` 작업으로도 메모리를 읽을 수 있지만, 현재는 한 워드씩만 읽고 트레이서와 트레이스이 프로세스 간 여러 컨텍스트 전환이 필요하기 때문에 훨씬 느립니다.

ELF 섹션을 파싱하는 과정은 디스크의 바이너리 파일에서 ELF 파일 형식 구조를 읽고 해석하는 것을 포함합니다. ELF 헤더에는 섹션 헤더 테이블에 대한 포인터가 포함되어 있습니다. 각 섹션 헤더는 이름(별도의 문자열 테이블에 저장됨), 오프셋, 크기를 포함하여 섹션에 대한 메타데이터를 담고 있습니다. 예를 들어 `.PyRuntime`과 같은 특정 섹션을 찾으려면, 이러한 헤더를 탐색하여 섹션 이름을 일치시켜야 합니다. 섹션 헤더는 파일 내에 해당 섹션이 존재하는 오프셋을 제공하며, 이 오프셋은 바이너리가 메모리에 로드될 때 런타임 주소를 계산하는 데 사용될 수 있습니다.

ELF 파일 형식에 대한 자세한 내용은 `ELF specification <https://en.wikipedia.org/wiki/Executable_and_Linkable_Format>`에서 읽어보실 수 있습니다.

macOS (Mach-O)

macOS에서 PyRuntime 구조를 찾으려면 다음 단계를 따르세요:

  1. task_for_pid() 를 호출하여 대상 프로세스의 mach_port_t 태스크 포트를 가져오세요. 이 핸들은 mach_vm_read_overwrite 및 mach_vm_region 과 같은 API를 사용하여 메모리를 읽는 데 필요합니다.

  2. 메모리 영역을 스캔하여 Python 실행 파일 또는 libpython 을 포함하는 영역을 찾으세요.

  3. 디스크에서 바이너리 파일을 로드하고 Mach-O 헤더를 파싱하여 __DATA 세그먼트에서 PyRuntime 이라는 이름을 가진 섹션을 찾으세요. macOS에서는 심볼 이름이 자동으로 언더스코어(underscore)로 접두사가 붙기 때문에, PyRuntime 심볼은 심볼 테이블에 _PyRuntime 으로 나타나지만, 섹션 이름은 영향을 받지 않습니다.

다음은 예제 구현입니다:

def find_py_runtime_macos(pid: int) -> int:
    # 1단계: 프로세스의 메모리에 접근합니다.
    handle = get_memory_access_handle(pid)

    # 2단계: 메모리에서 Python 실행 파일을 찾으려고 시도합니다.
    binary_path, base_address = find_mapped_binary(
        handle, name_contains="python"
    )

    # 3단계: 실행 파일을 찾지 못한 경우 libpython으로 폴백합니다.
    if binary_path is None:
        binary_path, base_address = find_mapped_binary(
            handle, name_contains="libpython"
        )

    # 4단계: Mach-O 헤더를 파싱하여 __DATA,__PyRuntime 섹션 오프셋을 얻습니다.
    section_offset = parse_macho_section_offset(
        binary_path, "__DATA", "__PyRuntime"
    )

    # 5단계: 메모리 내 PyRuntime 주소를 계산하여 반환합니다.
    return base_address + section_offset

macOS에서 다른 프로세스의 메모리에 접근하려면 Mach-O 전용 API와 파일 형식을 사용해야 합니다. 첫 번째 단계는 task_for_pid() 를 통해 task_port 핸들을 얻는 것이며, 이는 대상 프로세스의 메모리 공간에 접근할 수 있게 해줍니다. 이 핸들은 mach_vm_read_overwrite() 와 같은 API를 통해 메모리 작업을 가능하게 합니다.

프로세스 메모리는 mach_vm_region()``을 사용하여 가상 메모리 공간을 스캔하여 검사할 수 있으며, ``proc_regionfilename()``은 각 메모리 영역에 로드된 바이너리 파일을 식별하는 데 도움이 됩니다. Python 바이너리나 라이브러리를 찾으면, 해당 Mach-O 헤더를 파싱하여 ``PyRuntime 구조를 찾게 됩니다.

Mach-O 형식은 코드와 데이터를 세그먼트와 섹션으로 구성합니다. PyRuntime 구조는 __DATA 세그먼트 내에 __PyRuntime``이라는 이름을 가진 섹션에 존재합니다. 실제 런타임 주소 계산은 바이너리의 기본 주소 역할을 하는 ``__TEXT 세그먼트를 찾고, 그런 다음 대상 섹션을 포함하는 __DATA 세그먼트를 찾는 것을 포함합니다. 최종 주소는 기본 주소에 Mach-O 헤더의 적절한 섹션 오프셋을 결합하여 계산됩니다.

macOS에서 다른 프로세스의 메모리에 접근하려면 일반적으로 높은 권한이 필요하다는 점에 유의하세요. 즉, 루트 접근 권한이거나 디버깅 프로세스에 부여된 특별한 보안 권한이 필요합니다.

Windows (PE)

Windows에서 PyRuntime 구조를 찾으려면 다음 단계를 따르세요:

  1. ToolHelp API를 사용하여 대상 프로세스에 로드된 모든 모듈을 열거하세요. 이는 CreateToolhelp32Snapshot …, Module32First …, 및 `Module32Next …`와 같은 함수를 사용하여 수행됩니다.

  2. python.exe 또는 :file:`python{XY}.dll`에 해당하는 모듈을 식별하고, 여기에 기록된 기본 주소를 기록합니다.

  3. PyRuntim 섹션을 찾습니다. PE 형식은 섹션 이름에 8자 제한이 있기 때문에 (IMAGE_SIZEOF_SHORT_NAME``으로 정의됨), 원래 이름 ``PyRuntime``이 잘렸습니다. 이 섹션에는 ``PyRuntime 구조가 포함되어 있습니다.

  4. 섹션의 상대 가상 주소(RVA)를 검색하여 모듈의 기본 주소에 더합니다.

다음은 예제 구현입니다:

def find_py_runtime_windows(pid: int) -> int:
    # 단계 1: 메모리에서 Python 실행 파일을 찾으려고 시도합니다.
    binary_path, base_address = find_loaded_module(
        pid, name_contains="python"
    )

    # 단계 2: 실행 파일을 찾지 못한 경우 shared pythonXY.dll로 폴백합니다.
    if binary_path is None:
        binary_path, base_address = find_loaded_module(
            pid, name_contains="python3"
        )

    # 단계 3: PE 섹션 헤더를 파싱하여 PyRuntime 섹션의 RVA를 가져옵니다. 섹션 이름은 PE 형식에 의해 정의된 8자 제한 때문에 "PyRuntim"으로 나타납니다.
    section_rva = parse_pe_section_offset(binary_path, "PyRuntim")

    # 단계 4: 메모리에서 PyRuntime 주소 계산
    return base_address + section_rva

Windows에서 다른 프로세스의 메모리에 접근하려면 CreateToolhelp32Snapshot() 및 Module32First()/Module32Next() 와 같은 Windows API 함수를 사용하여 로드된 모듈을 열거해야 합니다. OpenProcess() 함수는 대상 프로세스의 메모리 공간에 접근할 핸들을 제공하며, 이를 통해 ReadProcessMemory() 를 사용하여 메모리 작업을 수행할 수 있습니다.

프로세스 메모리는 로드된 모듈을 열거하여 Python 바이너리나 DLL을 찾음으로써 검사할 수 있습니다. 발견되면, PyRuntime 구조를 찾기 위해 해당 PE 헤더를 파싱해야 합니다.

PE 형식은 코드와 데이터를 섹션으로 구성합니다. PyRuntime 구조는 “PyRuntim”이라는 섹션에 존재하며 (IE PE의 8자 이름 제한 때문에 “PyRuntime”에서 잘렸습니다), 실제 런타임 주소 계산에는 모듈 진입점으로부터 모듈의 기본 주소를 찾은 다음, PE 헤더에서 대상 섹션을 찾아야 합니다. 최종 주소는 기본 주소와 PE 섹션 헤더의 섹션 가상 주소를 결합하여 계산됩니다.

Windows에서 다른 프로세스의 메모리에 접근하려면 일반적으로 적절한 권한이 필요하다는 점에 유의하십시오. 이는 관리자 액세스이거나 디버깅 프로세스에 부여된 SeDebugPrivilege 권한이어야 합니다.

_Py_DebugOffsets 읽기¶

PyRuntime 구조의 주소가 결정되면, 다음 단계는 PyRuntime 블록의 시작 부분에 위치한 _Py_DebugOffsets 구조를 읽는 것입니다.

이 구조는 인터프리터 및 스레드 상태 메모리를 안전하게 읽는 데 필요한 버전별 필드 오프셋을 제공합니다. 이러한 오프셋은 CPython 버전에 따라 다르며, 호환성을 확인하기 위해 사용 전에 반드시 확인해야 합니다.

디버그 오프셋을 읽고 확인하려면 다음 단계를 따르십시오:

  1. 대상 프로세스의 PyRuntime 주소부터 시작하여 _Py_DebugOffsets 구조와 동일한 바이트 수를 커버하도록 메모리를 읽습니다. 이 구조는 PyRuntime 메모리 블록의 맨 처음에 위치합니다. 레이아웃은 CPython의 내부 헤더에 정의되어 있어 주어진 마이너 버전 내에서는 동일하게 유지되지만, 메이저 버전에서는 변경될 수 있습니다.

  2. 구조가 유효한 데이터가 포함되어 있는지 확인합니다:

    • cookie 필드는 예상되는 디버그 마커와 일치해야 합니다.

    • version 필드는 디버거가 사용하는 Python 인터프리터의 버전과 일치해야 합니다.

    • 디버거나 대상 프로세스 중 하나가 사전 릴리스 버전(예: 알파, 베타 또는 릴리스 후보)을 사용하는 경우, 버전이 정확히 일치해야 합니다.

    • free_threaded 필드는 디버거와 대상 프로세스 모두에서 동일한 값을 가져야 합니다.

  3. 구조가 유효하면, 포함된 오프셋을 사용하여 메모리 내 필드를 찾을 수 있습니다. 확인 중 오류가 발생하면, 잘못된 형식의 메모리 읽기를 방지하기 위해 디버거가 작업을 중지해야 합니다.

다음은 _Py_DebugOffsets 를 읽고 확인하는 예시 구현입니다:

def read_debug_offsets(pid: int, py_runtime_addr: int) -> DebugOffsets:
    # Step 1: Read memory from the target process at the PyRuntime address
    data = read_process_memory(
        pid, address=py_runtime_addr, size=DEBUG_OFFSETS_SIZE
    )

    # Step 2: Deserialize the raw bytes into a _Py_DebugOffsets structure
    debug_offsets = parse_debug_offsets(data)

    # Step 3: Validate the contents of the structure
    if debug_offsets.cookie != EXPECTED_COOKIE:
        raise RuntimeError("Invalid or missing debug cookie")
    if debug_offsets.version != LOCAL_PYTHON_VERSION:
        raise RuntimeError(
            "Mismatch between caller and target Python versions"
        )
    if debug_offsets.free_threaded != LOCAL_FREE_THREADED:
        raise RuntimeError("Mismatch in free-threaded configuration")

    return debug_offsets

경고

프로세스 일시 중지 권장

경합 상태를 방지하고 메모리 일관성을 보장하기 위해, 내부 인터프리터 상태를 읽거나 쓰는 작업을 수행하기 전에 대상 프로세스를 일시 중지하는 것이 강력히 권장됩니다. 파이썬 런타임은 정상적인 실행 중에 스레드를 생성하거나 파괴하는 것과 같은 인터프리터 데이터 구조를 동시에 수정할 수 있습니다. 이는 무효한 메모리 읽기나 쓰기를 초래할 수 있습니다.

디버거는 ptrace`를 사용하여 프로세스에 연결하거나 `SIGSTOP 시그널을 보내어 실행을 일시 중지할 수 있습니다. 디버거 측 메모리 작업이 완료된 후에만 실행을 재개해야 합니다.

참고

프로파일러나 샘플링 기반 디버거와 같은 일부 도구는 일시 중지 없이 실행 중인 프로세스에 작동할 수 있습니다. 이러한 경우, 도구는 부분적으로 업데이트되거나 불일치하는 메모리를 처리하도록 명시적으로 설계되어야 합니다. 대부분의 디버거 구현에서, 프로세스 일시 중지 여전히 가장 안전하고 견고한 접근 방식입니다.

인터프리터 및 스레드 상태 찾기¶

원격 파이썬 프로세스에서 코드를 주입하고 실행하기 전에, 디버거는 실행을 예약할 스레드를 선택해야 합니다. 이는 원격 코드 주입을 수행하는 데 사용되는 제어 필드가 _PyRemoteDebuggerSupport 구조체에 위치하며, 이 구조체는 PyThreadState 객체에 임베딩되어 있기 때문입니다. 이 필드들은 디버거가 주입된 스크립트의 실행을 요청하기 위해 수정합니다.

PyThreadState 구조체는 파이썬 인터프리터 내부에서 실행되는 스레드를 나타냅니다. 이 구조체는 스레드의 평가 컨텍스트를 유지하며, 디버거 조정에 필요한 필드를 포함합니다. 따라서 유효한 PyThreadState 를 찾는 것은 원격 실행을 트리거하기 위한 핵심 전제 조건입니다.

스레드는 일반적으로 그 역할이나 ID를 기반으로 선택됩니다. 대부분의 경우 메인 스레드가 사용되지만, 일부 도구는 네이티브 스레드 ID를 통해 특정 스레드를 대상으로 할 수 있습니다. 대상 스레드가 선택되면, 디버거는 메모리에서 인터프리터와 연결된 스레드 상태 구조체 둘 다를 찾아야 합니다.

관련 내부 구조체는 다음과 같이 정의됩니다:

  • PyInterpreterState 는 격리된 파이썬 인터프리터 인스턴스를 나타냅니다. 각 인터프리터는 자체 가져온 모듈 세트, 내장 상태 및 스레드 상태 목록을 유지합니다. 대부분의 파이썬 애플리케이션이 단일 인터프리터를 사용하지만, CPython은 동일한 프로세스에 여러 인터프리터를 지원합니다.

  • PyThreadState 는 인터프리터 내에서 실행되는 스레드를 나타냅니다. 여기에는 실행 상태와 디버거가 사용하는 제어 필드가 포함됩니다.

스레드를 찾는 방법:

  1. 오프셋 runtime_state.interpreters_head``를 사용하여 ``PyRuntime 구조체에서 첫 번째 인터프리터의 주소를 얻습니다. 이것은 활성 인터프리터의 연결 리스트 진입점입니다.

  2. 오프셋 interpreter_state.threads_main 을 사용하여 선택된 인터프리터와 관련된 메인 스레드 상태에 접근합니다. 일반적으로 대상의 가장 신뢰할 수 있는 스레드입니다.

  3. 선택적으로, 오프셋 interpreter_state.threads_head``를 사용하여 모든 스레드 상태의 연결 리스트를 반복할 수 있습니다. 각 ``PyThreadState 구조체에는 대상 스레드 ID와 비교될 수 있는 native_thread_id 필드가 포함되어 있어 특정 스레드를 찾을 수 있습니다.

  4. 일단 유효한 PyThreadState 를 찾았다면, 해당 주소는 디버거 제어 필드를 작성하거나 실행 일정을 잡는 등 프로토콜의 후속 단계에서 사용될 수 있습니다.

다음은 메인 스레드 상태를 찾는 예제 구현입니다:

def find_main_thread_state(
    pid: int, py_runtime_addr: int, debug_offsets: DebugOffsets,
) -> int:
    # Step 1: PyRuntime에서 interpreters_head를 읽습니다.
    interp_head_ptr = (
        py_runtime_addr + debug_offsets.runtime_state.interpreters_head
    )
    interp_addr = read_pointer(pid, interp_head_ptr)
    if interp_addr == 0:
        raise RuntimeError("대상 프로세스에서 인터프리터가 발견되지 않았습니다")

    # Step 2: 인터프리터에서 threads_main 포인터를 읽습니다.
    threads_main_ptr = (
        interp_addr + debug_offsets.interpreter_state.threads_main
    )
    thread_state_addr = read_pointer(pid, threads_main_ptr)
    if thread_state_addr == 0:
        raise RuntimeError("메인 스레드 상태를 사용할 수 없습니다")

    return thread_state_addr

다음 예제는 네이티브 스레드 ID를 사용하여 스레드를 찾는 방법을 보여줍니다:

def find_thread_by_id(
    pid: int,
    interp_addr: int,
    debug_offsets: DebugOffsets,
    target_tid: int,
) -> int:
    # threads_head에서 시작하여 연결 리스트를 순회합니다.
    thread_ptr = read_pointer(
        pid,
        interp_addr + debug_offsets.interpreter_state.threads_head
    )

    while thread_ptr:
        native_tid_ptr = (
            thread_ptr + debug_offsets.thread_state.native_thread_id
        )
        native_tid = read_int(pid, native_tid_ptr)
        if native_tid == target_tid:
            return thread_ptr
        thread_ptr = read_pointer(
            pid,
            thread_ptr + debug_offsets.thread_state.next
        )

    raise RuntimeError("주어진 ID의 스레드가 발견되지 않았습니다")

유효한 스레드 상태가 확인되면, 디버거는 다음 섹션에서 설명하는 바와 같이 해당 제어 필드를 수정하고 실행을 예약하는 작업을 진행할 수 있습니다.

제어 정보 작성¶

유효한 PyThreadState 구조가 확인되면, 디버거는 명시된 Python 스크립트의 실행을 예약하기 위해 내부 제어 필드를 수정할 수 있습니다. 이 제어 필드는 인터프리터에 의해 주기적으로 확인되며, 올바르게 설정되면 평가 루프의 안전 지점에서 원격 코드 실행을 트리거합니다.

각 PyThreadState``는 디버거와 인터프리터 간의 통신에 사용되는 ``_PyRemoteDebuggerSupport 구조를 포함합니다. 이 필드의 위치는 _Py_DebugOffsets 구조에 의해 정의되며 다음과 같습니다:

  • debugger_script_path: Python 소스 파일(.py)의 전체 경로를 담고 있는 고정 크기 버퍼입니다. 실행이 트리거될 때 타겟 프로세스에서 이 파일에 접근하고 읽을 수 있어야 합니다.

  • debugger_pending_call: 정수 플래그입니다. 이 값을 1 로 설정하면 스크립트가 실행 준비가 되었음을 인터프리터에 알립니다.

  • eval_breaker: 실행 중에 인터프리터가 확인하는 필드입니다. 이 필드의 비트 5를 (_PY_EVAL_PLEASE_STOP_BIT, 값 1U << 5) 설정하면 인터프리터가 일시 정지하고 디버거 활동を 확인하게 됩니다.

주입을 완료하려면, 디버거는 다음 단계를 수행해야 합니다:

  1. 전체 스크립트 경로를 debugger_script_path 버퍼에 작성합니다.

  2. debugger_pending_call 을 1 로 설정합니다.

  3. eval_breaker``의 현재 값을 읽고, 비트 5 (``_PY_EVAL_PLEASE_STOP_BIT)를 설정한 다음, 업데이트된 값을 다시 작성합니다. 이는 인터프리터에 디버거 활동 확인을 알리는 신호입니다.

다음은 예제 구현입니다:

def inject_script(
    pid: int,
    thread_state_addr: int,
    debug_offsets: DebugOffsets,
    script_path: str
) -> None:
    # _PyRemoteDebuggerSupport의 기본 오프셋을 계산합니다.
    support_base = (
        thread_state_addr +
        debug_offsets.debugger_support.remote_debugger_support
    )

    # 단계 1: 스크립트 경로를 debugger_script_path에 씁니다.
    script_path_ptr = (
        support_base +
        debug_offsets.debugger_support.debugger_script_path
    )
    write_string(pid, script_path_ptr, script_path)

    # 단계 2: debugger_pending_call을 1로 설정합니다.
    pending_ptr = (
        support_base +
        debug_offsets.debugger_support.debugger_pending_call
    )
    write_int(pid, pending_ptr, 1)

    # 단계 3: eval_breaker에 _PY_EVAL_PLEASE_STOP_BIT을 설정합니다 (비트 5, 값 1 << 5).
    eval_breaker_ptr = (
        thread_state_addr +
        debug_offsets.debugger_support.eval_breaker
    )
    breaker = read_int(pid, eval_breaker_ptr)
    breaker |= (1 << 5)
    write_int(pid, eval_breaker_ptr, breaker)

이 필드들이 설정되면, 디버거는 프로세스를 재개할 수 있습니다 (일시 중지된 경우). 인터프리터는 다음 안전한 평가 지점에서 요청을 처리하고, 디스크에서 스크립트를 로드한 후, 실행합니다.

스크립트 파일이 실행 중인 동안 대상 프로세스에 계속 존재하고 액세스 가능하도록 보장하는 것은 디버거의 책임입니다.

참고

스크립트 실행은 비동기적입니다. 스크립트 파일은 주입된 직후 삭제할 수 없습니다. 디버거는 파일 제거 전에 주입된 스크립트가 관찰 가능한 효과를 생성할 때까지 기다려야 합니다. 이 효과는 스크립트가 수행하도록 설계된 작업에 따라 달라집니다. 예를 들어, 디버거는 원격 프로세스가 소켓에 다시 연결될 때까지 기다린 후에 스크립트를 제거할 수 있습니다. 이러한 효과가 관찰되면, 파일이 더 이상 필요하지 않다고 가정해도 안전합니다.

요약¶

원격 프로세스에서 파이썬 스크립트를 주입하고 실행하는 방법:

  1. 대상 프로세스의 메모리에서 PyRuntime 구조를 찾습니다.

  2. PyRuntime 시작 부분에서 _Py_DebugOffsets 구조를 읽고 유효성을 검사합니다.

  3. 오프셋을 사용하여 유효한 PyThreadState 를 찾습니다.

  4. 파이썬 스크립트 경로를 debugger_script_path 에 작성합니다.

  5. debugger_pending_call 플래그를 1 로 설정합니다.

  6. eval_breaker 필드에 _PY_EVAL_PLEASE_STOP_BIT 을 설정합니다.

  7. 프로세스를 재개합니다 (일시 중지된 경우). 스크립트는 다음 안전한 평가 지점에서 실행됩니다.

보안 및 위협 모델¶

원격 디버깅 프로토콜은 GDB 및 LLDB와 같은 네이티브 디버거가 사용하는 것과 동일한 운영 체제 기본 기능에 의존합니다. 프로세스에 연결하려면 디버거가 필요로 하는 동일한 권한 이 필요합니다. 예를 들어, Linux의 ptrace/Yama LSM, macOS의 task_for_pid, Windows의 SeDebugPrivilege 등이 있습니다. 파이썬은 새로운 권한 상승 경로를 도입하지 않습니다. 공격자가 이미 프로세스에 연결하는 데 필요한 권한을 가지고 있다면, 그들은 GDB를 사용하여 메모리를 읽거나 코드를 주입하는 데 동일하게 사용할 수 있습니다.

이 기능에서 보안 취약점으로 간주되는 것과 그렇지 않은 것을 정의하는 원칙은 다음과 같습니다:

연결(Attach)에는 OS 레벨의 권한이 필요합니다

지원되는 모든 플랫폼에서 운영 체제는 교차 프로세스 메모리 액세스를 권한 검사(CAP_SYS_PTRACE, root, 또는 관리자 권한) 뒤에 배치합니다. 이러한 권한이 이미 획득된 후에만 문제가 있음이 입증되는 보고서는 OS 보안 경계가 이미 넘어갔기 때문에 CPython의 취약점은 아닙니다.

손상된 프로세스를 읽을 때 발생하는 충돌이나 메모리 오류는 취약점이 아닙니다

대상 프로세스에서 내부 인터프리터 상태를 읽는 도구는 해당 메모리가 잘 형성되었다는 것을 신뢰해야 합니다. 대상 프로세스가 손상되었거나 공격자에 의해 제어되는 경우, 디버거나 프로파일러는 충돌하거나, 쓰레기 출력을 생성하거나, 예측할 수 없게 동작할 수 있습니다. 이는 모든 ptrace 기반 디버거가 수용하는 것과 동일한 위험입니다. 이러한 범주의 버그(버퍼 오버플로, 세그멘테이션 오류, 또는 손상된 상태를 읽는 것으로 유발된 정의되지 않은 동작)는 보안 문제로 취급되지 않지만, 견고성을 개선하는 수정 사항은 환영합니다.

대상 프로세스에서 취약점은 스코프 외에 존재합니다.

디버깅하는 Python 프로세스가 이미 손상된 경우, 공격자는 이미 해당 프로세스에서 실행을 제어합니다. 그러한 시작점에서 추가적인 영향을 시연하는 것은 원격 디버깅 프로토콜의 취약점을 구성하지 않습니다.

PYTHON_DISABLE_REMOTE_DEBUG 사용 시점¶

환경 변수 PYTHON_DISABLE_REMOTE_DEBUG (그리고 동등한 -X disable_remote_debug 플래그)는 오퍼레이터가 프로토콜의 인-프로세스 측면을 방어 심도 조치로 비활성화할 수 있도록 합니다. 이는 프로세스의 디버깅이나 프로파일링이 예상되지 않아 공격 표면을 줄이는 것이 우선순위인 강화되었거나 샌드박스 환경의 배포 환경에서 유용할 수 있습니다. 비록 OS 수준의 권한 검사가 이미 비특권적 접근을 방지하지만 말입니다.

이 변수를 설정하는 것은 다른 OS 수준 디버깅 인터페이스(ptrace, /proc, task_for_pid 등)에는 영향을 미치지 않습니다. 이 인터페이스들은 자체적인 권한 모델에 따라 여전히 사용할 수 있습니다.
