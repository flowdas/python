Free 스레딩을 위한 C API 확장 지원¶

Starting with the 3.13 release, CPython has support for running with the global interpreter lock (GIL) disabled in a configuration called free threading. This document describes how to adapt C API extensions to support free threading.

C에서 자유 스레딩 빌드 식별¶

CPython C API는 Py_GIL_DISABLED 매크로를 노출합니다. 자유 스레딩 빌드에서는 1 로 정의되며, 일반 빌드에서는 정의되지 않습니다. 이 매크로를 사용하여 자유 스레딩 빌드에서만 실행되는 코드를 활성화할 수 있습니다.

#ifdef Py_GIL_DISABLED
/* 자유 스레딩 빌드에서만 실행되는 코드 */
#endif

참고

Windows에서는 이 매크로가 자동으로 정의되지 않으며, 빌드 시 컴파일러에 지정해야 합니다. sysconfig.get_config_var() 함수를 사용하면 현재 실행 중인 인터프리터가 매크로를 가지고 있었는지 확인할 수 있습니다.

모듈 초기화¶

확장 모듈은 GIL 비활성화 상태에서 실행을 지원한다는 것을 명시적으로 표시해야 합니다. 그렇지 않으면 확장 모듈을 임포트할 때 경고가 발생하고 런타임에 GIL이 활성화됩니다.

확장 모듈이 GIL 비활성화 지원을 나타내는 방법은 확장이 다단계 초기화 또는 단일 단계 초기화를 사용하는지에 따라 두 가지가 있습니다.

다단계 초기화¶

다단계 초기화 <multi-phase-initialization>`를 사용하는 확장 모듈 (예: :c:func:`PyModuleDef_Init, PyModExport_* 내보내기 훅, PyModule_FromSlotsAndSpec())은 모듈 정의에 Py_mod_gil 슬롯을 추가해야 합니다. 확장 자체가 CPython의 이전 버전을 지원하는 경우, PY_VERSION_HEX 검사로 슬롯을 보호해야 합니다.

static struct PyModuleDef_Slot module_slots[] = {
    ...
#if PY_VERSION_HEX >= 0x030D0000
    {Py_mod_gil, Py_MOD_GIL_NOT_USED},
#endif
    {0, NULL}
};

단일 단계 초기화¶

레거시 single-phase initialization 을 사용하는 확장 모듈(즉, PyModule_Create() 를 사용하는 경우)은 GIL 비활성 상태에서 실행 지원 여부를 나타내기 위해 PyUnstable_Module_SetGIL() 을 호출해야 합니다. 이 함수는 자유 스레딩 빌드에만 정의되어 있으므로, 일반 빌드에서 컴파일 오류를 방지하기 위해 #ifdef Py_GIL_DISABLED 로 호출을 보호해야 합니다.

static struct PyModuleDef moduledef = {
    PyModuleDef_HEAD_INIT,
    ...
};

PyMODINIT_FUNC
PyInit_mymodule(void)
{
    PyObject *m = PyModule_Create(&moduledef);
    if (m == NULL) {
        return NULL;
    }
#ifdef Py_GIL_DISABLED
    PyUnstable_Module_SetGIL(m, Py_MOD_GIL_NOT_USED);
#endif
    return m;
}

기본 API 지침¶

대부분의 C API는 스레드 안전하지만, 예외가 일부 있습니다.

  • 구조체 필드: Python C API 객체 또는 구조체의 필드에 직접 접근하는 것은 해당 필드가 동시에 수정될 수 있는 경우 스레드 안전하지 않습니다.

  • 매크로: PyList_GET_ITEM, :c:macro:`PyList_SET_ITEM`와 같이 :c:func:`PySequence_Fast`가 반환하는 객체를 사용하는 :c:macro:`PySequence_Fast_GET_SIZE`와 같은 매크로는 오류 검사나 잠금 처리를 수행하지 않습니다. 이 매크로들은 컨테이너 객체가 동시에 수정될 수 있는 경우 스레드 안전하지 않습니다.

  • 빌린 참조: borrowed references 섹션을 참조하십시오.

컨테이너 스레드 안전성¶

PyListObject, PyDictObject, 및 :c:struct:`PySetObject`와 같은 컨테이너는 자유 스레딩 빌드에서 내부 잠금 처리를 수행합니다. 예를 들어, :c:func:`PyList_Append`는 항목 추가 전에 리스트를 잠급니다.

PyDict_Next¶

주목할 만한 예외는 딕셔너리에 잠금 처리를 하지 않는 :c:func:`PyDict_Next`입니다. 딕셔너리를 반복하는 동안 해당 딕셔너리가 동시에 수정될 수 있는 경우, :c:macro:`Py_BEGIN_CRITICAL_SECTION`을 사용하여 보호해야 합니다.

Py_BEGIN_CRITICAL_SECTION(dict);
PyObject *key, *value;
Py_ssize_t pos = 0;
while (PyDict_Next(dict, &pos, &key, &value)) {
    ...
}
Py_END_CRITICAL_SECTION();

빌린 참조¶

일부 C API 함수는 :term:`borrowed references <borrowed reference>`를 반환합니다. 컨테이너 객체가 동시에 수정될 경우, 이지는 스레드 안전하지 않습니다. 예를 들어, 리스트가 동시에 수정될 수 있는 경우 :c:func:`PyList_GetItem`을 사용하는 것은 안전하지 않습니다.

다음 표는 일부 빌린 참조 API와 :term:`strong references <strong reference>`를 반환하는 대체 API를 나열합니다.

빌린 참조를 반환하는 모든 API가 문제가 되는 것은 아닙니다. 예를 들어, 튜플은 불변하기 때문에 :c:func:`PyTuple_GetItem`은 안전합니다. 마찬가지로, 위의 API 사용 사례가 모두 문제가 되는 것은 아닙니다. 예를 들어, :c:func:`PyDict_GetItem`은 함수 호출에서 키워드 인수 딕셔너리를 구문 분석할 때 자주 사용되며, 해당 키워드 인수 딕셔너리는 본질적으로 사적(다른 스레드에서 접근 불가)이므로, 이 컨텍스트에서 빌린 참조를 사용하는 것은 안전합니다.

이 중 일부 함수는 Python 3.13에서 추가되었습니다. 이전 버전의 Python을 위해 이러한 함수의 구현을 제공하려면 pythoncapi-compat 패키지를 사용할 수 있습니다.

메모리 할당 API¶

Python의 메모리 관리 C API는 세 가지 다른 allocation domains: “raw”, “mem”, 및 “object”에서 함수를 제공합니다. 스레드 안전성을 위해 자유 스레딩 빌드는 객체 도메인을 사용하여 Python 객체만 할당해야 하며, 모든 Python 객체는 해당 도메인을 사용하여 할당되어야 합니다. 이는 이전 Python 버전과는 다르며, 이전 버전에서는 단지 모범 사례일 뿐 하드 요구 사항은 아니었습니다.

참고

확장 모듈에서 PyObject_Malloc() 사용 사례를 검색하고 할당된 메모리가 Python 객체에 사용되는지 확인하십시오. PyObject_Malloc() 대신 :c:func:`PyMem_Malloc`을 사용하여 버퍼를 할당하십시오.

스레드 상태 및 GIL API¶

Python은 스레드 상태 및 GIL을 관리하기 위한 다양한 함수와 매크로를 제공합니다. 예를 들면:

  • PyThreadState_Ensure(), PyThreadState_EnsureFromView(), 및 PyThreadState_Release()

  • PyGILState_Ensure() 및 PyGILState_Release()

  • PyEval_SaveThread() 및 PyEval_RestoreThread()

  • Py_BEGIN_ALLOW_THREADS 및 Py_END_ALLOW_THREADS

:term:`GIL`이 비활성화된 경우에도 스레드 상태를 관리하기 위해 이 함수들을 여전히 사용해야 합니다. 예를 들어, Python 외부에서 스레드를 생성한 경우, 스레드가 유효한 Python 스레드 상태를 갖도록 보장하려면 Python API를 호출하기 전에 반드시 :c:func:`PyThreadState_Ensure`를 호출해야 합니다.

:c:func:`PyEval_SaveThread`나 :c:macro:`Py_BEGIN_ALLOW_THREADS`를 I/O나 록 획득과 같은 블로킹 작업 주변에 계속 호출하여 다른 스레드들이 :term:`cyclic garbage collector <garbage collection>`를 실행할 수 있도록 허용해야 합니다.

내부 확장 상태 보호¶

귀하의 확장에는 이전에 GIL에 의해 보호되던 내부 상태가 있을 수 있습니다. 이 상태를 보호하기 위해 록을 추가해야 할 수 있습니다. 접근 방식은 확장 프로그램마다 달라지겠지만, 일부 일반적인 패턴은 다음과 같습니다:

  • 캐시: 전역 캐시는 공유 상태의 일반적인 원인입니다. 캐시를 보호하기 위해 록을 사용하거나, 캐시가 성능에 중요하지 않은 경우 자유 스레드 빌드에서 이를 비활성화하는 것을 고려해 보세요.

  • 전역 상태: 전역 상태는 록으로 보호되거나 스레드 로컬 스토리지로 이동해야 할 수 있습니다. C11 및 C++11은 thread_-local storage_ <https://en.cppreference.com/w/c/language/storage_duration> 를 위해 thread_local 또는 __Thread_local 을 제공합니다.

임계 영역¶

자유 스레드 빌드에서는 CPython이 “임계 영역”이라는 메커니즘을 제공하여, 그렇지 않으면 GIL에 의해 보호될 데이터에 대한 보호를 제공합니다. 확장 프로그램 작성자가 내부 임계 영역 구현과 직접 상호 작용하지 않을 수도 있지만, 특정 C API 함수를 사용하거나 자유 스레드 빌드에서 공유 상태를 관리할 때 그 동작을 이해하는 것이 중요합니다.

임계 영역이란 무엇입니까?¶

개념적으로, 임계 영역은 단순한 뮤텍스를 기반으로 구축된 교착 상태 회피 계층 역할을 합니다. 각 스레드는 활성 임계 영역의 스택을 유지합니다. 스레드가 임계 영역과 관련된 록을 획득해야 할 때 (예를 들어, :c:func:`PyDict_SetItem`과 같은 스레드 안전한 C API 함수를 호출할 때 암시적으로, 또는 매크로를 사용하여 명시적으로), 이는 기본 뮤텍스를 획득하려고 시도합니다.

임계 영역 사용하기¶

임계 영역을 사용하는 주요 API는 다음과 같습니다:

  • Py_BEGIN_CRITICAL_SECTION 및 Py_END_CRITICAL_SECTION - 단일 객체 잠금용

  • Py_BEGIN_CRITICAL_SECTION2 및 Py_END_CRITICAL_SECTION2 - 두 개 객체 동시 잠금용

이 매크로들은 반드시 쌍을 이룰 때 사용해야 하며, 새로운 로컬 범위를 설정하기 때문에 같은 C 범위 내에 나타나야 합니다. 이 매크로들은 비-자유 스레드 빌드에서는 매크로 역할(no-ops)을 하므로, 두 빌드 유형을 모두 지원해야 하는 코드에 안전하게 추가할 수 있습니다.

임계 영역의 일반적인 사용 사례는 내부 속성에 접근하는 동안 객체를 잠그는 것입니다. 예를 들어, 확장 유형에 내부 카운트 필드가 있는 경우, 해당 필드를 읽거나 쓸 때 임계 영역을 사용할 수 있습니다:

// 카운트를 읽고, 내부 카운트 값에 대한 새 참조를 반환합니다.
PyObject *result;
Py_BEGIN_CRITICAL_SECTION(obj);
result = Py_NewRef(obj->count);
Py_END_CRITICAL_SECTION();
return result;

// 카운트를 쓰고, new_count로부터 참조를 소모합니다.
Py_BEGIN_CRITICAL_SECTION(obj);
obj->count = new_count;
Py_END_CRITICAL_SECTION();

임계 영역 작동 방식¶

전통적인 록과 달리, 임계 영역은 전체 지속 시간 동안 독점적인 접근을 보장하지 않습니다. 스레드가 임계 영역을 보유하는 동안 블로킹될 것인 경우 (예: 다른 록을 획득하거나 I/O를 수행하는 경우), 임계 영역은 일시 중단되어 모든 록이 해제된 후, 블로킹 작업이 완료되면 다시 재개됩니다.

이 동작은 스레드가 블로킹 호출을 할 때 GIL이 하는 일과 유사합니다. 주요 차이점은 다음과 같습니다:

  • 임계 영역은 전역적이기보다는 객체별로 작동합니다.

  • 임계 영역은 각 스레드 내에서 스택 규율을 따릅니다 (“begin”과 “end” 매크로는 쌍을 이루고 동일한 범위 내에 있어야 하므로 이를 강제합니다).

  • 임계 영역은 잠재적인 블로킹 작업 주변에서 록을 자동으로 해제하고 다시 획득합니다.

교착 상태 회피¶

임계 영역은 두 가지 방식으로 교착 상태를 방지하는 데 도움이 됩니다:

  1. 스레드가 이미 다른 스레드에 의해 획득된 록을 획득하려고 시도하는 경우, 모든 활성 중요 섹션을 일시 중지하고, 해당 록이 해제될 때까지 대기합니다.

  2. 블로킹 작업이 완료되면, 가장 상위 중요 섹션만 먼저 재획득됩니다.

이는 중첩된 중요 섹션에 의존하여 여러 객체를 한 번에 록 할 수 없다는 의미입니다. 내부 중요 섹션이 외부를 중지시킬 수 있기 때문입니다. 대신, :c:macro:`Py_BEGIN_CRITICAL_SECTION2`를 사용하여 두 객체를 동시에 록 하세요.

위에서 설명한 록들은 PyMutex 기반의 록에만 해당합니다. 중요 섹션 구현은 POSIX 뮤텍스와 같이 사용될 수 있는 다른 록 메커니즘에 대해서는 알거나 영향을 미치지 않습니다. 또한, 모든 :c:type:`PyMutex`에 블로킹하는 것은 중요 섹션이 중지되게 하지만, 중요 섹션의 일부인 뮤텍스만 해제됩니다. :c:type:`PyMutex`를 중요 섹션 없이 사용하는 경우 해제되지 않으며, 따라서 데드락 방지 메커니즘을 적용하지 못합니다.

주요 고려 사항¶

  • 중요 섹션은 록을 일시적으로 해제할 수 있어 다른 스레드가 보호되는 데이터를 수정할 수 있게 합니다. 블로킹을 유발할 수 있는 작업 이후 데이터 상태를 가정하는 것에 주의하십시오.

  • 록은 일시적으로 해제(중지)될 수 있으므로, 중요 섹션에 진입한다고 해서 해당 섹션의 전체 기간 동안 보호된 자원에 대한 독점적인 접근이 보장되는 것은 아닙니다. 중요 섹션 내의 코드가 블로킹하는 다른 함수(예: 다른 록을 획득하거나 블로킹 I/O를 수행)를 호출하면, 해당 스레드가 중요 섹션을 통해 보유하고 있던 모든 록이 해제됩니다. 이는 GIL이 블로킹 호출 중에 해제될 수 있는 방식과 유사합니다.

  • 어떤 시점에서도 가장 최근에 진입한(가장 상위의) 중요 섹션과 관련된 록만 보유가 보장됩니다. 외부 및 중첩된 중요 섹션의 록은 중지되었을 수 있습니다.

  • 이 API로는 최대 두 개의 객체만 동시에 록 할 수 있습니다. 더 많은 객체를 록 해야 한다면 코드를 재구성해야 합니다.

  • 중요 섹션은 같은 객체를 두 번 록 하려고 시도해도 데드락이 발생하지는 않지만, 이 사용 사례에는 목적에 맞게 설계된 재진입 록보다 효율적이지 않습니다.

  • :c:macro:`Py_BEGIN_CRITICAL_SECTION2`를 사용할 때, 객체의 순서는 정확성에 영향을 미치지 않습니다(구현이 데드락을 방지하기 때문입니다). 하지만 일관된 순서로 객체를 록 하는 것이 좋습니다.

  • 중요 섹션 매크로가 주로 위에서 설명된 데드락 시나리오에 취약한 내부 CPython 작업에 대한 접근을 보호하기 위한 것임을 기억하십시오. 순전히 내부 확장 상태를 보호하는 경우에는 표준 뮤텍스나 다른 동기화 프라이미티브가 더 적절할 수 있습니다.

객체별 록 (ob_mutex)¶

자유 스레딩 빌드에서는 각 Python 객체가 ob_mutex 필드를 가지고 있으며, 이는 PyMutex 타입입니다. 이 뮤텍스는 중요 섹션 API (Py_BEGIN_CRITICAL_SECTION / Py_END_CRITICAL_SECTION)에 사용되도록 예약되어 있습니다.

경고

PyMutex_Lock(&obj->ob_mutex)`를 사용하여 `ob_mutex`를 직접 록 하지 마십시오. 같은 뮤텍스에 대해 직접적인 `PyMutex_Lock 호출과 중요 섹션 API를 혼합하면 데드락을 유발할 수 있습니다.

사용자 코드에서 특정 객체 타입에 중요 섹션을 사용하지 않더라도, CPython 내부에서는 모든 Python 객체에 중요 섹션 API를 사용할 수 있습니다.

확장이 자체 록이 필요한 경우, 객체 구조체에 별도의 PyMutex 필드(또는 다른 동기화 프라이미티브)를 추가하십시오. :c:type:`PyMutex`는 매우 가벼우므로 추가하는 데 비용이 거의 없습니다.

자유 스레딩 빌드를 위한 확장 빌드¶

C API 확장은 자유 스레딩 빌드를 위해 특별히 빌드해야 합니다. 휠, 공유 라이브러리 및 바이너리는 t 접미사로 표시됩니다.

  • pypa/manylinux 는 t 접미사가 있는 자유 스레딩 빌드를 지원하며, 예시로 python3.14t 가 있습니다.

  • `pypa/cibuildwheel <https://github.com/pypa/cibuildwheel>`_는 Python 3.14 이상 버전의 자유 스레딩 빌드를 위해 휠을 빌드하는 것을 지원합니다.

제한된 C API 및 안정 ABI¶

자유 스레딩 빌드는 현재 제한된 C API 또는 안정 ABI를 지원하지 않습니다. 확장 모듈을 빌드하기 위해 setuptools 를 사용하고 현재 py_limited_api=True 로 설정한 경우, 자유 스레딩 빌드로 빌드할 때 제한된 API를 제외하려면 py_limited_api=not sysconfig.get_config_var("Py_GIL_DISABLED") `를 사용할 수 있습니다.

참고

별도의 휠을 자유 스레딩 빌드를 위해 빌드해야 합니다. 현재 안정 ABI를 사용하는 경우, 여러 비-자유 스레딩 파이썬 버전을 위한 단일 휠을 계속 빌드할 수 있습니다.

Windows¶

공식 Windows 설치 프로그램의 제한으로 인해, 소스에서 확장 모듈을 빌드할 때 Py_GIL_DISABLED=1 을 수동으로 정의해야 합니다.
