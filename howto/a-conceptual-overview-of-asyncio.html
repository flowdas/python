:mod:`!asyncio`에 대한 개념적 개요¶

이 HOWTO 문서는 :mod:`asyncio`가 근본적으로 어떻게 작동하는지에 대한 견고한 개념 모델을 구축하는 데 도움을 주어, 권장되는 패턴 뒤에 숨겨진 방법과 이유를 이해하는 데 도움을 주기 위해 작성되었습니다.

직접 :mod:`!asyncio`의 일부 핵심 개념이 궁금할 수 있습니다. 이 기사가 끝날 때쯤이면 다음 질문들에 자신 있게 답할 수 있게 될 것입니다:

  • 객체가 await되는 동안 내부적으로 어떤 일이 발생하나요?

  • :mod:`!asyncio`는 CPU 시간이 필요하지 않은 작업(네트워크 요청 또는 파일 읽기와 같은)과 필요한 작업(n-팩토리얼 계산과 같은)을 어떻게 구별하나요?

  • 비동기 슬립이나 데이터베이스 요청과 같은 연산의 비동기 버전을 작성하는 방법입니다.

개념적 개요 1부: 고수준¶

1부에서는 asyncio 의 주요 고수준 구성 요소인 이벤트 루프, 코루틴 함수, 코루틴 객체, 태스크, 그리고 await 에 대해 다룰 것입니다.

이벤트 루프¶

:mod:`!asyncio`의 모든 것은 이벤트 루프와 관련하여 발생합니다. 이는 가장 중요한 부분이지만, 그보다는 백그라운드에서 리소스를 관리하고 조정하는 것을 선호합니다. 오케스트라 지휘자와 같습니다. 명시적으로 일부 권한이 부여되지만, 실제로 무언가를 수행할 수 있는 많은 능력은 밴드 멤버들의 존중과 협력에서 비롯됩니다.

더 기술적인 용어로, 이벤트 루프는 실행해야 할 작업들의 모음을 담고 있습니다. 일부 작업은 사용자 본인이 직접 추가하고, 일부는 :mod:`!asyncio`에 의해 간접적으로 추가됩니다. 이벤트 루프는 백로그 작업에서 작업을 가져와 함수를 호출하는 것과 유사하게 작업을 실행하거나(또는 “제어권을 주고”), 그런 다음 해당 작업이 실행됩니다. 일시 중지되거나 완료되면 이벤트 루프로 제어권을 반환합니다. 그러면 이벤트 루프는 풀에서 다른 작업을 선택하여 이를 호출합니다. 작업들의 모음을 대략적인 큐로 생각할 수 있습니다. 작업들은 추가된 후 일반적으로(하지만 항상은 아닐지라도) 한 번에 하나씩 처리됩니다. 이 과정은 이벤트 루프가 끊임없이 순환하면서 무기한 반복됩니다. 더 이상 실행 대기 중인 작업이 없으면, 이벤트 루프는 쉬는 것이 충분히 똑똑하며 불필요하게 CPU 사이클을 낭비하는 것을 방지하고, I/O 작업이 완료되거나 타이머가 만료되는 경우와 같이 할 일이 더 있을 때 돌아옵니다.

효율적인 실행은 작업들이 잘 공유하고 협력하는 것에 달려 있습니다. 게으른 작업은 제어권을 독점하여 다른 작업들이 기아 상태에 놓이게 만들 수 있으며, 전반적인 이벤트 루프 접근법을 거의 무용지물로 만듭니다.

import asyncio

# 이 코드는 이벤트 루프를 생성하고
# 작업 컬렉션을 무한히 순환합니다.
event_loop = asyncio.new_event_loop()
event_loop.run_forever()

비동기 함수와 코루틴¶

이것은 기본적이고 지루한 파이썬 함수입니다:

def hello_printer():
    print(
        "Hi, I am a lowly, simple printer, though I have all I "
        "need in life -- \nfresh paper and my dearly beloved octopus "
        "partner in crime."
    )

일반 함수를 호출하면 해당 로직 또는 본문이 호출됩니다:

>>> hello_printer()
Hi, I am a lowly, simple printer, though I have all I need in life --
fresh paper and my dearly beloved octopus partner in crime.

async def 객체가 생성되고 반환됩니다.

async def loudmouth_penguin(magic_number: int):
    print(
     "I am a super special talking penguin. Far cooler than that printer. "
     f"By the way, my lucky number is: {magic_number}."
    )

비동기 함수 loudmouth_penguin 를 호출해도 print문이 실행되지 않고, 대신 코루틴 객체가 생성됩니다:

>>> loudmouth_penguin(magic_number=3)
<coroutine object loudmouth_penguin at 0x104ed2740>

“코루틴 함수”와 “코루틴 객체”라는 용어는 종종 (코루틴으로) 혼동됩니다. 이것은 혼란스러울 수 있습니다! 이 문건에서 코루틴은 특히 코루틴 객체, 또는 더 정확하게는 types.CoroutineType (네이티브 코루틴)의 인스턴스를 의미합니다. 코루틴은 또한 :class:`collections.abc.Coroutine`의 인스턴스로 존재할 수 있다는 점에 유의하십시오. (이는 타입 검사에서 중요합니다.)

코루틴은 함수의 본문 또는 로직을 나타냅니다. 코루틴은 명시적으로 시작되어야 합니다. 다시 말해, 코루틴을 단순히 생성하는 것만으로는 시작되지 않습니다. 주목할 점은 이 코루틴이 함수의 본문 내의 여러 지점에서 일시 중지되고 재개될 수 있다는 것입니다. 바로 이 일시 중지 및 재개 능력이 비동기 동작을 가능하게 합니다!

Coroutines and coroutine functions were built by leveraging the functionality of generators and generator functions. Recall, a generator function is a function that yields, like this one:

def get_random_number():
    # 이것은 나쁜 난수 생성기입니다!
    print("Hi")
    yield 1
    print("Hello")
    yield 7
    print("Howdy")
    yield 4
    ...

코루틴 함수와 유사하게, 제너레이터 함수를 호출해도 실행되지 않습니다. 대신, 제너레이터 객체가 생성됩니다:

>>> get_random_number()
<generator object get_random_number at 0x1048671c0>

빌트인 함수 next() 를 사용하여 제너레이터의 다음 yield 로 진행할 수 있습니다. 다시 말해, 제너레이터는 실행된 후 일시 중지됩니다. 예를 들면 다음과 같습니다:

>>> generator = get_random_number()
>>> next(generator)
Hi
1
>>> next(generator)
Hello
7

태스크¶

대략적으로, <asyncio-task-obj> 태스크`는 이벤트 루프에 연결된 코루틴(코루틴 함수는 아님)입니다. 태스크는 또한 콜백 함수 목록을 유지하며, 이는 await 를 논의할 때 그 중요성이 명확해집니다.

태스크를 생성하면 자동으로 실행이 예약됩니다 (이벤트 루프의 할 일 목록에 콜백을 추가함으로써, 즉 작업들의 모음). 태스크를 생성하는 권장 방법은 :func:`asyncio.create_task`를 사용하는 것입니다.

asyncio`는 태스크를 이벤트 루프와 자동으로 연결합니다. 이러한 자동 연결은 단순성을 위해 :mod:!asyncio`에 의도적으로 설계되었습니다. 이것이 없으면, 작업을 생성하려는 모든 코루틴 함수에 이벤트 루프 객체를 추적하고 전달해야 하므로 코드에 불필요한 복잡성을 더해야 합니다.

coroutine = loudmouth_penguin(magic_number=5)
# 이것은 Task 객체를 생성하고 이벤트 루프를 통해 실행을 예약합니다.
task = asyncio.create_task(coroutine)

이전에는 이벤트 루프를 수동으로 생성하여 무한히 실행되도록 설정했습니다. 실제로는 이벤트 루프 관리를 담당하고 제공된 코루틴이 완료된 후 진행하도록 보장하는 :func:`asyncio.run`을 사용하는 것이 권장되며 흔히 볼 수 있습니다. 예를 들어, 많은 비동기 프로그램은 다음과 같은 설정을 따릅니다:

import asyncio

async def main():
    # 모든 엉뚱하고 광란적인 비동기 작업을 수행합니다...
    ...

if __name__ == "__main__":
    asyncio.run(main())
    # 프로그램은 코루틴 main()이 완료될 때까지 다음 print 문에 도달하지 않습니다.
    print("coroutine main()이 완료되었습니다!")

태스크 자체는 이벤트 루프에 추가되지 않고, 태스크에 대한 콜백만 추가된다는 점을 인지하는 것이 중요합니다. 이는 생성한 태스크 객체가 이벤트 루프에 의해 호출되기 전에 가비지 컬렉션되는 경우에 중요합니다. 예를 들어, 이 프로그램을 고려해 보세요:

1async def hello():
2    print("hello!")
3
4async def main():
5    asyncio.create_task(hello())
6    # 한동안 실행되면서 제어권을 이벤트 루프에 양도하는 다른 비동기 명령어...
7    ...
8
9asyncio.run(main())

작업 객체를 생성한 라인 5에는 참조가 없기 때문에, 이벤트 루프가 이를 호출하기 전에 가비지 컬렉션 될 수 있습니다. 코루틴 main() 의 후반 지침들은 제어권을 이벤트 루프로 넘겨 다른 작업을 호출할 수 있게 합니다. 이벤트 루프가 궁극적으로 태스크를 실행하려고 할 때 실패하고 태스크 객체가 존재하지 않다는 것을 발견할 수 있습니다! 코루틴이 태스크에 대한 참조를 유지하고 있더라도, 해당 태스크가 완료되기 전에 코루틴이 완료되면 이런 일이 발생할 수도 있습니다. 코루틴이 종료되면 지역 변수는 범위를 벗어나 가비지 컬렉션의 대상이 될 수 있습니다. 실제로는 asyncio 와 Python 가비지 컬렉터가 이러한 일이 발생하지 않도록 매우 열심히 작동합니다. 하지만 그렇다고 해서 무모해도 된다는 의미는 아닙니다!

await¶

:keyword:`await`는 두 가지 방식으로 흔히 사용되는 파이썬 키워드입니다:

await task
await coroutine

중요하게도, await 의 동작은 기다리는 객체의 유형에 따라 달라집니다.

태스크 기다리기¶

태스크를 기다리면 현재 태스크나 코루틴의 제어권이 이벤트 루프로 넘어갑니다. 제어권을 포기하는 과정에서 몇 가지 중요한 것들이 발생합니다. 다음 코드 예제를 사용하여 설명하겠습니다:

async def plant_a_tree():
    dig_the_hole_task = asyncio.create_task(dig_the_hole())
    await dig_the_hole_task

    # 나무를 심는 것과 관련된 다른 지침들.
    ...

이 예제에서, 이벤트 루프가 코루틴 plant_a_tree() 의 시작 지점으로 제어권을 넘겼다고 가정해 봅시다. 위에서 볼 수 있듯이, 코루틴은 태스크를 생성하고 이를 기다립니다. await dig_the_hole_task 지침은 콜백(plant_a_tree() 를 재개할 것)을 dig_the_hole_task 객체의 콜백 목록에 추가합니다. 그런 다음, 이 지침은 제어권을 이벤트 루프로 넘깁니다. 시간이 지난 후, 이벤트 루프는 dig_the_hole_task 에 제어권을 넘기고 태스크는 필요한 모든 작업을 마칩니다. 태스크가 완료되면, 다양한 콜백들을 이벤트 루프에 추가하며, 이 경우 plant_a_tree() 를 재개하는 호출을 합니다.

일반적으로, 기다리던 태스크(dig_the_hole_task)가 완료되면, 원래의 태스크나 코루틴(plant_a_tree())은 재개되기 위해 이벤트 루프의 할 일 목록에 다시 추가됩니다.

이것은 기본적이지만 신뢰할 수 있는 사고 모델입니다. 실제로는 제어권 인계가 약간 더 복잡하지만, 그 차이는 크지 않습니다. 파트 2에서는 이것이 가능한 세부 사항을 살펴보겠습니다.

코루틴 기다리기¶

태스크와 달리, 코루틴을 기다리는 것은 이벤트 루프로 제어권을 넘기지 않습니다! 코루틴을 먼저 태스크로 래핑한 후 기다리면 제어권을 넘기게 됩니다. await coroutine 의 동작은 일반적인 동기식 파이썬 함수를 호출하는 것과 효과적으로 동일합니다. 다음 프로그램을 고려해 보세요:

import asyncio

async def coro_a():
   print("I am coro_a(). Hi!")

async def coro_b():
   print("I am coro_b(). I sure hope no one hogs the event loop...")

async def main():
   task_b = asyncio.create_task(coro_b())
   num_repeats = 3
   for _ in range(num_repeats):
      await coro_a()
   await task_b

asyncio.run(main())

코루틴 main()``의 첫 번째 구문은 ``task_b``를 생성하고 이벤트 루프를 통해 실행하도록 예약합니다. 그런 다음, ``coro_a()``가 반복적으로 기다려집니다. 제어권이 이벤트 루프로 절대 넘어가지 않기 때문에, ``coro_b()``의 출력 전에 세 ``coro_a() 호출의 모든 출력이 표시되는 것입니다:

I am coro_a(). Hi!
I am coro_a(). Hi!
I am coro_a(). Hi!
I am coro_b(). I sure hope no one hogs the event loop...

만약 await coro_a() 를 await asyncio.create_task(coro_a()) 로 변경하면 동작이 달라집니다. 코루틴 main() 는 해당 구문을 통해 이벤트 루프에 제어권을 넘깁니다. 그러면 이벤트 루프는 백로그 작업을 순차적으로 처리하며, task_b 를 호출한 다음 coro_a() 를 감싼 태스크를 호출하여 코루틴 main() 을 재개합니다.

I am coro_b(). I sure hope no one hogs the event loop...
I am coro_a(). Hi!
I am coro_a(). Hi!
I am coro_a(). Hi!

await coroutine``의 이러한 동작은 많은 사람들을 혼란스럽게 할 수 있습니다! 이 예제는 ``await coroutine``만 사용하면 다른 태스크로부터 제어권을 의도치 않게 독점하여 이벤트 루프를 효과적으로 멈출 수 있음을 강조합니다. :func:`asyncio.run`은 ``debug=True 플래그를 통해 이러한 발생을 감지하는 데 도움이 될 수 있으며, 이는 :ref:`디버그 모드 <asyncio-debug-mode>`를 활성화합니다. 다른 것들 중에서도 100ms 이상 실행을 독점하는 모든 코루틴을 로그합니다.

디자인은 await 사용과 관련된 개념적 명료함 중 일부를 성능 향상을 위해 의도적으로 포기했습니다. 태스크가 어웨이트될 때마다, 제어는 콜 스택의 맨 위까지 이벤트 루프로 전달되어야 합니다. 그런 다음, 이벤트 루프는 자신의 내부 상태를 관리하고 다음 작업을 재개하기 위해 처리 로직을 순회해야 합니다. 사소하게 들릴 수 있지만, 많은 await 가 있는 대규모 프로그램에서는 그 오버헤드가 무시할 수 없는 성능 저하를 초래할 수 있습니다.

개념적 개요 파트 2: 내부 작동 원리¶

파트 2에서는 asyncio 가 제어 흐름을 관리하는 메커니즘에 대해 자세히 설명합니다. 바로 마법이 일어나는 부분입니다. 이 섹션을 마치면 await 가 내부적으로 무엇을 하는지, 그리고 자신만의 비동기 연산자를 어떻게 만드는지 알게 될 것입니다.

코루틴의 내부 작동 원리¶

:mod:`!asyncio`는 파이썬의 네 가지 구성 요소를 활용하여 제어 흐름을 전달합니다.

coroutine.send(arg) 는 코루틴을 시작하거나 재개하는 데 사용되는 메서드입니다. 코루틴이 일시 중지되었다가 이제 재개되는 경우, 인자 arg 는 원래 일시 중지시켰던 yield 문의 반환 값으로 전송될 것입니다. 코루틴이 처음 사용되는 경우(재개되는 경우와 달리), arg 는 None 이어야 합니다.

 1class Rock:
 2    def __await__(self):
 3        value_sent_in = yield 7
 4        print(f"Rock.__await__ resuming with value: {value_sent_in}.")
 5        return value_sent_in
 6
 7async def main():
 8    print("Beginning coroutine main().")
 9    rock = Rock()
10    print("Awaiting rock...")
11    value_from_rock = await rock
12    print(f"Coroutine received value: {value_from_rock} from rock.")
13    return 23
14
15coroutine = main()
16intermediate_result = coroutine.send(None)
17print(f"Coroutine paused and returned intermediate value: {intermediate_result}.")
18
19print(f"Resuming coroutine and sending in value: 42.")
20try:
21    coroutine.send(42)
22except StopIteration as e:
23    returned_value = e.value
24print(f"Coroutine main() finished and provided value: {returned_value}.")

yield, 평소대로 실행을 일시 중지하고 호출자에게 제어를 반환합니다. 위 예제에서, 3번째 줄의 yield 는 11번째 줄의 ... = await rock 에 의해 호출됩니다. 더 광범위하게 말하자면, await 는 주어진 객체의 __await__() 메서드를 호출합니다. await 는 또한 한 가지 매우 특별한 일을 합니다. 받은 모든 yield 를 호출 체인 위로 전파하거나(“전달”) 합니다. 이 경우, 이는 16번째 줄의 ... = coroutine.send(None) 로 돌아갑니다.

코루틴은 21번째 줄의 coroutine.send(42) 호출을 통해 재개됩니다. 코루틴은 3번째 줄에서 yield (또는 일시 중지)되었던 곳에서 다시 시작하여 본문의 나머지 문장들을 실행합니다. 코루틴이 완료되면, 반환 값과 함께 StopIteration 예외를 value 속성에 첨부하며 발생시킵니다.

해당 스니펫은 다음과 같은 결과가 나옵니다:

Beginning coroutine main().
Awaiting rock...
Coroutine paused and returned intermediate value: 7.
Resuming coroutine and sending in value: 42.
Rock.__await__ resuming with value: 42.
Coroutine received value: 42 from rock.
Coroutine main() finished and provided value: 23.

여기서 잠시 멈춰서 제어 흐름과 값들이 전달된 다양한 방식들을 따라왔는지 확인해 볼 필요가 있습니다. 많은 중요한 개념들이 다루어졌으며 이해가 확실한지 확인하는 것이 중요합니다.

코루틴에서 값을 생성하거나(또는 실질적으로 제어를 넘겨)할 수 있는 유일한 방법은 __await__ 메서드에서 yield 하는 객체를 await 하는 것입니다. 이것은 이상하게 들릴 수 있습니다. 당신은 다음과 같이 생각할지 모릅니다:

1. What about a yield directly within the coroutine function? The coroutine function becomes an async generator function, a different beast entirely.

2. What about a yield from within the coroutine function to a (plain) generator? That causes the error: SyntaxError: yield from not allowed in a coroutine. This was intentionally designed for the sake of simplicity – mandating only one way of using coroutines. Despite that, yield from and await effectively do the same thing. Initially yield was barred as well, but was re-accepted to allow for async generators.

Future¶

:ref:`future <asyncio-future-obj>`는 계산의 상태와 결과를 나타내기 위해 사용되는 객체입니다. 이 용어는 아직 오지 않았거나 아직 발생하지 않은 무언가라는 아이디어에서 비롯되었으며, 이 객체는 그 무언가를 주시하는 방법입니다.

Future는 몇 가지 중요한 속성을 가지고 있습니다. 하나는 상태(state)로, “pending”, “cancelled”, 또는 “done” 중 하나일 수 있습니다. 다른 하나는 결과(result)로, 상태가 done으로 전환될 때 설정됩니다. 코루틴과 달리, Future는 수행해야 할 실제 계산을 나타내지 않습니다. 대신, 마치 상태 표시등(빨간색, 노란색 또는 녹색)이나 지표처럼, 해당 계산의 상태와 결과를 나타냅니다.

asyncio.Task 는 이러한 다양한 기능을 얻기 위해 asyncio.Future 를 상속합니다. 이전 섹션에서는 태스크가 콜백 목록을 저장한다고 했지만, 이는 완전히 정확하지 않았습니다. 실제로 이 로직을 구현하는 것은 Future 클래스이며, Task 가 이를 상속받습니다.

Futures may also be used directly (not via tasks). Tasks mark themselves as done when their coroutine is complete. Futures are much more versatile and will be marked as done when you say so. In this way, they’re the flexible interface for you to make your own conditions for waiting and resuming.

A homemade asyncio.sleep¶

We’ll go through an example of how you could leverage a future to create your own variant of asynchronous sleep (async_sleep) which mimics asyncio.sleep().

This snippet registers a few tasks with the event loop and then awaits the task created by asyncio.create_task, which wraps the async_sleep(3) coroutine. We want that task to finish only after three seconds have elapsed, but without preventing other tasks from running.

async def other_work():
    print("I like work. Work work.")

async def main():
    # Add a few other tasks to the event loop, so there's something
    # to do while asynchronously sleeping.
    work_tasks = [
        asyncio.create_task(other_work()),
        asyncio.create_task(other_work()),
        asyncio.create_task(other_work())
    ]
    print(
        "Beginning asynchronous sleep at time: "
        f"{datetime.datetime.now().strftime("%H:%M:%S")}."
    )
    await asyncio.create_task(async_sleep(3))
    print(
        "Done asynchronous sleep at time: "
        f"{datetime.datetime.now().strftime("%H:%M:%S")}."
    )
    # asyncio.gather effectively awaits each task in the collection.
    await asyncio.gather(*work_tasks)

Below, we use a future to enable custom control over when that task will be marked as done. If future.set_result() (the method responsible for marking that future as done) is never called, then this task will never finish. We’ve also enlisted the help of another task, which we’ll see in a moment, that will monitor how much time has elapsed and, accordingly, call future.set_result().

async def async_sleep(seconds: float):
    future = asyncio.Future()
    time_to_wake = time.time() + seconds
    # Add the watcher-task to the event loop.
    watcher_task = asyncio.create_task(_sleep_watcher(future, time_to_wake))
    # Block until the future is marked as done.
    await future

Below, we use a rather bare YieldToEventLoop() object to yield from its __await__ method, ceding control to the event loop. This is effectively the same as calling asyncio.sleep(0), but this approach offers more clarity, not to mention it’s somewhat cheating to use asyncio.sleep when showcasing how to implement it!

As usual, the event loop cycles through its tasks, giving them control and receiving control back when they pause or finish. The watcher_task, which runs the coroutine _sleep_watcher(...), will be invoked once per full cycle of the event loop. On each resumption, it’ll check the time and if not enough has elapsed, then it’ll pause once again and hand control back to the event loop. Once enough time has elapsed, _sleep_watcher(...) marks the future as done and completes by exiting its infinite while loop. Given this helper task is only invoked once per cycle of the event loop, you’d be correct to note that this asynchronous sleep will sleep at least three seconds, rather than exactly three seconds. Note this is also true of asyncio.sleep.

class YieldToEventLoop:
    def __await__(self):
        yield

async def _sleep_watcher(future, time_to_wake):
    while True:
        if time.time() >= time_to_wake:
            # This marks the future as done.
            future.set_result(None)
            break
        else:
            await YieldToEventLoop()

Here is the full program’s output:

$ python custom-async-sleep.py
Beginning asynchronous sleep at time: 14:52:22.
I like work. Work work.
I like work. Work work.
I like work. Work work.
Done asynchronous sleep at time: 14:52:25.

You might feel this implementation of asynchronous sleep was unnecessarily convoluted. And, well, it was. The example was meant to showcase the versatility of futures with a simple example that could be mimicked for more complex needs. For reference, you could implement it without futures, like so:

async def simpler_async_sleep(seconds):
    time_to_wake = time.time() + seconds
    while True:
        if time.time() >= time_to_wake:
            return
        else:
            await YieldToEventLoop()
