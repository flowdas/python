무료 스레딩을 위한 Stable ABI(abi3t)로 마이그레이션¶

Starting with the 3.15 release, CPython supports a variant of the Stable ABI that supports free-threaded Python: Stable ABI for Free-Threaded Builds, or abi3t for short. This document describes how to adapt C API extensions to support free threading.

이유¶

Stable ABI를 사용하는 일반적인 이유는 라이브러리 버전마다 빌드하고 배포해야 하는 아티팩트의 수를 줄이기 위함입니다.

Stable ABI가 없으면, 지원하려는 각 CPython 기능 버전에 대해 별도의 공유 라이브러리, 그리고 일반적으로 wheel 배포판을 빌드해야 합니다. 예를 들어, 다음 표의 각 태그는 별도의 라이브러리/휠을 나타냅니다:

이 방법을 사용하지 않는 이유¶

Stable ABI에는 두 가지 주요 단점이 있습니다.

첫째, Stable ABI는 성능보다 호환성을 우선시하기 때문에 확장 기능이 더 느려질 수 있습니다. 이러한 차이는 일반적으로 눈에 띄지 않으며, “1단계” CPython 버전을 위한 Stable ABI 빌드와 몇 가지 버전별 빌드를 같은 소스를 사용하여 빌드함으로써 종종 완화할 수 있습니다.

둘째, 모든 C API가 사용 가능한 것은 아닙니다. 확장은 Stable ABI용으로 포팅되어야 하며, 이 과정은 어려울 수 있거나 드문 경우 불가능할 수도 있습니다.

구체적으로, abi3t 는 CPython 3.15에서 추가된 API를 필요로 합니다. 동일한 소스에서 이전 버전의 CPython을 위해 확장 기능을 빌드하려면 두 가지 주요 옵션이 있습니다:

  • 전처리기 조건 사용.

    이 가이드를 따를 때, 관심 있는 CPython 버전에서 빌드를 깨뜨리는 변경을 수행하라는 지시를 받을 때는 항상 #ifdef Py_TARGET_ABI3T 블록을 사용하십시오. 기존 코드는 #else 블록에 유지하십시오.

    수동으로 작성된 C 확장 기능의 경우, :pep:`697`에서 도입된 추가 사항 덕분에 이 접근 방식은 CPython 3.12까지 합리적입니다. 3.11 이하와의 호환성을 유지하는 것은 코드 생성기(예를 들어, Cython)가 사용할 만한 가치가 있을 수 있습니다.

  • 이전 버전의 지원을 중단할 수 있을 때까지 abi3t 로 포팅하지 말고, 각 CPython 버전을 위한 확장 기능을 계속 별도로 빌드해야 합니다.

    이것은 유효한 접근 방식입니다. 모든 확장 기능이 지금 당장 abi3t 로 전환할 필요는 없습니다.

사전 요구 사항¶

이 가이드는 C(또는 C++)에 직접 작성된 확장 기능을 가지고 있으며, 이를 abi3t 로 포팅하려는 것을 가정합니다.

확장 기능이 코드 생성기(예: Cython) 또는 언어 바인딩(예: PyO3)을 사용하는 경우, 해당 도구가 abi3t 를 지원할 때까지 기다리는 것이 가장 좋습니다. 만약 그러한 도구를 직접 유지 관리한다면, 여기에 있는 지침을 여러분의 도구에 맞게 조정할 수 있을 것입니다.

Non-free-threaded Stable ABI¶

확장 기능은 Stable ABI(abi3t)를 지원해야 합니다. 그렇지 않다면, 먼저 포팅하거나, 이 가이드를 따르되 가이드에서 언급하지 않는 문제를 수정할 준비를 해야 합니다.

Free-threading 지원¶

기술적으로는 필수 전제 조건은 아니지만, abi3t 로 포팅하기 전에 확장 기능을 free-threading을 위해 준비하는 것이 거의 확실하게 필요합니다. 지침은 Free 스레딩을 위한 C API 확장 지원 를 참조하십시오.

더 보기

자유 스레딩 사용 지원을 위한 확장 모듈 포팅 <https://py-free-threading.github.io/porting/>: 확장 작성자를 위한 커뮤니티 유지 관리 포팅 가이드입니다.

확장 모듈 격리¶

모듈은 :ref:`multi-phase initialization <multi-phase-initialization>`을 사용해야 하며, 격리되거나 프로세스당 최대 한 번만 로드되도록 제한해야 합니다. 그렇지 않은 경우, 먼저 :ref:`isolating-extensions-howto`를 따르십시오. (바로 가속이 필요한 경우 :ref:`opt-out section <isolating-extensions-optout>`을 참조합니다.)

가변 크기 데이터 타입 피하기¶

확장 기능에서 가변 크기 타입( Py_tp_itemsize 또는 PyTypeObject.tp_itemsize 사용)을 정의하는 경우, abi3t 3.15로 포팅할 수 없습니다.

빌드 설정¶

빌드 도구(setuptools, meson-python, scikit-build-core 등)를 사용하는 경우, abi3t``를 선택하는 방법을 해당 문서에서 검색하십시오. 현재까지는 모든 도구가 이를 갖춘 것은 아니지만, 도구가 지원한다면 사용하십시오. 다음 내용을 ``#include <Python.h> 바로 뒤에 임시로 추가하여 올바른 플래그가 설정되었는지 확인할 수 있습니다:

#if Py_TARGET_ABI3T+0 <= 0x30f0000
#error "abi3t define is not set!"
#endif

이는 “abt3t define is not set”과 다른 오류를 발생시킬 것입니다.

참고

빌드 도구가 아직 abi3t 를 지원하지 않는 경우, Python.h 를 포함하기 전에 다음 매크로를 설정하십시오:

#define Py_TARGET_ABI3T 0x30f0000

또는 컴파일러 플래그로 지정합니다. 예를 들면:

-DPy_TARGET_ABI3T=0x30f0000

확장이 이 설정으로 빌드되면 CPython 3.15 이상과 호환됩니다.

이 매크로를 수동으로 설정하는 경우, 나중에 결과 확장 기능의 이름과 태그도 수동으로 지정해야 합니다. 이는 아래 :ref:`abi3t-migration-tagging`에서 다룹니다.

이 가이드는 일련의 변경 사항을 요구합니다. 각 변경 사항 후에는 확장 모듈이 원래의 (non-abi3t) 구성에서 계속 빌드되는지 확인하고, 이상적으로는 지원하는 모든 파이썬 버전에 대해 테스트를 실행하십시오. 이렇게 하면 포팅하는 과정에서 아무것도 깨지지 않도록 보장할 수 있습니다.

모듈 내보내기 훅¶

아직 이 단계를 수행하지 않았다면, 확장 모듈은 `:ref:`모듈 초기화 함수 <extension-pyinit>`인 `:samp:`PyInit_{<module_name>}`을 정의합니다. 이 기능을 `:pep:`793`에서 CPython 3.15에 추가된 `:ref:`모듈 내보내기 훅 <extension-export-hook>`인 `:samp:`PyModExport_{<module name>}`으로 포팅해야 합니다.

기존 초기화 함수는 다음과 같아야 합니다 (<modname> 및 <moddef> 에 사용자 이름이 포함됨):

PyMODINIT_FUNC
PyInit_<modname>(void)
{
    return PyModuleDef_Init(&<moddef>);
}

return 앞에 코드가 있는 경우, 이를 :c:macro:`Py_mod_create 또는 Py_mod_exec 슬롯 함수로 옮기십시오. 관련 정보는 :ref:`PyInit 문서 <extension-pyinit>`를 참조하십시오.

함수는 PyModuleDef 객체 (위 코드의 <moddef>)를 참조합니다. 그 정의는 다음과 유사해야 하며, 값은 다를 수 있고 일부 필드는 이름이 없거나 생략될 수 있습니다:

static PyModuleDef <moddef> = {
    PyModuleDef_HEAD_INIT,
    .m_name = "my_module",
    .m_doc = "my docstring",
    .m_size = sizeof(my_state_struct),
    .m_methods = my_methods,
    .m_slots = my_slots,
    .m_traverse = my_traverse,
    .m_clear = my_clear,
    .m_free = my_free,
};

이 정의와 PyInit 함수를 제거하고 (또는 후방 호환성을 유지하기 위해 #ifndef Py_TARGET_ABI3T 블록에 넣고) 다음으로 대체하십시오:

PyABIInfo_VAR(abi_info);

static PySlot my_slot_array[] = {
    PySlot_STATIC_DATA(Py_mod_abi, &abi_info),
    PySlot_STATIC_DATA(Py_mod_name, "my_module"),
    PySlot_STATIC_DATA(Py_mod_doc, "my docstring"),
    PySlot_SIZE(Py_mod_state_size, sizeof(my_state_struct)),
    PySlot_STATIC_DATA(Py_mod_methods, my_methods),
    PySlot_STATIC_DATA(Py_mod_slots, my_slots),
    PySlot_FUNC(Py_mod_state_traverse, my_traverse),
    PySlot_FUNC(Py_mod_state_clear, my_clear),
    PySlot_FUNC(Py_mod_state_free, my_free),
    PySlot_END
};

PyMODEXPORT_FUNC
PyModExport_<modname>(void)
{
    return my_slot_array;
}

누락된 필드는 무시하고 (새로운 Py_mod_abi 제외), 사용자의 값을 대체하십시오.

이 API에 대한 자세한 내용은 PySlot 및 내보내기 훅 문서를 참조하십시오.

연관 PyModuleDef¶

새로운 API는 PyModuleDef 구조를 사용하지 않으므로, 결과 모듈과 연결되는 정의가 없습니다. 이는 다음 함수들의 동작을 변경합니다:

  • PyModule_GetDef()

  • PyType_GetModuleByDef()

코드를 검색하여 이것들을 확인하십시오. 사용하지 않으면 이 섹션은 건너뛰어도 됩니다.

이 함수들은 일반적으로 두 가지 목적으로 사용됩니다:

  1. 모듈이 생성될 때 사용된 정의를 가져옵니다. 새로운 API를 사용하면 더 이상 불가능합니다. 모듈은 더 이상 정의에 대한 참조를 유지하지 않으므로, 관련 데이터를 전달하는 다른 방법을 마련해야 합니다.

  1. 주어진 모듈 객체가 “본인의 것”인지 확인합니다. 이 사용 사례는 이제 :ref:`모듈 토큰 <ext-module-token>`이 제공합니다. 이는 모듈을 식별하는 불투명(opaque) 포인터입니다. 토큰을 사용하려면 다음과 같이 고유한 정적 변수를 선언하거나 재사용하십시오:

    static char my_token;

    그리고 모듈의 PySlot 배열에 새 항목으로 포인터를 추가하십시오:

    static PySlot my_slot_array[] = {
   ...
   PySlot_STATIC_DATA(Py_mod_token, &my_token),
   PySlot_END
}

    그런 다음, 다음과 같은 PyModule_GetDef() 호출을 사용에서 전환하십시오:

    PyModuleDef *def = PyModule_GetDef(module);

    PyModule_GetToken() (출력 인수를 사용하며 예외로 실패할 수 있음)으로:

    void *token;
if (PyModule_GetToken(module, &token) < 0) {
    /* 오류 처리 */
}

    그리고 다음과 같은 PyType_GetModuleByDef() 호출에서 벗어나서:

    PyObject *module = PyType_GetModuleByDef(type, my_def);
/* 오류 처리; module 사용 */

    PyType_GetModuleByToken() (강한 참조를 반환):

    PyObject *module = PyType_GetModuleByToken(type, my_token);
/* 오류 처리; module 사용 */
Py_XDECREF(module);

PyObject 불투명성¶

PyObject 및 PyVarObject 구조는 abi3t 에서 불투명합니다.

멤버에 액세스하는 것은 금지됩니다. 이렇게 할 경우, 설명서에 언급된 getter/setter 함수로 전환하십시오:

  • PyObject.ob_type

  • PyObject.ob_refcnt

  • PyVarObject.ob_size

또한, PyObject 구조의 size 는 컴파일러가 알 수 없습니다. 이는 다른 CPython 빌드 간에 변경될 수 있습니다.

참고

런타임에는 크기가 사용 가능하지만(예: Python 코드에서 sys.getsizeof(object())), 여기에서 포인터 오프셋을 계산하려는 유혹을 저항해야 합니다. 객체 메모리 레이아웃은 향후 abi3t 구현에서 변경될 수 있습니다.

사용자 정의 타입 정의¶

:c:type:`!PyObject`가 불투명하기 때문에, 사용자 정의 타입을 정의하는 기존 방식은 더 이상 작동하지 않습니다:

typedef struct {
   PyObject_HEAD  // 알려지지 않은 크기가 있는 ‘PyObject ob_base;'로 확장됩니다.

   int my_data;
} CustomObject;

static PyType_Spec CustomType_spec = {
   ...
   .basicsize = sizeof(CustomObject),
   ...
};

아마도 클래스 정의 전체와 클래스의 데이터를 액세스하는 모든 코드를 재작성해야 할 것입니다. 이것이 사용자가 abi3t 를 지원하기 위해 지원해야 할 가장 큰 변경 사항이 될 가능성이 높습니다.

이러한 각 유형에 대해, 전체 인스턴스를 위해 struct 를 정의하는 대신, 추가 필드, 즉 슈퍼클래스가 아닌 클래스에만 특이한 필드를 포함하는 struct 를 정의하십시오:

typedef struct {
   int my_data;
} CustomObjectData;

이름을 변경하십시오. 구조체를 사용하는 거의 모든 코드가 변경해야 합니다 (특히, 새 구조체로의 포인터는 PyObject*``로 캐스팅할 수 없습니다). 이름을 변경하면 사용 사례가 컴파일러 오류로 강조 표시됩니다. (``typeof, C++ auto 또는 유형 이름을 지정하는 것을 피하는 유사한 방법을 사용하는 경우 이 기능은 작동하지 않습니다. 매우 주의하고, 정의되지 않은 동작을 감지하기 위해 도구를 실행하는 것을 고려하십시오.)

그런 다음, 클래스를 생성하려면 음수 basicsize 를 사용하여 전체 인스턴스 크기 대신 “추가” 저장 공간임을 나타내십시오:

static PyType_Spec CustomType_spec = {
   ...
   .basicsize = -sizeof(CustomObjectData), /* 마이너스 기호를 참고하십시오 */
   ...
};

Py_tp_members 를 사용하면, 각 멤버에 Py_RELATIVE_OFFSET 플래그를 설정하고 offset 을 새 구조체에 대해 상대적으로 지정해야 합니다.

커스텀 유형 데이터 액세스¶

다음은 어려운 부분입니다. 이 구조체에 액세스해야 하는 모든 코드에서, PyObject *``에서 ``CustomObjectData * 포인터를 검색하기 위해 추가적인 PyObject_GetTypeData() 호출이 필요합니다:

PyObject *obj = ...;
CustomObjectData *data = PyObject_GetTypeData(obj, cls);

이 호출에는 클래스의 유형 객체 (cls)가 필요하다는 점에 유의하십시오.

클래스가 하위 클래스화(즉, Py_TPFLAGS_BASETYPE 플래그를 사용하지 않는 경우), cls 는 Py_TYPE(obj) 가 됩니다. 그렇지 않은 경우, PyObject_GetTypeData() 와 함께 Py_TYPE 을 사용하지 마십시오: 관련 없는 하위 클래스에 예약된 메모리를 반환할 수 있습니다! 예를 들어, 사용자가 다음과 같은 하위 클래스를 만드는 경우:

class Sub(YourCustomClass):
   __slots__ = ('a', 'b')

그러면 Py_TYPE(obj) 는 YourCustomClass 이며, 기본 메모리는 다음과 같이 보일 수 있습니다:

╭─ PyObject *obj
│              ╭─ the pointer you want
│              │                    ╭─ PyObject_GetTypeData(obj, Py_TYPE(obj))
▼              ▼                    ▼
┌──────────┬───┬────────────────┬───┬─────────────┬───┬─────────────┐
│ PyObject │...│ CustomTypeData │...│ PyObject *a │...│ PyObject *b │
└──────────┴───┴────────────────┴───┴─────────────┴───┴─────────────┘

(줄임표는 가능한 패딩을 나타냅니다. 이 메모리 레이아웃은 보장되지 않습니다. 파이썬의 미래 버전에서는 다른 패딩을 추가하거나 구조체의 순서를 변경할 수 있습니다.)

올바른 클래스를 가져오는 두 가지 주요 방법이 있습니다:

  • 인스턴스 메서드의 경우, 구현은 PyCMethod 시그니처(및 PyMethodDef.ml_flags 의 METH_METHOD 비트)를 사용할 수 있으며, defining_class 인자로 클래스를 가져올 수 있습니다.

  • 그렇지 않은 경우, Py_tp_token 슬롯을 사용하여 클래스에 고유한 정적 토큰을 부여하고 다음을 사용하십시오:

    PyTypeObject cls;
if (PyType_GetBaseByToken(Py_TYPE(obj), my_tp_token, &cls) < 0) {
    /* 에러 처리 */
}
CustomObjectData *data = PyObject_GetTypeData(obj, cls);

    유형 토큰은 이 가이드에서 전에 다룬 모듈 토큰과 유사하게 작동합니다.

빌드 시간 조건문 피하기¶

확장 모듈을 빌드 하는 데 사용된 파이썬 버전을 식별하는 API를 코드로 확인하십시오. 이는 더 이상 확장 모듈이 실행되는 파이썬 버전과 일치하지 않으므로, 이 정보를 사용하는 코드는 종종 변경해야 합니다. 확인해야 할 매크로는 다음과 같습니다:

  • PY_VERSION_HEX, PY_MAJOR_VERSION, PY_MINOR_VERSION:

    • 런타임 버전을 얻으려면 :c:data:`Py_Version`을 사용하십시오:

    • 사용 가능한 C API를 확인하려면 :c:macro:`Py_TARGET_ABI3T`를 사용하십시오. 이 매크로는 지원되는 최소 버전을 설정합니다.

  • Py_GIL_DISABLED: abi3t 하에서 이 매크로는 항상 정의됩니다. 자유 스레드 지원 파이썬에서 작동하는 코드는 GIL이 활성화되어도 작동해야 합니다 (GIL은 런타임에 활성화될 수 있으므로), 그리고 일반적으로 작동합니다 (특정 이유로 한 번에 하나 이상의 :term:`attached thread state <attached thread state>`가 필요한 경우가 아니라면).

추가 코드 변경 사항¶

컴파일러 오류나 경고가 여전히 남아 있다면, 수정 방법을 찾아보십시오. 아쉽게도 이 가이드는 제한적이며, 확장 모듈이 필요로 할 수 있는 모든 가능한 코드 변경을 다룰 수는 없습니다.

다른 확장 모듈 작성자가 겪을 수 있는 문제가 발견되면, 이 가이드에 대해 문제 보고 (또는 풀 리퀘스트 전송)를 고려하십시오.

현재 버전의 abi3t 용으로 해당 문제를 수정할 수 없을 수 있습니다. 이 경우, 보고해 주시면 다음 버전 CPython에서 우선순위가 높아지는 데 도움이 될 수 있습니다.

태깅 및 배포¶

abi3t 지원 빌드 도구를 사용하고 있다면 확장 프로그램이 준비되었지만, 올바르게 빌드되었는지 확인해 보시는 것이 좋습니다.

abi3t 로 빌드된 확장 프로그램은 다음 확장자를 가져야 합니다:

  • Windows의 경우: .pyd (다른 모든 확장 프로그램과 같습니다);

  • .so 접미사를 사용하는 Linux, macOS 및 기타 시스템의 경우: .abi3t.so (아님: .cpython-315t.so 또는 .abi3.so). 자유 스레딩 및 비-자유 스레딩 빌드 모두 .abi3t.so 확장 프로그램을 로드한다는 점에 유의하십시오;

  • 기타 시스템: 배포처에 문의하고, 필요하다면 이 가이드를 업데이트하십시오.

확장 프로그램을 휠 로 배포하는 경우, 다음 태그를 사용하십시오:

  • Python 태그: cp3XX, 여기서 XX 는 확장 프로그램이 빌드된 최소 Python 버전입니다. (예를 들어, Py_TARGET_ABI3T 를 0x30f0000 로 설정한 경우 cp315 입니다. 더 많은 값은 안정 ABI용으로 컴파일하기 을 참조하십시오.)

  • ABI 태그: abi3.abi3t. 이는 비-자유 스레딩 및 자유 스레딩 빌드 모두를 지원함을 나타내는 압축 태그 세트 입니다.

예를 들어, 휠 파일명은 다음과 같을 수 있습니다:

myproject-1.0-cp315-abi3.abi3t-macosx_11_0_arm64.whl

파일명이나 태그가 잘못된 경우, 수정하십시오.

테스트¶

여러 버전의 CPython과 호환되는 확장 프로그램을 빌드할 때, 지원하는 모든 버전(예: 3.15, 3.16 등)으로 항상 테스트*해야 합니다. 안정적인 ABI는 *ABI 호환성만을 보장합니다. 의도적인 동작 변경(:pep:`387`에서 다루는 내용 포함)과 버그가 있을 수 있습니다.

자유 스레딩 및 비-자유 스레딩 빌드 모두에서 테스트를 실행하는 것을 확인하십시오.

통과했다면, 축하드립니다! abi3t 확장 프로그램입니다.
