파이썬 2.3 메서드 결정 순서¶

참고

이것은 공식 문서의 부록으로 제공된 역사적 문서입니다. 여기서 논의되는 메서드 결정 순서는 Python 2.3에서 도입 되었지만, Python 3를 포함하여 이후 버전에서도 사용됩니다.

Michele Simionato 작성. (https://www.phyast.pitt.edu/~micheles/)

개요:

이 문서는 Python 2.3에서 사용되는 C3 메서드 결정 순서에 대해 이해하고자 하는 Python 프로그래머를 위한 것입니다. 초보자에게 의도된 것은 아니지만, 많은 예시를 통해 상당히 교육적입니다. 저는 동일한 범위를 가진 다른 공개적으로 사용 가능한 문서를 알지 못하므로, 유용할 것입니다.

면책 조항:

저는 이 문서를 Python Software Foundation에 Python 2.3 라이선스 기준으로 기증합니다. 평소와 같이, 저는 독자에게 다음 내용이 정확해야 한다고 경고하지만, 어떠한 보증도 하지 않습니다. 독자님의 자체적인 위험과 책임하에 사용하십시오!

감사의 글:

저의 지원을 보내준 Python 메일링 리스트의 모든 분들께. 다양한 부정확한 점을 지적하고 로컬 우선순위 순서에 대한 부분을 추가하게 해준 Paul Foley님께. reStructuredText 서식 지정에 도움을 준 David Goodger님께. 편집에 도움을 준 David Mertz님께. 마지막으로, 이 문서를 공식 Python 2.3 홈페이지에 열정적으로 추가해주신 Guido van Rossum님께.

시작¶

Felix qui potuit rerum cognoscere causas – Virgilius

Everything started with a post by Samuele Pedroni to the Python development mailing list [1]. In his post, Samuele showed that the Python 2.2 method resolution order is not monotonic and he proposed to replace it with the C3 method resolution order. Guido agreed with his arguments and therefore now Python 2.3 uses C3. The C3 method itself has nothing to do with Python, since it was invented by people working on Dylan and it is described in a paper intended for lispers [2]. The present paper gives a (hopefully) readable discussion of the C3 algorithm for Pythonistas who want to understand the reasons for the change.

우선, 제가 말씀드릴 내용은 Python 2.2에서 도입된 새 스타일 클래스 에만 적용된다는 점을 말씀드립니다: 클래식 클래스 는 기존의 메서드 결정 순서를 유지하며, 깊이 우선 탐색 후 왼쪽에서 오른쪽 순서를 따릅니다. 따라서 클래식 클래스의 경우 기존 코드를 깨뜨리는 부분은 없습니다. 그리고 원칙적으로 Python 2.2 새 스타일 클래스의 경우 코드가 깨질 가능성이 있다 하더라도, C3 결정 순서가 Python 2.2의 메서드 결정 순서와 다른 경우는 매우 드물어서 실제 코드 깨짐은 예상되지 않습니다. 그러므로:

무서워하지 마세요!

게다가, 다중 상속을 강력하게 사용하거나 복잡한 계층 구조가 없다면 C3 알고리즘을 이해할 필요가 없으며, 이 논문을 쉽게 건너뛸 수 있습니다. 반면에, 다중 상속이 어떻게 작동하는지 정말 알고 싶다면 이 논문이 적합합니다. 좋은 소식은 예상했던 것만큼 복잡하지 않다는 것입니다.

몇 가지 기본적인 정의부터 시작해 보겠습니다.

  1. 복잡한 다중 상속 계층에 속한 클래스 C의 경우, 메서드가 재정의되는 순서, 즉 C의 조상들의 순서를 지정하는 것은 까다로운 작업입니다.

  2. 클래스 자체를 포함하여 클래스 C의 조상들의 목록을 가장 가까운 조상부터 가장 먼 조상까지의 순서로 배열한 것을 클래스 우선순위 목록 또는 C의 선형화 라고 합니다.

  3. 메서드 결정 순서 (MRO)는 선형화를 구성하는 규칙들의 집합입니다. 파이썬 문헌에서는 “C의 MRO”라는 관용구가 종종 클래스 C의 선형화와 동의어로 사용됩니다.

  4. 예를 들어, 단일 상속 계층의 경우, C가 C1의 서브클래스이고 C1이 C2의 서브클래스라면, C의 선형화는 단순히 [C, C1 , C2] 목록입니다. 하지만 다중 상속 계층에서는 지역 우선순위 순서 와 단조성 을 모두 존중하는 선형화를 구성하는 것이 더 까다롭습니다.

  5. 지역 우선순위 순서는 나중에 논하겠습니다만, 여기서는 단조성의 정의를 할 수 있습니다. MRO는 다음이 참일 때 단조롭다고 합니다: 만약 C의 선형화에서 C1이 C2보다 먼저 오면, C의 모든 서브클래스의 선형화에서도 C1이 C2보다 먼저 와야 한다. 그렇지 않으면, 단순히 새 클래스를 파생시키는 사소한 작업이라도 메서드의 결정 순서를 변경하여 매우 미묘한 버그를 유발할 수 있습니다. 이러한 현상이 발생하는 예시는 나중에 보여드리겠습니다.

  6. 모든 클래스가 선형화를 가질 수 있는 것은 아닙니다. 복잡한 계층 구조에서는 원하는 모든 속성을 존중하는 방식으로 클래스를 파생시키는 것이 불가능한 경우가 있습니다.

여기서는 이러한 상황의 예시를 보여드리겠습니다. 다음 계층 구조를 고려해 보세요:

>>> O = object
>>> class X(O): pass
>>> class Y(O): pass
>>> class A(X,Y): pass
>>> class B(Y,X): pass

이것은 다음과 같은 상속 그래프로 표현할 수 있으며, 여기서 저는 O로 새로운 스타일 클래스의 모든 계층 구조의 시작점인 “object” 클래스를 표시했습니다:

 -----------
|           |
|    O      |
|  /   \    |
 - X    Y  /
   |  / | /
   | /  |/
   A    B
   \   /
     ?

이 경우, A에서 B를 파생시킬 수 없습니다. 왜냐하면 X가 A에서 Y보다 먼저 오지만, B에서는 Y가 X보다 먼저 오기 때문에 C의 메서드 결정 순서가 모호해지기 때문입니다.

Python 2.3은 이 상황에서 예외를 발생시켜 (TypeError: MRO conflict among bases Y, X) 초보 개발자가 모호한 계층 구조를 만드는 것을 금지합니다. 반면 Python 2.2는 예외를 발생시키지 않고 임시적인 순서(이 경우 CABXYO)를 선택합니다.

C3 메서드 결정 순서¶

다음 논의에 유용할 몇 가지 간단한 표기법을 소개하겠습니다. 여기서는 이 줄임 표기법을 사용하겠습니다:

C1 C2 ... CN

은 [C1, C2, … , CN] 목록을 나타냅니다.

이 목록의 머리 는 첫 번째 요소입니다:

head = C1

반면에 꼬리 는 목록의 나머지를 나타냅니다:

tail = C2 ... CN.

또한 다음 표기법도 사용하겠습니다:

C + (C1 C2 ... CN) = C C1 C2 ... CN

이는 [C] + [C1, C2, … ,CN] 목록의 합을 나타냅니다.

이제 Python 2.3에서 MRO가 어떻게 작동하는지 설명할 수 있습니다.

B1, B2, … , BN 베이스 클래스들로부터 상속받는 다중 상속 계층의 클래스 C를 고려해 보겠습니다. 우리는 클래스 C의 선형화 L[C]를 계산하고자 합니다. 규칙은 다음과 같습니다:

C의 선형화는 C에 부모들의 선형화 합치와 부모들의 리스트 합친 것과 같습니다.

기호 표기법으로:

L[C(B1 ... BN)] = C + merge(L[B1] ... L[BN], B1 ... BN)

특히 C가 부모가 없는 object 클래스라면, 선형화는 자명합니다:

L[object] = object.

하지만 일반적으로는 다음 규정에 따라 merge를 계산해야 합니다:

첫 번째 리스트의 헤드, 즉 L[B1][0]를 가져옵니다. 이 헤드가 다른 어떤 리스트의 테일에 포함되어 있지 않다면, 이를 C의 선형화에 추가하고 merge의 리스트에서 제거합니다. 그렇지 않다면, 다음 리스트의 헤드를 보고 좋은 헤드인지 확인한 다음 가져옵니다. 이 작업을 모든 클래스가 제거되거나 좋은 헤드를 찾을 수 없게 될 때까지 반복합니다. 이 경우, merge를 구성하는 것은 불가능하며, Python 2.3은 클래스 C를 생성하는 것을 거부하고 예외를 발생시킵니다.

이 규정은 merge 연산이 순서성을 보존 함을 보장합니다. 단, 순서성을 보존할 수 없는 경우(위에서 논의된 심각한 순서 불일치 예시와 같이)는 merge를 계산할 수 없습니다.

만약 C가 부모가 하나뿐인 경우(단일 상속) merge 계산은 자명합니다. 이 경우:

L[C(B)] = C + merge(L[B],B) = C + L[B]

하지만 다중 상속의 경우 상황이 더 복잡하며, 몇 가지 예시 없이는 규칙을 이해하기 어렵다고 생각하지 않습니다 😉

예제¶

첫 번째 예제입니다. 다음과 같은 위계 구조를 고려해 주십시오:

>>> O = object
>>> class F(O): pass
>>> class E(O): pass
>>> class D(O): pass
>>> class C(D,F): pass
>>> class B(D,E): pass
>>> class A(B,C): pass

이 경우 상속 그래프는 다음과 같이 그릴 수 있습니다:

                          6
                         ---
Level 3                 | O |                  (more general)
                      /  ---  \
                     /    |    \                      |
                    /     |     \                     |
                   /      |      \                    |
                  ---    ---    ---                   |
Level 2        3 | D | 4| E |  | F | 5                |
                  ---    ---    ---                   |
                   \  \ _ /       |                   |
                    \    / \ _    |                   |
                     \  /      \  |                   |
                      ---      ---                    |
Level 1            1 | B |    | C | 2                 |
                      ---      ---                    |
                        \      /                      |
                         \    /                      \ /
                           ---
Level 0                 0 | A |                (more specialized)
                           ---

O, D, E, F의 선형화는 자명합니다:

L[O] = O
L[D] = D O
L[E] = E O
L[F] = F O

B의 선형화는 다음과 같이 계산할 수 있습니다:

L[B] = B + merge(DO, EO, DE)

D가 좋은 헤드임을 확인할 수 있으므로, 이를 가져오고 merge(O,EO,E) 를 계산하는 것으로 축소됩니다. 이제 O는 순서 EO의 테일에 포함되어 있으므로 좋은 헤드가 아닙니다. 이 경우 규칙은 다음 순서로 건너뛰어야 한다고 말합니다. 다음으로 E가 좋은 헤드임 것을 확인하고, 이를 가져와 merge(O,O) 를 계산하는 것으로 축소되며, 이는 O를 반환합니다. 따라서:

L[B] = B D E O

동일한 절차를 사용하면 다음을 얻을 수 있습니다:

L[C] = C + merge(DO,FO,DF)
     = C + D + merge(O,FO,F)
     = C + D + F + merge(O,O)
     = C D F O

이제 계산할 수 있습니다:

L[A] = A + merge(BDEO,CDFO,BC)
     = A + B + merge(DEO,CDFO,C)
     = A + B + C + merge(DEO,DFO)
     = A + B + C + D + merge(EO,FO)
     = A + B + C + D + E + merge(O,FO)
     = A + B + C + D + E + F + merge(O,O)
     = A B C D E F O

이 예제에서는 선형화가 상속 레벨에 따라 매우 보기 좋은 순서로 정렬되어 있습니다. 즉, 하위 레벨(더 전문화된 클래스)이 더 높은 우선순위를 갖습니다(상속 그래프 참조). 그러나 이것이 일반적인 경우는 아닙니다.

독자들에게 저의 두 번째 예제에 대한 선형화를 계산하는 숙제를 드립니다:

>>> O = object
>>> class F(O): pass
>>> class E(O): pass
>>> class D(O): pass
>>> class C(D,F): pass
>>> class B(E,D): pass
>>> class A(B,C): pass

이전 예제와 유일한 차이점은 B(D,E)가 B(E,D)로 변경되었다는 점입니다. 하지만 이런 작은 수정만으로도 위계의 순서가 완전히 바뀝니다:

                           6
                          ---
Level 3                  | O |
                       /  ---  \
                      /    |    \
                     /     |     \
                    /      |      \
                  ---     ---    ---
Level 2        2 | E | 4 | D |  | F | 5
                  ---     ---    ---
                   \      / \     /
                    \    /   \   /
                     \  /     \ /
                      ---     ---
Level 1            1 | B |   | C | 3
                      ---     ---
                       \       /
                        \     /
                          ---
Level 0                0 | A |
                          ---

계층 구조의 두 번째 레벨에 있는 클래스 E가 첫 번째 레벨에 있는 클래스 C 앞에 온다는 점에 주목하세요. 즉, E는 높은 레벨에 있더라도 C보다 더 전문화되어 있습니다.

게으른 프로그래머는 Python 2.2에서 MRO를 직접 얻을 수 있는데, 이 경우 Python 2.3의 선형화와 일치하기 때문입니다. 클래스 A의 mro() 메서드를 호출하기만 하면 됩니다:

>>> A.mro()
[<class 'A'>, <class 'B'>, <class 'E'>,
<class 'C'>, <class 'D'>, <class 'F'>,
<class 'object'>]

마지막으로, 심각한 순서 불일치를 포함하는 첫 번째 섹션에서 논의된 예제를 생각해 보겠습니다. 이 경우, O, X, Y, A, B의 선형화를 계산하는 것은 간단합니다:

L[O] = 0
L[X] = X O
L[Y] = Y O
L[A] = A X Y O
L[B] = B Y X O

하지만 A와 B로부터 상속받는 클래스 C의 선형화를 계산하는 것은 불가능합니다:

L[C] = C + merge(AXYO, BYXO, AB)
     = C + A + merge(XYO, BYXO, B)
     = C + A + B + merge(XYO, YXO)

이 시점에서 우리는 X가 YXO의 테일에 있고 Y가 XYO의 테일에 있기 때문에 리스트 XYO와 YXO를 merge할 수 없습니다. 따라서 좋은 헤드가 없으며 C3 알고리즘은 중지됩니다. Python 2.3은 오류를 발생시키고 클래스 C의 생성을 거부합니다.

잘못된 메서드 결정 순서¶

MRO가 국소 우선순서 순서나 단조성과 같은 근본적인 속성을 깨뜨릴 때 나쁩니다. 이 섹션에서는 클래식 클래스에 대한 MRO와 Python 2.2의 새로운 스타일 클래스에 대한 MRO 모두 나쁘다는 것을 보여드리겠습니다.

국소 우선순서 순서로 시작하는 것이 더 쉽습니다. 다음 예제를 고려해 보십시오:

>>> F=type('Food',(),{'remember2buy':'spam'})
>>> E=type('Eggs',(F,),{'remember2buy':'eggs'})
>>> G=type('GoodFood',(F,E),{}) # under Python 2.3 this is an error!

상속 다이어그램

             O
             |
(buy spam)   F
             | \
             | E   (buy eggs)
             | /
             G

      (buy eggs or spam ?)

클래스 G는 F와 E를 상속받으며, F가 E 보다 먼저 옵니다: 따라서 우리는 속성 G.remember2buy 가 F.remember2buy 로부터 상속되고 E.remember2buy 로부터 상속되지 않을 것이라고 예상합니다: 그럼에도 불구하고 Python 2.2는

>>> G.remember2buy
'eggs'

이는 로컬 선행 순서(local precedence ordering)를 깨뜨리는 것입니다. 왜냐하면 로컬 선행 목록, 즉 G의 부모 목록의 순서가 Python 2.2의 G 선형화 과정에서 보존되지 않기 때문입니다:

L[G,P22]= G E F object   # F는 E *다음*입니다

F가 E의 슈퍼클래스이기 때문에 F가 E보다 덜 특화되었다고 주장할 수 있습니다. 그럼에도 불구하고 로컬 선행 순서의 파괴는 상당히 직관적이지 않고 오류가 발생하기 쉽습니다. 특히 구형 스타일의 클래스와는 다르기 때문에 더욱 그렇습니다:

>>> class F: remember2buy='spam'
>>> class E(F): remember2buy='eggs'
>>> class G(F,E): pass
>>> G.remember2buy
'spam'

이 경우 MRO는 GFEF이며 로컬 선행 순서가 유지됩니다.

일반적으로는 이전과 같은 계층 구조를 피해야 합니다. 왜냐하면 F가 E를 오버라이드해야 하는지 아니면 그 반대인지 불분명하기 때문입니다. Python 2.3은 클래스 G 생성 시 예외를 발생시켜 이 모호성을 해결하며, 사실상 프로그래머가 모호한 계층 구조 생성을 막습니다. 그 이유는 merge::를 할 때 C3 알고리즘이 실패하기 때문입니다.

merge(FO,EFO,FE)

계산될 수 없습니다. 왜냐하면 F는 EFO의 꼬리(tail)에 있고 E는 FE의 꼬리(tail)에 있기 때문입니다.

진짜 해결책은 모호하지 않은 계층 구조를 설계하는 것입니다. 즉, G를 F와 E(더 특수한 것부터)로부터 파생시키고 F와 E로부터 파생시키지 않는 것입니다. 이 경우 MRO는 의심할 여지 없이 GEF입니다.

           O
           |
           F (spam)
         / |
(eggs)   E |
         \ |
           G
             (eggs, no doubt)

Python 2.3은 프로그래머가 좋은 계층 구조(또는 적어도 오류가 덜 발생할 수 있는 구조)를 작성하도록 강제합니다.

관련하여 말씀드리자면, Python 2.3 알고리즘이 부모 목록의 클래스 중복을 인식할 만큼 충분히 똑똑하다는 점을 말씀드리겠습니다:

>>> class A(object): pass
>>> class C(A,A): pass # error
Traceback (most recent call last):
  File "<stdin>", line 1, in ?
TypeError: duplicate base class A

이 상황에서 Python 2.2(클래식 클래스 및 새로운 스타일 클래스 모두)는 예외를 발생시키지 않습니다.

마지막으로, 이 예제를 통해 우리가 배운 두 가지 교훈을 말씀드리겠습니다:

  1. 이름과는 달리, MRO는 메서드뿐만 아니라 속성의 해결 순서를 결정합니다;

  2. Python 개발자들에게 인기 있는 음식은 스팸입니다! (이미 알고 계시겠지만 ;-))

로컬 선행 순서 이슈에 대해 논의했으니, 이제 단조성(monotonicity) 이슈를 살펴보겠습니다. 저의 목표는 클래식 클래스의 MRO와 Python 2.2 새로운 스타일 클래스의 MRO 모두 단조롭지 않다는 것을 보여주는 것입니다.

클래식 클래스의 MRO가 비단조롭다는 것을 증명하는 것은 매우 간단합니다. 다이아몬드 다이어그램을 살펴보는 것만으로 충분합니다:

   C
  / \
 /   \
A     B
 \   /
  \ /
   D

불일치를 쉽게 알 수 있습니다:

L[B,P21] = B C        # B가 C보다 선행합니다 : B의 메서드가 우선합니다
L[D,P21] = D A C B C  # B가 C보다 후행합니다  : C의 메서드가 우선합니다!

한편으로는, Python 2.2와 2.3 MRO에는 문제가 없습니다. 둘 다 제공합니다:

L[D] = D A B C

Guido points out in his essay [3] that the classic MRO is not so bad in practice, since one can typically avoids diamonds for classic classes. But all new style classes inherit from object, therefore diamonds are unavoidable and inconsistencies shows up in every multiple inheritance graph.

Python 2.2의 MRO는 단조성을 깨뜨리기 어렵게 만들지만, 불가능하지는 않습니다. 다음 예제는 Samuele Pedroni가 originally 제공한 것이며, Python 2.2의 MRO가 비단조롭다는 것을 보여줍니다:

>>> class A(object): pass
>>> class B(object): pass
>>> class C(object): pass
>>> class D(object): pass
>>> class E(object): pass
>>> class K1(A,B,C): pass
>>> class K2(D,B,E): pass
>>> class K3(D,A):   pass
>>> class Z(K1,K2,K3): pass

다음은 C3 MRO에 따른 선형화(linearizations)입니다. (독자는 연습 문제로 이 선형화 과정들을 확인하고 상속 다이어그램을 그려보시기 바랍니다 😉):

L[A] = A O
L[B] = B O
L[C] = C O
L[D] = D O
L[E] = E O
L[K1]= K1 A B C O
L[K2]= K2 D B E O
L[K3]= K3 D A O
L[Z] = Z K1 K2 K3 D A B C E O

Python 2.2는 A, B, C, D, E, K1, K2, K3에 대해 정확히 같은 선형화를 제공하지만, Z에 대해서는 다른 선형화를 제공합니다:

L[Z,P22] = Z K1 K3 A K2 D B C E O

이 선형화는 잘못*되었습니다. 왜냐하면 A가 D보다 먼저 오지만 K3의 선형화에서는 A가 D보다 *나중에 오기 때문입니다. 다시 말해, K3에서는 D가 파생시킨 메서드가 A가 파생시킨 메서드를 덮어쓰지만, Z에서는 [K3의 서브클래스]가 여전히 K3인 경우 A가 파생시킨 메서드가 D가 파생시킨 메서드를 덮어씁니다! 이것은 단조성 위반입니다. 게다가 Z의 Python 2.2 선형화는 로컬 우선순위 순서와도 일치하지 않습니다. 왜냐하면 클래스 Z의 로컬 우선순위 목록은 [K1, K2, K3]이며 (K2가 K3보다 먼저 오지만), Z의 선형화에서는 K2가 K3보다 나중에 오기 때문입니다. 이러한 문제들이 2.2 규칙이 C3 규칙에 의해 폐기된 이유를 설명합니다.

끝¶

이 섹션은 이전 섹션들을 건너뛰고 바로 끝까지 점프한 조급한 독자를 위한 것입니다. 또한 두뇌를 사용하고 싶지 않은 게으른 프로그래머를 위한 것이기도 합니다. 마지막으로, 어느 정도의 오만함을 지닌 프로그래머를 위한 것입니다. 그렇지 않다면/그녀는 다중 상속 계층 구조에서 C3 메서드 결정 순서에 대한 논문을 읽고 있지 않을 것입니다 😉 이 세 가지 미덕을 모두 갖춘 (그리고 개별적으로 아닌) 이들은 상을 받을 자격이 있습니다. 그 상은 당신의 두뇌에 위험을 주지 않으면서 2.3 MRO를 계산할 수 있게 해주는 짧은 Python 2.2 스크립트입니다. 단순히 마지막 줄을 변경하여 이 논문에서 논의한 다양한 예시를 가지고 놀아 보세요.

#<mro.py>

"""Samuele Pedroni가 작성한 C3 알고리즘입니다 (가독성 향상은 제가 담당했습니다)."""

class __metaclass__(type):
    "모든 클래스는 보기 좋게 출력되도록 메타마법적으로 수정됩니다."
    __repr__ = lambda cls: cls.__name__

class ex_2:
    "심각한 순서 불일치" #From Guido
    class O: pass
    class X(O): pass
    class Y(O): pass
    class A(X,Y): pass
    class B(Y,X): pass
    try:
        class Z(A,B): pass # Python 2.2에서는 Z(A,B)를 생성합니다
    except TypeError:
        pass # Python 2.3에서는 Z(A,B)를 생성할 수 없습니다

class ex_5:
    "첫 번째 예제"
    class O: pass
    class F(O): pass
    class E(O): pass
    class D(O): pass
    class C(D,F): pass
    class B(D,E): pass
    class A(B,C): pass

class ex_6:
    "두 번째 예제"
    class O: pass
    class F(O): pass
    class E(O): pass
    class D(O): pass
    class C(D,F): pass
    class B(E,D): pass
    class A(B,C): pass

class ex_9:
    "Python 2.2 MRO와 C3의 차이점" #From Samuele
    class O: pass
    class A(O): pass
    class B(O): pass
    class C(O): pass
    class D(O): pass
    class E(O): pass
    class K1(A,B,C): pass
    class K2(D,B,E): pass
    class K3(D,A): pass
    class Z(K1,K2,K3): pass

def merge(seqs):
    print '\n\nCPL[%s]=%s' % (seqs[0][0],seqs),
    res = []; i=0
    while 1:
      nonemptyseqs=[seq for seq in seqs if seq]
      if not nonemptyseqs: return res
      i+=1; print '\n',i,'round: candidates...',
      for seq in nonemptyseqs: # 시퀀스 헤드들 중에서 병합 후보를 찾습니다.
          cand = seq[0]; print ' ',cand,
          nothead=[s for s in nonemptyseqs if cand in s[1:]]
          if nothead: cand=None #후보를 거부합니다
          else: break
      if not cand: raise "Inconsistent hierarchy"
      res.append(cand)
      for seq in nonemptyseqs: # 후보를 제거합니다
          if seq[0] == cand: del seq[0]

def mro(C):
    "C3에 따라 클래스 우선순위 목록(mro)을 계산합니다."
    return merge([[C]]+map(mro,C.__bases__)+[list(C.__bases__)])

def print_mro(C):
    print '\nMRO[%s]=%s' % (C,mro(C))
    print '\nP22 MRO[%s]=%s' % (C,C.mro())

print_mro(ex_9.Z)

#</mro.py>
