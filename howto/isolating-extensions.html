확장 모듈 격리하기¶

개요

전통적으로 Python 확장 모듈에 속한 상태는 프로세스 전체 범위의 C static 변수에 보관되어 왔습니다. 이 문서는 이러한 프로세스별 상태의 문제점을 설명하고 더 안전한 방법인 모듈별 상태를 보여줍니다.

이 문서는 또한 가능할 때 모듈별 상태로 전환하는 방법을 설명합니다. 이 전환에는 해당 상태를 위한 공간 할당, 정적 타입에서 힙 타입으로의 잠재적 전환, 그리고 무엇보다 중요한 모듈별 상태에 코드로 접근하는 것이 포함됩니다.

이 문서를 읽어야 할 사람¶

이 지침서는 파이썬 자체를 라이브러리로 사용하는 애플리케이션에서 해당 확장을 더 안전하게 사용할 수 있도록 하려는 C-API 확장 개발자를 위해 작성되었습니다.

배경 지식¶

인터프리터 는 파이썬 코드가 실행되는 환경입니다. 여기에는 구성(예: import 경로)과 런타임 상태(예: 임포트된 모듈 세트)가 포함되어 있습니다.

Python은 하나의 프로세스에서 여러 인터프리터를 실행하는 것을 지원합니다. 고려해야 할 두 가지 경우가 있습니다. 즉, 사용자가 인터프리터를 다음과 같이 실행할 수 있습니다:

  • 순차적으로, 여러 Py_InitializeEx()/Py_FinalizeEx() 사이클을 거쳐, 그리고

  • 동시에, Py_NewInterpreter()/:c:func:`Py_EndInterpreter`를 사용하여 “서브 인터프리터”를 관리하면서 실행할 수 있습니다.

두 경우 모두 (그리고 이들의 조합) 라이브러리 내부에 Python을 임베딩할 때 가장 유용합니다. 라이브러리는 일반적으로 자신들을 사용하는 애플리케이션에 대해 가정해서는 안 되며, 여기에는 프로세스 전체의 “메인 Python 인터프리터”라는 가정을 포함합니다.

역사적으로 파이썬 확장 모듈은 이 사용 사례를 잘 처리하지 못했습니다. 많은 확장 모듈(심지어 일부 stdlib 모듈도)은 C의 static 변수를 사용하기가 매우 쉬워서 프로세스별 전역 상태를 사용합니다. 따라서 특정 인터프리터에만 있어야 할 데이터가 인터프리터들 사이에서 공유되는 경우가 발생합니다. 확장 개발자가 주의하지 않으면, 같은 프로세스에서 모듈이 여러 인터프리터에 로드될 때 충돌을 일으키는 엣지 케이스를 도입하기가 매우 쉽습니다.

안타깝게도 인터프리터별 상태를 얻기는 쉽지 않습니다. 확장 작성자들은 개발할 때 여러 인터프리터를 염두에 두지 않는 경향이 있으며, 현재로서는 그 동작을 테스트하는 것이 번거롭습니다.

모듈별 상태 입문¶

인터프리터별 상태에 초점을 맞추는 대신, Python의 C API는 더 세부적인 모듈별 상태를 더 잘 지원하도록 발전하고 있습니다. 이는 C 레벨 데이터가 모듈 객체 에 연결되어야 함을 의미합니다. 각 인터프리터는 자체 모듈 객체를 생성하여 데이터를 분리하게 합니다. 격리를 테스트하기 위해, 단일 확장과 관련된 여러 모듈 객체도 단일 인터프리터에 로드될 수 있습니다.

모듈별 상태는 수명 주기 및 리소스 소유권을 생각하기 쉽게 만듭니다. 확장 모듈은 모듈 객체가 생성될 때 초기화되고, 해제될 때 정리됩니다. 이러한 점에서, 모듈은 다른 PyObject* 와 같습니다. 생각하거나 잊을 “인터프리터 종료 시” 훅이 없습니다.

“전역”의 여러 종류의 사용 사례가 있다는 점에 유의하십시오: 프로세스별, 인터프리터별, 스레드별 또는 작업별 상태가 있습니다. 모듈별 상태가 기본값이지만, 이들은 여전히 가능합니다. 그러나 이들을 예외적인 경우로 처리해야 합니다: 필요하다면, 추가적인 주의와 테스트를 기울여야 합니다. (이 가이드에서는 다루지 않습니다.)

격리된 모듈 객체¶

확장 모듈을 개발할 때 염두에 두어야 할 핵심 사항은 단일 공유 라이브러리로부터 여러 모듈 객체가 생성될 수 있다는 것입니다. 예를 들어:

>>> import sys
>>> import binascii
>>> old_binascii = binascii
>>> del sys.modules['binascii']
>>> import binascii  # 새 모듈 객체 생성
>>> old_binascii == binascii
False

경험칙으로, 두 모듈은 완전히 독립적이어야 합니다. 모듈에 특정적인 모든 객체와 상태는 모듈 객체 내부에 캡슐화되어야 하며, 다른 모듈 객체와 공유되어서는 안 되고, 모듈 객체가 할당 해제될 때 정리되어야 합니다. 이것은 단지 경험칙일 뿐이므로 예외는 가능합니다 (Managing Global State 참조), 하지만 이 경우 더 많은 생각과 엣지 케이스에 대한 주의가 필요할 것입니다.

일부 모듈은 덜 엄격한 제한이 필요할 수도 있지만, 격리된 모듈은 다양한 사용 사례 전반에 걸쳐 작동하는 명확한 기대치와 가이드를 설정하기 쉽게 만듭니다.

예상을 벗어난 엣지 케이스¶

격리된 모듈은 일부 놀라운 엣지 케이스를 생성한다는 점에 유념하십시오. 가장 주목할 만한 점은, 각 모듈 객체가 일반적으로 다른 유사한 모듈과 클래스 및 예외를 공유하지 않는다는 것입니다. 위의 example above 를 계속하면, old_binascii.Error 와 binascii.Error 가 별개의 객체라는 점에 유념하십시오. 다음 코드에서는 예외가 포착되지 않습니다:

>>> old_binascii.Error == binascii.Error
False
>>> try:
...     old_binascii.unhexlify(b'qwertyuiop')
... except binascii.Error:
...     print('boo')
...
Traceback (most recent call last):
  File "<stdin>", line 2, in <module>
binascii.Error: Non-hexadecimal digit found

이는 예상된 동작입니다. 순수 Python 모듈 역시 같은 방식으로 동작하는 것을 알 수 있습니다: 이것은 Python 작동 방식의 일부입니다.

목표는 확장 모듈을 C 레벨에서 안전하게 만드는 것이지, 해커들이 직관적으로 행동하게 만드는 것이 아닙니다. sys.modules 를 “수동으로” 변경하는 것은 해킹에 해당합니다.

여러 인터프리터로 모듈 안전하게 만들기¶

전역 상태 관리¶

파이썬 모듈에 연결된 상태는 모듈 자체에 국한되지 않고 전체 프로세스(또는 모듈보다 “더 전역적인” 다른 무언가)에 국한될 때가 있습니다. 예를 들어:

  • readline 모듈은 터미널 전체를 관리합니다.

  • 회로 기판에서 실행되는 모듈이 온보드 LED를 제어하려고 합니다.

이러한 경우, 파이썬 모듈은 소유 하기보다는 전역 상태에 대한 접근 을 제공해야 합니다. 가능하다면, 모듈을 작성하여 여러 복사본이 상태에 독립적으로 접근할 수 있도록 하십시오(파이썬이나 다른 언어의 다른 라이브러리와 함께). 이것이 불가능한 경우, 명시적인 잠금(locking)을 고려하십시오.

프로세스 전역 상태를 사용해야 하는 경우, 여러 인터프리터 간의 문제를 피하는 가장 간단한 방법은 모듈이 프로세스당 한 번만 로드되도록 명시적으로 방지하는 것입니다. :ref:`절연 확장 옵트아웃<isolating-extensions-optout>`을 참조하십시오.

모듈별 상태 관리¶

모듈별 상태를 사용하려면 :ref:`다단계 확장 모듈 초기화<multi-phase-initialization>`를 사용하십시오. 이는 모듈이 여러 인터프리터를 올바르게 지원한다는 신호를 보냅니다.

PyModuleDef.m_size`를 양수 값으로 설정하여 모듈에 국한된 저장 공간을 요청할 수 있습니다. 보통 이는 모듈의 C 수준 상태 전체를 저장하는 데 사용할 수 있는 모듈별 ``struct``의 크기로 설정됩니다. 특히, C 코드가 기능하는 데 필요한 클래스(예외 포함, 정적 타입 제외)와 설정(예: ``csv``의 :py:data:`~csv.field_size_limit)에 대한 포인터를 여기에 넣어야 합니다.

참고

다른 방법은 모듈의 __dict__ 에 상태를 저장하는 것이지만, 사용자가 파이썬 코드에서 __dict__ 를 수정할 때 충돌하는 것을 피해야 합니다. 이는 일반적으로 C 수준의 오류 및 타입 검사를 의미하며, 이는 잘못하기 쉬워 충분히 테스트하기 어렵습니다.

하지만 모듈 상태가 C 코드에서 필요하지 않은 경우, __dict__ 에만 저장하는 것은 좋은 생각입니다.

모듈 상태가 PyObject 포인터를 포함하는 경우, 모듈 객체는 해당 객체에 대한 참조를 유지해야 하며 모듈 수준 훅 m_traverse, m_clear 및 m_free 를 구현해야 합니다. 이것은 클래스의 tp_traverse, tp_clear 및 tp_free 와 유사하게 작동합니다. 이를 추가하면 약간의 작업이 필요하여 코드가 더 길어집니다. 이는 깨끗하게 언로드될 수 있는 모듈에 대한 대가입니다.

모듈별 상태를 가진 모듈의 예시는 현재 `xxlimited <https://github.com/python/cpython/blob/master/Modules/xxlimited.c>`__에서 사용할 수 있습니다. 예시 모듈 초기화는 파일 하단에 나와 있습니다.

선택해제: 프로세스당 단일 모듈 객체로 제한¶

음수가 아닌 PyModuleDef.m_size 는 모듈이 여러 인터프리터를 올바르게 지원한다는 신호입니다. 모듈이 아직 이 경우에 해당하지 않는다면, 모듈이 프로세스당 한 번만 로드 가능하도록 명시적으로 만들 수 있습니다. 예를 들면:

// 프로세스 전체 플래그
static int loaded = 0;

// 스레드 안전성을 제공하기 위한 뮤텍스 (free-threaded Python에만 필요)
static PyMutex modinit_mutex = {0};

static int
exec_module(PyObject* module)
{
    PyMutex_Lock(&modinit_mutex);
    if (loaded) {
        PyMutex_Unlock(&modinit_mutex);
        PyErr_SetString(PyExc_ImportError,
                        "cannot load module more than once per process");
        return -1;
    }
    loaded = 1;
    PyMutex_Unlock(&modinit_mutex);
    // ... 나머지 초기화
}

모듈의 PyModuleDef.m_clear 함수가 향후 재초기화를 준비할 수 있다면, loaded 플래그를 지워야 합니다. 이 경우, 모듈은 동시 로 여러 인스턴스가 존재함을 지원하지 않지만, 예를 들어 파이썬 런타임 종료 (Py_FinalizeEx()) 후 및 재초기화 (Py_Initialize()) 후 로드가 지원되기는 합니다.

함수에서의 모듈 상태 접근¶

모듈 수준 함수에서 상태에 접근하는 것은 간단합니다. 함수는 첫 번째 인수로 모듈 객체를 받으며, 상태를 추출하려면 PyModule_GetState 를 사용하면 됩니다.

static PyObject *
func(PyObject *module, PyObject *args)
{
    my_struct *state = (my_struct*)PyModule_GetState(module);
    if (state == NULL) {
        return NULL;
    }
    // ... 나머지 로직
}

참고

PyModule_GetState may return NULL without setting an exception if there is no module state, i.e. PyModuleDef.m_size was zero. In your own module, you’re in control of m_size, so this is easy to prevent.

힙 형¶

전통적으로 C 코드에서 정의된 형은 정적(static) 입니다. 즉, 코드에 직접 정의되고 PyType_Ready() 를 사용하여 초기화되는 static PyTypeObject 구조체입니다.

이러한 형들은 프로세스 전반에 걸쳐 반드시 공유됩니다. 이들을 모듈 객체 간에 공유하려면, 해당 객체가 소유하거나 접근하는 모든 상태에 주의를 기울여야 합니다. 가능한 문제를 제한하기 위해, 정적 형은 파이썬 레벨에서 불변입니다. 예를 들어, str.myattribute = 123 와 같이 설정할 수 없습니다.

진정으로 불변인 객체를 인터프리터 간에 공유하는 것은 문제가 없습니다. 단, 가변 객체에 대한 접근을 제공하지 않는다는 전제가 필요합니다. 그러나 CPython에서는 모든 파이썬 객체가 가변적인 구현 세부 사항, 즉 참조 횟수를 가지고 있습니다. refcount 변경은 GIL에 의해 보호됩니다. 따라서 인터프리터 전반에 걸쳐 파이썬 객체를 공유하는 코드는 암시적으로 CPython의 현재 프로세스 전체 GIL에 의존하게 됩니다.

정적 형처럼 불변하고 프로세스 전역적인 객체는 “자신들의” 모듈 상태에 접근할 수 없습니다. 이러한 유형의 메서드가 모듈 상태에 접근해야 하는 경우, 해당 유형은 힙 할당 유형(heap-allocated type), 또는 줄여서 힙 형(heap type) 으로 변환되어야 합니다. 이는 파이썬의 class 문으로 생성된 클래스와 더 가깝습니다.

새 모듈의 경우, 기본적으로 힙 형을 사용하는 것이 좋은 관례입니다.

정적 형을 힙 형으로 변경하기¶

정적 형은 힙 형으로 변환될 수 있지만, 힙 형 API는 정적 형으로부터의 “손실 없는” 변환을 위해 설계된 것이 아니라는 점에 유의해야 합니다. 즉, 주어진 정적 형과 정확히 동일하게 작동하는 유형을 만드는 것은 아닙니다. 따라서 새로운 API에서 클래스 정의를 다시 작성할 때 몇 가지 세부 사항(예: pickle 가능성 또는 상속된 슬롯)이 의도치 않게 변경될 가능성이 높습니다. 중요한 세부 사항은 항상 테스트해야 합니다.

특히 다음 두 가지 사항을 유의하십시오 (하지만 이것이 전체 목록은 아님에 유의하십시오):

  • 정적 형과 달리, 힙 형 객체는 기본적으로 가변입니다. Py_TPFLAGS_IMMUTABLETYPE 플래그를 사용하여 가변성을 방지하십시오.

  • 힙 형은 기본적으로 tp_new 를 상속받기 때문에, 파이썬 코드로 인스턴스화하는 것이 가능해질 수 있습니다. 이는 Py_TPFLAGS_DISALLOW_INSTANTIATION 플래그로 방지할 수 있습니다.

힙 형 정의하기¶

힙 형은 PyType_Spec 구조체, 즉 클래스의 설명이나 “청사진”을 채우고, :c:func:`PyType_FromModuleAndSpec`을 호출하여 새로운 클래스 객체를 구성함으로써 생성할 수 있습니다.

참고

:c:func:`PyType_FromSpec`과 같은 다른 함수들도 힙 형을 생성할 수 있지만, :c:func:`PyType_FromModuleAndSpec`은 모듈을 클래스와 연관 지어 메서드에서 모듈 상태에 접근할 수 있게 합니다.

해당 클래스는 일반적으로 모듈 상태(C에서 안전하게 접근하기 위함)와 모듈의 __dict__ (파이썬 코드에서 접근하기 위함) 둘 다 에 저장되어야 합니다.

가비지 수집 프로토콜¶

힙 형의 인스턴스는 자신의 유형에 대한 참조를 가집니다. 이는 모든 인스턴스가 파괴되기 전에 유형이 파괴되지 않도록 보장하지만, 가비지 컬렉터가 풀어야 하는 참조 순환을 초래할 수도 있습니다.

메모리 누수를 방지하려면, 힙 형의 인스턴스는 가비지 컬렉션 프로토콜을 구현해야 합니다. 즉, 힙 형은 다음을 해야 합니다:

  • Py_TPFLAGS_HAVE_GC 플래그를 가지고 있어야 합니다.

  • Py_tp_traverse 를 사용하여 순회 함수를 정의하고, 이 함수는 유형을 방문합니다 (예: Py_VISIT(Py_TYPE(self)) 사용).

추가 고려 사항에 대해서는 :c:macro:`Py_TPFLAGS_HAVE_GC`와 :c:member:`~PyTypeObject.tp_traverse`의 문서를 참조하십시오.

힙 형을 정의하는 API는 유기적으로 성장했으며, 현재 상태에서는 사용하기가 다소 어색합니다. 다음 섹션들은 일반적인 문제들을 통해 안내해 드릴 것입니다.

Python 3.8 이하 버전의 tp_traverse¶

tp_traverse 에서 유형을 방문해야 하는 요구사항은 Python 3.9에 추가되었습니다. Python 3.8 이하를 지원하는 경우, 순회 함수에서 유형을 방문해서는 안 되므로, 더 복잡해야 합니다:

static int my_traverse(PyObject *self, visitproc visit, void *arg)
{
    if (Py_Version >= 0x03090000) {
        Py_VISIT(Py_TYPE(self));
    }
    return 0;
}

안타깝게도 :c:data:`Py_Version`은 Python 3.11에 추가되었습니다. 대안으로 다음을 사용하십시오:

  • :c:macro:`PY_VERSION_HEX`가 안정적인 ABI를 사용하지 않는 경우, 또는

  • sys.version_info ( PySys_GetObject() 및 :c:func:`PyArg_ParseTuple`을 통해 접근).

tp_traverse 위임¶

If your traverse function delegates to the tp_traverse of its base class (or another type), ensure that Py_TYPE(self) is visited only once. Note that only heap type are expected to visit the type in tp_traverse.

예를 들어, 사용자 트래버스 함수가 다음을 포함하는 경우:

base->tp_traverse(self, visit, arg)

…그리고 base\가 정적 형식일 수 있다면, 다음도 포함해야 합니다:

if (base->tp_flags & Py_TPFLAGS_HEAPTYPE) {
    // 힙형의 tp_traverse가 이미 Py_TYPE(self)를 방문했습니다
} else {
    if (Py_Version >= 0x03090000) {
        Py_VISIT(Py_TYPE(self));
    }
}

tp_new 및 :c:member:`~PyTypeObject.tp_clear`에서는 형식의 참조 횟수를 처리할 필요가 없습니다.

tp_dealloc 정의¶

형에 사용자 정의 tp_dealloc 함수가 있는 경우, 다음을 수행해야 합니다:

  • 필드가 무효화되기 전에 PyObject_GC_UnTrack()\를 호출하고, 그리고

  • 형의 참조 횟수를 감소시켜야 합니다.

tp_free\이 호출되는 동안 형을 유효하게 유지하려면, 인스턴스가 해제된 후에 형의 참조 횟수를 감소시켜야 합니다. 예를 들어:

static void my_dealloc(PyObject *self)
{
    PyObject_GC_UnTrack(self);
    ...
    PyTypeObject *type = Py_TYPE(self);
    type->tp_free(self);
    Py_DECREF(type);
}

기본 tp_dealloc 함수가 이를 수행하므로, 형이 tp_dealloc 을 덮어쓰지 않았다면 추가할 필요가 없습니다.

tp_free 덮어쓰지 않기¶

힙형의 tp_free 슬롯은 :c:func:`PyObject_GC_Del`로 설정되어야 합니다. 이것이 기본값입니다. 덮어쓰지 마십시오.

PyObject_New 피하기¶

GC로 추적되는 객체는 GC를 인지하는 함수를 사용하여 할당해야 합니다.

PyObject_New() 또는 :c:func:`PyObject_NewVar`를 사용하는 경우:

  • 가능하다면, 형식의 tp_alloc 슬롯을 가져와 호출해야 합니다. 즉, TYPE *o = PyObject_New(TYPE, typeobj)\를 다음으로 대체하십시오:

    TYPE *o = typeobj->tp_alloc(typeobj, 0);

    o = PyObject_NewVar(TYPE, typeobj, size)\를 동일하게 대체하되, 0 대신 size를 사용하십시오.

  • 위 방법이 불가능한 경우 (예: 사용자 정의 tp_alloc 내부), PyObject_GC_New() 또는 PyObject_GC_NewVar()\를 호출하십시오:

    TYPE *o = PyObject_GC_New(TYPE, typeobj);

TYPE *o = PyObject_GC_NewVar(TYPE, typeobj, size);

클래스로부터의 모듈 상태 접근¶

PyType_FromModuleAndSpec()\로 정의된 형식 객체가 있는 경우, :c:func:`PyType_GetModule`을 호출하여 연결된 모듈을 얻을 수 있고, 이후 :c:func:`PyModule_GetState`를 호출하여 모듈의 상태를 얻을 수 있습니다.

번거로운 오류 처리 상용구 코드를 줄이려면, 이 두 단계를 PyType_GetModuleState()\로 결합하여 다음의 결과를 얻을 수 있습니다:

my_struct *state = (my_struct*)PyType_GetModuleState(type);
if (state == NULL) {
    return NULL;
}

일반 메서드로부터의 모듈 상태 접근¶

클래스 메서드에서 모듈 수준의 상태에 접근하는 것은 다소 복잡하지만, Python 3.9에 도입된 API 덕분에 가능합니다. 상태를 얻으려면 먼저 정의하는 클래스 (defining class)를 얻은 다음, 그 클래스에서 모듈 상태를 얻어야 합니다.

가장 큰 장애물은 메서드가 정의된 클래스 (the class a method was defined in)나 간단히 그 메서드의 “정의하는 클래스”를 얻는 것입니다. 정의하는 클래스는 자신이 속한 모듈에 대한 참조를 가질 수 있습니다.

정의하는 클래스를 Py_TYPE(self) \와 혼동하지 마십시오. 메서드가 형식의 서브 클래스 에서 호출되는 경우, Py_TYPE(self) \는 해당 서브 클래스를 참조하며, 이는 자신의 것과 다른 모듈에 정의되었을 수 있습니다.

참고

다음 Python 코드가 개념을 보여줄 수 있습니다. Base.get_defining_class\는 심지어 type(self) == Sub\이더라도 Base\를 반환합니다:

class Base:
    def get_type_of_self(self):
        return type(self)

    def get_defining_class(self):
        return __class__

class Sub(Base):
    pass

메서드가 “정의하는 클래스”를 얻으려면, METH_METHOD | METH_FASTCALL | METH_KEYWORDS calling convention 와 해당 PyCMethod 시그니처를 사용해야 합니다:

PyObject *PyCMethod(
    PyObject *self,               // 메서드가 호출된 객체
    PyTypeObject *defining_class, // 정의하는 클래스
    PyObject *const *args,        // 인수들의 C 배열
    Py_ssize_t nargs,             // "args"의 길이
    PyObject *kwnames)            // NULL 또는 키워드 인수 딕셔너리

정의하는 클래스를 알게 되면, PyType_GetModuleState()\를 호출하여 연결된 모듈의 상태를 얻으십시오.

예를 들면

static PyObject *
example_method(PyObject *self,
        PyTypeObject *defining_class,
        PyObject *const *args,
        Py_ssize_t nargs,
        PyObject *kwnames)
{
    my_struct *state = (my_struct*)PyType_GetModuleState(defining_class);
    if (state == NULL) {
        return NULL;
    }
    ... // 나머지 로직
}

PyDoc_STRVAR(example_method_doc, "...");

static PyMethodDef my_methods[] = {
    {"example_method",
      (PyCFunction)(void(*)(void))example_method,
      METH_METHOD|METH_FASTCALL|METH_KEYWORDS,
      example_method_doc}
    {NULL},
}

슬롯 메서드, 게터 및 세터에서의 모듈 상태 접근¶

참고

이는 Python 3.11에서 새롭게 도입되었습니다.

슬롯 메서드는, :c:member:`~PyNumberMethods.nb_add`의 :py:attr:`~object.__add__`나 초기화를 위한 :c:member:`~PyTypeObject.tp_new`와 같은 특별한 메서드에 대한 빠르고 간결한 C 구현체이며, :c:type:`PyCMethod`와 달리 정의하는 클래스를 전달할 수 없습니다. :c:type:`PyGetSetDef`로 정의된 게터 및 세터도 마찬가지입니다.

이러한 경우에 모듈 상태에 접근하려면 PyType_GetModuleByDef() 함수를 사용하고 모듈 정의를 전달해야 합니다. 모듈을 얻은 후, :c:func:`PyModule_GetState`를 호출하여 상태를 가져옵니다:

PyObject *module = PyType_GetModuleByDef(Py_TYPE(self), &module_def);
my_struct *state = (my_struct*)PyModule_GetState(module);
if (state == NULL) {
    return NULL;
}

PyType_GetModuleByDef() works by searching the method resolution order (i.e. all superclasses) for the first superclass that has a corresponding module.

참고

매우 예외적인 경우(동일한 정의에서 생성된 여러 모듈에 걸친 상속 체인)에는, :c:func:`!PyType_GetModuleByDef`가 실제 정의하는 클래스의 모듈을 반환하지 않을 수 있습니다. 하지만, 항상 동일한 정의를 가진 모듈을 반환하여 호환되는 C 메모리 레이아웃을 보장합니다.

모듈 상태의 생명주기¶

모듈 객체가 가비지 수집되면, 해당 모듈의 상태도 해제됩니다. 모듈 상태의 (일부) 포인터마다 모듈 객체에 대한 참조를 보유해야 합니다.

일반적으로 이는 문제가 되지 않습니다. 왜냐하면 :c:func:`PyType_FromModuleAndSpec`으로 생성된 타입과 그 인스턴스는 모듈에 대한 참조를 보유하기 때문입니다. 하지만 외부 라이브러리의 콜백과 같이 다른 곳에서 모듈 상태를 참조할 때는 참조 횟수에 주의해야 합니다.

열린 이슈¶

모듈별 상태 및 힙 타입과 관련된 몇 가지 이슈는 여전히 열려 있습니다.

상황 개선에 대한 논의는 `discuss forum under c-api tag <https://discuss.python.org/c/core-dev/c-api/30>`에서 하는 것이 가장 좋습니다.

클래스별 범위¶

현재는 (Python 3.11 기준) CPython 구현 세부 사항에 의존하지 않고 개별 타입 에 상태를 첨부하는 것이 불가능합니다(이 세부 사항은 미래에 변경될 수 있습니다. 어쩌면, 역설적으로, 클래스별 범위를 위한 적절한 해결책을 허용하도록 변경될 수도 있습니다).

힙 타입으로의 손실 없는 변환¶

힙 타입 API는 정적 타입으로부터 “손실 없는” 변환을 위해 설계되지 않았습니다. 즉, 주어진 정적 타입과 정확히 똑같이 작동하는 타입을 생성하는 것이 아닙니다.
