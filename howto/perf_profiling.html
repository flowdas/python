Python의 perf map 호환 프로파일러 지원¶

`The Linux perf profiler <https://perf.wiki.kernel.org>`_와 `samply <https://github.com/mstange/samply>`_는 애플리케이션의 성능을 프로파일링하고 관련된 정보를 얻을 수 있는 강력한 도구입니다. 두 도구 모두 생성하는 데이터를 분석하는 데 도움을 주는 활발한 생태계를 가지고 있습니다.

이러한 프로파일러를 Python 애플리케이션에 사용하는 주요 문제점은 네이티브 심볼, 즉 C로 작성된 함수와 절차의 이름에 대한 정보만 얻는다는 것입니다. 이는 코드의 Python 함수 이름과 파일명이 프로파일러 출력에 표시되지 않음을 의미합니다.

Python 3.12부터 인터프리터는 Python 함수가 호환 가능한 프로파일러의 출력에 나타날 수 있도록 특별한 모드로 실행될 수 있습니다. 이 모드가 활성화되면, 모든 Python 함수 실행 전에 인터프리터가 작은 코드를 즉석에서 컴파일하여 삽입하고, :doc:`perf map files <../c-api/perfmaps>`를 사용하여 이 코드 조각과 관련된 Python 함수 간의 관계를 프로파일러에 알려줄 것입니다.

참고

프로파일링 지원은 Linux 및 macOS의 특정 아키텍처에서 사용할 수 있습니다. Perf는 Linux에서 사용 가능하며, samply는 Linux와 macOS 모두에서 사용할 수 있습니다. samply의 macOS 지원은 Python 3.15부터 사용할 수 있습니다. 시스템이 지원되는지 확인하려면 configure 빌드 단계의 출력이나 python -m sysconfig | grep HAVE_PERF_TRAMPOLINE 의 출력을 확인하십시오.

예를 들어, 다음 스크립트를 고려해 보십시오:

def foo(n):
    result = 0
    for _ in range(n):
        result += 1
    return result

def bar(n):
    foo(n)

def baz(n):
    bar(n)

if __name__ == "__main__":
    baz(1000000)

`perf`를 사용하여 9999 헤르츠에서 CPU 스택 트레이스를 샘플링할 수 있습니다:

$ perf record -F 9999 -g -o perf.data python my_script.py

samply 프로파일러 사용¶

samply는 perf의 대안으로 사용될 수 있는 모던한 프로파일러입니다. 이 프로파일러는 Python이 생성하는 것과 동일한 perf map 파일을 사용하므로 Python 프로파일링 지원과 호환됩니다. samply는 perf를 사용할 수 없는 macOS에서 특히 유용합니다.

samply를 Python과 함께 사용하려면, 먼저 https://github.com/mstange/samply의 지침에 따라 설치한 다음, 다음을 실행하십시오:

$ samply record PYTHONPERFSUPPORT=1 python my_script.py

이것은 프로파일링 데이터를 대화형으로 분석할 수 있는 웹 인터페이스를 열어줍니다. samply의 장점은 프로파일링 데이터를 분석하기 위한 현대적인 웹 기반 인터페이스를 제공하며 리눅스와 macOS 모두에서 작동한다는 것입니다.

macOS에서는 samply 지원을 위해 Python 3.15 이상이 필요합니다. 또한 macOS에서 samply는 macOS의 제한 사항으로 인해 서명된 Python 실행 파일의 프로파일링을 할 수 없습니다. 직접 컴파일했거나 서명되지 않았거나 로컬로 서명된(예: Homebrew로 설치된 모든 것) Python 바이너리로 프로파일링을 할 수 있습니다. macOS에서 실행 중인 프로세스에 연결하려면, samply 바이너리를 자체 서명하기 위해 `samply setup`을 한 번 실행하십시오 (그리고 samply가 업데이트될 때마다 실행해야 합니다).

perf 프로파일링 지원 활성화 방법¶

perf 프로파일링 지원은 환경 변수 PYTHONPERFSUPPORT`를 사용하거나 :option:-X perf <-X>` 옵션을 사용하여 처음부터 활성화하거나, 또는 sys.activate_stack_trampoline() 및 :func:`sys.deactivate_stack_trampoline`을 사용하여 동적으로 활성화할 수 있습니다.

sys 함수는 -X 옵션보다 우선하며, -X 옵션은 환경 변수보다 우선합니다.

예시: 환경 변수 사용:

$ PYTHONPERFSUPPORT=1 perf record -F 9999 -g -o perf.data python my_script.py
$ perf report -g -i perf.data

예시: -X 옵션 사용:

$ perf record -F 9999 -g -o perf.data python -X perf my_script.py
$ perf report -g -i perf.data

예시: 파일 example.py`에서 :mod:`sys API 사용:

import sys

sys.activate_stack_trampoline("perf")
do_profiled_stuff()
sys.deactivate_stack_trampoline()

non_profiled_stuff()

…그런 다음:

$ perf record -F 9999 -g -o perf.data python ./example.py
$ perf report -g -i perf.data

최상의 결과를 얻는 방법¶

최상의 결과를 얻으려면 프레임 포인터를 활성화 상태로 유지하십시오. 지원되는 GCC-호환 툴체인은 CPython을 -fno-omit-frame-pointer 및 유사한 플래그로 빌드합니다 (자세한 내용은 --without-frame-pointers`를 참조하십시오). 이 플래그들은 프로파일러가 DWARF 디버그 정보가 아닌 프레임 포인터만 사용하여 언와인딩할 수 있도록 허용합니다. 이는 ``perf` 지원을 위해 삽입된 코드가 동적으로 생성되므로 DWARF 디버깅 정보가 없기 때문입니다.

시스템이 이 플래그로 컴파일되었는지 확인하려면 다음을 실행하십시오:

$ python -m sysconfig | grep 'no-omit-frame-pointer'

출력이 없으면 인터프리터가 프레임 포인터로 컴파일되지 않았음을 의미하므로, perf 의 출력에서 Python 함수를 표시하지 못할 수 있습니다.

프레임 포인터 없이 작업하는 방법¶

프레임 포인터 없이 컴파일된 Python 인터프리터로 작업하는 경우에도 계속 perf 프로파일러를 사용할 수 있지만, Python이 매번 Python 함수 호출에 대해 언와인딩 정보를 실시간으로 생성해야 하므로 오버헤드가 약간 높을 것입니다. 또한, perf 는 스택을 언와인딩하기 위해 DWARF 디버그 정보를 사용해야 하므로 더 많은 시간이 걸립니다.

이 모드를 활성화하려면 환경 변수 PYTHON_PERF_JIT_SUPPORT`를 사용하거나 :option:-X perf_jit <-X>` 옵션을 사용할 수 있으며, 이는 perf 프로파일러의 JIT 모드를 활성화합니다.

참고

perf 도구의 버그 때문에, JIT 모드는 v6.8보다 높은 버전의 perf 에서만 작동합니다. 이 수정 사항은 도구의 v6.7.2 버전으로도 백포트되었습니다.

perf 도구의 버전을 확인할 때 (perf version 을 실행하여 확인할 수 있습니다) 어떤 배포판에서는 - 문자를 포함한 일부 사용자 지정 버전 번호를 추가할 수 있다는 점에 유의해야 합니다. 즉, perf 6.7-3 이 반드시 perf 6.7.3 을 의미하는 것은 아닙니다.

perf JIT 모드를 사용할 때는 perf report``를 실행하기 전에 추가 단계가 필요합니다. JIT 정보를 ``perf.data 파일에 주입하려면 perf inject 명령을 호출해야 합니다:

$ perf record -F 9999 -g -k 1 --call-graph dwarf -o perf.data python -Xperf_jit my_script.py
$ perf inject -i perf.data --jit --output perf.jit.data
$ perf report -g -i perf.jit.data

또는 환경 변수를 사용하는 방법:

$ PYTHON_PERF_JIT_SUPPORT=1 perf record -F 9999 -g --call-graph dwarf -o perf.data python my_script.py
$ perf inject -i perf.data --jit --output perf.jit.data
$ perf report -g -i perf.jit.data

perf inject --jit 명령어는 perf.data 파일을 읽고, Python이 생성한 perf 덤프 파일(/tmp/perf-$PID.dump``에 위치)을 자동으로 가져와 모든 JIT 정보를 통합하는 ``perf.jit.data 파일을 생성합니다. 또한 현재 디렉터리에 Python에 의해 생성된 모든 JIT 트램폴린에 대한 ELF 이미지를 담은 jitted-XXXX-N.so 파일을 많이 생성해야 합니다.

경고

--call-graph dwarf 를 사용할 경우, perf 도구는 프로파일링 중인 프로세스의 스택 스냅샷을 찍어 해당 정보를 perf.data 파일에 저장합니다. 기본적으로 스택 덤프의 크기는 8192바이트이지만, 쉼표 뒤에 --call-graph dwarf,16384 와 같이 값을 전달하여 크기를 변경할 수 있습니다.

스택 덤프 크기는 중요합니다. 크기가 너무 작으면 perf 가 스택을 언와인드(unwind)하지 못하여 결과가 불완전해질 수 있습니다. 반면에 크기가 너무 크면 오버헤드가 높아져서 perf 가 원하는 만큼 프로세스를 자주 샘플링할 수 없습니다.

스택 크기는 특히 낮은 최적화 수준(예: -O0)으로 컴파일된 Python 코드를 프로파일링할 때 중요합니다. 이러한 빌드는 더 큰 스택 프레임을 갖는 경향이 있기 때문입니다. Python을 -O0 으로 컴파일했는데 프로파일링 결과에서 Python 함수가 보이지 않는다면, 스택 덤프 크기를 최대값인 65528바이트로 늘려보세요:

$ perf record -F 9999 -g -k 1 --call-graph dwarf,65528 -o perf.data python -Xperf_jit my_script.py

컴파일 플래그가 다르면 스택 크기에 큰 영향을 미칠 수 있습니다:

  • -O0``으로 빌드된 경우, 일반적으로 ``-O1 또는 그 이상의 최적화 수준으로 빌드된 경우보다 스택 프레임이 훨씬 더 큽니다.

  • 최적화(-O1, -O2 등)를 추가하면 일반적으로 스택 크기가 줄어듭니다.

  • 프레임 포인터(-fno-omit-frame-pointer)는 일반적으로 더 신뢰할 수 있는 스택 언와인딩을 제공합니다.
