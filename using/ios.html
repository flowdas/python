7. iOS에서 파이썬 사용하기¶

저자:

러스셀 키스-메지 (2024-03)

iOS의 Python은 데스크톱 플랫폼의 Python과는 다릅니다. 데스크톱 플랫폼에서 Python은 일반적으로 시스템 리소스로 설치되어 해당 컴퓨터의 모든 사용자가 사용할 수 있습니다. 사용자는 python 실행 파일을 실행하고 대화형 프롬프트에 명령을 입력하거나, 파이썬 스크립트를 실행하여 Python과 상호 작용합니다.

iOS에서는 시스템 리소스로 설치하는 개념 자체가 없습니다. 유일한 소프트웨어 배포 단위는 “앱”입니다. 또한 python 실행 파일을 실행하거나 Python REPL과 상호 작용할 수 있는 콘솔도 없습니다.

결과적으로, iOS에서 Python을 사용할 수 있는 유일한 방법은 임베디드 모드입니다. 즉, 네이티브 iOS 애플리케이션을 작성하고 libPython 을 사용하여 Python 인터프리터를 임베딩하며, Python embedding API 을 사용하여 Python 코드를 호출하는 방식입니다. 전체 Python 인터프리터, 표준 라이브러리 및 모든 Python 코드는 이제 iOS App Store를 통해 배포할 수 있는 독립적인 번들로 패키징됩니다.

Python으로 iOS 앱 개발을 처음 실험해 보려면, BeeWare 및 `Kivy <https://kivy.org>`__와 같은 프로젝트가 훨씬 더 접근하기 쉬운 사용자 경험을 제공할 것입니다. 이러한 프로젝트들은 iOS 프로젝트를 실행하는 것과 관련된 복잡성을 관리하므로, 사용자는 순전히 Python 코드 자체만 다루면 됩니다.

7.1. iOS에서의 런타임 Python¶

7.1.1. iOS 버전 호환성¶

최소 지원 iOS 버전은 --host 옵션을 사용하여 configure 시점에 지정됩니다. 기본적으로 iOS용으로 컴파일된 경우, Python은 최소 지원 iOS 버전 13.0으로 컴파일됩니다. 다른 최소 iOS 버전을 사용하려면, --host 인수로 버전 번호를 제공해야 합니다. 예를 들어, --host=arm64-apple-ios15.4-simulator 는 배포 대상이 15.4인 ARM64 시뮬레이터 빌드를 컴파일합니다.

7.1.2. 플랫폼 식별¶

iOS에서 실행할 때, sys.platform 은 ios 로 보고됩니다. 이 값은 앱이 시뮬레이터에서 실행되는지 물리 장치에서 실행되는지와 관계없이 iPhone 또는 iPad에서 반환됩니다.

iOS 버전, 장치 모델, 그리고 장치가 시뮬레이터인지 여부를 포함하는 특정 런타임 환경에 대한 정보는 platform.ios_ver() 를 사용하여 얻을 수 있습니다. platform.system() 은 장치에 따라 iOS 또는 iPadOS 를 보고합니다.

os.uname() 은 커널 수준의 세부 정보를 보고하며, 그 결과는 Darwin 이라는 이름을 보고합니다.

7.1.3. 표준 라이브러리 가용성¶

Python 표준 라이브러리는 iOS에서 몇 가지 주목할 만한 누락 및 제한 사항이 있습니다. 자세한 내용은 :ref:`API availability guide for iOS <mobile-availability>`를 참조하십시오.

7.1.4. 바이너리 확장 모듈¶

플랫폼으로서의 iOS에 관한 주목할 만한 차이점 중 하나는 App Store 배포가 애플리케이션 패키징에 엄격한 요구 사항을 부과한다는 것입니다. 이러한 요구 사항 중 하나는 바이너리 확장 모듈이 어떻게 배포되는지를 규율합니다.

iOS App Store는 iOS 앱의 모든 바이너리 모듈이 적절한 메타데이터를 가진 프레임워크 내에 포함된 동적 라이브러리여야 하며, 패키징된 앱의 Frameworks 폴더에 저장되어야 한다고 요구합니다. 프레임워크당 단일 바이너리만 존재할 수 있으며, Frameworks 폴더 외부에 실행 가능한 바이너리 자료가 존재할 수 없습니다.

이는 sys.path``에 있는 모든 위치에서 바이너리 확장 모듈을 로드할 수 있도록 허용하는 일반적인 Python의 배포 접근 방식과 충돌합니다. App Store 정책 준수를 보장하려면, iOS 프로젝트는 모든 Python 패키지를 후처리하여 적절한 메타데이터 및 서명이 포함된 개별 독립 실행형 프레임워크로 ``.so 바이너리 모듈을 변환해야 합니다. 이 후처리를 수행하는 방법에 대한 자세한 내용은 프로젝트에 Python 추가하기 가이드를 참조하십시오.

Python이 새로운 위치에서 바이너리를 발견하도록 돕기 위해, sys.path``에 있는 원래의 ``.so 파일은 .fwork 파일로 대체됩니다. 이 파일은 앱 번들에 상대적인 프레임워크 바이너리 위치를 포함하는 텍스트 파일입니다. 프레임워크가 원래 위치로 되돌아갈 수 있도록 하려면, 프레임워크는 앱 번들에 상대적인 .fwork 파일의 위치를 포함하는 .origin 파일을 포함해야 합니다.

예를 들어, from foo.bar import _whiz``를 가져오는 경우를 고려해 보겠습니다. 여기서 ``_whiz``는 바이너리 모듈 ``sources/foo/bar/_whiz.abi3.so``로 구현되었으며, ``sources``는 애플리케이션 번들에 상대적인 ``sys.path``에 등록된 위치입니다. 이 모듈은 *반드시* ``Frameworks/foo.bar._whiz.framework/foo.bar._whiz``로 배포되어야 합니다 (모듈의 전체 import 경로에서 프레임워크 이름을 생성). 그리고 ``.framework 디렉터리에 있는 Info.plist 파일이 바이너리를 프레임워크로 식별해야 합니다. foo.bar._whiz 모듈은 원래 위치에 sources/foo/bar/_whiz.abi3.fwork 마커 파일로 표현되며, 이 파일은 Frameworks/foo.bar._whiz/foo.bar._whiz 경로를 포함합니다. 이 프레임워크는 또한 Frameworks/foo.bar._whiz.framework/foo.bar._whiz.origin``을 포함하며, 여기에는 ``.fwork 파일의 경로가 포함됩니다.

iOS에서 실행할 때, Python 인터프리터는 .fwork 파일을 읽고 가져올 수 있는 AppleFrameworkLoader 를 설치합니다. 가져온 후, 바이너리 모듈의 __file__ 속성은 .fwork 파일의 위치를 보고합니다. 그러나 로드된 모듈에 대한 ModuleSpec 은 프레임워크 폴더 내의 바이너리 위치를 origin 을 통해 보고합니다.

7.1.5. 컴파일러 스텁 바이너리¶

Xcode는 iOS에 대한 명시적인 컴파일러를 노출하지 않습니다. 대신, 전체 컴파일러 경로를 해결하는 xcrun 스크립트를 사용합니다 (예: iPhone 기기의 clang``을 얻기 위한 ``xcrun --sdk iphoneos clang). 그러나 이 스크립트를 사용하면 두 가지 문제가 발생합니다:

  • xcrun 의 출력에는 기기별 경로가 포함되어, 사용자 간에 공유할 수 없는 sysconfig 모듈이 생성됩니다.

  • 이는 공백을 포함하는 CC/CPP/LD/AR 정의를 초래합니다. 많은 C 생태계 도구는 명령어 라인에서 첫 번째 공백을 기준으로 경로를 분할하여 컴파일러 실행 파일의 경로를 얻을 수 있다고 가정합니다. 이는 xcrun 을 사용할 때 해당되지 않습니다.

이러한 문제를 피하기 위해 Python은 이러한 도구의 스텁을 제공했습니다. 이 스텁은 기본적으로 xcrun 도구를 감싼 셸 스크립트 래퍼로, 컴파일된 iOS 프레임워크와 함께 배포되는 bin 폴더에 배포됩니다. 이 스크립트들은 재배치 가능하며 항상 적절한 로컬 시스템 경로로 해결됩니다. 프레임워크를 동반하는 bin 폴더에 이 스크립트들을 포함함으로써, sysconfig 모듈의 내용이 최종 사용자가 자체 모듈을 컴파일하는 데 유용하게 됩니다. iOS용 서드파티 Python 모듈을 컴파일할 때는 이러한 스텁 바이너리가 경로에 있는지 확인해야 합니다.

7.2. iOS용 Python 설치하기¶

7.2.1. iOS 앱 빌드 도구¶

iOS를 위해 빌드하려면 Apple의 Xcode 도구가 필요합니다. 가장 최신 안정 버전의 Xcode를 사용하는 것이 강력히 권장됩니다. 이를 위해서는 Apple이 구형 macOS 버전에 대해 Xcode를 유지 관리하지 않기 때문에, 가장 최신이거나 그 다음으로 최신에 출시된 macOS 버전 사용이 필요합니다. Xcode Command Line Tools만으로는 iOS 개발에 충분하지 않습니다. 전체 Xcode 설치가 필요합니다.

iOS 시뮬레이터에서 코드를 실행하려면 iOS 시뮬레이터 플랫폼을 설치해야 합니다. Xcode를 처음 실행할 때 iOS 시뮬레이터 플랫폼을 선택하라는 메시지가 나타날 것입니다. 또는 Xcode 설정(Settings) 창의 ‘플랫폼(Platforms)’ 탭에서 iOS 시뮬레이터 플랫폼을 추가할 수 있습니다.

7.2.2. iOS 프로젝트에 파이썬 추가하기¶

파이썬은 Swift 또는 Objective C 중 어느 것을 사용하든 모든 iOS 프로젝트에 추가할 수 있습니다. 다음 예제에서는 Objective C를 사용합니다. 만약 Swift를 사용한다면, `PythonKit <https://github.com/pvieito/PythonKit>`__와 같은 라이브러리가 도움이 될 수 있습니다.

iOS Xcode 프로젝트에 파이썬을 추가하는 방법:

  1. 파이썬 XCFramework``를 빌드하거나 얻으십시오. Python ``XCFramework``를 빌드하는 방법은 :source:`Apple/iOS/README.md` (CPython 소스 배포판)의 지침을 참조하십시오. 최소한 ``arm64-apple-ios``를 지원하고, ``arm64-apple-ios-simulator 또는 x86_64-apple-ios-simulator 중 하나를 지원하는 빌드가 필요합니다.

  2. XCframework 를 iOS 프로젝트로 끌어다 놓으십시오. 다음 지침에서는 XCframework 를 프로젝트 루트에 넣었다고 가정하지만, 필요에 따라 경로를 조정하여 원하는 다른 위치를 사용할 수도 있습니다.

  3. 응용 프로그램 코드를 Xcode 프로젝트에 폴더로 추가하십시오. 다음 지침에서는 사용자 코드가 프로젝트 루트에 app 이라는 이름의 폴더에 있다고 가정합니다. 필요에 따라 경로를 조정하여 다른 위치를 사용할 수 있습니다. 이 폴더가 앱 타겟과 연결되어 있는지 확인하십시오.

  4. Xcode 프로젝트의 최상위 노드를 선택한 다음, 측면 표시줄에 표시되는 타겟 이름을 선택하여 앱 타겟을 선택하십시오.

  5. “General” 설정에서 “Frameworks, Libraries and Embedded Content” 아래에 “Embed & Sign”이 선택된 Python.xcframework 를 추가하십시오.

  6. “Build Settings” 탭에서 다음을 수정하십시오:

    • 빌드 옵션

      • 사용자 스크립트 샌드박싱: 아니요

      • 테스트 가능 활성화: 예

    • 검색 경로

      • 프레임워크 검색 경로: $(PROJECT_DIR)

      • 헤더 검색 경로: "$(BUILT_PRODUCTS_DIR)/Python.framework/Headers"

    • Apple Clang - 경고 - 모든 언어

      • 프레임워크 헤더에 따옴표 포함 인클루드: 아니요

  7. 파이썬 표준 라이브러리를 처리하고, 자체 파이썬 바이너리 종속성을 처리하는 빌드 단계를 추가하십시오. “Build Phases” 탭에서 “Embed Frameworks” 단계 이전, “Copy Bundle Resources” 단계 이후 에 새 “Run Script” 빌드 단계를 추가하십시오. 이 단계를 “Process Python libraries”라고 이름을 지정하고, “Based on dependency analysis” 체크박스를 비활성화한 다음, 스크립트 내용은 다음과 같이 설정하십시오:

    set -e
source $PROJECT_DIR/Python.xcframework/build/build_utils.sh
install_python Python.xcframework app

    XCframework를 프로젝트 루트가 아닌 다른 곳에 배치한 경우, 첫 번째 인수의 경로를 수정하십시오.

  8. 임베디드 모드에서 파이썬 인터프리터를 초기화하고 사용하는 Objective C 코드를 추가하십시오. 다음 사항들을 확인해야 합니다:

    • UTF-8 모드 (PyPreConfig.utf8_mode)가 활성화 되어 있어야 합니다;

    • 버퍼링된 stdio (PyConfig.buffered_stdio)는 비활성화 되어 있어야 합니다;

    • 바이트코드 작성 (PyConfig.write_bytecode)은 비활성화 되어 있어야 합니다;

    • 시그널 핸들러 (PyConfig.install_signal_handlers)가 활성화 되어 있어야 합니다;

    • 시스템 로깅 (PyConfig.use_system_logger)이 활성화 되어 있어야 합니다 (선택 사항이지만 강력히 권장됨; 기본적으로 활성화되어 있음);

    • 인터프리터의 PYTHONHOME`이 앱 번들의 ``python` 하위 폴더를 가리키도록 구성되어야 하며;

    • 인터프리터의 :envvar:`PYTHONPATH`에는 다음이 포함되어야 합니다:

      • 앱 번들의 python/lib/python3.X 하위 폴더,

      • 앱 번들의 python/lib/python3.X/lib-dynload 하위 폴더, 그리고

      • 앱 번들의 app 하위 폴더

    앱의 번들 위치는 [[NSBundle mainBundle] resourcePath] 를 사용하여 알 수 있습니다.

이 지침의 7단계와 8단계는 순수 파이썬 응용 프로그램 코드만 포함하는 단일 폴더, 즉 app 이라는 이름의 폴더가 있다고 가정합니다. 앱에 서드파티 바이너리 모듈이 있는 경우 추가 단계가 필요합니다:

  • 서드파티 바이너리를 포함하는 모든 폴더가 앱 타겟과 연결되어 있거나, 7단계에서 명시적으로 복사되어야 함을 확인해야 합니다. 7단계에서는 특정 빌드가 대상으로 하는 플랫폼에 적합하지 않은 모든 바이너리(예: 시뮬레이터를 대상으로 하는 앱을 빌드하는 경우 디바이스 바이너리 삭제)를 제거해야 합니다.

  • 서드파티 패키지를 위해 별도의 폴더를 사용하는 경우, 해당 폴더를 7단계의 install_python 호출 끝 부분과 8단계의 PYTHONPATH 구성에 추가했는지 확인해야 합니다.

  • 만약 서드파티 패키지를 포함하는 폴더 중 .pth 파일을 포함하는 폴더가 있다면, 해당 폴더를 PYTHONPATH`나 :attr:`sys.path`에 직접 추가하기보다는 *사이트 디렉터리*로 추가해야 합니다 ( :meth:`site.addsitedir 사용).

7.2.3. Python 패키지 테스트하기¶

CPython 소스 트리에는 iOS 시뮬레이터에서 CPython 테스트 스위트를 실행하는 데 사용되는 :source:`테스트베드 프로젝트 <Apple/iOS/testbed>`가 포함되어 있습니다. 이 테스트베드는 iOS에서 파이썬 라이브러리의 테스트 스위트를 실행하는 데 사용할 수 있는 테스트베드 프로젝트로도 사용될 수 있습니다.

iOS XCFramework를 빌드하거나 획득한 후(자세한 내용은 Apple/iOS/README.md 참조), Python iOS 테스트베드 프로젝트의 복제본을 생성하세요. 만약 XCframework를 빌드하는 데 Apple 빌드 스크립트를 사용했다면, 다음을 실행할 수 있습니다:

$ python cross-build/iOS/testbed clone --app <path/to/module1> --app <path/to/module2> app-testbed

또는, 직접 소스한 XCframework를 사용하여 다음을 실행함으로써 실행할 수 있습니다:

$ python Apple/testbed clone --platform iOS --framework <path/to/Python.xcframework> --app <path/to/module1> --app <path/to/module2> app-testbed

--app 플래그로 지정된 모든 폴더는 복제된 테스트베드 프로젝트에 복사됩니다. 생성된 테스트베드는 app-testbed 폴더에 위치합니다. 이 예시에서, module1``과 ``module2``는 런타임 시 임포트 가능한 모듈입니다. 프로젝트에 추가 종속성이 있는 경우, ``app-testbed/Testbed/app_packages 폴더에 설치할 수 있습니다 (pip install --target app-testbed/Testbed/app_packages 또는 이와 유사한 명령 사용).

그런 다음 app-testbed 폴더를 사용하여 앱의 테스트 스위트를 실행할 수 있습니다. 예를 들어, module1.tests 가 테스트 스위트의 진입점이었다면, 다음을 실행할 수 있습니다:

$ python app-testbed run -- module1.tests

이것은 데스크톱 Python 빌드에서 python -m module1.tests 를 실행하는 것과 동일합니다. -- 뒤의 모든 인수는 테스트베드에 마치 데스크톱 기기에서 python -m 의 인자인 것처럼 전달됩니다.

또한, 다음을 실행하여 테스트베드 프로젝트를 Xcode에서 열 수도 있습니다:

$ open app-testbed/iOSTestbed.xcodeproj

이것은 디버깅을 위한 전체 Xcode 도구 세트를 사용할 수 있게 해줍니다.

테스트 스위트를 실행하는 데 사용되는 인수는 테스트 계획의 일부로 정의됩니다. 테스트 계획을 수정하려면, 프로젝트 트리에서 테스트 계획 노드를 선택하고(“root node의 첫 번째 자식”이어야 함), “Configurations” 탭을 선택합니다. “Arguments Passed On Launch” 값을 수정하여 테스트 인수를 변경합니다.

테스트 계획은 병렬 테스트도 비활성화하며, 디버거 구성 정보를 제공하기 위해 Testbed.lldbinit 파일을 사용하도록 지정합니다. 기본 디버거 구성은 SIGINT, SIGUSR1, SIGUSR2, 및 SIGXFSZ 시그널에 대한 자동 브레이크포인트를 비활성화합니다.

7.3. App Store 준수¶

서드파티 iOS 기기로 앱을 배포하는 유일한 메커니즘은 앱을 iOS App Store에 제출하는 것이며, 배포 목적으로 제출된 앱은 애플의 앱 검토 과정을 통과해야 합니다. 이 과정에는 제출된 애플리케이션 번들을 문제 코드에 대해 검사하는 일련의 자동화된 검증 규칙이 포함됩니다. 이 검증 단계를 통과할 수 있도록 하려면 수행해야 하는 몇 가지 단계가 있습니다.

7.3.1. 표준 라이브러리 내의 비호환 코드¶

Python 표준 라이브러리에는 이러한 자동 검사 규칙을 위반하는 것으로 알려진 코드가 포함되어 있습니다. 이러한 위반 사항은 거짓 양성처럼 보일 수 있지만, 애플의 검토 규칙은 이의를 제기할 수 없으므로, 앱이 App Store 검토를 통과하려면 Python 표준 라이브러리를 수정하는 것이 필요합니다.

Python 소스 트리는 App Store 검토 과정에서 문제가 될 수 있는 모든 코드를 제거할 :source:`패치 파일 <Mac/Resources/app-store-compliance.patch>`를 포함하고 있습니다. 이 패치는 iOS용으로 빌드할 때 자동으로 적용됩니다.

7.3.2. 개인 정보 보호 매니페스트¶

2025년 4월, Apple은 특정 타사 라이브러리 <https://developer.apple.com/support/third-party-SDK-requirements>`를 제공하도록 요구사항을 도입했습니다. 그 결과, 영향을 받는 라이브러리를 사용하는 바이너리 모듈이 있는 경우, 해당 라이브러리에 대해 `.xcprivacy`` 파일을 제공해야 합니다. OpenSSL은 이 요구사항의 영향을 받는 라이브러리 중 하나이지만, 다른 라이브러리도 있습니다.

만약 mymodule.so`이라는 바이너리 모듈을 생성하고 위 7단계에서 설명한 Xcode 빌드 스크립트를 사용한다면, `mymodule.so 옆에 mymodule.xcprivacy 파일을 배치할 수 있으며, 바이너리 모듈이 프레임워크로 변환될 때 프라이버시 매니페스트가 필요한 위치에 설치됩니다.
