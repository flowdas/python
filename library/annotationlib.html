annotationlib — 어노테이션을 인트로스펙션하기 위한 기능¶

Added in version 3.14.

소스 코드: Lib/annotationlib.py

annotationlib 모듈은 모듈, 클래스 및 함수에 있는 :term:`주석 <annotation>`을 인트로스펙션하기 위한 도구를 제공합니다.

어노테이션은 지연 평가되는 것이며, 어노테이션이 생성될 때 아직 정의되지 않은 객체에 대한 전방 참조를 포함하는 경우가 많습니다. 이 모듈은 전방 참조 및 기타 엣지 케이스가 있는 경우에도 어노테이션을 신뢰할 수 있는 방식으로 검색하는 데 사용될 수 있는 일련의 저수준 도구를 제공합니다.

이 모듈은 세 가지 주요 형식(Format 참조)으로 어노테이션을 검색하는 것을 지원하며, 각각은 서로 다른 사용 사례에 가장 적합합니다:

  • :attr:`~Format.VALUE`는 어노테이션을 평가하고 그 값을 반환합니다. 이는 다루기 가장 간단하지만, 예를 들어 어노테이션에 정의되지 않은 이름에 대한 참조가 포함된 경우 오류를 발생시킬 수 있습니다.

  • FORWARDREF`는 확인할 수 없는 어노테이션에 대해 :class:`ForwardRef 객체를 반환하여 평가하지 않고도 어노테이션을 검사할 수 있도록 합니다. 이는 해결되지 않은 전방 참조를 포함할 수 있는 어노테이션으로 작업해야 할 때 유용합니다.

  • :attr:`~Format.STRING`은 어노테이션을 문자열로 반환하며, 소스 파일에 표시되는 방식과 유사합니다. 이는 어노테이션을 읽기 쉬운 방식으로 표시하려는 문서 생성기에게 유용합니다.

get_annotations() 함수는 어노테이션을 검색하기 위한 주요 진입점입니다. 함수, 클래스 또는 모듈이 주어지면, 요청된 형식의 어노테이션 딕셔너리를 반환합니다. 이 모듈은 또한 어노테이션을 평가하는 데 사용되는 annotate function`과 직접 작업하는 기능을 제공하며, 여기에는 :func:`get_annotate_from_class_namespace 및 call_annotate_function`과 같은 기능과, :term:`evaluate functions <evaluate function>`과 작업하기 위한 :func:`call_evaluate_function() 함수가 포함됩니다.

조심

이 모듈의 대부분의 기능은 임의 코드를 실행할 수 있습니다. 더 많은 정보는 :ref:`보안 섹션 <annotationlib-security>`을 참조하십시오.

어노테이션 의미론¶

어노테이션이 평가되는 방식은 파이썬 3의 역사에 따라 변경되었으며, 현재는 여전히 :ref:`future import <future>`에 의존하고 있습니다. 어노테이션에는 다음과 같은 실행 모델이 존재했습니다:

  • 스톡 의미론 (Python 3.0부터 3.13까지 기본값; PEP 3107 및 PEP 526 참조): 어노테이션은 소스 코드에서 발견되는 대로 즉시 평가됩니다.

  • 문자열 형식의 어노테이션 (Python 3.7부터 from __future__ import annotations 와 함께 사용; PEP 563 참조): 어노테이션은 문자열로만 저장됩니다.

  • 지연 평가 (Python 3.14부터 기본값; PEP 649 및 PEP 749 참조): 어노테이션은 접근될 때만 지연적으로 평가됩니다.

예를 들어, 다음 프로그램을 고려하십시오:

def func(a: Cls) -> None:
    print(a)

class Cls: pass

print(func.__annotations__)

이는 다음과 같이 동작합니다:

  • 스톡 의미론(Python 3.13 이하)에서는, Cls 가 해당 시점에 정의되지 않은 이름이기 때문에 func 가 정의된 줄에서 NameError 가 발생합니다.

  • 문자열 형식의 어노테이션(from __future__ import annotations 를 사용한 경우)에서는, {'a': 'Cls', 'return': 'None'} 을 출력합니다.

  • 지연 평가(Python 3.14 이상)에서는, {'a': <class 'Cls'>, 'return': None} 을 출력합니다.

스톡 의미론은 함수 어노테이션이 Python 3.0에서 처음 도입되었을 때(PEP 3107) 사용되었는데, 이것이 가장 간단하고 명확한 어노테이션 구현 방식이었기 때문입니다. 변수 어노테이션이 Python 3.6에서 도입되었을 때(PEP 526)도 동일한 실행 모델이 사용되었습니다. 하지만 스톡 의미론은 어노테이션을 타입 힌트로 사용할 때 문제를 일으켰는데, 예를 들어 어노테이션이 발견될 때 아직 정의되지 않은 이름에 참조해야 할 필요성이 있었습니다. 또한, 모듈 임포트 시 어노테이션을 실행하는 데 성능 문제가 있었습니다. 따라서 Python 3.7에서는 PEP 563 이 from __future__ import annotations 구문을 사용하여 어노테이션을 문자열로 저장할 수 있는 기능을 도입했습니다. 당시 계획은 이 동작을 궁극적으로 기본값으로 만드는 것이었지만, 문제가 발생했습니다. 즉, 문자열 형식의 어노테이션은 런타임에 어노테이션을 인트로스펙션하는 사용자에게는 처리하기가 더 어려웠습니다. PEP 649 라는 대안적 제안이 세 번째 실행 모델인 지연 평가를 도입했으며, 이는 Python 3.14에서 구현되었습니다. from __future__ import annotations 가 있는 경우에도 문자열 형식의 어노테이션이 여전히 사용되지만, 이 동작은 결국 제거될 예정입니다.

클래스들¶

class annotationlib.Format¶

어노테이션이 반환될 수 있는 형식들을 설명하는 이 열거형의 멤버 또는 그에 해당하는 정수 값은 :func:`get_annotations 및 이 모듈의 다른 함수, 그리고 __annotate__ 함수에 전달될 수 있습니다.

VALUE = 1¶

값은 어노테이션 표현식을 평가한 결과입니다.

VALUE_WITH_FAKE_GLOBALS = 2¶

어노테이션 함수가 가짜 전역 변수가 있는 특수 환경에서 평가되고 있음을 알리는 데 사용되는 특수 값입니다. 이 값을 전달할 때, 어노테이션 함수는 Format.VALUE 형식과 동일한 값을 반환하거나, 이 환경에서 실행을 지원하지 않음을 알리기 위해 :exc:`NotImplementedError`를 발생시켜야 합니다. 이 형식은 내부적으로만 사용되며, 이 모듈의 함수에 전달해서는 안 됩니다.

FORWARDREF = 3¶

값은 정의된 값에 대한 실제 어노테이션 값( Format.VALUE 형식 기준)과 정의되지 않은 값에 대한 ForwardRef 프록시입니다. 실제 객체는 ForwardRef 프록시 객체에 대한 참조를 담을 수 있습니다.

STRING = 4¶

값은 소스 코드에 나타나는 어노테이션의 텍스트 문자열이며, 공백 정규화 및 상수 값 최적화를 포함하되 이에 국한되지 않는 수정 사항까지 포함합니다.

이러한 문자열의 정확한 값은 향후 버전의 파이썬에서 변경될 수 있습니다.

Added in version 3.14.

class annotationlib.ForwardRef¶

어노테이션 내 전방 참조를 위한 프록시 객체입니다.

이 클래스의 인스턴스는 FORWARDREF 형식이 사용되고 어노테이션이 해석할 수 없는 이름을 포함할 때 반환됩니다. 이는 예를 들어 클래스가 정의되기 전에 전방 참조가 어노테이션에 사용될 때 발생할 수 있습니다.

__forward_arg__¶

A string containing the code that was evaluated to produce the ForwardRef. The string may not be exactly equivalent to the original source.

evaluate(*, owner=None, globals=None, locals=None, type_params=None, format=Format.VALUE)¶

순방향 참조(forward reference)를 평가하여 그 값을 반환합니다.

format 인자가 VALUE (기본값)인 경우, 순방향 참조가 해결할 수 없는 이름을 참조하면 NameError 와 같은 예외가 발생할 수 있습니다. 이 메서드의 인수는 그렇지 않으면 정의되지 않을 이름에 대한 바인딩을 제공하는 데 사용될 수 있습니다. format 인자가 FORWARDREF 인 경우, 이 메서드는 절대 예외를 발생시키지 않지만, ForwardRef 인스턴스를 반환할 수 있습니다. 예를 들어, 순방향 참조 객체가 list[undefined] 코드를 포함하고, 여기서 undefined 는 정의되지 않은 이름이라면, FORWARDREF 형식으로 평가할 때 list[ForwardRef('undefined')] 를 반환합니다. 만약 format 인자가 STRING 이라면, 이 메서드는 __forward_arg__ 를 반환합니다.

owner 매개변수는 본 메서드에 스코프 정보를 전달하는 선호되는 메커니즘을 제공합니다. :class:`~ForwardRef`의 소유자는 :class:`~ForwardRef`가 파생되는 어노테이션을 포함하는 객체이며, 예를 들면 모듈 객체, 타입 객체, 또는 함수 객체입니다.

The globals, locals, and type_params parameters provide a more precise mechanism for influencing the names that are available when the ForwardRef is evaluated. globals and locals are passed to eval(), representing the global and local namespaces in which the name is evaluated. The type_params parameter is relevant for objects created using the native syntax for generic classes and functions. It is a tuple of type parameters that are in scope while the forward reference is being evaluated. For example, if evaluating a ForwardRef retrieved from an annotation found in the class namespace of a generic class C, type_params should be set to C.__type_params__.

get_annotations`가 반환하는 :class:`~ForwardRef() 인스턴스는 해당 인스턴스가 유래한 스코프에 대한 참조를 유지하므로, 추가 인자 없이 이 메서드를 호출하는 것만으로도 해당 객체를 평가하기에 충분할 수 있습니다. 다른 수단을 통해 생성된 ForwardRef 인스턴스는 스코프에 대한 정보가 없을 수 있으므로, 성공적으로 평가하려면 이 메서드에 인수를 전달하는 것이 필요할 수 있습니다.

owner , globals , locals , 또는 type_params 가 제공되지 않고 ForwardRef 가 출처에 대한 정보를 포함하지 않는 경우, 빈 globals 및 locals 딕셔너리가 사용됩니다.

Added in version 3.14.

함수:¶

annotationlib.annotations_to_string(annotations)¶

런타임 값을 포함하는 어노테이션 딕셔너리를 문자열만 포함하는 딕셔너리로 변환합니다. 값이 이미 문자열이 아닌 경우, type_repr`를 사용하여 변환됩니다. 이는 :attr:`~Format.STRING() 형식을 지원하지만 어노테이션을 생성하는 코스에 접근할 수 없는 사용자 제공 어노테이트 함수용 헬퍼로 사용됩니다.

예를 들어, 이 것은 기능적 문법을 통해 생성된 typing.TypedDict 클래스에 대해 :attr:`~Format.STRING`을 구현하는 데 사용됩니다.

>>> from typing import TypedDict
>>> Movie = TypedDict("movie", {"name": str, "year": int})
>>> get_annotations(Movie, format=Format.STRING)
{'name': 'str', 'year': 'int'}

Added in version 3.14.

annotationlib.call_annotate_function(annotate, format, *, owner=None)¶

Call the annotate function annotate with the given format, a member of the Format enum, and return the annotations dictionary produced by the function.

이 헬퍼 함수는 컴파일러가 함수, 클래스 및 모듈을 위해 생성한 annotate 함수가 직접 호출될 때 VALUE 형식만 지원하기 때문에 필요합니다. 다른 형식을 지원하려면, 이 함수는 클래스가 구성되는 동안 어노테이션을 부분적으로 평가해야 하는 기능 구현에 유용한 빌딩 블록인 특수한 환경에서 annotate 함수를 호출합니다.

owner 는 어노테이션 함수를 소유하는 객체이며, 일반적으로 함수, 클래스 또는 모듈입니다. 제공되면, 더 많은 정보를 담는 ForwardRef 객체를 생성하기 위해 FORWARDREF 형식에서 사용됩니다.

더 보기

:PEP:`PEP 649 <649#the-stringizer-and-the-fake-globals-environment>`는 이 함수에서 사용된 구현 기술에 대한 설명을 포함하고 있습니다.

Added in version 3.14.

annotationlib.call_evaluate_function(evaluate, format, *, owner=None)¶

Call the evaluate function evaluate with the given format, a member of the Format enum, and return the value produced by the function. This is similar to call_annotate_function(), but the latter always returns a dictionary mapping strings to annotations, while this function returns a single value.

이는 타입 에일리어스와 타입 파라미터와 관련된 지연 평가 요소에 대해 생성된 evaluate 함수와 함께 사용하도록 의도되었습니다:

  • typing.TypeAliasType.evaluate_value(), 타입 에일리어스의 값

  • typing.TypeVar.evaluate_bound(), 타입 변수의 바운드

  • typing.TypeVar.evaluate_constraints(), 타입 변수의 제약 조건

  • typing.TypeVar.evaluate_default(), 타입 변수의 기본값

  • typing.ParamSpec.evaluate_default(), 파라미터 명세의 기본값

  • typing.TypeVarTuple.evaluate_default(), 타입 변수 튜플의 기본값

owner 는 evaluate 함수를 소유하는 객체이며, 타입 에일리어스 또는 타입 변수 객체가 될 수 있습니다.

format 은 값을 반환할 형식(format)을 제어하는 데 사용될 수 있습니다. 가능한 값은 다음과 같습니다:

>>> type Alias = undefined
>>> call_evaluate_function(Alias.evaluate_value, Format.VALUE)
Traceback (most recent call last):
...
NameError: name 'undefined' is not defined
>>> call_evaluate_function(Alias.evaluate_value, Format.FORWARDREF)
ForwardRef('undefined')
>>> call_evaluate_function(Alias.evaluate_value, Format.STRING)
'undefined'

Added in version 3.14.

annotationlib.get_annotate_from_class_namespace(namespace)¶

Retrieve the annotate function from a class namespace dictionary namespace. Return None if the namespace does not contain an annotate function. This is primarily useful before the class has been fully created (e.g., in a metaclass); after the class exists, the annotate function can be retrieved with cls.__annotate__. See below for an example using this function in a metaclass.

Added in version 3.14.

annotationlib.get_annotations(obj, *, globals=None, locals=None, eval_str=False, format=Format.VALUE)¶

객체에 대한 어노테이션 딕셔너리를 계산합니다.

obj 는 호출 가능 객체(callable), 클래스, 모듈 또는 __annotate__ 또는 __annotations__ 속성을 가진 다른 객체일 수 있습니다. 다른 객체를 전달하면 TypeError 가 발생합니다.

format 매개변수는 어노테이션이 반환되는 형식을 제어하며, Format 열거형의 멤버 또는 그 정수 등가 값이어야 합니다. 다양한 형식은 다음과 같이 작동합니다:

  • VALUE: object.__annotations__`가 먼저 시도되며, 존재하지 않으면 :attr:!object.__annotate__` 함수가 존재하는 경우 호출됩니다.

  • FORWARDREF: object.__annotations__`가 존재하고 성공적으로 평가될 수 있는 경우, 이것이 사용됩니다. 그렇지 않으면 :attr:!object.__annotate__` 함수가 호출됩니다. 둘 다 존재하지 않으면 :attr:`!object.__annotations__`를 다시 시도하며, 접근 중 발생하는 모든 오류를 다시 발생시킵니다.

    • :attr:`!object.__annotate__`를 호출할 때, 먼저 :attr:`~Format.FORWARDREF`로 호출됩니다. 이것이 구현되지 않은 경우, :attr:`~Format.VALUE_WITH_FAKE_GLOBALS`가 지원되는지 확인하고 가짜 전역 환경에서 그것을 사용합니다. 이 두 형식 중 어느 것도 지원되지 않으면, :attr:`~Format.VALUE`를 사용하는 방식으로 폴백됩니다. :attr:`~Format.VALUE`가 실패하면, 이 호출에서 발생한 오류가 발생합니다.

  • STRING: object.__annotate__`가 존재하는 경우, 이것이 먼저 호출되며, 그렇지 않으면 :attr:!object.__annotations__`를 사용하고 :func:`annotations_to_string`를 사용하여 문자열화합니다.

    • :attr:`!object.__annotate__`를 호출할 때, 먼저 :attr:`~Format.STRING`으로 호출됩니다. 이것이 구현되지 않은 경우, :attr:`~Format.VALUE_WITH_FAKE_GLOBALS`가 지원되는지 확인하고 가짜 전역 환경에서 그것을 사용합니다. 이 두 형식 중 어느 것도 지원되지 않으면, :attr:`~Format.VALUE`를 사용하여 폴백하며, 결과는 :func:`annotations_to_string`를 사용하여 변환됩니다. :attr:`~Format.VALUE`가 실패하면, 이 호출에서 발생한 오류가 발생합니다.

딕셔너리를 반환합니다. :func:`!get_annotations`는 호출될 때마다 새로운 딕셔너리를 반환합니다. 동일한 객체에 대해 두 번 호출하면 두 개의 다르고 동등한 딕셔너리가 반환됩니다.

이 함수는 여러 세부 사항을 처리합니다:

  • If eval_str is true, values of type str will be un-stringized using eval(). This is intended for use with stringized annotations (from __future__ import annotations). It is an error to set eval_str to true with formats other than Format.VALUE.

  • obj 에 annotations 딕셔너리가 없으면 빈 딕셔너리를 반환합니다. (함수와 메서드는 항상 annotations 딕셔너리를 가집니다. 클래스, 모듈 및 다른 타입의 콜러블은 그렇지 않을 수 있습니다.)

  • 클래스에서 상속된 annotations와 메타 클래스의 annotations를 무시합니다. 클래스에 자체 annotations 딕셔너리가 없으면 빈 딕셔너리를 반환합니다.

  • 객체 멤버 및 딕셔너리 값 접근은 안전을 위해 모두 getattr() 및 dict.get() 을 사용합니다.

eval_str controls whether or not values of type str are replaced with the result of calling eval() on those values:

  • eval_str가 true인 경우, str 타입의 값에 대해 eval`을 호출합니다. (참고: :func:()!get_annotations`는 예외를 잡지 않습니다. 만약 eval`이 예외를 발생시키면, :func:()!get_annotations` 호출 지점을 지나 스택이 풀리게 됩니다.)

  • eval_str 가 false인 경우(기본값), str 타입의 값은 변경되지 않습니다.

globals 와 locals 가 eval() 에 전달됩니다. 더 자세한 내용은 eval() 의 문서를 참조하십시오. globals 또는 locals 가 None 인 경우, 이 함수는 type(obj) 에 따라 해당 값을 문맥별 기본값으로 대체할 수 있습니다:

  • obj 가 모듈인 경우, globals 는 obj.__dict__ 를 기본값으로 사용합니다.

  • obj*가 클래스인 경우, *globals*는 ``sys.modules[obj.__module__].__dict__``가, *locals*는 *obj 클래스의 이름 공간을 기본값으로 사용합니다.

  • obj 가 콜러블인 경우, globals 는 obj.__globals__ 를 기본값으로 사용합니다. 비록 obj 가 래핑된 함수( functools.update_wrapper() 사용)이거나 functools.partial 객체일지라도, 비래핑 함수가 발견될 때까지 언랩(unwrapped)됩니다.

어떤 객체의 annotations 딕셔너리에 접근하는 가장 좋은 방법은 :func:`!get_annotations`를 호출하는 것입니다. annotations 모범 사례에 대한 더 자세한 내용은 :ref:`annotations-howto`를 참조하십시오.

>>> def f(a: int, b: str) -> float:
...     pass
>>> get_annotations(f)
{'a': <class 'int'>, 'b': <class 'str'>, 'return': <class 'float'>}

Added in version 3.14.

annotationlib.type_repr(value)¶

임의의 파이썬 값을 STRING 포맷을 사용하기에 적합한 형식으로 변환합니다. 이 함수는 대부분의 객체에 대해 :func:`repr`을 호출하지만, 타입 객체와 같이 일부 객체에 대해서는 특수 처리를 합니다.

이것은 STRING 포맷을 지원하지만 annotations를 생성하는 코드에 접근할 수 없는 사용자가 제공하는 annotate 함수를 위한 헬퍼 역할을 합니다. 또한 annotations에서 흔히 접하는 값을 포함하는 다른 객체에 대해 사용자 친화적인 문자열 표현을 제공하는 데 사용될 수도 있습니다.

Added in version 3.14.

예제들¶

메타 클래스에서의 annotations 사용¶

metaclass <metaclasses>`는 클래스 생성 중에 클래스 본문의 annotations를 검사하거나 심지어 수정하고 싶을 수 있습니다. 그렇게 하려면 클래스 네임스페이스 딕셔너리에서 annotations를 검색해야 합니다. ``from __future__ import annotations``로 생성된 클래스의 annotations는 딕셔너리의 ``__annotations__` 키에 있을 것입니다. annotations가 있는 다른 클래스의 경우, get_annotate_from_class_namespace`를 사용하여 annotate 함수를 얻을 수 있고, :func:`call_annotate_function`를 사용하여 이를 호출하고 annotations를 검색할 수 있습니다. :attr:`~Format.FORWARDREF() 포맷을 사용하는 것이 일반적으로 가장 좋습니다. 왜냐하면 이것이 클래스가 생성될 때 아직 해결될 수 없는 이름을 annotations가 참조할 수 있도록 허용하기 때문입니다.

annotations를 수정하려면, 원래 annotate 함수를 호출하고, 필요한 조정 사항을 적용하고, 결과를 반환하는 래퍼 annotate 함수를 만드는 것이 가장 좋습니다.

아래는 클래스에서 모든 typing.ClassVar annotations를 필터링하여 클래스 본문에서 별도의 어트리뷰트에 넣는 메타 클래스의 예제입니다:

import annotationlib
import typing

class ClassVarSeparator(type):
   def __new__(mcls, name, bases, ns):
      if "__annotations__" in ns:  # from __future__ import annotations
         annotations = ns["__annotations__"]
         classvar_keys = {
            key for key, value in annotations.items()
            # 단순성을 위해 문자열 비교를 사용합니다. 더 강력한 해결책은
            # annotationlib.ForwardRef.evaluate를 사용할 수 있습니다.
            if value.startswith("ClassVar")
         }
         classvars = {key: annotations[key] for key in classvar_keys}
         ns["__annotations__"] = {
            key: value for key, value in annotations.items()
            if key not in classvar_keys
         }
         wrapped_annotate = None
      elif annotate := annotationlib.get_annotate_from_class_namespace(ns):
         annotations = annotationlib.call_annotate_function(
            annotate, format=annotationlib.Format.FORWARDREF
         )
         classvar_keys = {
            key for key, value in annotations.items()
            if typing.get_origin(value) is typing.ClassVar
         }
         classvars = {key: annotations[key] for key in classvar_keys}

         def wrapped_annotate(format):
            annos = annotationlib.call_annotate_function(annotate, format, owner=typ)
            return {key: value for key, value in annos.items() if key not in classvar_keys}

      else:  # 어노테이션 없음
         classvars = {}
         wrapped_annotate = None
      typ = super().__new__(mcls, name, bases, ns)

      if wrapped_annotate is not None:
         # 원래의 __annotate___를 ClassVar를 제거하는 래퍼로 감쌉니다.
         typ.__annotate__ = wrapped_annotate
      typ.classvars = classvars  # ClassVar를 별도의 어트리뷰트로 저장합니다.
      return typ

사용자 정의 호출 가능한 어노테이트 함수 만들기¶

사용자 정의 :term:`annotate functions <annotate function>`는 함수, 클래스, 모듈에 자동으로 생성되는 함수일 수 있습니다. 또는, 클래스를 통해 제공되는 캡슐화를 이용할 수도 있으며, 이 경우 모든 :term:`callable`은 :term:`annotate function`으로 사용될 수 있습니다.

VALUE, STRING, 또는 FORWARDREF 포맷을 직접 제공하려면, :term:`annotate function`은 다음 어트리뷰트를 제공해야 합니다:

  • 지원되는 형식으로 호출할 때 NotImplementedError 를 발생시키지 않는 서명이 __call__(format, /) -> dict 인 호출 가능 객체입니다.

직접 지원되지 않는 경우 STRING 또는 VALUE_WITH_FAKE_GLOBALS 형식을 제공하려면, :term:`annotate functions <annotate function>`은 다음 어트리뷰트를 제공해야 합니다:

  • VALUE_WITH_FAKE_GLOBALS 로 호출할 때 NotImplementedError 를 발생시키지 않는 서명이 __call__(format, /) -> dict 인 호출 가능 객체입니다.

  • 어노테이트 함수에 컴파일된 코드를 담고 있는 code object __code__ 입니다.

  • 선택 사항: 만약 __code__ 가 표현하는 함수가 위치 기본값을 사용하는 경우, 해당 함수가 가진 위치 기본값의 튜플인 __kwdefaults__ 입니다.

  • 선택 사항: 만약 __code__ 가 표현하는 함수가 키워드 기본값을 사용하는 경우, 해당 함수가 가진 키워드 기본값의 dict인 __defaults__ 입니다.

  • 선택 사항: 나머지 모든 :ref:`function attributes <inspect-types>`입니다.

class Annotate:
    called_formats = []

    def __call__(self, format=None, /, *, _self=None):
        # 가짜 글로벌 변수(fake globals)로 호출될 때는 `_self`가 실제 self 값을,
        # `self`가 형식을 나타냅니다.
        if _self is not None:
            self, format = _self, self

        self.called_formats.append(format)
        if format <= 2:  # VALUE 또는 VALUE_WITH_FAKE_GLOBALS
            return {"x": MyType}
        raise NotImplementedError

    __code__ = __call__.__code__
    __defaults__ = (None,)
    __kwdefaults__ = property(lambda self: dict(_self=self))

    __globals__ = {}
    __builtins__ = {}
    __closure__ = None

다음과 같이 호출할 수 있습니다:

>>> from annotationlib import call_annotate_function, Format
>>> call_annotate_function(Annotate(), format=Format.STRING)
{'x': 'MyType'}

또는 객체의 어노테이트 함수로 사용됩니다:

>>> from annotationlib import get_annotations, Format
>>> class C:
...   pass
>>> C.__annotate__ = Annotate()
>>> get_annotations(Annotate(), format=Format.STRING)
{'x': 'MyType'}

STRING 형식의 제한 사항¶

STRING 형식은 주석의 소스 코드를 근사화하기 위한 것이지만, 사용된 구현 전략으로 인해 정확한 소스 코드를 복구하는 것이 항상 가능한 것은 아닙니다.

첫째, 문자열화기는 코멘트, 공백, 괄호화, 컴파일러에 의해 단순화되는 연산을 포함하여 컴파일된 코드에 존재하지 않는 어떠한 정보도 복구할 수 없습니다.

둘째, 문자열화기는 특정 스코프에서 참조되는 이름과 관련된 거의 모든 연산을 가로챌 수 있지만, 상수만으로 완전히 작동하는 연산은 가로챌 수 없습니다. 결과적으로, 이는 신뢰할 수 없는 코드에 STRING 형식을 요청하는 것이 안전하지 않다는 것을 의미합니다. 파이썬은 전역 변수나 내장 기능에 대한 접근 없이도 임의의 코드 실행을 달성할 수 있을 만큼 강력합니다. 예를 들면:

>>> def f(x: (1).__class__.__base__.__subclasses__()[-1].__init__.__builtins__["print"]("Hello world")): pass
...
>>> annotationlib.get_annotations(f, format=annotationlib.Format.STRING)
Hello world
{'x': 'None'}

참고

이 특정 예제는 작성 시점에서는 작동하지만, 구현 세부 사항에 의존하며 미래에도 작동함이 보장되지 않습니다.

ast 모듈로 표현되는 파이썬에 존재하는 다양한 종류의 표현식 중에서, 일부 표현식은 지원되어 STRING 형식으로 일반적으로 원래의 소스 코드를 복구할 수 있습니다; 다른 표현식은 지원되지 않아 잘못된 출력이나 오류가 발생할 수 있습니다.

다음은 지원됩니다 (때로는 주의 사항과 함께):

  • ast.BinOp

  • ast.UnaryOp

    • ast.Invert (~), ast.UAdd (+), 그리고 ast.USub (-)가 지원됩니다

    • ast.Not (not)는 지원되지 않습니다

  • ast.Dict (** 언패킹을 사용할 때를 제외하고)

  • ast.Set

  • ast.Compare

    • :class:`ast.Eq`와 :class:`ast.NotEq`가 지원됩니다

    • ast.Lt, ast.LtE, ast.Gt, 그리고 :class:`ast.GtE`가 지원되지만, 피연산자가 뒤집힐 수 있습니다

    • ast.Is, ast.IsNot, ast.In, 그리고 :class:`ast.NotIn`은 지원되지 않습니다

  • ast.Call (** 언패킹을 사용할 때를 제외하고)

  • ast.Constant (비록 상수의 정확한 표현은 아니지만; 예를 들어, 문자열의 이스케이프 시퀀스는 손실되며, 16진수 숫자는 10진수로 변환됩니다)

  • ast.Attribute (값이 상수가 아니라고 가정함)

  • ast.Subscript (값이 상수가 아니라고 가정함)

  • ast.Starred (* 언패킹)

  • ast.Name

  • ast.List

  • ast.Tuple

  • ast.Slice

다음은 지원되지 않지만, 문자열화기에 의해 발견되면 정보성 오류를 발생시킵니다:

  • ast.FormattedValue (f-문자열; !r 과 같은 변환 지정자를 사용하면 오류가 감지되지 않습니다)

  • ast.JoinedStr (f-문자열)

다음은 지원되지 않으며 잘못된 출력을 초래합니다:

  • ast.BoolOp (and 및 or)

  • ast.IfExp

  • ast.Lambda

  • ast.ListComp

  • ast.SetComp

  • ast.DictComp

  • ast.GeneratorExp

다음은 주석 스코프에서 허용되지 않으므로 관련이 없습니다:

  • ast.NamedExpr (:=)

  • ast.Await

  • ast.Yield

  • ast.YieldFrom

FORWARDREF 형식의 제한 사항¶

FORWARDREF 형식은 가능한 한 실제 값을 생성하는 것을 목표로 하며, 해석할 수 없는 모든 것은 ForwardRef 객체로 대체됩니다. 이 형식은 STRING 형식과 광범위하게 동일한 제한 사항의 영향을 받습니다. 리터럴에 대해 연산을 수행하거나 지원되지 않는 표현식 유형을 사용하는 주석은 FORWARDREF 형식을 사용하여 평가될 때 예외를 발생시킬 수 있습니다.

지원되지 않는 표현식에 대한 동작의 몇 가지 예는 다음과 같습니다:

>>> from annotationlib import get_annotations, Format
>>> def zerodiv(x: 1 / 0): ...
>>> get_annotations(zerodiv, format=Format.STRING)
Traceback (most recent call last):
  ...
ZeroDivisionError: division by zero
>>> get_annotations(zerodiv, format=Format.FORWARDREF)
Traceback (most recent call last):
  ...
ZeroDivisionError: division by zero
>>> def ifexp(x: 1 if y else 0): ...
>>> get_annotations(ifexp, format=Format.STRING)
{'x': '1'}

주석을 검사하는 것의 보안 영향¶

이 모듈의 많은 기능은 주석과 관련된 코드를 실행하는 것을 포함하며, 이는 임의의 동작을 수행할 수 있습니다. 예를 들어, get_annotations`는 임의의 :term:`annotate function`를 호출할 수 있으며, :meth:`ForwardRef.evaluate`는 임의의 문자열에 대해 :func:`eval`을 호출할 수 있습니다. 주석에 포함된 코드는 임의의 시스템 호출을 하거나, 무한 루프에 진입하거나, 다른 모든 연산을 수행할 수 있습니다. 이는 또한 :attr:`~object.__annotations__() 속성에 대한 모든 접근과, typing.get_type_hints`와 같이 주석과 함께 작동하는 :mod:`typing() 모듈의 다양한 함수에도 해당됩니다.

이것을 유감스럽게 느끼게 하는 모든 보안 문제 역시 신뢰할 수 없는 어노테이션을 포함할 수 있는 코드를 가져오는 직후에 적용됩니다. 코드를 가져오는 것은 임의의 작업을 항상 유발할 수 있습니다. 그러나 어노테이션을 검사하기 위해 문자열이나 기타 입력을 신뢰할 수 없는 소스에서 받아와 any의 API에 전달하는 것은 안전하지 않습니다. 예를 들어, __annotations__ 딕셔너리를 편집하거나 ForwardRef 객체를 직접 생성하는 경우입니다.
