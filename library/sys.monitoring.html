sys.monitoring — 실행 이벤트 모니터링¶

Added in version 3.12.

참고

sys.monitoring 은 독립적인 모듈이 아니라 sys 모듈 내의 네임스페이스이며, import sys.monitoring 은 ModuleNotFoundError 로 실패할 것입니다. 대신, 단순히 import sys 를 실행한 후 sys.monitoring 을 사용하십시오.

이 네임스페이스는 이벤트 모니터링을 활성화하고 제어하는 데 필요한 함수와 상수에 대한 액세스를 제공합니다.

프로그램이 실행됨에 따라, 실행을 감시하는 데 관심이 있을 수 있는 이벤트가 발생합니다. sys.monitoring 이름 공간은 관심 이벤트가 발생할 때 콜백을 받을 수단을 제공합니다.

모니터링 API는 세 가지 구성 요소로 구성됩니다:

  • Tool identifiers

  • Events

  • 콜백

도구 식별자¶

도구 식별자는 정수와 연결된 이름입니다. 도구 식별자는 도구들이 서로 간섭하는 것을 방지하고 여러 도구가 동시에 작동할 수 있도록 사용됩니다. 현재 도구들은 완전히 독립적이며 서로를 모니터링하는 데 사용될 수 없습니다. 이 제한은 향후 해제될 수 있습니다.

이벤트를 등록하거나 활성화하기 전에, 도구는 식별자를 선택해야 합니다. 식별자는 0부터 5까지의 범위 내의 정수입니다.

도구 등록 및 사용¶

sys.monitoring.use_tool_id(tool_id: int, name: str, /) → None¶

tool_id 를 사용하기 전에 호출되어야 합니다. tool_id 는 0부터 5까지의 범위여야 합니다. tool_id 가 사용 중일 경우 ValueError 를 발생시킵니다.

sys.monitoring.clear_tool_id(tool_id: int, /) → None¶

tool_id 와 관련된 모든 이벤트와 콜백 함수를 등록 취소합니다.

sys.monitoring.free_tool_id(tool_id: int, /) → None¶

도구가 더 이상 tool_id 를 필요로 하지 않을 때 호출해야 합니다. tool_id 를 해제하기 전에 clear_tool_id() 를 호출할 것입니다.

sys.monitoring.get_tool(tool_id: int, /) → str | None¶

tool_id 가 사용 중이면 해당 도구의 이름을 반환하고, 그렇지 않으면 None 을 반환합니다. tool_id 는 0부터 5까지의 범위여야 합니다.

모든 ID는 이벤트와 관련하여 VM에 의해 동일하게 처리되지만, 다음 ID들은 도구들의 상호 작용을 더 쉽게 만들기 위해 사전 정의되어 있습니다:

sys.monitoring.DEBUGGER_ID = 0
sys.monitoring.COVERAGE_ID = 1
sys.monitoring.PROFILER_ID = 2
sys.monitoring.OPTIMIZER_ID = 5

이벤트¶

다음 이벤트들이 지원됩니다:

sys.monitoring.events.BRANCH_LEFT¶

조건 분기가 왼쪽으로 이동합니다.

“왼쪽” 및 “오른쪽” 분기를 어떻게 표시할지는 도구에 달렸습니다. 프로그램이 진행되는 동안 일관적이라는 점을 제외하고는 어떤 분기가 “왼쪽”이고 어떤 분기가 “오른쪽”인지에 대한 보장은 없습니다.

sys.monitoring.events.BRANCH_RIGHT¶

조건 분기가 오른쪽으로 이동합니다.

sys.monitoring.events.CALL¶

Python 코드의 호출(이벤트가 호출 전에 발생합니다).

sys.monitoring.events.C_RAISE¶

어떤 호출 가능성으로부터 발생한 예외(Python 함수 제외, 이벤트가 종료 후에 발생합니다).

sys.monitoring.events.C_RETURN¶

어떤 호출 가능성으로부터의 반환(Python 함수 제외, 이벤트가 반환 후에 발생합니다).

sys.monitoring.events.EXCEPTION_HANDLED¶

예외가 처리됩니다.

sys.monitoring.events.INSTRUCTION¶

VM 명령이 실행되려고 합니다.

sys.monitoring.events.JUMP¶

제어 흐름 그래프에서 조건에 관계없는 점프가 발생합니다.

sys.monitoring.events.LINE¶

이전 명령어와 다른 줄 번호를 가진 명령을 실행하려고 합니다.

sys.monitoring.events.PY_RESUME¶

Python 함수의 재개(제너레이터 및 코루틴 함수용), throw() 호출 제외.

sys.monitoring.events.PY_RETURN¶

Python 함수로부터의 반환(반환 직전 발생하며, 호출자의 프레임이 스택에 놓이게 됩니다).

sys.monitoring.events.PY_START¶

파이썬 함수 호출의 시작 지점입니다 (호출 직후, 호출된 함수의 프레임이 스택에 있습니다).

sys.monitoring.events.PY_THROW¶

파이썬 함수가 throw() 호출을 통해 재개됩니다.

sys.monitoring.events.PY_UNWIND¶

예외 언와인딩(unwinding) 중 파이썬 함수에서 종료됩니다. 여기에는 함수 내부에서 직접 발생하여 전파되도록 허용된 예외가 포함됩니다.

sys.monitoring.events.PY_YIELD¶

파이썬 함수에서 값을 반환합니다 (yield) (yield 직전에 발생하며, 호출된 함수의 프레임이 스택에 있습니다).

sys.monitoring.events.RAISE¶

STOP_ITERATION 이벤트를 유발하는 경우를 제외하고 예외가 발생합니다.

sys.monitoring.events.RERAISE¶

예를 들어, finally 블록 끝에서 예외가 재발생합니다.

sys.monitoring.events.STOP_ITERATION¶

인위적인 StopIteration 이 발생합니다. the STOP_ITERATION event 이벤트를 참조하십시오.

향후 더 많은 이벤트가 추가될 수 있습니다.

이러한 이벤트들은 sys.monitoring.events 네임스페이스의 속성입니다. 각 이벤트는 2의 거듭제곱 형태의 정수 상수로 표현됩니다. 이벤트 세트를 정의하려면 개별 이벤트를 비트별 OR 연산자로 결합하기만 하면 됩니다. 예를 들어, PY_RETURN 및 PY_START 이벤트를 모두 지정하려면 PY_RETURN | PY_START 표현식을 사용합니다.

sys.monitoring.events.NO_EVENTS¶

`0`에 대한 별칭으로, 사용자가 다음과 같은 명시적인 비교를 수행할 수 있게 합니다:

if get_events(DEBUGGER_ID) == NO_EVENTS:
    ...

이 이벤트를 설정하면 모든 이벤트가 비활성화됩니다.

지역 이벤트¶

지역 이벤트는 프로그램의 정상적인 실행과 관련되며 명확하게 정의된 위치에서 발생합니다. 모든 지역 이벤트는 위치별로 비활성화할 수 있습니다. 지역 이벤트는 다음과 같습니다:

  • PY_START

  • PY_RESUME

  • PY_RETURN

  • PY_YIELD

  • CALL

  • LINE

  • INSTRUCTION

  • JUMP

  • BRANCH_LEFT

  • BRANCH_RIGHT

  • STOP_ITERATION

사용 지원 중단된 이벤트¶

  • BRANCH

BRANCH 이벤트는 3.14 버전에서 지원이 중단됩니다. BRANCH_LEFT 및 BRANCH_RIGHT 이벤트를 사용하면 개별적으로 비활성화할 수 있어 성능이 훨씬 좋습니다.

부속 이벤트¶

부속 이벤트는 다른 이벤트처럼 모니터링할 수 있지만, 다른 이벤트에 의해 제어됩니다:

  • C_RAISE

  • C_RETURN

C_RETURN 및 C_RAISE 이벤트는 CALL 이벤트에 의해 제어됩니다. C_RETURN 및 C_RAISE 이벤트는 해당 CALL 이벤트가 모니터링되고 있을 때만 감지됩니다.

기타 이벤트¶

기타 이벤트는 프로그램의 특정 위치와 반드시 연결되어 있는 것은 아니며 위치별로 개별적으로 비활성화할 수 없습니다.

모니터링할 수 있는 다른 이벤트는 다음과 같습니다:

  • PY_THROW

  • PY_UNWIND

  • RAISE

  • EXCEPTION_HANDLED

  • RERAISE

버전 3.15에서 변경: 다른 이벤트는 이제 코드 객체별로 켜고 끌 수 있습니다. 콜백에서 :data:`DISABLE`을 반환하면 전체 코드 객체(현재 도구에 대해)에 대한 이벤트가 비활성화됩니다.

STOP_ITERATION 이벤트¶

PEP 380 은 제너레이터 또는 코루틴에서 값을 반환할 때 StopIteration 예외가 발생한다고 명시합니다. 그러나 이는 값을 반환하기 위한 매우 비효율적인 방법이므로 CPython 3.12 이상과 같은 일부 파이썬 구현에서는 다른 코드에서 가시적이지 않은 경우 예외를 발생시키지 않습니다.

도구가 제너레이터 및 코루틴의 속도를 늦추지 않으면서 실제 예외를 모니터링할 수 있도록, STOP_ITERATION 이벤트가 제공됩니다. :monitoring-event:`STOP_ITERATION`은 지역적으로 비활성화될 수 있습니다.

STOP_ITERATION 이벤트와 StopIteration 예외를 위한 RAISE 이벤트는 동등함을 참고하십시오. 그리고 이벤트 생성 시 서로 교환 가능하게 취급됩니다. 구현체는 성능상의 이유로 STOP_ITERATION`을 선호하지만, :exc:`StopIteration`을 포함하는 :monitoring-event:`RAISE 이벤트를 생성할 수도 있습니다.

이벤트 켜고 끄기¶

이벤트를 모니터링하려면, 이벤트를 켜고 해당되는 콜백을 등록해야 합니다. 이벤트는 전역적으로 및/또는 특정 코드 객체에 대해 이벤트를 설정하여 켜거나 끌 수 있습니다. 이벤트는 전역적과 지역적으로 모두 켜져 있는 경우에도 한 번만 트리거됩니다.

전역 이벤트 설정¶

이벤트는 모니터링되는 이벤트 세트를 수정함으로써 전역적으로 제어할 수 있습니다.

sys.monitoring.get_events(tool_id: int, /) → int¶

활성 이벤트 전체를 나타내는 int 를 반환합니다.

sys.monitoring.set_events(tool_id: int, event_set: int, /) → None¶

event_set 에 설정된 모든 이벤트를 활성화합니다. tool_id 가 사용 중이 아니면 ValueError 를 발생시킵니다.

기본적으로 활성 이벤트는 없습니다.

코드 객체별 이벤트¶

이벤트는 코드 객체별로도 제어할 수 있습니다. 아래에 정의된 함수 중 types.CodeType`을 받도록 정의된 함수는 Python에서 정의되지 않은 함수로부터 유사한 객체를 받을 준비가 되어 있어야 합니다 (참고: :ref:`c-api-monitoring).

sys.monitoring.get_local_events(tool_id: int, code: CodeType, /) → int¶

code 에 대한 모든 로컬 이벤트 을 반환합니다.

sys.monitoring.set_local_events(tool_id: int, code: CodeType, event_set: int, /) → None¶

Activates all the local events for code which are set in event_set. Raises a ValueError if tool_id is not in use.

이벤트 비활성화¶

sys.monitoring.DISABLE¶

현재 코드 위치의 이벤트를 비활성화하기 위해 콜백 함수에서 반환될 수 있는 특수 값입니다.

:ref:`로컬 이벤트 <monitoring-event-local>`은 콜백 함수에서 :data:`sys.monitoring.DISABLE`를 반환하여 특정 코드 위치에 대해 비활성화할 수 있습니다. 이는 설정된 이벤트나 동일한 이벤트에 대한 다른 코드 위치를 변경하지 않습니다.

:ref:`다른 이벤트 <monitoring-event-global>`은 콜백 함수에서 :data:`sys.monitoring.DISABLE`를 반환하여 코드 객체별로 비활성화할 수 있습니다. 이는 전체 코드 객체(현재 도구)에 대한 이벤트를 비활성화합니다.

특정 위치에 대한 이벤트를 비활성화하는 것은 높은 성능의 모니터링에 매우 중요합니다. 예를 들어, 디버거가 몇 개의 중단점 외의 모든 모니터링을 비활성화하는 경우, 오버헤드 없이 디버거 하에서 프로그램을 실행할 수 있습니다.

sys.monitoring.restart_events() → None¶

모든 도구에 대해 :data:`sys.monitoring.DISABLE`로 비활성화된 모든 이벤트를 활성화합니다.

콜백 함수 등록¶

sys.monitoring.register_callback(tool_id: int, event: int, func: Callable | None, /) → Callable | None¶

주어진 tool_id 로 event 에 대한 콜러블 func 를 등록합니다.

만약 주어진 tool_id 와 event 에 대해 다른 콜백이 등록되어 있었다면, 해당 콜백을 등록 해제하고 반환합니다. 그렇지 않으면 register_callback() 은 None 을 반환합니다.

인자 func 를 가진 감사 이벤트 sys.monitoring.register_callback 을 발생시킵니다.

함수는 sys.monitoring.register_callback(tool_id, event, None) 를 호출하여 등록 해제할 수 있습니다.

콜백 함수는 언제든지 등록하거나 등록 해제할 수 있습니다.

이벤트가 전역적으로나 로컬적으로 모두 켜져 있더라도 콜백은 한 번만 호출됩니다. 따라서, 코드에 의해 이벤트가 전역 이벤트와 로컬 이벤트 모두로 켜질 수 있는 경우, 콜백은 두 가지 트리거 중 하나를 처리하도록 작성되어야 합니다.

콜백 함수 인자¶

sys.monitoring.MISSING¶

호출에 인자가 없음을 나타내기 위해 콜백 함수에 전달되는 특수 값입니다.

활성 이벤트가 발생하면 등록된 콜백 함수가 호출됩니다. :data:`DISABLE`가 아닌 객체를 반환하는 콜백 함수는 아무런 효과가 없습니다. 서로 다른 이벤트는 다음과 같이 콜백 함수에 다른 인자를 제공합니다:

  • PY_START 및 PY_RESUME`:

    func(code: CodeType, instruction_offset: int) -> object

  • PY_RETURN 및 PY_YIELD`:

    func(code: CodeType, instruction_offset: int, retval: object) -> object

  • CALL, C_RAISE 및 C_RETURN (arg0 는 특히 MISSING 일 수 있습니다):

    func(code: CodeType, instruction_offset: int, callable: object, arg0: object) -> object

    code 는 호출이 이루어지는 코드 객체를 나타내고, callable 은 호출되려는 (그리고 따라서 이벤트를 유발하는) 객체입니다. 인자가 없으면 arg0 는 sys.monitoring.MISSING 으로 설정됩니다.

    인스턴스 메서드의 경우, *callable*`은 :data:`*arg0*`이 인스턴스(즉, 메서드의 `self 인자)로 설정된 클래스에서 발견되는 함수 객체가 됩니다.

  • RAISE, RERAISE, EXCEPTION_HANDLED, PY_UNWIND, PY_THROW 및 STOP_ITERATION`:

    func(code: CodeType, instruction_offset: int, exception: BaseException) -> object

  • LINE:

    func(code: CodeType, line_number: int) -> object

  • BRANCH_LEFT, BRANCH_RIGHT 및 JUMP`:

    func(code: CodeType, instruction_offset: int, destination_offset: int) -> object

    destination_offset 은 다음에 코드가 실행될 위치임을 유의하십시오.

  • INSTRUCTION:

    func(code: CodeType, instruction_offset: int) -> object
