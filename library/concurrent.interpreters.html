concurrent.interpreters — 프로세스 내 다중 인터프리터¶

Added in version 3.14.

소스 코드: Lib/concurrent/interpreters

concurrent.interpreters\ 모듈은 하위 수준 _interpreters\ 모듈 위에 더 높은 수준의 인터페이스를 구성합니다.

이 모듈은 주로 인터프리터를 관리하고 그 안에서 작업을 수행하기 위한 기본적인 API를 제공하는 데 목적이 있습니다(별칭으로 “서브인터프리터”). 작업은 주로 인터프리터(현재 스레드에서)로 전환하여 해당 실행 컨텍스트에서 함수를 호출하는 것을 포함합니다.

동시성을 위해서는 인터프리터 자체(그리고 이 모듈)가 제공하는 것은 격리화 외에는 거의 없습니다. 격리화 자체는 유용하지 않습니다. 실제 동시성은 :mod:`threads <threading>`를 통해 별도로 사용할 수 있습니다. `아래의 <interp-concurrency>`를 참조하세요.

주요 세부 정보¶

더 자세히 살펴보기 전에, 여러 인터프리터를 사용하는 것에 관해 기억해야 할 몇 가지 세부 사항이 있습니다:

  • isolated, by default

  • 암시적인 스레드가 없습니다.

  • 모든 PyPI 패키지가 아직 여러 인터프리터에서 사용 가능한 것은 아닙니다.

소개¶

“인터프리터”는 사실상 Python 런타임의 실행 컨텍스트입니다. 프로그램 실행에 필요한 모든 상태를 포함합니다. 여기에는 임포트 상태 및 내장 함수와 같은 것이 포함됩니다. (메인 스레드만 있는 경우에도, 각 스레드는 현재 인터프리터 외에 현재 예외 및 바이트코드 평가 루프와 관련된 추가적인 런타임 상태를 가지고 있습니다.)

인터프리터의 개념과 기능은 Python 버전 2.2부터 일부였지만, 이 기능은 C-API를 통해서만 사용 가능했고 잘 알려져 있지 않았으며, `isolation <interp-isolation>`은 버전 3.12까지 상대적으로 불완전했습니다.

다수 인터프리터 및 격리화¶

Python 구현체는 동일한 프로세스 내에서 여러 인터프리터 사용을 지원할 수 있습니다. CPython은 이 지원을 합니다. 각 인터프리터는 효과적으로 다른 인터프리터와 격리되어 있습니다(제한된 수의 주의 깊게 관리된 프로세스 전역 예외 항목을 제외하고).

이 격리화는 주로 프로그램의 뚜렷한 논리적 구성 요소 간의 강력한 분리 지점으로 유용하며, 여기에서 해당 구성 요소들이 어떻게 상호 작용할지 세심하게 제어하고 싶을 때 유용합니다.

참고

동일한 프로세스 내의 인터프리터는 메모리 접근에 대한 제한이 거의 없기 때문에 기술적으로 서로 엄격하게 고립될 수 없습니다. Python 런타임은 격리를 최선을 다해 구현하지만, 확장 모듈이 이를 쉽게 위반할 수 있습니다. 따라서 다른 객체의 데이터에 접근해서는 안 되는 보안이 민감한 상황에서는 여러 인터프리터를 사용하지 마십시오.

인터프리터 내에서 실행하기¶

다른 인터프리터에서 실행하려면, 현재 스레드에서 해당 인터프리터로 전환한 다음 함수를 호출해야 합니다. 런타임은 현재 인터프리터의 상태를 사용하여 해당 함수를 실행합니다. concurrent.interpreters 모듈은 인터프리터를 생성하고 관리하며, 전환 및 호출 작업을 위한 기본적인 API를 제공합니다.

이 작업에 대해 다른 스레드 자동 시작은 없습니다. 하지만, 이를 위한 a helper <interp-call-in-thread>가 있습니다. 또한, 내장 :func:`exec`를 인터프리터에서 호출하는 전용 헬퍼도 있습니다.

exec() (또는 eval())가 인터프리터에서 호출되면, 인터프리터의 __main__ 모듈을 “globals” 네임스페이스로 사용하여 실행됩니다. 모듈과 관련되지 않은 함수도 마찬가지입니다. 이는 명령줄에서 호출된 스크립트가 __main__ 모듈에서 실행되는 방식과 같습니다.

동시성 및 병렬성¶

앞서 언급했듯이, 인터프리터 자체는 동시성을 제공하지 않습니다. 인터프리터는 런타임이 현재 스레드에서 사용할 격리된 실행 컨텍스트를 엄격하게 나타냅니다. 이러한 격리는 프로세스와 유사하게 만들지만, 여전히 스레드처럼 프로세스 내 효율성을 누립니다.

그럼에도 불구하고, 인터프리터는 본질적으로 특정 동시성 형태를 지원합니다. 이러한 격리성에서 강력한 부가 효과가 나옵니다. 이는 비동기(async)나 스레드로는 얻을 수 없는 다른 접근 방식을 가능하게 합니다. 이는 CSP나 액터 모델과 비슷한 동시성 모델이며, 상대적으로 추론하기 쉽습니다.

이러한 동시성 모델은 단일 스레드에서 인터프리터 간에 왔다 갔다 하며 활용할 수 있습니다. 스택리스(Stackless) 방식으로 말입니다. 그러나 이 모델은 인터프리터를 여러 스레드와 결합할 때 더 유용합니다. 이는 주로 새로운 스레드를 시작하고, 다른 인터프리터로 전환하여 원하는 작업을 수행하는 것을 포함합니다.

Python의 각 실제 스레드는, 심지어 메인 스레드에서만 실행되더라도, 자체의 현재 실행 컨텍스트를 가지고 있습니다. 여러 스레드는 동일한 인터프리터나 서로 다른 인터프리터를 사용할 수 있습니다.

높은 수준에서 볼 때, 스레드와 인터프리터의 조합은 선택적 공유가 가능한 스레드와 같습니다.

주요 장점으로는, 인터프리터가 :term:`GIL`를 공유하지 않도록 충분히 격리되어 있어, 스레드를 여러 인터프리터와 결합하면 완전한 다중 코어 병렬성을 구현할 수 있다는 점이 있습니다. (이는 Python 3.12부터 가능했습니다.)

인터프리터 간 통신¶

실제적으로, 여러 인터프리터는 그들 사이에 통신할 방법이 있을 때만 유용합니다. 이는 일반적으로 어떤 형태의 메시지 전달을 포함하지만, 신중하게 관리된 방식으로 데이터를 공유하는 것을 의미할 수도 있습니다.

이를 염두에 두고, concurrent.interpreters 모듈은 create_queue`를 통해 이용 가능한 :class:`queue.Queue() 구현을 제공합니다.

“공유” 객체¶

인터프리터 간에 실제로 공유되는 모든 데이터는 :term:`GIL`이 제공하는 스레드 안전성을 잃게 됩니다. 확장 모듈에서 이를 처리하기 위한 다양한 옵션이 있습니다. 하지만, Python 코드 관점에서 보면 스레드 안전성의 부족은 몇 가지 예외를 제외하고는 객체가 실제로 공유될 수 없다는 것을 의미합니다. 대신, 복사본이 생성되어야 하며, 이는 가변 객체가 동기화 상태를 유지하지 못한다는 것을 의미합니다.

기본적으로, 대부분의 객체는 다른 인터프리터로 전달될 때 :mod:`pickle`로 복사됩니다. 거의 모든 불변 내장 객체는 직접 공유되거나 효율적으로 복사됩니다. 예를 들어:

  • None

  • bool (True`와 :const:`False)

  • bytes

  • str

  • int

  • float

  • tuple (유사하게 지원되는 객체의 튜플)

인터프리터 간에 실제로 가변 데이터를 공유하는 Python 타입은 소수입니다:

  • memoryview

  • Queue

참조¶

모듈은 다음 함수들을 정의합니다:

concurrent.interpreters.list_all()¶

Return a list of Interpreter objects, one for each existing interpreter.

concurrent.interpreters.get_current()¶

현재 실행 중인 인터프리터를 위한 Interpreter 객체를 반환합니다.

concurrent.interpreters.get_main()¶

메인 인터프리터를 위한 Interpreter 객체를 반환합니다. 이는 런타임이 REPL 또는 명령줄에서 제공된 스크립트를 실행하기 위해 생성한 인터프리터입니다. 일반적으로 유일합니다.

concurrent.interpreters.create()¶

새로운 (유휴) Python 인터프리터를 초기화하고 해당 Interpreter 객체를 반환합니다.

concurrent.interpreters.create_queue()¶

새로운 크로스-인터프리터 큐를 초기화하고 해당 Queue 객체를 반환합니다.

인터프리터 객체¶

class concurrent.interpreters.Interpreter(id)¶

현재 프로세스 내의 단일 인터프리터입니다.

일반적으로 :class:`Interpreter`는 직접 호출해서는 안 됩니다. 대신 :func:`create`를 사용하거나 다른 모듈 함수를 사용하십시오.

id¶

(읽기 전용)

내부 인터프리터의 ID입니다.

whence¶

(읽기 전용)

인터프리터가 어디에서 왔는지 설명하는 문자열입니다.

is_running()¶

인터프리터가 현재 자신의 __main__ 모듈에서 코드를 실행 중이면 True 를, 그렇지 않으면 False 를 반환합니다.

close()¶

인터프리터를 초기화하고 파괴합니다.

prepare_main(ns=None, **kwargs)¶

인터프리터의 __main__ 모듈에 객체를 바인딩합니다.

일부 객체는 실제로 공유되며 일부는 효율적으로 복사되지만, 대부분은 :mod:`pickle`을 통해 복사됩니다. :ref:`interp-object-sharing`을 참조하십시오.

exec(code, /, dedent=True)¶

주어진 소스 코드를 인터프리터에서 실행합니다 (현재 스레드에서).

call(callable, /, *args, **kwargs)¶

인터프리터에서 주어진 함수를 호출하는 결과를 반환합니다 (현재 스레드에서).

call_in_thread(callable, /, *args, **kwargs)¶

인터프리터에서 주어진 함수를 실행합니다 (새로운 스레드에서).

예외 처리¶

exception concurrent.interpreters.InterpreterError¶

:exc:`Exception`의 서브 클래스인 이 예외는 인터프리터 관련 오류가 발생했을 때 발생합니다.

exception concurrent.interpreters.InterpreterNotFoundError¶

:exc:`InterpreterError`의 서브 클래스인 이 예외는 대상 인터프리터가 더 이상 존재하지 않을 때 발생합니다.

exception concurrent.interpreters.ExecutionFailed¶

:exc:`InterpreterError`의 서브 클래스인 이 예외는 실행 코드가 포착되지 않은 예외를 발생시켰을 때 발생합니다.

excinfo¶

다른 인터프리터에서 발생한 예외의 기본 스냅샷입니다.

exception concurrent.interpreters.NotShareableError¶

객체가 다른 인터프리터로 전송될 수 없을 때 발생하는 :exc:`TypeError`의 서브 클래스인 이 예외입니다.

인터프리터 간 통신¶

class concurrent.interpreters.Queue(id)¶

queue.Queue 인터페이스를 구현하는 저수준의 크로스-인터프리터 큐를 감쌉니다. 내부 큐는 :func:`create_queue`를 통해서만 생성할 수 있습니다.

일부 객체는 실제로 공유되며 일부는 효율적으로 복사되지만, 대부분은 :mod:`pickle`을 통해 복사됩니다. :ref:`interp-object-sharing`을 참조하십시오.

id¶

(읽기 전용)

큐의 ID입니다.

exception concurrent.interpreters.QueueEmptyError¶

큐가 비어 있을 때 Queue.get() 및 :meth:`!Queue.get_nowait`에서 발생하는 :exc:`queue.Empty`의 서브 클래스인 이 예외입니다.

exception concurrent.interpreters.QueueFullError¶

큐가 가득 찼을 때 Queue.put() 및 :meth:`!Queue.put_nowait`에서 발생하는 :exc:`queue.Full`의 서브 클래스인 이 예외입니다.

기본 사용법¶

인터프리터를 생성하고 코드 실행하기:

from concurrent import interpreters

interp = interpreters.create()

# 현재 OS 스레드에서 실행.

interp.exec('print("foo!")')

interp.exec("""if True:
    print('foo!')
    """)

from textwrap import dedent
interp.exec(dedent("""
    print('foo!')
    """))

def run(arg):
    return arg

res = interp.call(run, 'foo!')
print(res)

def run():
    print('foo!')

interp.call(run)

# 새 OS 스레드에서 실행.

t = interp.call_in_thread(run)
t.join()
