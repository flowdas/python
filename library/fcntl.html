fcntl — fcntl과 ioctl 시스템 호출¶

이 모듈은 파일 기술자에 대한 파일과 I/O 제어를 수행합니다. fcntl()과 ioctl() 유닉스 루틴에 대한 인터페이스입니다. 자세한 설명은 fcntl(2)과 ioctl(2) 유닉스 매뉴얼 페이지를 참조하십시오.

가용성: Unix, not WASI.

이 모듈의 모든 함수는 첫 번째 인자로 파일 기술자 fd를 받아들입니다. 이것은 sys.stdin.fileno()에 의해 반환된 것과 같은 정수 파일 기술자이거나 sys.stdin 자체와 같은 io.IOBase 객체일 수 있습니다. 이 객체는 실제 파일 기술자를 반환하는 fileno()를 제공합니다.

버전 3.3에서 변경: 이 모듈의 연산은 IOError를 발생시켰는데, 이제는 OSError를 발생시킵니다.

버전 3.8에서 변경: fcntl 모듈에는 이제 os.memfd_create() 파일 기술자를 봉인(seal)하기 위한 F_ADD_SEALS, F_GET_SEALS 및 F_SEAL_* 상수가 포함됩니다.

버전 3.9에서 변경: macOS에서, fcntl 모듈은 파일 기술자에서 파일의 경로를 얻는 F_GETPATH 상수를 노출합니다. 리눅스(>=3.15)에서, fcntl 모듈은 열린 파일 기술 록(open file description locks)으로 작업할 때 사용되는 F_OFD_GETLK, F_OFD_SETLK 및 F_OFD_SETLKW 상수를 노출합니다.

버전 3.10에서 변경: Linux >= 2.6.11에서, fcntl 모듈은 F_GETPIPE_SZ 및 F_SETPIPE_SZ 상수를 노출하여 각각 파이프의 크기 확인 및 수정이 가능하게 합니다.

버전 3.11에서 변경: FreeBSD에서는 fcntl 모듈이 F_DUP2FD 및 F_DUP2FD_CLOEXEC 상수를 노출하며, 이는 파일 기술자를 복제하는 데 사용되며, 후자는 추가로 FD_CLOEXEC 플래그를 설정합니다.

버전 3.12에서 변경: Linux 4.5 이상에서는 fcntl 모듈이 FICLONE 및 FICLONERANGE 상수를 노출합니다. 이는 일부 파일 시스템에서 파일의 일부 데이터를 다른 파일과 공유할 수 있도록 재링크(relinking)하는 것을 허용합니다(예: btrfs, OCFS2, XFS). 이 동작은 일반적으로 “copy-on-write”라고 지칭됩니다.

버전 3.13에서 변경: Linux 2.6.32 이상에서는 fcntl 모듈이 F_GETOWN_EX, F_SETOWN_EX, F_OWNER_TID, F_OWNER_PID, F_OWNER_PGRP 상수를 노출하며, 이는 특정 스레드, 프로세스 또는 프로세스 그룹에 I/O 가용성 신호를 직접 전달할 수 있게 합니다. Linux 4.13 이상에서는 fcntl 모듈이 F_GET_RW_HINT, F_SET_RW_HINT, F_GET_FILE_RW_HINT, F_SET_FILE_RW_HINT 및 RWH_WRITE_LIFE_* 상수를 노출하여, 주어진 노드(inode) 또는 특정 열린 파일 기술을 통해 쓰기에 대한 상대적인 예상 수명에 대해 커널에 정보를 제공할 수 있게 합니다. Linux 5.1 이상 및 NetBSD에서는 fcntl 모듈이 F_ADD_SEALS 및 F_GET_SEALS 작업과 함께 사용하기 위한 F_SEAL_FUTURE_WRITE 상수를 노출합니다. FreeBSD에서는 fcntl 모듈이 F_READAHEAD, F_ISUNIONSTACK 및 F_KINFO 상수를 노출합니다. macOS 및 FreeBSD에서는 fcntl 모듈이 F_RDAHEAD 상수를 노출합니다. NetBSD 및 AIX에서는 fcntl 모듈이 F_CLOSEM 상수를 노출합니다. NetBSD에서는 fcntl 모듈이 F_MAXFD 상수를 노출합니다. macOS 및 NetBSD에서는 fcntl 모듈이 F_GETNOSIGPIPE 및 F_SETNOSIGPIPE 상수를 노출합니다.

버전 3.14에서 변경: Linux 6.1 이상에서는 fcntl 모듈이 파일과 동일한 파일을 가리키는 파일 기술자를 조회하기 위한 F_DUPFD_QUERY 를 노출합니다.

이 모듈은 다음 함수를 정의합니다:

fcntl.fcntl(fd, cmd, arg=0, /)¶

파일 기술자 fd\에 대해 cmd 연산을 수행합니다( fileno() 메서드를 제공하는 파일 객체도 허용됩니다). cmd\에 사용되는 값은 운영 체제에 따라 달라지며, 관련 C 헤더 파일에서 사용된 것과 같은 이름을 사용하여 fcntl 모듈에서 상수로 사용할 수 있습니다. 인자 arg\는 정수 값, bytes-like object, 또는 문자열일 수 있습니다. arg\의 타입과 크기는 관련 C 문서에 지정된 작업의 인자 타입과 크기와 일치해야 합니다.

arg\가 정수일 경우, 함수는 C fcntl() 호출의 정수 반환 값을 반환합니다.

인자가 바이트열 객체인 경우, 이는 struct.pack()\으로 생성된 예시와 같은 바이너리 구조를 나타냅니다. 문자열 값은 UTF-8 인코딩을 사용하여 바이너리로 인코딩됩니다. 바이너리 데이터는 주소가 C fcntl() 호출에 전달되는 버퍼로 복사됩니다. 성공적인 호출 후 반환 값은 버퍼 내용이며, bytes 객체로 변환됩니다. 반환된 객체의 길이는 arg 인자의 길이와 동일합니다.

fcntl() 호출이 실패하면, OSError가 발생합니다.

참고

arg \의 타입이나 크기가 작업 인자의 타입이나 크기(예를 들어, 포인터가 예상되는 경우 정수가 전달되거나, 운영 체제가 버퍼로 반환하는 정보가 arg 의 크기보다 큰 경우)와 일치하지 않으면, 이는 세그멘테이션 위반이나 더 미묘한 데이터 손상을 초래할 가능성이 큽니다.

인자 fd, cmd, arg로 감사 이벤트 fcntl.fcntl을 발생시킵니다.

버전 3.14에서 변경: Add support of arbitrary bytes-like objects, not only bytes.

버전 3.15에서 변경: 바이트와 같은 객체의 크기는 더 이상 1024바이트로 제한되지 않습니다.

fcntl.ioctl(fd, request, arg=0, mutate_flag=True, /)¶

이 함수는 인자 처리가 훨씬 더 복잡하다는 점을 제외하면, fcntl() 함수와 같습니다.

request 매개 변수는 플랫폼에 따라 32비트 또는 64비트에 맞는 값으로 제한됩니다. request 인자로 사용하기에 흥미로운 추가 상수들은 관련 C 헤더 파일에서 사용된 이름과 동일한 이름으로 termios 모듈에서 찾을 수 있습니다.

매개 변수 arg \는 정수, 바이트와 같은 객체, 또는 문자열일 수 있습니다. arg 의 타입과 크기는 관련 C 문서에 지정된 연산의 매개 변수 타입과 크기에 일치해야 합니다.

만약 arg 가 읽기-쓰기 버퍼 인터페이스를 지원하지 않거나 mutate_flag 가 false인 경우, fcntl() 함수와 동일하게 작동합니다.

arg 가 읽기-쓰기 버퍼 인터페이스를 지원하고 (bytearray 같은) mutate_flag 가 true인 경우 (기본값), 버퍼는 (실질적으로) 하위 ioctl() 시스템 호출로 전달되며, 후자의 반환 코드는 호출한 Python에 다시 전달되고, 버퍼의 새로운 내용은 ioctl() 의 동작을 반영합니다. 이것은 약간의 단순화이며, 제공된 버퍼가 1024바이트보다 짧은 경우 먼저 1024바이트 길이의 정적 버퍼에 복사된 다음 ioctl() 에 전달되고 다시 제공된 버퍼에 복사됩니다.

ioctl() 호출이 실패하면, OSError 예외가 발생합니다.

참고

arg \의 타입이나 크기가 작업 인자의 타입이나 크기(예를 들어, 포인터가 예상되는 경우 정수가 전달되거나, 운영 체제가 버퍼로 반환하는 정보가 arg 의 크기보다 큰 경우)와 일치하지 않으면, 이는 세그멘테이션 위반이나 더 미묘한 데이터 손상을 초래할 가능성이 큽니다.

예제:

>>> import array, fcntl, struct, termios, os
>>> os.getpgrp()
13341
>>> struct.unpack('h', fcntl.ioctl(0, termios.TIOCGPGRP, "  "))[0]
13341
>>> buf = array.array('h', [0])
>>> fcntl.ioctl(0, termios.TIOCGPGRP, buf, 1)
0
>>> buf
array('h', [13341])

인자 fd, request, arg로 감사 이벤트 fcntl.ioctl을 발생시킵니다.

버전 3.14에서 변경: GIL은 시스템 호출 중에 항상 해제됩니다. EINTR로 실패하는 시스템 호출은 자동으로 재시도됩니다.

버전 3.15에서 변경: 변경되지 않은 바이트와 같은 객체의 크기는 더 이상 1024바이트로 제한되지 않습니다.

fcntl.flock(fd, operation, /)¶

파일 기술자 fd(fileno() 메서드를 제공하는 파일 객체도 허용됩니다)에 대한 잠금 연산 operation을 수행합니다. 자세한 내용은 유닉스 매뉴얼 flock(2)를 참조하십시오. (일부 시스템에서는, 이 함수가 fcntl()를 사용하여 에뮬레이트됩니다.)

flock() 호출이 실패하면, OSError 예외가 발생합니다.

인자 fd, operation으로 감사 이벤트 fcntl.flock을 발생시킵니다.

fcntl.lockf(fd, cmd, len=0, start=0, whence=0, /)¶

이것은 본질에서 fcntl() 잠금 호출에 대한 래퍼입니다. fd는 잠그거나 잠금 해제할 파일의 파일 기술자이고 (fileno() 메서드를 제공하는 파일 객체도 허용됩니다), cmd는 다음 값 중 하나입니다:

fcntl.LOCK_UN¶

기존 잠금 해제.

fcntl.LOCK_SH¶

공유 잠금 획득.

fcntl.LOCK_EX¶

배타적 잠금 획득.

fcntl.LOCK_NB¶

다른 세 개의 LOCK_* 상수와 비트별 OR 하여 요청을 비 블로킹으로 만듭니다.

LOCK_NB가 사용되고 잠금을 얻을 수 없을 때, OSError가 발생하고 errno 어트리뷰트가 EACCES나 EAGAIN으로 설정됩니다 (운영 체제에 따라 다릅니다; 이식성을 위해서 두 값을 모두 확인하십시오). 적어도 일부 시스템에서, LOCK_EX는 파일 기술자가 쓰기 위해 열린 파일을 참조할 때만 사용할 수 있습니다.

len은 잠글 바이트 수, start는 whence가 정의하는 기준으로 잠금이 시작되는 바이트 오프셋이며 whence는 io.IOBase.seek()에서와 같은데, 구체적으로 다음과 같습니다:

  • 0 – 파일의 시작에 상대적 (os.SEEK_SET)

  • 1 – 현재 버퍼 위치에 상대적 (os.SEEK_CUR)

  • 2 – 파일의 끝에 상대적 (os.SEEK_END)

start의 기본값은 파일 시작 부분에서 시작한다는 의미인 0입니다. len의 기본값은 파일 끝까지 잠그는 것을 의미하는 0입니다. whence의 기본값도 0입니다.

인자 fd, cmd, len, start, whence로 감사 이벤트 fcntl.lockf를 발생시킵니다.

예제 (모두 SVR4 호환 시스템에서):

import struct, fcntl, os

f = open(...)
rv = fcntl.fcntl(f, fcntl.F_SETFL, os.O_NDELAY)

lockdata = struct.pack('hhllhh', fcntl.F_WRLCK, 0, 0, 0, 0, 0)
rv = fcntl.fcntl(f, fcntl.F_SETLKW, lockdata)

첫 번째 예제에서 반환 값 변수 rv는 정숫값을 저장합니다; 두 번째 예제에서는 bytes 객체를 저장합니다. lockdata 변수에 대한 구조체 배치는 시스템 종속적입니다 — 그래서 flock() 호출을 사용하는 것이 더 좋을 수 있습니다.
