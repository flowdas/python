profiling.tracing — 결정론적 프로파일러¶

Added in version 3.15.

소스 코드: Lib/profiling/tracing/

profiling.tracing 모듈은 파이썬 프로그램의 결정론적 프로파일링을 제공합니다. 이 모듈은 모든 함수 호출, 함수 반환 및 예외 이벤트를 모니터링하고 각 이벤트에 대한 정확한 타이밍을 기록합니다. 이 접근 방식은 정확한 호출 횟수와 프로그램 실행에 대한 완벽한 가시성을 제공하여 개발 및 테스트 시나리오에 이상적입니다.

참고

이 모듈은 하위 호환성을 위해 cProfile``로도 사용 가능합니다. ``cProfile 이름은 향후 모든 파이썬 버전에서 계속 작동할 것입니다. 코드베이스에 맞는 임포트 방식을 사용하세요:

# 권장 (새 스타일)
import profiling.tracing
profiling.tracing.run('my_function()')

# 작동함 (하위 호환)
import cProfile
cProfile.run('my_function()')

결정론적 프로파일링이란 무엇입니까?¶

:dfn:`결정론적 프로파일링(Deterministic profiling)`은 프로그램 실행 중 모든 함수 호출, 함수 반환 및 예외 이벤트를 포착합니다. 프로파일러는 이러한 이벤트 사이의 정확한 시간 간격을 측정하여 프로그램의 동작에 대한 정확한 통계를 제공합니다.

:ref:`통계적 프로파일링 <profiling-sampling>`은 호출 스택을 주기적으로 샘플링하여 시간 지출을 추정하는 것과 달리, 결정론적 프로파일링은 모든 이벤트를 기록합니다. 이는 통계적 근사치 대신 정확한 호출 횟수를 얻을 수 있음을 의미합니다. 단점은 모든 이벤트를 계측하는 것이 오버헤드를 초래하여 프로그램 실행을 늦출 수 있다는 것입니다.

파이썬의 인터프리터적 특성 덕분에 결정론적 프로파일링이 실용적입니다. 인터프리터는 이미 함수 호출 및 반환에 대한 이벤트를 디스패치하기 때문에, 프로파일러는 코드 수정 없이도 이 메커니즘에 후킹할 수 있습니다. 오버헤드는 인터프리터의 본질적인 비용에 비해 적당하여, 결정론적 프로파일링은 대부분의 개발 워크플로우에 적합합니다.

결정론적 프로파일링은 다음과 같은 질문에 답하는 데 도움이 됩니다:

  • 이 함수는 몇 번 호출되었나요?

  • 제 프로그램의 전체 호출 그래프는 무엇인가요?

  • 특정 함수는 어떤 함수들에 의해 호출되나요?

  • 예상치 못한 함수 호출이 발생하고 있나요?

호출 횟수 통계는 버그(놀라운 횟수)와 인라인 확장 기회(높은 호출 횟수)를 식별할 수 있습니다. 내부 시간 통계는 최적화가 필요한 “핫 루프(hot loops)”를 드러냅니다. 누적 시간 통계는 알고리즘 비효율성을 파악하는 데 도움이 됩니다. 이 프로파일러에서 누적 시간 처리는 재귀 구현과 반복 구현을 직접 비교할 수 있게 합니다.

명령줄 인터페이스¶

profiling.tracing 모듈은 스크립트 인수로 사용되어 다른 스크립트나 모듈을 프로파일링할 수 있습니다:

python -m profiling.tracing [-o output_file] [-s sort_order] (-m module | script.py)

이것은 지정된 스크립트 또는 모듈을 프로파일러 아래에서 실행하고 표준 출력(또는 파일에 저장)에 결과를 출력합니다.

-o <output_file>¶

프로파일 결과를 표준 출력이 아닌 파일에 작성합니다. 출력 파일은 나중에 분석을 위해 pstats 모듈로 읽을 수 있습니다.

-s <sort_order>¶

출력을 지정된 키로 정렬합니다. 이는 cumulative, time, calls, 또는 name`과 같은 :meth:`pstats.Stats.sort_stats`가 인식하는 모든 정렬 키 값을 받습니다. 이 옵션은 :option:-o <profiling.tracing -o>`가 지정되지 않았을 때만 적용됩니다.

-m <module>¶

스크립트 대신 모듈을 프로파일링합니다. 해당 모듈은 표준 임포트 메커니즘을 사용하여 위치를 찾습니다.

Added in version 3.7: cProfile`을 위한 `-m`` 옵션입니다.

Added in version 3.8: profile`을 위한 `-m`` 옵션입니다.

프로그래밍 방식 사용 예시¶

프로파일링에 대한 더 많은 제어를 위해서는 모듈의 함수와 클래스를 직접 사용해야 합니다.

기본 프로파일링¶

가장 간단한 접근 방식은 run() 함수를 사용하는 것입니다:

import profiling.tracing
profiling.tracing.run('my_function()')

이것은 주어진 코드 문자열을 프로파일링하고 표준 출력에 요약 내용을 출력합니다. 나중에 분석할 결과를 저장하려면:

profiling.tracing.run('my_function()', 'output.prof')

Profile 클래스 사용¶

Profile 클래스는 세밀한 제어를 제공합니다:

import profiling.tracing
import pstats
from io import StringIO

pr = profiling.tracing.Profile()
pr.enable()
# ... 프로파일할 코드 ...
pr.disable()

# 결과 출력
s = StringIO()
ps = pstats.Stats(pr, stream=s).sort_stats(pstats.SortKey.CUMULATIVE)
ps.print_stats()
print(s.getvalue())

Profile 클래스는 컨텍스트 관리자로도 작동합니다:

import profiling.tracing

with profiling.tracing.Profile() as pr:
    # ... 프로파일할 코드 ...

pr.print_stats()

모듈 참조¶

profiling.tracing.run(command, filename=None, sort=-1)¶

명령어의 실행을 프로파일링하고 결과를 출력하거나 저장합니다.

이 함수는 exec`를 사용하여 *command* 문자열을 ``__main__`() 모듈의 네임스페이스에서 실행합니다:

exec(command, __main__.__dict__, __main__.__dict__)

filename 을 제공하지 않으면, 함수는 pstats.Stats 인스턴스를 생성하고 표준 출력에 요약 내용을 인쇄합니다. filename 을 제공하면, 원시 프로파일 데이터가 나중에 pstats 로 분석되도록 해당 파일에 저장됩니다.

sort 인수는 인쇄되는 출력의 정렬 순서를 지정하며, :meth:`pstats.Stats.sort_stats`가 인식하는 모든 값을 받을 수 있습니다.

profiling.tracing.runctx(command, globals, locals, filename=None, sort=-1)¶

명시적인 네임스페이스를 사용하여 명령어 실행을 프로파일링합니다.

run`과 유사하지만, ``__main__`() 모듈의 네임스페이스를 사용하는 대신 지정된 globals 및 locals 매핑을 사용하여 명령을 실행합니다:

exec(command, globals, locals)
class profiling.tracing.Profile(timer=None, timeunit=0.0, subcalls=True, builtins=True)¶

실행 통계를 수집하는 프로파일러 객체입니다.

선택적 타이머 인수는 사용자 정의 타이밍 함수를 지정합니다. 제공되지 않으면, 프로파일러는 플랫폼에 맞는 기본 타이머를 사용합니다. 사용자 정의 타이머를 제공할 때는 반드시 현재 시간을 나타내는 단일 숫자를 반환해야 합니다. 타이머가 정수를 반환하는 경우, 한 시간 단위의 지속 시간을 지정하기 위해 timeunit 을 사용하십시오 (예: 밀리초의 경우 0.001).

subcalls 인수는 프로파일러가 함수 간의 호출 관계를 추적하는지 여부를 제어합니다. builtins 인수는 내장 함수가 프로파일링되는지 여부를 제어합니다.

버전 3.8에서 변경: 컨텍스트 관리자 지원을 추가했습니다.

enable()¶

프로파일링 데이터를 수집합니다.

disable()¶

프로파일링 데이터 수집을 중지합니다.

create_stats()¶

데이터 수집을 중지하고 결과를 현재 프로파일로 내부적으로 기록합니다.

print_stats(sort=-1)¶

현재 프로파일로부터 pstats.Stats 객체를 생성하고 결과를 표준 출력으로 인쇄합니다.

sort 인수는 정렬 순서를 지정합니다. :meth:`pstats.Stats.sort_stats`와 동일한 값을 사용하여 단일 키 또는 다단계 정렬을 위한 키 튜플을 허용합니다.

Added in version 3.13: 정렬 키 튜플을 위한 지원입니다.

dump_stats(filename)¶

현재 프로파일 데이터를 filename 에 작성합니다. 이 파일은 나중에 분석하기 위해 pstats.Stats 에서 읽을 수 있습니다.

run(cmd)¶

:func:`exec`를 통해 명령 문자열을 프로파일합니다.

runctx(cmd, globals, locals)¶

지정된 네임스페이스를 사용하여 :func:`exec`를 통해 명령 문자열을 프로파일합니다.

runcall(func, /, *args, **kwargs)¶

함수 호출을 프로파일합니다. func 가 반환하는 모든 것을 반환합니다:

result = pr.runcall(my_function, arg1, arg2, keyword=value)

참고

프로파일링은 프로파일링된 코드가 정상적으로 반환하는 것을 요구합니다. 해석기가 프로파일링 중에 종료되는 경우(예: :func:`sys.exit`를 통해), 결과는 사용할 수 없습니다.

사용자 정의 타이머 사용¶

Profile 클래스는 사용자 정의 타이밍 함수를 받아, 벽시계 시간이나 CPU 시간과 같이 실행의 다양한 측면을 측정하도록 합니다. 타이밍 함수를 생성자에 전달하십시오:

pr = profiling.tracing.Profile(my_timer_function)

타이머 함수는 현재 시간을 나타내는 단일 숫자를 반환해야 합니다. 정수를 반환하는 경우, 하나의 단위를 나타내는 지속 시간도 timeunit 을 지정해야 합니다:

# 타이머가 밀리초 단위로 시간을 반환합니다.
pr = profiling.tracing.Profile(my_ms_timer, 0.001)

최적의 성능을 위해 타이머 함수는 가능한 한 빨라야 합니다. 프로파일러가 이 함수를 자주 호출하므로, 타이머 오버헤드는 프로파일링 오버헤드에 직접적인 영향을 미칩니다.

time 모듈은 사용자 정의 타이머로 사용하기에 적합한 여러 함수를 제공합니다:

  • 고해상도 벽시계 시간의 time.perf_counter()

  • CPU 시간(슬립 제외)을 위한 time.process_time()

  • 단조 시계 시간을 위한 time.monotonic()

제한 사항¶

결정론적 프로파일링은 타이밍 정확도와 관련된 본질적인 한계를 가집니다.

기본 타이머는 일반적으로 약 1밀리초의 해상도를 가집니다. 측정은 이 해상도보다 더 정확할 수 없습니다. 충분한 측정에서는 타이밍 오류가 평균화되는 경향이 있지만, 개별 측정은 부정확할 수 있습니다.

이벤트가 발생하는 시점과 프로파일러가 타임스탬프를 캡처하는 시점 사이에 지연 시간도 있습니다. 마찬가지로, 사용자 코드가 다시 실행되기 전에 타임스탬프를 읽은 후에도 지연 시간(그리고 저장됩니다)이 있습니다. 자주 호출되는 함수는 이 지연 시간을 누적하며, 이로 인해 실제보다 느리게 보이는 현상이 발생할 수 있습니다. 이 오류는 일반적으로 호출당 클럭 틱보다 작지만, 여러 번 호출되는 함수에게는 상당히 커질 수 있습니다.

profiling.tracing 모듈(및 그 cProfile 별칭)은 낮은 오버헤드를 가진 C 확장을 기반으로 구현되었으므로, 이러한 타이밍 문제는 사용이 중단된 순수 Python profile 모듈보다 덜 두드러집니다.
