실행자¶

소스 코드: Lib/asyncio/runners.py

이 섹션에서는 asyncio 코드를 실행하기 위한 높은 수준의 asyncio 원시를 설명합니다.

이들은 공통적이고 널리 퍼진 시나리오에 대한 비동기 코드 사용을 단순화하는 것을 목표로 하는 event loop 위에 구축되어 있습니다.

asyncio 프로그램 실행하기¶

asyncio.run(coro, *, debug=None, loop_factory=None)¶

asyncio 이벤트 루프에서 coro 를 실행하고 결과를 반환합니다.

인자는 모든 어웨이터블 객체일 수 있습니다.

이 함수는 awaitable을 실행하며, asyncio 이벤트 루프와 비동기 제너레이터의 파이널리제이션\ 및 실행기 닫기를 관리합니다.

이 함수는 다른 asyncio 이벤트 루프가 같은 스레드에서 실행 중일 때 호출될 수 없습니다.

debug\이 True\면, 이벤트 루프는 디버그 모드로 실행됩니다. False\는 디버그 모드를 명시적으로 비활성화합니다. None\는 전역 디버그 모드\ 설정을 존중하는 데 사용됩니다.

loop_factory\가 None\이 아닌 경우, 새 이벤트 루프를 생성하는 데 사용됩니다. 그렇지 않은 경우 asyncio.new_event_loop()\가 사용됩니다. 루프는 끝날 때 닫힙니다. 이 함수는 asyncio 프로그램의 메인 진입 지점으로 사용해야 하며, 이상적으로는 한 번만 호출해야 합니다. 정책 대신 loop_factory\를 사용하여 이벤트 루프를 구성하는 것이 좋습니다. asyncio.EventLoop\를 전달하면 정책 시스템 없이 asyncio를 실행할 수 있습니다.

실행기에는 5분의 시간 초과 시간이 주어집니다. 실행기가 해당 시간 내에 완료되지 않으면 경고가 발생하고 실행기가 닫힙니다.

예제:

async def main():
    await asyncio.sleep(1)
    print('hello')

asyncio.run(main())

Added in version 3.7.

버전 3.9에서 변경: loop.shutdown_default_executor()\를 사용하도록 업데이트되었습니다.

버전 3.10에서 변경: :debug\는 전역 디버그 모드 설정을 존중하기 위해 기본적으로 None\입니다.

버전 3.12에서 변경: loop_factory\ 매개변수가 추가되었습니다.

버전 3.14에서 변경: coro\는 모든 awaitable 객체일 수 있습니다.

참고

asyncio 정책 시스템은 사용 중단되었으며 Python 3.16에서 제거될 예정입니다. 그때부터는 이벤트 루프를 구성하기 위해 명시적인 loop_factory\가 필요합니다.

Runner 컨텍스트 관리자¶

class asyncio.Runner(*, debug=None, loop_factory=None)¶

같은 컨텍스트에서 multiple\한 비동기 함수 호출을 단순화하는 컨텍스트 관리자입니다.

때로는 여러 최상위 비동기 함수가 같은 event loop 및 contextvars.Context\에서 호출되어야 합니다.

debug\이 True\면, 이벤트 루프는 디버그 모드로 실행됩니다. False\는 디버그 모드를 명시적으로 비활성화합니다. None\는 전역 디버그 모드\ 설정을 존중하는 데 사용됩니다.

loop_factory\는 루프 생성을 오버라이드하는 데 사용될 수 있습니다. 생성된 루프를 현재 루프로 설정하는 것은 loop_factory\의 책임입니다. 기본적으로 loop_factory\가 None\이면 asyncio.new_event_loop()\가 사용되고 asyncio.set_event_loop()\를 통해 현재 이벤트 루프로 설정됩니다.

기본적으로, asyncio.run()\ 예제는 러너 사용법으로 재작성될 수 있습니다:

async def main():
    await asyncio.sleep(1)
    print('hello')

with asyncio.Runner() as runner:
    runner.run(main())

Added in version 3.11.

run(coro, *, context=None)¶

임베디드 이벤트 루프에서 coro\를 실행합니다.

인자는 모든 어웨이터블 객체일 수 있습니다.

인자가 코루틴인 경우, Task로 래핑됩니다.

선택적 키워드 전용 context\ 인자는 코드가 실행될 사용자 정의 contextvars.Context\를 지정할 수 있도록 합니다. context가 None\이면 러너의 기본 컨텍스트가 사용됩니다.

awaitable의 결과를 반환하거나 예외를 발생시킵니다.

이 함수는 다른 asyncio 이벤트 루프가 같은 스레드에서 실행 중일 때 호출될 수 없습니다.

버전 3.14에서 변경: coro\는 모든 awaitable 객체일 수 있습니다.

close()¶

러너를 종료합니다.

비동기 제너레이터를 파이널라이즈하고, 기본 실행기를 종료하며, 이벤트 루프를 닫고 임베디드 contextvars.Context\를 해제합니다.

get_loop()¶

러너 인스턴스와 연관된 이벤트 루프를 반환합니다.

참고

Runner\는 지연 초기화 전략을 사용하며, 생성자는 내부 낮은 수준의 구조체들을 초기화하지 않습니다.

임베디드 loop\와 context\는 with\ 블록 진입 또는 run()\ 또는 get_loop()\의 첫 번째 호출 시 생성됩니다.

키보드 중단 처리¶

Added in version 3.11.

signal.SIGINT\가 Ctrl-C\에 의해 발생하면, 기본적으로 메인 스레드에서 KeyboardInterrupt\ 예외가 발생합니다. 하지만 이것은 asyncio\와 함께 작동하지 않는데, 이는 asyncio 내부를 중단시킬 수 있고 프로그램을 종료하는 것을 지연시킬 수 있기 때문입니다.

이 문제를 완화하기 위해, asyncio\는 signal.SIGINT\를 다음과 같이 처리합니다:

  1. :meth:asyncio.Runner.run`은 사용자 코드를 실행하기 전에 사용자 지정 :const:`signal.SIGINT 처리기를 설치하고, 함수를 종료할 때 제거합니다.`

  2. :class:~asyncio.Runner`는 전달된 코루틴의 실행을 위한 메인 태스크를 생성합니다.

  3. :const:signal.SIGINT`가 `:kbd:`Ctrl-C`에 의해 발생하면, 사용자 지정 신호 처리기가 :meth:`asyncio.Task.cancel`를 호출하여 메인 태스크를 취소하고, 이 과정에서 메인 태스크 내부에 :exc:`asyncio.CancelledError`가 발생합니다. 이는 파이썬 스택을 풀백(unwind)시키고, ``try/except` 및 try/finally 블록을 리소스 정리용으로 사용할 수 있게 합니다. 메인 태스크가 취소된 후, asyncio.Runner.run`은 :exc:`KeyboardInterrupt`를 발생시킵니다.()

  4. 사용자는 :meth:`asyncio.Task.cancel`에 의해 중단될 수 없는 순환 루프를 작성할 수 있으며, 이 경우 두 번째 `:kbd:`Ctrl-C`는 메인 태스크를 취소하지 않고 즉시 :exc:`KeyboardInterrupt`를 발생시킵니다.
