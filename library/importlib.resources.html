importlib.resources – 패키지 리소스 읽기, 열기, 및 액세스¶

소스 코드: Lib/importlib/resources/__init__.py

Added in version 3.7.

이 모듈은 파이썬의 임포트 시스템을 활용하여 패키지(packages) 안에 있는 리소스(resources)\에 대한 액세스를 제공합니다.

“리소스”는 Python에서 모듈이나 패키지와 연관된 파일 같은 리소스입니다. 리소스는 패키지 내에 직접 포함되거나, 해당 패키지 내의 하위 디렉토리에 포함되거나, 패키지 외부의 모듈 옆에 위치할 수 있습니다. 리소스는 텍스트 또는 바이너리일 수 있습니다. 결과적으로, 패키지의 Python 모듈 소스(.py), 컴파일된 아티팩트(pycache), 그리고 설치된 아티팩트(디렉터리 내의 :func:`reserved filenames <os.path.isreserved>`와 같은 파일들)는 기술적으로 해당 패키지의 사실상의 리소스입니다. 그러나 실제로는 리소스는 패키지 작성자가 특별히 노출하는 비(非)Python 아티팩트가 주로 해당됩니다.

리소스는 바이너리 모드 또는 텍스트 모드 둘 중 하나로 열거나 읽을 수 있습니다.

리소스는 디렉터리 내부의 파일과 유사하지만, 이것이 단지 은유라는 점을 염두에 두는 것이 중요합니다. 리소스와 패키지는 파일 시스템에 물리적인 파일과 디렉터리로 존재할 필요가 없습니다. 예를 들어, 패키지와 그 리소스를 :py:mod:`zipimport`를 사용하여 zip 파일에서 가져올 수 있습니다.

경고

importlib.resources`는 내장 :func:`open 함수와 동일한 보안 모델을 따릅니다. 이 모듈의 함수에 신뢰할 수 없는 입력을 전달하는 것은 안전하지 않습니다.

참고

이 모듈의 독립형 역 이식은 `using importlib.resources <https://importlib-resources.readthedocs.io/en/latest/using.html>`_ 및 `migrating from pkg_resources to importlib.resources <https://importlib-resources.readthedocs.io/en/latest/migration.html>`_에서 더 많은 정보를 제공합니다.

리소스 읽기를 지원하려는 Loaders <importlib.abc.Loader>`는 :class:`importlib.resources.abc.ResourceReader`에 의해 지정된 ``get_resource_reader(fullname)` 메서드를 구현해야 합니다.

class importlib.resources.Anchor¶

리소스의 앵커를 나타내며, module object 이거나 문자열 형태의 모듈 이름일 수 있습니다. Union[str, ModuleType] 으로 정의됩니다.

importlib.resources.files(anchor: Anchor | None = None)¶

패키지의 리소스 컨테이너(디렉터리를 생각하십시오)와 그 리소스(파일을 생각하십시오)를 나타내는 Traversable 객체를 반환합니다. Traversable은 다른 컨테이너를 포함할 수 있습니다 (서브 디렉터리를 생각하십시오).

anchor 는 선택적 Anchor 입니다. 앤커가 패키지인 경우 리소스는 그 패키지에서 해석됩니다. 모듈인 경우, 리소스는 해당 모듈 옆(같은 패키지 또는 패키지 루트)에서 해석됩니다. 앤커가 생략된 경우, 호출자의 모듈이 사용됩니다.

Added in version 3.9.

버전 3.12에서 변경: package 매개 변수는 anchor 로 이름이 변경되었습니다. package 도 여전히 받아들여지지만, 사용이 중단되었습니다.

버전 3.15에서 변경: package 매개 변수는 완전히 제거되었습니다. 이제 anchor 는 비(非)패키지 모듈일 수 있으며, 생략될 경우 호출자의 모듈을 기본값으로 사용합니다. Python 3.15부터는 더 이상 package 가 허용되지 않습니다. 구 버전 Python에서는 호환 인터페이스를 위해 앤커를 위치별로 전달하거나 importlib_resources >= 5.10 을 사용하도록 고려하십시오.

importlib.resources.as_file(traversable)¶

importlib.resources.files`에서 가져온 파일 또는 디렉터리를 나타내는 :class:`~importlib.resources.abc.Traversable() 객체를 주면, with 문에서 사용할 컨텍스트 관리자를 반환합니다. 컨텍스트 관리자는 pathlib.Path 객체를 제공합니다.

컨텍스트 관리자를 종료하면 예를 들어 zip 파일에서 리소스를 추출할 때 생성된 임시 파일이나 디렉터리가 정리됩니다.

Traversable 메서드(read_text 등)가 충분하지 않고 파일 시스템의 실제 파일 또는 디렉터리가 필요하면 as_file 을 사용하십시오.

Added in version 3.9.

버전 3.12에서 변경: 디렉터리를 나타내는 traversable 에 대한 지원이 추가되었습니다.

기능별 API¶

간소화되고 하위 호환 가능한 헬퍼 집합이 제공됩니다. 이를 통해 일반적인 작업을 단일 함수 호출에서 수행할 수 있습니다.

다음 모든 함수에 대해:

  • anchor 는 files() 에서와 같이 Anchor 입니다. files 와 달리, 이는 생략할 수 없습니다.

  • path_names 는 앤커를 기준으로 한 리소스 경로 이름의 구성 요소입니다. 예를 들어, info.txt 라는 이름을 가진 리소스의 텍스트를 얻으려면 다음을 사용하십시오:

    importlib.resources.read_text(my_module, "info.txt")

    Traversable.joinpath 와 마찬가지로, 개별 구성 요소는 경로 구분 기호로 슬래시(/)를 사용해야 합니다. 예를 들어, 다음은 모두 동일합니다:

    importlib.resources.read_binary(my_module, "pics/painting.png")
importlib.resources.read_binary(my_module, "pics", "painting.png")

    하위 호환성 문제로 인해, 텍스트를 읽는 함수는 여러 path_names 가 제공되는 경우 명시적 encoding 인수가 필요합니다. 예를 들어, info/chapter1.txt 의 텍스트를 얻으려면 다음을 사용하십시오:

    importlib.resources.read_text(my_module, "info", "chapter1.txt",
                              encoding='utf-8')
importlib.resources.open_binary(anchor, *path_names)¶

이름 지정된 리소스를 바이너리 읽기로 엽니다.

anchor 및 path_names 에 대한 자세한 내용은 도입부 을 참조하십시오.

이 함수는 BinaryIO 객체, 즉 읽기용 바이너리 스트림을 반환합니다.

이 함수는 다음과 거의 동일합니다:

files(anchor).joinpath(*path_names).open('rb')

버전 3.13에서 변경: 여러 path_names 를 수용합니다.

importlib.resources.open_text(anchor, *path_names, encoding='utf-8', errors='strict')¶

이름 지정된 리소스를 텍스트 읽기로 엽니다. 기본적으로 내용은 엄격한 UTF-8로 읽힙니다.

anchor 및 path_names 에 대한 자세한 내용은 도입부 을 참조하십시오. encoding 및 errors 는 내장 open() 에서와 동일한 의미를 갖습니다.

하위 호환성 문제로 인해, path_names*가 여러 개인 경우 *encoding 인수를 명시적으로 제공해야 합니다. 이 제한은 Python 3.15에서 제거될 예정입니다.

이 함수는 TextIO 객체, 즉 읽기용 텍스트 스트림을 반환합니다.

이 함수는 다음과 거의 동일합니다:

files(anchor).joinpath(*path_names).open('r', encoding=encoding)

버전 3.13에서 변경: 여러 path_names 를 수용합니다. encoding 및 errors 는 키워드 인수(keyword arguments)로 제공해야 합니다.

importlib.resources.read_binary(anchor, *path_names)¶

이름 지정된 리소스의 내용을 :class:`bytes`로 읽고 반환합니다.

anchor 및 path_names 에 대한 자세한 내용은 도입부 을 참조하십시오.

이 함수는 다음과 거의 동일합니다:

files(anchor).joinpath(*path_names).read_bytes()

버전 3.13에서 변경: 여러 path_names 를 수용합니다.

importlib.resources.read_text(anchor, *path_names, encoding='utf-8', errors='strict')¶

이름 지정된 리소스의 내용을 :class:`str`로 읽고 반환합니다. 기본적으로 내용은 엄격한 UTF-8로 읽힙니다.

anchor 및 path_names 에 대한 자세한 내용은 도입부 을 참조하십시오. encoding 및 errors 는 내장 open() 에서와 동일한 의미를 갖습니다.

하위 호환성 문제로 인해, path_names*가 여러 개인 경우 *encoding 인수를 명시적으로 제공해야 합니다. 이 제한은 Python 3.15에서 제거될 예정입니다.

이 함수는 다음과 거의 동일합니다:

files(anchor).joinpath(*path_names).read_text(encoding=encoding)

버전 3.13에서 변경: 여러 path_names 를 수용합니다. encoding 및 errors 는 키워드 인수(keyword arguments)로 제공해야 합니다.

importlib.resources.path(anchor, *path_names)¶

resource 에 대한 실제 파일 시스템 경로를 제공합니다. 이 함수는 with 문에서 사용할 컨텍스트 관리자를 반환합니다. 컨텍스트 관리자는 pathlib.Path 객체를 제공합니다.

컨텍스트 관리자를 종료하면 예를 들어 리소스를 zip 파일에서 추출해야 할 때 생성된 임시 파일이 정리됩니다.

예를 들어, stat() 메서드는 실제 파일 시스템 경로가 필요합니다. 다음과 같이 사용할 수 있습니다:

with importlib.resources.path(anchor, "resource.txt") as fspath:
    result = fspath.stat()

anchor 및 path_names 에 대한 자세한 내용은 도입부 을 참조하십시오.

이 함수는 다음과 거의 동일합니다:

as_file(files(anchor).joinpath(*path_names))

버전 3.13에서 변경: 여러 path_names 를 수용합니다.

importlib.resources.is_resource(anchor, *path_names)¶

이름이 지정된 리소스가 존재하면 True 를 반환하고, 그렇지 않으면 False 를 반환합니다. 이 함수는 디렉터리를 리소스로 간주하지 않습니다.

anchor 및 path_names 에 대한 자세한 내용은 도입부 을 참조하십시오.

이 함수는 다음과 거의 동일합니다:

files(anchor).joinpath(*path_names).is_file()

버전 3.13에서 변경: 여러 path_names 를 수용합니다.

importlib.resources.contents(anchor, *path_names)¶

패키지 또는 경로 내의 이름이 지정된 항목에 대한 이터러블을 반환합니다. 이 이터러블은 리소스의 이름(예: 파일)과 리소스가 아닌 것(예: 디렉터리)을 :class:`str`로 반환합니다. 이터러블은 하위 디렉터리로 재귀하지 않습니다.

anchor 및 path_names 에 대한 자세한 내용은 도입부 을 참조하십시오.

이 함수는 다음과 거의 동일합니다:

for resource in files(anchor).joinpath(*path_names).iterdir():
    yield resource.name

버전 3.11부터 폐지됨: 다음과 같이 iterdir() 를 사용하는 것이 좋습니다. 이 방법은 결과에 대한 더 많은 제어와 풍부한 기능을 제공합니다.
