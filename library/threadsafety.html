스레드 안전성 보장¶

이 페이지는 Python의 자유 스레딩 빌드에 포함된 기본 유형에 대한 스레드 안전성 보증을 문서화합니다. 여기에 설명된 보증은 `:term:`GIL`이 비활성화된(자유 스레딩 모드) 상태에서 Python을 사용할 때 적용됩니다. GIL이 활성화되면, 대부분의 작업은 암묵적으로 직렬화됩니다.

무료 스레딩 Python에서 스레드 안전 코드를 작성하는 일반적인 안내는 :ref:`freethreading-python-howto`를 참조하십시오.

스레드 안전성 수준¶

C API 문서는 각 함수의 스레드 안전성 보장을 설명하기 위해 다음 수준을 사용합니다. 수준은 가장 안전하지 않은 것부터 가장 안전한 순서로 나열되어 있습니다.

호환되지 않음¶

외부 동기화가 있어도 안전하게 사용할 수 없는 함수 또는 연산입니다. 비호환 코드는 일반적으로 전역 상태에 동기화되지 않은 방식으로 접근하며, 프로그램 실행 주기 동안 단일 스레드에서만 호출되어야 합니다.

예: 시그널 핸들러 또는 환경 변수와 같이 프로세스 전체의 상태를 수정하는 함수로, 외부 락킹(locking)을 포함하여 어떤 스레드에서 발생한 동시 호출이라도 런타임 또는 다른 라이브러리와 충돌할 수 있습니다.

호환 가능¶

호출자가 적절한 외부 동기화 (예: 호출하는 동안 lock 보유)를 제공하는 경우 여러 스레드에서 호출하는 것이 안전한 함수 또는 연산입니다. 그러한 동기화가 없으면, 동시 호출은 race conditions 또는 :term:`data races <데이터 경쟁>`를 생성할 수 있습니다.

예: 내부 상태가 록에 의해 보호되지 않는 객체에서 읽거나 쓰는 함수. 호출자는 두 스레드가 같은 객체에 동시에 액세스하지 않도록 보장해야 합니다.

개별 객체에 안전¶

외부 동기화 없이 여러 스레드에서 호출하기 안전한 함수 또는 작업. 각 스레드가 다른 객체에 대해 작동하는 경우에 한합니다. 두 스레드가 동시에 함수를 호출할 수 있지만, 동일한 객체(또는 기본 상태를 공유하는 객체)를 인수로 전달해서는 안 됩니다.

예: 비원자적 쓰기를 사용하여 구조체 필드를 수정하는 함수. 두 스레드는 각자 자신의 구조체 인스턴스에 대해 안전하게 함수를 호출할 수 있지만, 같은 인스턴스에 대한 동시 호출에는 외부 동기화가 필요합니다.

공유 객체에 안전¶

같은 객체에 대한 동시 사용에 안전한 함수 또는 작업. 구현은 공유 가능한 가변 상태를 보호하기 위해 내부 동기화(예: per-object locks 또는 critical sections)를 사용하므로, 호출자가 자체 록킹을 공급할 필요가 없습니다.

예: :c:func:`PyList_GetItemRef`는 동일한 :c:type:`PyListObject`에서 여러 스레드에 의해 호출될 수 있습니다. 이는 접근을 직렬화하기 위해 내부 동기화를 사용합니다.

원자적 (Atomic)¶

다른 스레드를 기준으로 :term:`atomic <atomic operation>`로 보이는 함수 또는 작업. 이는 다른 스레드의 관점에서 즉시 실행됩니다. 이는 가장 강력한 형태의 스레드 안전성입니다.

예: :c:func:`PyMutex_IsLocked`는 뮤텍스 상태를 원자적으로 읽어 수행하며, 모든 스레드에서 언제든지 호출될 수 있습니다.

리스트 객체에 대한 스레드 안전성¶

:class:`list`에서 단일 요소를 읽는 것은 :term:`atomic <atomic operation>`입니다:

lst[i]   # list.__getitem__

다음 메서드들은 리스트를 순회하며 각 항목의 atomic 읽기를 사용하여 기능을 수행합니다. 이는 동시 수정에 의해 영향을 받은 결과를 반환할 수 있음을 의미합니다:

item in lst
lst.index(item)
lst.count(item)

위의 모든 작업은 :term:`per-object locks <per-object lock>`을 획득하는 것을 피합니다. 이들은 동시 수정을 차단하지 않습니다. 록을 보유하는 다른 작업들도 중간 상태를 관찰하는 것을 차단하지 않을 것입니다.

이후의 모든 작업은 :term:`per-object lock`을 사용하여 차단합니다.

lst[i] = x 를 통해 단일 항목을 쓰는 것은 여러 스레드에서 호출하기 안전하며, 리스트를 손상시키지 않습니다.

다음 작업들은 새 객체를 반환하며 다른 스레드에게 :term:`atomic <atomic operation>`으로 보입니다:

lst1 + lst2    # 두 리스트를 새 리스트로 이어붙입니다
x * lst        # lst를 x번 반복하여 새 리스트로 만듭니다
list.copy()     # 리스트의 얕은 복사본을 반환합니다

다음 메서드들은 이동이 필요 없는 단일 요소에만 작동하므로 :term:`atomic <atomic operation>`입니다:

lst.append(x)  # 리스트 끝에 추가하며, 이동이 필요하지 않습니다
lst.pop()      # 리스트 끝에서 요소를 제거하며, 이동이 필요하지 않습니다

clear() 메서드 또한 :term:`atomic <atomic operation>`입니다. 다른 스레드는 요소가 제거되는 것을 관찰할 수 없습니다.

sort() 메서드는 :term:`atomic <atomic operation>`이 아닙니다. 다른 스레드는 정렬 중 중간 상태를 관찰할 수 없지만, 정렬 기간 동안 리스트는 비어 있는 것처럼 보입니다.

다음 작업들은 여러 요소를 제자리에서 수정하기 때문에 lock-free 작업이 중간 상태를 관찰할 수 있도록 허용할 수 있습니다:

lst.insert(idx, item)  # 요소를 이동시킵니다
lst.pop(idx)           # idx가 리스트 끝이 아니며, 요소를 이동시킵니다
lst *= x               # 요소를 제자리에서 복사합니다

remove() 메서드는 요소 비교가 임의의 Python 코드( :meth:`~object.__eq__`을 통해)를 실행할 수 있으므로 동시 수정을 허용할 수 있습니다.

만약 그 이터러블이 :class:`list(), tuple, set, frozenset, dict 또는 딕셔너리 뷰 객체 (또는 그 자식 클래스가 아닐 경우)라면, extend 작업은 이터러블에 대한 동시 수정으로부터 안전합니다. 그렇지 않은 경우, 다른 스레드에 의해 동시 수정될 수 있는 이터레이터가 생성됩니다. 이는 `lst += iterable`를 사용하여 리스트를 다른 이터러블과 인플레이스 연결하는 경우에도 동일하게 적용됩니다.

마찬가지로, lst[i:j] = iterable 을 사용하여 리스트 슬라이스에 할당하는 것은 여러 스레드에서 호출해도 안전하지만, iterable 는 list 일 때만 (그 자식 클래스는 제외) 잠금 처리됩니다.

여러 액세스나 반복을 수반하는 작업은 절대 원자적이지 않습니다. 예를 들어:

# 원자적이지 않음: 읽기-수정-쓰기
lst[i] = lst[i] + 1

# 원자적이지 않음: 확인 후 실행
if lst:
    item = lst.pop()

# 스레드에 안전하지 않음: 수정 중 반복
for item in lst:
    process(item)  # 다른 스레드가 lst를 수정할 수 있음

쓰레드 간에 list 인스턴스를 공유할 때는 외부 동기화를 고려하십시오.

딕셔너리 객체의 스레드 안전성¶

dict 생성자를 사용하여 딕셔너리를 생성하는 것은 인수가 dict 또는 tuple`일 때 원자적입니다. :meth:`dict.fromkeys 메서드를 사용할 때는 인수가 dict, tuple, set 또는 :class:`frozenset`일 때 딕셔너리 생성이 원자적입니다.

The following operations and functions are lock-free and atomic.

d[key]       # dict.__getitem__
d.get(key)   # dict.get
key in d     # dict.__contains__
len(d)       # dict.__len__

All other operations from here on hold the per-object lock.

단일 항목을 쓰거나 제거하는 것은 여러 스레드에서 호출해도 안전하며 딕셔너리를 손상시키지 않습니다:

d[key] = value        # 쓰기
del d[key]            # 삭제
d.pop(key)            # 제거 및 반환
d.popitem()           # 제거 및 마지막 항목 반환
d.setdefault(key, v)  # 누락된 경우 삽입

이러한 비교 연산은 :class:`str(), int, 및 :class:`float`와 같은 내장 유형의 경우, C에서 :meth:`~object.__eq__`를 구현하므로, 기본 잠금은 비교 중에 해제되지 않으므로 이는 문제가 되지 않습니다.

The following operations return new objects and hold the per-object lock for the duration of the operation:

d.copy()      # 딕셔너리의 얕은 복사본을 반환합니다.
d | other     # 두 딕셔너리를 새 딕셔너리로 병합합니다.
d.keys()      # 새로운 dict_keys 뷰 객체를 반환합니다.
d.values()    # 새로운 dict_values 뷰 객체를 반환합니다.
d.items()     # 새로운 dict_items 뷰 객체를 반환합니다.

clear() 메서드는 그 기간 동안 락을 유지합니다. 다른 스레드는 제거되는 요소를 관찰할 수 없습니다.

다음 연산들은 두 딕셔너리 모두에 락을 겁니다. update() 와 |= 의 경우, 이는 다른 피연산자가 표준 딕셔너리 이터레이터를 사용하는 dict 일 때만 적용됩니다 (반복을 재정의하는 자식 클래스는 제외). 등가 비교의 경우, 이는 dict 와 그 자식 클래스에 적용됩니다:

d.update(other_dict)  # other_dict가 dict일 경우 두 딕셔너리 모두 잠김
d |= other_dict       # other_dict가 dict일 경우 두 딕셔너리 모두 잠김
d == other_dict       # dict 및 자식 클래스의 경우 두 딕셔너리 모두 잠김

모든 비교 연산은 :meth:`~object.__eq__`를 사용하여 값도 비교하므로, 내장되지 않은 유형의 경우 비교 중에 락이 해제될 수 있습니다.

fromkeys`는 반복 가능한 객체가 정확히 :class:`dict(), set, 또는 :class:`frozenset`일 때 새로운 딕셔너리와 이터러블 모두에 락을 겁니다 (자식 클래스는 제외).

dict.fromkeys(a_dict)      # 둘 다 잠금
dict.fromkeys(a_set)       # 둘 다 잠금
dict.fromkeys(a_frozenset) # 둘 다 잠금

비-딕셔너리 이터러블로부터 업데이트하는 경우, 대상 딕셔너리만 잠깁니다. 이터러블은 다른 스레드에 의해 동시 수정될 수 있습니다:

d.update(iterable)        # iterable은 딕셔너리가 아닐 경우: d만 잠금
d |= iterable             # 딕셔너리가 아닐 경우: d만 잠금
dict.fromkeys(iterable)   # 딕셔너리/집합/불변집합이 아닐 경우: 결과만 잠금

여러 액세스 및 반복을 포함하는 작업은 원자적이지 않습니다:

# 원자적이지 않음: 읽기-수정-쓰기
d[key] = d[key] + 1

# 원자적이지 않음: 확인 후 실행 (TOCTOU)
if key in d:
    del d[key]

# 스레드 안전하지 않음: 수정 중 반복
for key, value in d.items():
    process(key)  # 다른 스레드가 d를 수정할 수 있음

확인 시간과 사용 시간 사이의 불일치 (TOCTOU) 문제를 피하려면, 원자적 연산을 사용하거나 예외 처리를 하십시오:

# 확인 후 삭제 대신 기본값과 함께 pop() 사용
d.pop(key, None)

# 또는 예외 처리
try:
    del d[key]
except KeyError:
    pass

다른 스레드에 의해 수정될 수 있는 딕셔너리를 안전하게 반복하려면, 복사본을 반복하세요:

# 안전하게 반복하기 위해 복사본 생성
for key, value in d.copy().items():
    process(key)

스레드 간에 dict 인스턴스를 공유할 때는 외부 동기화를 고려하십시오.

집합 객체의 스레드 안전성¶

len() 함수는 락이 필요 없으며 :term:`원자적 <atomic operation>`입니다.

다음 읽기 작업은 락이 필요 없습니다. 동시 수정 작업을 차단하지 않으며, 개체별 락을 보유하는 작업에서 중간 상태를 관찰할 수 있습니다:

elem in s    # set.__contains__

이 작업은 :class:`str(), int, 및 float`과 같은 내장 유형의 경우, :meth:!__eq__`는 비교 중 기본 락을 해제하지 않으므로 이는 문제가 아닙니다.

나머지 모든 작업은 개체별 락을 유지합니다.

요소 하나를 추가하거나 제거하는 것은 다중 스레드에서 호출하기 안전하며 세트를 손상시키지 않습니다:

s.add(elem)      # 요소 추가
s.remove(elem)   # 요소 제거하고, 누락된 경우 오류 발생
s.discard(elem)  # 요소가 존재하는 경우에만 제거
s.pop()          # 임의의 요소를 제거하고 반환

이러한 작업도 요소를 비교하므로, 위의 것과 동일한 __eq__() 고려 사항이 적용됩니다.

copy() 메서드는 새 객체를 반환하며, 항상 원자적으로 유지하기 위해 개체별 락을 유지합니다.

clear() 메서드는 자신의 기간 동안 락을 유지합니다. 다른 스레드는 요소가 제거되는 것을 관찰할 수 없습니다.

다음 작업은 오직 set 또는 :class:`frozenset`만을 피연산자로 받으며 항상 두 객체를 잠급니다:

s |= other                   # other는 set/frozenset이어야 함
s &= other                   # other는 set/frozenset이어야 함
s -= other                   # other는 set/frozenset이어야 함
s ^= other                   # other는 set/frozenset이어야 함
s & other                    # other는 set/frozenset이어야 함
s | other                    # other는 set/frozenset이어야 함
s - other                    # other는 set/frozenset이어야 함
s ^ other                    # other는 set/frozenset이어야 함

set.update(), set.union(), set.intersection() 및 :meth:`set.difference`는 여러 이터러블을 인자로 받을 수 있습니다. 이들은 모두 전달된 모든 이터러블을 반복하여 다음을 수행합니다:

  • set.update() 및 :meth:`set.union`은 다음 경우에만 두 객체를 잠급니다:

    다른 피연산자가 set, frozenset, 또는 :class:`dict`인 경우.

  • set.intersection() 및 :meth:`set.difference`는 항상 모든 객체를 잠그려고 시도합니다:

    모든 객체를 잠급니다.

:meth:`set.symmetric_difference`는 두 객체 모두를 잠그려고 시도합니다.

위 메서드의 업데이트 변형에도 몇 가지 차이점이 있습니다:

  • set.difference_update() 및 :meth:`set.intersection_update`는

    모든 객체를 하나씩 잠그하려고 시도합니다.

  • :meth:`set.symmetric_difference_update`는 오직 인수가

    set, frozenset, 또는 dict 유형일 경우에만 잠급니다.

다음 메서드들은 항상 두 객체를 잠그려고 시도합니다:

s.isdisjoint(other)          # 두 개 모두 잠금
s.issubset(other)            # 두 개 모두 잠금
s.issuperset(other)          # 두 개 모두 잠금

여러 액세스 및 반복을 포함하는 작업은 원자적이지 않습니다:

# 원자적이지 않음: 체크-후-실행
if elem in s:
      s.remove(elem)

# 스레드 안전하지 않음: 수정 중 반복
for elem in s:
      process(elem)  # 다른 스레드가 s를 수정할 수 있습니다.

스레드 간에 set 인스턴스를 공유할 때는 외부 동기화를 고려하세요. 자세한 내용은 자유 스레딩에 대한 파이썬 지원 를 참고하세요.

바이트 배열 객체의 스레드 안전성¶

len() 함수는 락이 필요 없으며 :term:`원자적 <atomic operation>`입니다.

이어붙이기와 비교는 버퍼 프로토콜을 사용하며, 이는 크기 조절을 방지하지만 개별 객체 잠금은 유지하지 않습니다. 이 작업들은 동시 수정의 중간 상태를 관찰할 수 있습니다:

ba + other    # 동시 쓰기 작업 관찰 가능
ba == other   # 동시 쓰기 작업 관찰 가능
ba < other    # 동시 쓰기 작업 관찰 가능

나머지 모든 작업은 개체별 락을 유지합니다.

단일 요소를 읽거나 슬라이스 하는 것은 여러 스레드에서 호출해도 안전합니다:

ba[i]        # bytearray.__getitem__
ba[i:j]      # slice

다음 작업들은 여러 스레드에서 호출해도 안전하며 바이트 배열을 손상시키지 않습니다:

ba[i] = x         # 단일 바이트 쓰기
ba[i:j] = values  # 슬라이스 쓰기
ba.append(x)      # 단일 바이트 추가
ba.extend(other)  # 반복 가능한 객체로 확장
ba.insert(i, x)   # 단일 바이트 삽입
ba.pop()          # 마지막 바이트 제거하고 반환
ba.pop(i)         # 인덱스에 있는 바이트 제거하고 반환
ba.remove(x)      # 첫 번째 발생 제거
ba.reverse()      # 제자리 역순화
ba.clear()        # 모든 바이트 제거

슬라이스 대입은 values 가 bytearray 인 경우 두 객체 모두에 잠금을 겁니다:

ba[i:j] = other_bytearray  # 둘 다 잠금됨

다음 작업들은 새 객체를 반환하고 해당 기간 동안 개별 객체 잠금을 유지합니다:

ba.copy()     # 얕은 복사본 반환
ba * n        # 새로운 바이트 배열로 반복

멤버십 테스트는 이 시간 동안 잠금을 유지합니다:

x in ba       # bytearray.__contains__

다른 모든 바이트 배열 메서드 (예: find(), replace(), split(), decode() 등)는 해당 기간 동안 개별 객체 잠금을 유지합니다.

여러 액세스 및 반복을 포함하는 작업은 원자적이지 않습니다:

# 원자적이지 않음: 체크-후-실행
if x in ba:
    ba.remove(x)

# 스레드 안전하지 않음: 수정 중 반복
for byte in ba:
    process(byte)  # 다른 스레드가 ba를 수정할 수 있습니다.

다른 스레드에 의해 수정될 수 있는 바이트 배열을 안전하게 반복하려면 사본을 반복하세요:

# 안전하게 반복하려면 사본을 만드세요
for byte in ba.copy():
    process(byte)

스레드 간에 bytearray 인스턴스를 공유할 때는 외부 동기화를 고려하세요. 자세한 내용은 자유 스레딩에 대한 파이썬 지원 를 참고하세요.

MemoryView 객체의 스레드 안전성¶

memoryview 객체는 내부 객체의 내부 데이터에 복사 없이 접근할 수 있게 합니다. 스레드 안전성은 메모리뷰 자체와 내부 버퍼 내보내기에 달려 있습니다.

memoryview 구현은 free-threaded build`에서 자체 내보내기를 추적하기 위해 원자적 작업을 사용합니다. 메모리뷰를 생성하고 해제하는 것은 스레드 안전합니다. 속성 접근(예: :attr:`~memoryview.shape, format)은 메모리뷰의 수명 주기 동안 불변인 필드를 읽으므로, 메모리뷰가 해제되지 않는 한 동시 읽기는 안전합니다.

그러나 메모리뷰를 통해 액세스되는 실제 데이터는 내부 객체가 소유합니다. 이 데이터에 대한 동시 액세스는 내부 객체가 이를 지원하는 경우에만 안전합니다:

  • :class:`bytes`와 같은 불변 객체의 경우, 여러 메모리뷰를 통한 동시 읽기는 안전합니다.

  • :class:`bytearray`와 같은 가변 객체의 경우, 외부 동기화 없이 여러 스레드에서 같은 메모리 영역을 읽고 쓰는 것은 안전하지 않으며 데이터 손상(data corruption)을 초래할 수 있습니다. 읽기 전용 메모리뷰도 내부 객체가 다른 스레드에서 수정되면 데이터 경쟁(data races)을 방지하지 못하다는 점에 유의하세요.

# 안전하지 않음: 같은 버퍼에 동시 쓰기
data = bytearray(1000)
view = memoryview(data)
# 스레드 1: view[0:500] = b'x' * 500
# 스레드 2: view[0:500] = b'y' * 500

# 안전함: 동시 액세스를 위해 록 사용
import threading
lock = threading.Lock()
data = bytearray(1000)
view = memoryview(data)

with lock:
    view[0:500] = b'x' * 500

메모리뷰가 내보내지고 있는 동안 내부 객체를 크기 조정하거나 재할당하는 작업(예: bytearray.resize() 호출)은 :exc:`BufferError`를 발생시킵니다. 이는 스레딩 여부와 관계없이 적용됩니다.
