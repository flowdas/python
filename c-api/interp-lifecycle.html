인터프리터 초기화와 최종화¶

초기화 전에 인터프리터를 구성하는 방법에 대한 자세한 내용은 :ref:`Python Initialization Configuration <init-config>`를 참조하십시오.

Python 초기화 전¶

파이썬을 내장한 응용 프로그램에서는 다른 파이썬/C API 함수를 사용하기 전에 Py_Initialize() 함수를 호출해야 합니다. 몇 가지 함수와 :ref:`전역 구성 변수 <global-conf-vars>`를 제외하고는 그러합니다.

다음 함수들은 Python이 초기화되기 전에 안전하게 호출될 수 있습니다:

  • 인터프리터를 초기화하는 함수들:

    • Py_Initialize()

    • Py_InitializeEx()

    • Py_InitializeFromConfig()

    • Py_BytesMain()

    • Py_Main()

    • 참조: 파이썬 초기화 구성 에서 다루는 런타임 사전 초기화 함수들

  • 구성 함수:

    • PyImport_AppendInittab()

    • PyImport_ExtendInittab()

    • PyInitFrozenExtensions()

    • PyMem_SetAllocator()

    • PyMem_SetupDebugHooks()

    • PyObject_SetArenaAllocator()

    • Py_SetProgramName()

    • Py_SetPythonHome()

    • :ref:`init-config`에서 다루는 구성 함수들

  • 정보 제공 함수:

    • Py_IsInitialized()

    • PyMem_GetAllocator()

    • PyObject_GetArenaAllocator()

    • Py_GetBuildInfo()

    • Py_GetCompiler()

    • Py_GetCopyright()

    • Py_GetPlatform()

    • Py_GetVersion()

    • Py_IsInitialized()

  • 유틸리티:

    • Py_DecodeLocale()

    • :ref:`init-config`에서 다루는 상태 보고 및 유틸리티 함수들

  • 메모리 할당자으 :<system>

    • PyMem_RawMalloc()

    • PyMem_RawRealloc()

    • PyMem_RawCalloc()

    • PyMem_RawFree()

  • 동기화 :<system>

    • PyMutex_Lock()

    • PyMutex_Unlock()

참고

표면적으로는 위에 나열된 일부 함수들과 유사하지만, 다음 함수들은 인터프리터가 초기화되기 전에 호출해서는 안 됩니다: Py_EncodeLocale(), PyEval_InitThreads(), 및 Py_RunMain().

전역 구성 변수들¶

Python은 다양한 기능과 옵션을 제어하기 위한 전역 구성을 위한 변수를 가지고 있습니다. 기본적으로, 이 플래그들은 :ref:`명령 줄 옵션 <using-on-interface-options>`에 의해 제어됩니다.

플래그가 옵션에 의해 설정되면, 해당 플래그의 값은 옵션이 설정된 횟수입니다. 예를 들어, -b 는 Py_BytesWarningFlag 를 1로 설정하고 -bb 는 Py_BytesWarningFlag 를 2로 설정합니다.

int Py_BytesWarningFlag¶

이 API는 하위 호환성을 위해 유지됩니다. 대신, 파이썬 초기화 구성 <init-config>`을 참조하여 :c:member:`PyConfig.bytes_warning 설정이 사용되어야 합니다.

bytes 또는 :class:`bytearray`를 :class:`str`와 비교할 때 경고를 발생시키거나, :class:`bytes`를 :class:`int`와 비교할 때 경고를 발생시킵니다. 2 이상일 경우 오류를 발생시킵니다.

-b 옵션으로 설정됩니다.

버전 3.12에서 폐지되었고, 버전 3.16에서 제거됩니다.

int Py_DebugFlag¶

이 API는 하위 호환성을 위해 유지됩니다: 대신 PyConfig.parser_debug 설정을 사용하는 것이 좋습니다. 자세한 내용은 :ref:`Python Initialization Configuration <init-config>`를 참고하십시오.

파서 디버깅 출력을 <0xEC><0xBC><0xAD>니다 (전문가 전용, 컴파일 옵션에 따라 다름).

-d 옵션과 PYTHONDEBUG 환경 변수로 설정됩니다.

버전 3.12에서 폐지되었고, 버전 3.16에서 제거됩니다.

0이 아닌 값으로 설정하면, 파이썬은 소스 모듈을 임포트할 때 .pyc 파일을 쓰려고 시도하지 않습니다.

-B 옵션과 PYTHONDONTWRITEBYTECODE 환경 변수로 설정됩니다.

버전 3.12에서 폐지되었고, 버전 3.16에서 제거됩니다.

이 API는 하위 호환성을 위해 유지됩니다: 대신 PyConfig.pathconfig_warnings 설정을 사용하는 것이 좋습니다. 자세한 내용은 :ref:`Python Initialization Configuration <init-config>`를 참고하십시오.

인터프리터 초기화 및 종료¶

void Py_Initialize()¶
상의 안정 ABI.

Python 인터프리터를 초기화합니다. Python을 내장하는 애플리케이션에서는 다른 Python/C API 함수를 사용하기 전에 이것을 호출해야 합니다. 몇 가지 예외는 Before Python Initialization 를 참조하십시오.

이것은 로드된 모듈 테이블 (sys.modules)을 초기화하고 기본 모듈인 builtins, __main__ 및 sys 를 생성합니다. 또한 모듈 검색 경로 (sys.path)도 초기화합니다. sys.argv 는 설정하지 않으며, 이에는 Python Initialization Configuration API를 사용하십시오. 두 번째 호출 시 (먼저 Py_FinalizeEx() 를 호출하지 않은 경우)는 아무 작업도 하지 않습니다. 반환 값은 없으며 초기화에 실패하면 치명적인 오류입니다.

Python Initialization Configuration 를 사용자 정의하려면 Py_InitializeFromConfig() 를 사용하십시오.

참고

Windows에서 콘솔 모드를 O_TEXT 에서 O_BINARY 로 변경하며, 이는 C Runtime을 사용하는 비(非)Python용 콘솔에도 영향을 미칩니다.

void Py_InitializeEx(int initsigs)¶
상의 안정 ABI.

이 함수는 initsigs 가 1 일 경우 Py_Initialize() 와 같은 방식으로 작동합니다. 만약 initsigs 가 0 이면, 신호 핸들러의 초기화 등록을 건너뛰며, 이는 CPython이 더 큰 애플리케이션의 일부로 임베드될 때 유용할 수 있습니다.

Python Initialization Configuration 를 사용자 정의하려면 Py_InitializeFromConfig() 를 사용하십시오.

PyStatus Py_InitializeFromConfig(const PyConfig *config)¶

PyConfig를 사용한 초기화 에서 설명하는 바와 같이 config 구성에서 Python을 초기화합니다.

인터프리터를 미리 초기화하고, 실행 시간 구성 구조를 채우고, 반환된 상태 구조를 쿼리하는 방법에 대한 자세한 내용은 파이썬 초기화 구성 섹션을 참조하십시오.

int Py_IsInitialized()¶
상의 안정 ABI.

Python 인터프리터가 초기화되었으면 true (0이 아님) 를, 아니면 false (0) 를 반환합니다. Py_FinalizeEx() 가 호출된 후에는 다시 Py_Initialize() 가 호출될 때까지 이 값을 false 로 반환합니다.

버전 3.15에서 변경: 이 함수는 이제 site 모듈 임포트를 포함하여 초기화가 완전히 완료될 때만 true 를 반환합니다. 이전에는 Py_Initialize() 가 아직 실행 중인 동안 true 를 반환할 수 있었습니다.

int Py_IsFinalizing()¶
상의 안정 ABI 버전 3.13 이후로.

메인 Python 인터프리터가 종료 중이면 true (0이 아님) 를, 그렇지 않으면 false (0) 를 반환합니다.

Added in version 3.13.

int Py_FinalizeEx()¶
상의 안정 ABI 버전 3.6 이후로.

Undo all initializations made by Py_Initialize() and subsequent use of Python/C API functions, and destroy all sub-interpreters (see Py_NewInterpreter() below) that were created and not yet destroyed since the last call to Py_Initialize(). This is a no-op when called for a second time (without calling Py_Initialize() again first).

이것은 :c:func:`Py_Initialize`의 반대 작업이므로, 동일한 인터프리터가 활성화된 같은 스레드에서 호출해야 합니다. 즉, 메인 스레드와 메인 인터프리터를 의미합니다. 이는 :c:func:`Py_RunMain`이 실행되는 동안에는절대로 호출되어서는 안 됩니다.

일반적으로 반환 값은 0 입니다. 만약 최종화(버퍼링된 데이터를 플러시하는 과정) 중에 오류가 발생했다면, -1 이 반환됩니다.

Python 인터프리터가 할당한 모든 메모리를 해제하기 위해 최선을 다할 것임을 유의하십시오. 따라서, C-확장 모듈은 이후 Py_Initialize() 호출에서 사용하기 전에 이전에 할당된 모든 PyObject를 올바르게 정리해야 합니다. 그렇지 않으면 취약점과 잘못된 동작을 초래할 수 있습니다.

이 함수는 여러 가지 이유로 제공됩니다. 임베딩 애플리케이션은 자체적으로 응용 프로그램을 재시작할 필요 없이 Python을 재시작하고 싶을 수 있습니다. 동적 라이브러리(또는 DLL)에서 Python 인터프리터를 로드한 애플리케이션은 DLL을 언로드하기 전에 Python이 할당한 모든 메모리를 해제하길 원할 수도 있습니다. 또한, 개발자가 애플리케이션의 메모리 누수를 찾으면서 응용 프로그램을 종료하기 전에 Python이 할당한 모든 메모리를 해제하고 싶어 할 수도 있습니다.

버그 및 주의사항: 모듈 내 객체와 모듈의 파괴는 무작위 순서로 수행되므로, 다른 객체(심지어 함수)나 모듈에 의존하는 경우 소멸자 (__del__() 메서드)가 실패할 수 있습니다. Python에 의해 로드되는 동적 확장 모듈은 언로드되지 않습니다. Python 인터프리터가 할당한 적은 양의 메모리는 해제되지 않을 수 있습니다 (누수를 발견하면 보고하십시오). 객체 간 순환 참조로 묶인 메모리는 해제되지 않습니다. 인턴된 문자열은 참조 횟수와 관계없이 모두 할당 해제됩니다. 확장 모듈에 의해 할당된 일부 메모리가 해제되지 않을 수 있습니다. 초기화 루틴이 한 번 이상 호출되는 경우 일부 확장은 제대로 작동하지 않을 수 있습니다. 이는 애플리케이션이 Py_Initialize() 와 Py_FinalizeEx() 를 여러 번 호출할 때 발생할 수 있습니다. Py_FinalizeEx() 는 내부에서 자신으로부터 재귀적으로 호출되어서는 안 됩니다. 그러므로, atexit 핸들러, 객체 최종화기 또는 stdout 및 stderr 파일을 플러시하는 동안 실행될 수 있는 코드와 같이 인터프리터 종료 과정의 일부로 실행될 수 있는 어떤 코드에 의해서도 호출되어서는 안 됩니다.

인자 없이 감사 이벤트 cpython._PySys_ClearAuditHooks\를 발생시킵니다.

Added in version 3.6.

int Py_Main(int argc, wchar_t **argv)¶
상의 안정 ABI.

환경 및 명령줄에서 구성 설정을 읽는 기능을 구현하는 추가 동작과 전체 초기화/최종화 주기를 포함하는 표준 인터프리터의 메인 프로그램이며, 이후 명령 줄.

이것은 단순히 더 큰 응용 프로그램에 Python 런타임을 임베딩하는 것보다 완전한 CPython 명령줄 인터페이스를 지원하려는 프로그램을 위해 제공됩니다.

argc 및 argv 매개변수는 C 프로그램의 main() 함수에 전달되는 것과 유사하지만, argv 항목은 먼저 Py_DecodeLocale() 을 사용하여 wchar_t 로 변환된다는 점이 다릅니다. 또한, 인자 목록 항목이 전달된 문자열이 아닌 다른 문자열을 가리키도록 수정될 수 있다는 점에 유의하는 것이 중요합니다 (하지만 인자 목록이 가리키는 문자열의 내용은 변경되지 않습니다).

인자 목록이 유효한 Python 명령줄을 나타내지 않는 경우 반환 값은 2 이고, 그렇지 않은 경우에는 Py_RunMain() 과 동일합니다.

CPython 런타임 구성 API 중 <init-config> 섹션에 문서화된 내용(오류 처리는 제외하고)에 따르면, Py_Main 은 대략 다음 구문과 같습니다:

PyConfig config;
PyConfig_InitPythonConfig(&config);
PyConfig_SetArgv(&config, argc, argv);
Py_InitializeFromConfig(&config);
PyConfig_Clear(&config);

Py_RunMain();

일반적인 사용 사례에서는 임베딩 애플리케이션이 본 문서의 다른 부분에 설명된 대로 모든 설정을 적용하면서, Py_Initialize(), Py_InitializeEx() 또는 Py_InitializeFromConfig() 를 직접 호출하는 대신 이 함수를 호출합니다. 만약 이 함수가 선행 런타임 초기화 API 호출 이후 에 호출된다면, 어떤 환경 및 명령줄 구성 설정이 정확히 업데이트될지는 버전에 따라 다릅니다(이는 런타임이 처음 초기화되었을 때 이미 설정된 후 수정되는 것을 올바르게 지원하는 설정에 따라 달라지기 때문입니다).

int Py_RunMain(void)¶

완전히 구성된 CPython 런타임에서 메인 모듈을 실행합니다.

명령줄 또는 구성에서 지정된 명령 (PyConfig.run_command), 스크립트 (PyConfig.run_filename) 또는 모듈 (PyConfig.run_module)을 실행합니다. 이 값들 중 어느 것도 설정되지 않은 경우, __main__ 모듈의 전역 네임스페이스를 사용하여 대화형 Python 프롬프트(REPL)를 실행합니다.

PyConfig.inspect 가 설정되지 않은 경우 (기본값), 인터프리터가 정상적으로 종료되면 반환 값은 0 이며, 처리되지 않은 SystemExit 의 경우 종료 상태를 반환하거나, 기타 모든 처리되지 않은 예외의 경우 1 을 반환합니다.

PyConfig.inspect 가 설정된 경우 (예: -i 옵션을 사용하는 경우), 인터프리터가 종료될 때 반환하는 대신, 실행은 __main__ 모듈의 전역 네임스페이스를 사용하여 대화형 Python 프롬프트(REPL)에서 재개됩니다. 만약 인터프리터가 예외와 함께 종료되었다면, 이는 즉시 REPL 세션에서 발생합니다. 이후 함수 반환 값은 REPL 세션 이 종료되는 방식에 따라 결정되며, 상술한 대로 0, 1 또는 SystemExit 의 상태입니다.

이 함수는 항상 반환되기 전에 Python 인터프리터를 파이널라이즈합니다.

Py_RunMain()\를 사용하여 항상 격리 모드에서 실행되는 커스터마이징된 Python의 예는 :ref:`파이썬 구성 <init-python-config>`를 참조하십시오.

인터프리터 파이널라이제이션 관련 주의사항¶

interpreter shutdown`의 후기 단계에서, 데몬 스레드가 아닌 스레드에 대한 종료 대기(이는 :class:`KeyboardInterrupt`로 중단될 수 있음)를 시도하고 :mod:`atexit 함수가 실행된 후, 런타임은 파이널라이징 상태로 표시되어 Py_IsFinalizing() 및 :func:`sys.is_finalizing`가 true를 반환합니다. 이 지점에서는 최종화 스레드(최종화를 시작한 스레드; 일반적으로 메인 스레드임)만이 해당 스레트에 스레드 상태를 :term:`attach <attached thread state>`할 수 있습니다.

파이널라이징 중에 첨부하려는 다른 스레드는, 명시적으로 (예: PyThreadState_Ensure() 또는 Py_END_ALLOW_THREADS 를 통해) 또는 암묵적으로 (바이트코드 명령어 사이에서) 영구적으로 차단된 상태 에 들어갑니다. 일반적으로 이것은 무해하지만, 교착 상태를 야기할 수 있습니다. 예를 들어, 스레드가 락을 잡고 영구적으로 차단될 수 있으며, 이 경우 최종화 스레드는 해당 락을 절대 얻을 수 없습니다.

CPython 3.13 이전 버전에서는 스레드가 대기 상태에 머무르는 대신 종료되어 다른 문제를 야기했습니다 ( :c:func:`PyThread_exit_thread`의 경고 사항 참조).

오류인가요? 네. Python 3.15부터는 임시적으로 파이널라이제이션을 방지하여 이러한 문제를 피할 수 있게 하는 여러 C API가 존재합니다:

type PyInterpreterGuard¶
상의 안정 ABI (불투명한 구조체로) 버전 3.15 이후로.

불투명한 인터프리터 가드 구조입니다.

인터프리터 가드를 보유함으로써 호출하는 쪽은 해당 가드가 닫힐 때까지 해석기가 파이널라이징되지 않도록 보장할 수 있습니다 (:c:func:`PyInterpreterGuard_Close`를 통해).

가드가 유지되는 경우, 인터프리터를 파이널라이즈하려는 스레드는 가드가 닫힐 때까지 차단된 후에야 파이널라이징을 시작합니다. 파이널라이징이 시작된 후에는 스레드가 해당 인터프리터에 대한 가드를 영원히 획득할 수 없게 됩니다. 이는 인터프리터 가드를 닫는 것을 잊으면, 프로세스가 파이널라이징되는 동안 영구적으로 멈출 것임을 의미합니다!

인터프리터를 위한 가드를 보유하는 것은 마치 Python 객체에 대한 강한 참조 <strong reference>`를 보유하는 것과 유사하지만, 모든 가드가 해제된 후 자동으로 파이널화되지는 않습니다. 명시적인 :c:func:`Py_EndInterpreter 호출이 필요합니다.

Added in version 3.15.

PyInterpreterGuard *PyInterpreterGuard_FromCurrent(void)¶
상의 안정 ABI 버전 3.15 이후로.

현재 인터프리터의 파이널라이징 가드를 생성합니다. 이렇게 하면 가드가 닫힐 때까지 파이널라이징을 방지할 수 있습니다.

예를 들어:

// 임시로 파이널라이징을 방지합니다.
PyInterpreterGuard *guard = PyInterpreterGuard_FromCurrent();
if (guard == NULL) {
   // 이미 파이널라이징이 시작되었거나 메모리가 부족합니다.
   return NULL;
}

Py_BEGIN_ALLOW_THREADS;
// 여기에 중요한 처리를 수행합니다. 예를 들어, 파이널라이징 스레드가 얻을 수 있는 락을 안전하게 획득할 수 있습니다.
Py_END_ALLOW_THREADS;

// 이제 핵심 처리가 완료되었으므로, 인터프리터는 다시 파이널라이즈될 수 있습니다.
PyInterpreterGuard_Close(guard);

성공하면 현재 인터프리터에 대한 가드를 반환하고, 실패하면 예외가 설정된 상태로 NULL 을 반환합니다.

현재 인터프리터가 이미 파이널라이징을 시작했거나 프로세스가 메모리를 초과한 경우에만 이 함수는 실패합니다.

이 함수에서 반환된 가드 포인터는 반드시 :c:func:`PyInterpreterGuard_Close`를 사용하여 닫혀야 합니다. 그렇게 하지 않으면 Python 프로세스가 무한정 대기하게 됩니다.

호출자는 :term:`attached thread state`를 보유해야 합니다.

Added in version 3.15.

PyInterpreterGuard *PyInterpreterGuard_FromView(PyInterpreterView *view)¶
상의 안정 ABI 버전 3.15 이후로.

뷰를 통해 인터프리터에 대한 파이널라이징 가드를 생성합니다.

성공하면 이 함수는 view 가 나타내는 인터프리터에 대한 가드를 반환합니다. 뷰는 이 함수를 호출한 후에도 유효합니다. 가드는 반드시 PyInterpreterGuard_Close() 로 닫혀야 합니다.

인터프리터가 더 이상 존재하지 않거나, 이미 파이널라이징 중이거나, 메모리가 부족하면 이 함수는 예외를 설정하지 않고 NULL 을 반환합니다.

호출자가 :term:`attached thread state`를 보유할 필요가 없습니다.

Added in version 3.15.

void PyInterpreterGuard_Close(PyInterpreterGuard *guard)¶
상의 안정 ABI 버전 3.15 이후로.

인터프리터 가드를 닫아 다른 가드가 남아있지 않으면 인터프리터가 파이널라이징을 시작하도록 허용합니다. 만약 인터프리터 가드가 절대 닫히지 않으면, 인터프리터는 파이널라이징에 진입하려고 할 때 무한정 기다립니다!

인터프리터 가드가 닫힌 후에는 :c:func:`PyThreadState_Ensure`에서 사용될 수 없습니다. 그렇게 하면 정의되지 않은 동작이 발생합니다.

이 함수는 실패할 수 없고, 호출하는 쪽은 :term:`첨부된 스레드 상태 <attached thread state>`를 보유할 필요가 없습니다.

Added in version 3.15.

이것은 파이널라이징 중이거나 이미 파괴되었을 수 있는 인터프리터에 액세스하는 스레드 안전 방식입니다.

성공하면 현재 인터프리터에 대한 뷰를 반환하고, 실패하면 예외 설정과 함께 NULL 을 반환합니다.

void PyInterpreterView_Close(PyInterpreterView *view)¶

만약 인터프리터 뷰가 절대 닫히지 않으면, 뷰의 메모리는 해제되지 않지만 다른 결과는 없습니다. (반면, 가드를 닫는 것을 잊으면 메인 스레드가 파이널라이징 중 무한정 대기합니다.)

이 함수는 실패할 수 없고, 호출하는 쪽은 :term:`첨부된 스레드 상태 <attached thread state>`를 보유할 필요가 없습니다.

Added in version 3.15.

PyInterpreterView *PyInterpreterView_FromMain(void)¶
상의 안정 ABI 버전 3.15 이후로.

메인 인터프리터(Python 프로세스에서 첫 번째이자 기본 인터프리터입니다. PyInterpreterState_Main() 참조)의 뷰를 생성합니다.

성공하면 메인 인터프리터에 대한 뷰를 반환하고, 실패하면 예외 설정과 함께 NULL 을 반환합니다. 실패는 프로세스가 메모리를 초과했음을 나타냅니다.

Use this function when an interpreter pointer or view cannot be supplied by the caller, such as when a native threading library does not provide a void *arg parameter that could carry a PyInterpreterGuard or PyInterpreterView. In code that supports subinterpreters, prefer PyInterpreterView_FromCurrent() so the guard tracks the calling interpreter rather than the main one.

호출자가 :term:`attached thread state`를 보유할 필요가 없습니다.

Added in version 3.15.

void PySys_SetArgvEx(int argc, wchar_t **argv, int updatepath)¶
상의 안정 ABI.

이 API는 하위 호환성을 위해 유지 관리됩니다. 대신 PyConfig.argv, PyConfig.parse_argv 및 :c:member:`PyConfig.safe_path`를 설정해야 하며, 내용은 :ref:`파이썬 초기화 구성 <init-config>`를 참조하십시오.

argc 와 argv 를 기반으로 sys.argv 를 설정합니다. 이 매개 변수들은 프로그램의 main() 함수에 전달되는 것과 유사하지만, 첫 번째 항목은 파이썬 인터프리터를 호스팅하는 실행 파일 대신 실행될 스크립트 파일을 가리켜야 한다는 차이가 있습니다. 실행할 스크립트가 없다면, argv 의 첫 번째 항목은 빈 문자열일 수 있습니다. 이 함수가 sys.argv 를 초기화하는 데 실패하면, Py_FatalError() 를 사용하여 치명적인 조건이 신호됩니다.

updatepath 가 0이면 이것이 함수가 하는 모든 일입니다. updatepath 가 0이 아니면, 이 함수는 추가적으로 다음 알고리즘에 따라 sys.path 를 수정합니다:

  • 만약 기존 스크립트의 이름이 argv[0] 에 전달된다면, 해당 스크립트가 위치한 디렉터리의 절대 경로가 sys.path 앞에 추가됩니다.

  • 그렇지 않은 경우 (즉, argc 가 0 이거나 argv[0] 가 기존 파일 이름을 가리키지 않는 경우), 빈 문자열이 sys.path 앞에 추가되며, 이는 현재 작업 디렉터리(".")\를 추가하는 것과 같습니다.

Py_DecodeLocale() 를 사용하여 바이트 문자열을 디코딩하여 wchar_t* 문자열을 얻습니다.

또한 Python Initialization Configuration <init-config>`의 :c:member:`PyConfig.orig_argv 및 PyConfig.argv 멤버를 참조하십시오.

참고

파이썬 인터프리터를 단일 스크립트 실행 목적 이외에 포함하려는 애플리케이션에서는 updatepath 로 0 을 전달하는 것이 좋으며, 필요한 경우 sys.path 자체를 업데이트하십시오. CVE 2008-5983 \를 참조하십시오.

3.1.3 이전 버전에서는 다음과 같이 PySys_SetArgv() 를 호출한 후 첫 번째 sys.path 요소를 수동으로 제거하여 같은 효과를 얻을 수 있습니다:

PyRun_SimpleString("import sys; sys.path.pop(0)\\n");

Added in version 3.1.3.

버전 3.11에서 폐지되었고, 버전 3.16에서 제거됩니다.
