파이썬 프로세스의 다중 인터프리터¶

대부분의 사용 사례에서는 단일 Python 인터프리터만 임베드하지만, 동일한 프로세스, 어쩌면 동일한 스레드에 여러 독립적인 인터프리터를 생성해야 하는 경우도 있습니다. 서브인터프리터가 이를 가능하게 합니다.

“main” 인터프리터는 런타임이 초기화될 때 생성되는 첫 번째 인터프리터입니다. 보통 프로세스 내에 유일한 Python 인터프리터입니다. 서브인터프리터와는 달리, 메인 인터프리터는 시그널 처리와 같은 고유한 프로세스 전역 책임을 가집니다. 또한 런타임 초기화 중 실행에도 책임이 있으며, 런타임 마무리 중에는 보통 활성 인터프리터입니다. PyInterpreterState_Main() 함수는 해당 상태에 대한 포인터를 반환합니다.

PyThreadState_Swap() 함수를 사용하여 서브인터프리터 간에 전환할 수 있습니다. 다음 함수들을 사용하여 생성하고 파괴할 수 있습니다:

type PyInterpreterConfig¶

서브 인터프리터를 구성하는 데 사용되는 대부분의 매개변수를 포함하는 구조체입니다. 이 값은 :c:func:`Py_NewInterpreterFromConfig`에서만 사용되며 런타임에 수정되지 않습니다.

Added in version 3.12.

구조체 필드는 다음과 같습니다:

int use_main_obmalloc¶

이 값이 0 이면 서브 인터프리터가 자체 “객체” 할당자 상태를 사용합니다. 그렇지 않으면 메인 인터프리터의 것을 (공유) 사용합니다.

이 값이 0 이면 check_multi_interp_extensions 는 반드시 1 (0이 아님)이어야 합니다. 이 값이 1 이면 gil 은 PyInterpreterConfig_OWN_GIL 이 아니어야 합니다.

int allow_fork¶

이 값이 0 이면 런타임은 서브 인터프리터가 현재 활성화된 모든 스레드에서 프로세스를 포크하는 것을 지원하지 않습니다. 그렇지 않으면 포크는 제한이 없습니다.

subprocess 모듈은 포크가 허용되지 않을 때도 계속 작동한다는 점에 유의하십시오.

int allow_exec¶

이 값이 0 이면 런타임은 서브 인터프리터가 현재 활성화된 모든 스레드에서 exec을 통해 현재 프로세스를 대체하는 것을 지원하지 않습니다 (예: os.execv()). 그렇지 않으면 exec은 제한이 없습니다.

subprocess 모듈은 exec이 허용되지 않을 때도 계속 작동한다는 점에 유의하십시오.

int allow_threads¶

이 값이 0 이면 서브 인터프리터의 threading 모듈은 스레드를 생성하지 않습니다. 그렇지 않으면 스레드는 허용됩니다.

int allow_daemon_threads¶

이 값이 0 이면 서브 인터프리터의 threading 모듈은 데몬 스레드를 생성하지 않습니다. 그렇지 않으면 데몬 스레드가 허용됩니다 (단, allow_threads 가 0이 아닌 경우).

int check_multi_interp_extensions¶

이 값이 0 이면 서브 인터프리터가 현재 활성화된 모든 스레드에서 레거시 (단일 단계 초기화) 모듈을 포함한 모든 확장 모듈을 가져올 수 있습니다. 그렇지 않으면 다단계 초기화 확장 모듈 (참조: PEP 489)만 가져올 수 있습니다. (또한 Py_mod_multiple_interpreters 도 참조하십시오.)

이는 use_main_obmalloc 가 0 일 경우 1 (0이 아님)이어야 합니다.

int gil¶

이것은 서브 인터프리터의 GIL 작동 방식을 결정합니다. 다음 중 하나일 수 있습니다:

PyInterpreterConfig_DEFAULT_GIL¶

기본 선택값(PyInterpreterConfig_SHARED_GIL)을 사용합니다.

PyInterpreterConfig_SHARED_GIL¶

메인 인터프리터의 GIL을 (공유) 사용합니다.

PyInterpreterConfig_OWN_GIL¶

서브 인터프리터 자신의 GIL을 사용합니다.

이것이 PyInterpreterConfig_OWN_GIL 인 경우 PyInterpreterConfig.use_main_obmalloc 는 0 이어야 합니다.

PyStatus Py_NewInterpreterFromConfig(PyThreadState **tstate_p, const PyInterpreterConfig *config)¶

새로운 서브 인터프리터를 생성합니다. 이것은 파이썬 코드 실행을 위한 (거의) 완전히 별개의 환경입니다. 특히, 새로운 인터프리터는 기본 모듈인 builtins, __main__ 및 sys 를 포함하여 모든 가져온 모듈의 독립적이고 별개의 버전을 가집니다. 로드된 모듈의 목록 (sys.modules)과 모듈 검색 경로 (sys.path) 또한 별개입니다. 새로운 환경에는 sys.argv 변수가 없습니다. 새로운 표준 I/O 스트림 파일 객체인 sys.stdin, sys.stdout 및 sys.stderr 를 가집니다 (하지만 이들은 동일한 기본 파일 디스크립터를 참조합니다).

주어진 config 는 인터프리터가 초기화되는 옵션을 제어합니다.

Upon success, tstate_p will be set to the first thread state created in the new sub-interpreter. This thread state is attached. Note that no actual thread is created; see the discussion of thread states below. If creation of the new interpreter is unsuccessful, tstate_p is set to NULL; no exception is set since the exception state is stored in the attached thread state, which might not exist.

모든 다른 Python/C API 함수와 마찬가지로, 이 함수를 호출하기 전에 :term:`attached thread state`가 존재해야 하지만, 반환 시에는 분리될 수 있습니다. 성공하는 경우, 반환된 스레드 상태는 :term:`attached <attached thread state>`가 됩니다. 서브 인터프리터가 자체 :term:`GIL`을 가지고 생성된 경우, 호출 인터프리터의 :term:`attached thread state`는 분리됩니다. 함수가 반환되면, 새 인터프리터의 :term:`thread state`는 현재 스레드에 대해 :term:`attached <attached thread state>`가 될 것이고, 이전 인터프리터의 :term:`attached thread state`는 분리된 상태로 유지됩니다.

Added in version 3.12.

서브 인터프리터는 특정 기능이 제한되어 서로 격리되어 있을 때 가장 효과적입니다:

PyInterpreterConfig config = {
    .use_main_obmalloc = 0,
    .allow_fork = 0,
    .allow_exec = 0,
    .allow_threads = 1,
    .allow_daemon_threads = 0,
    .check_multi_interp_extensions = 1,
    .gil = PyInterpreterConfig_OWN_GIL,
};
PyThreadState *tstate = NULL;
PyStatus status = Py_NewInterpreterFromConfig(&tstate, &config);
if (PyStatus_Exception(status)) {
    Py_ExitStatusException(status);
}

이 구성은 잠시 사용될 뿐 수정되지 않음에 유의하십시오. 초기화 중에 구성의 값들은 다양한 PyInterpreterState 값으로 변환됩니다. 구성의 읽기 전용 사본이 :c:type:`PyInterpreterState`에 내부적으로 저장될 수 있습니다.

확장 모듈은 다음과 같이 (서브-)인터프리터 간에 공유됩니다:

  • 다단계 초기화를 사용하는 모듈의 경우, 예를 들어 :c:func:`PyModule_FromDefAndSpec`을 사용하면 각 인터프리터에 대해 별도의 모듈 객체가 생성되고 초기화됩니다. C 수준의 정적 및 전역 변수만 이러한 모듈 객체들 사이에서 공유됩니다.

  • 레거시 single-phase initialization <single-phase-initialization>`을 사용하는 모듈의 경우, 예를 들어 :c:func:`PyModule_Create`를 사용하면 특정 확장이 처음 임포트될 때 정상적으로 초기화되고, 모듈 딕셔너리의 (얕은) 사본이 저장됩니다. 같은 확장이 다른 (서브-)인터프리터에 의해 임포트될 때, 새 모듈이 초기화되고 이 복사본의 내용으로 채워지며; 확장 자체의 ``init` 함수는 호출되지 않습니다. 따라서 모듈 딕셔너리의 객체들은 (서브-)인터프리터 전반에 걸쳐 공유되어 원치 않는 동작을 유발할 수 있습니다 (아래 Bugs and caveats 참조).

    이는 인터프리터가 Py_FinalizeEx() 와 Py_Initialize() 를 호출하여 완전히 재초기화된 후에 확장이 임포트되는 경우와는 다릅니다. 그 경우에는 확장 자체의 initmodule 함수가 다시 호출됩니다. 다단계 초기화의 경우와 마찬가지로, 이는 이러한 모듈들 사이에서 C 수준의 정적 및 전역 변수만 공유됨을 의미합니다.

PyThreadState *Py_NewInterpreter(void)¶
상의 안정 ABI.

새 서브 인터프리터를 생성합니다. 이것은 본질적으로 기존 동작을 보존하는 config를 가진 Py_NewInterpreterFromConfig() 주위에 래퍼를 만든 것과 같습니다. 그 결과는 메인 인터프리터의 GIL을 공유하고, fork/exec를 허용하며, 데몬 스레드를 허용하고, 단일 단계 초기화 모듈을 허용하는 비격리형 서브 인터프리터가 됩니다.

void Py_EndInterpreter(PyThreadState *tstate)¶
상의 안정 ABI.

주어진 :term:`thread state`가 나타내는 (서브-)인터프리터를 파괴합니다. 주어진 스레드 상태는 :term:`attached <attached thread state>`여야 합니다. 호출이 반환되면, :term:`attached thread state`는 존재하지 않게 됩니다. 이 인터프리터와 관련된 모든 스레드 상태가 파괴됩니다.

:c:func:`Py_FinalizeEx`는 해당 시점에 명시적으로 파괴되지 않은 모든 서브 인터프리터를 파괴합니다.

인터프리터별 GIL¶

Added in version 3.12.

Py_NewInterpreterFromConfig() 를 사용하여 자체 GIL을 포함하여 다른 인터프리터와 완전히 격리된 서브 인터프리터를 생성할 수 있습니다. 이 격리의 가장 중요한 이점은 그러한 인터프리터가 다른 인터프리터에 의해 차단되지 않고, 또는 다른 인터프리터를 차단하지 않으면서 Python 코드를 실행할 수 있다는 점입니다. 따라서 단일 Python 프로세스는 Python 코드를 실행할 때 여러 CPU 코어를 진정으로 활용할 수 있습니다. 또한 이러한 격리는 단순히 스레드를 사용하는 것보다 다른 동시성 접근 방식을 장려합니다. ( PEP 554 및 PEP 684 참조.)

고립된 인터프리터를 사용하려면 격리를 유지하는 데 주의가 필요합니다. 특히, 스레드 안전성에 대한 보장 없이 어떤 객체나 변경 가능한 상태도 공유하지 않는 것이 중요합니다. 일반적으로 불변하는 객체(예: None, (1, 5))라 할지라도 참조 카운트 때문에 공유할 수 없습니다. 이를 해결하는 간단하지만 덜 효율적인 방법 중 하나는 모든 상태(또는 객체) 사용에 글로벌 락을 두는 것입니다. 대안으로, 효과적으로 불변 객체(정수나 문자열 같은)는 참조 카운트에도 불구하고, 이들을 불멸적인 방식으로 만들어서 안전하게 처리할 수 있습니다. 실제로, 이 방식은 내장 싱글턴, 작은 정수, 그리고 다수의 다른 내장 객체에 대해 이미 수행되었습니다.

격리를 유지하면 자유 스레딩이 수반하는 복잡성 없이 적절한 멀티 코어 컴퓨팅을 활용할 수 있습니다. 격리를 유지하지 못하면 경합 상태(races)와 디버깅하기 어려운 충돌을 포함하여 자유 스레딩의 전체 결과를 노출하게 됩니다.

그 외에도, 여러 격리된 인터프리터를 사용하는 주요 과제 중 하나는 어떻게 안전하게 (격리를 깨지 않고) 그리고 효율적으로 그들 간에 통신할 것인지입니다. 런타임과 stdlib에는 아직 표준적인 접근 방식이 제공되지 않습니다. 미래의 stdlib 모듈은 격리 상태 유지의 노력을 완화하고 인터프리터 간의 데이터 통신(및 공유)을 위한 효과적인 도구를 노출하는 데 도움이 될 것입니다.

버그 및 주의사항¶

서브 인터프리터(및 메인 인터프리터)는 같은 프로세스의 일부이기 때문에, 그들 간의 절연이 완벽하지 않습니다 — 예를 들어, os.close()\와 같은 저수준 파일 연산을 사용하면 서로의 열린 파일에 (실수로 혹은 악의적으로) 영향을 미칠 수 있습니다. (서브) 인터프리터 간에 확장이 공유되는 방식 때문에, 일부 확장이 제대로 작동하지 않을 수 있습니다; 이것은 특히 단단계 초기화나 (정적) 전역 변수를 사용할 때 특히 그렇습니다. 한 서브 인터프리터에서 생성된 객체를 다른 (서브) 인터프리터의 이름 공간에 삽입할 수 있습니다; 가능하면 피해야 합니다.

사용자 정의 함수, 메서드, 인스턴스 또는 클래스를 서브 인터프리터 간에 공유하는 것을 방지하기 위한 특별한 주의가 필요합니다. 이러한 객체에 의해 실행되는 임포트 작업은 잘못된 (서브) 인터프리터의 로드된 모듈 딕셔너리에 영향을 줄 수 있기 때문입니다. 위 사항들이 도달할 수 있는 객체를 공유하는 것도 똑같이 중요합니다.

또한, 이 기능을 PyGILState_* API와 결합하는 것은 섬세합니다. 왜냐하면 이 API들은 파이썬 스레드 상태와 OS 수준 스레드 간의 일대일 대응(bijection)을 가정하는데, 서브 인터프리터의 존재로 인해 그 가정이 깨지기 때문입니다. 따라서 일치하는 PyGILState_Ensure() 와 PyGILState_Release() 호출 사이에서 서브 인터프리터를 전환하지 않는 것이 강력히 권장됩니다. 게다가, 이 API를 사용하여 비-파이썬 생성 스레드에서 파이썬 코드 호출을 허용하는 확장(예: ctypes)은 서브 인터프리터를 사용할 때 작동하지 않을 가능성이 높습니다.

고수준 API¶

type PyInterpreterState¶
상의 안정 ABI (불투명한 구조체로).

이 자료 구조는 여러 협력 스레드에 의해 공유되는 상태를 나타냅니다. 동일한 인터프리터에 속하는 스레드들은 모듈 관리와 몇 가지 다른 내부 항목을 공유합니다. 이 구조체에는 공개 멤버가 없습니다.

서로 다른 인터프리터에 속하는 스레드들은 처음에는 이용 가능한 메모리, 열린 파일 디스크립터 등과 같은 프로세스 상태를 제외하고는 아무것도 공유하지 않습니다. 글로벌 인터프리터 록 역시 어떤 인터프리터에 속하든 모든 스레드에 의해 공유됩니다.

버전 3.12에서 변경: PEP 684 는 인터프리터별 GIL 가능성을 도입했습니다. Py_NewInterpreterFromConfig() 를 참조하십시오.

PyObject *PyInterpreterState_GetDict(PyInterpreterState *interp)¶
반환값: 빌린 참조. 상의 안정 ABI 버전 3.8 이후로.

인터프리터별 데이터를 저장할 수 있는 딕셔너리를 반환합니다. 이 함수가 NULL 을 반환하면 예외가 발생하지 않았으며, 호출자는 인터프리터별 딕셔너리가 사용 가능하지 않다고 가정해야 합니다.

이것은 확장 기능이 인터프리터별 상태 정보를 저장하는 데 사용해야 하는 `:c:func:`PyModule_GetState()`를 대체하는 것이 아닙니다.

반환되는 딕셔너리는 인터프리터에서 가져온 것이며, 인터프리터가 종료될 때까지 유효합니다.

Added in version 3.8.

typedef PyObject *(*_PyFrameEvalFunction)(PyThreadState *tstate, _PyInterpreterFrame *frame, int throwflag)¶

void _PyInterpreterState_SetEvalFrameAllowSpecialization(PyInterpreterState *interp, int allow_specialization)¶

사용자 정의 프레임 평가기(evaluator)가 설정되어 있는지 여부를 활성화하거나 비활성화합니다.

allow_specialization 이 0이 아니면, 사용자 정의 eval 프레임 함수가 설정되어 있더라도 적응형 특수화기(adaptive specializer)는 바이트코드를 계속 특수화합니다. allow_specialization 이 0이면, 사용자 정의 eval 프레임 함수를 설정해도 특수화는 비활성화됩니다. 표준 인터프리터 루프는 프레임 평가 API가 있는 동안 계속해서 deopt할 것이므로, 프레임 평가 함수는 특수화된 opcode를 처리해야 이점을 얻을 수 있습니다.

Added in version 3.15.

int _PyInterpreterState_IsSpecializationEnabled(PyInterpreterState *interp)¶

인터프리터에 대해 적응형 특수화가 활성화되어 있으면 0이 아닌 값을 반환합니다. 사용자 정의 eval 프레임 함수가 설정되지 않았거나 allow_specialization 이 활성화된 경우 특수화가 활성화됩니다.

Added in version 3.15.

상의 안정 ABI.

새로운 인터프리터 상태 객체를 생성합니다. :term:`attached thread state`는 필요하지 않지만, 이 함수에 대한 호출을 직렬화해야 하는 경우 선택적으로 존재할 수 있습니다.

인자 없이 auditing event cpython.PyInterpreterState_New 를 발생시킵니다.

void PyInterpreterState_Clear(PyInterpreterState *interp)¶

인터프리터 상태 객체의 모든 정보를 재설정합니다. 인터프리터에는 반드시 :term:`attached thread state`가 있어야 합니다.

인자 없이 auditing event cpython.PyInterpreterState_Clear 를 발생시킵니다.

void PyInterpreterState_Delete(PyInterpreterState *interp)¶

인터프리터 상태 객체를 파괴합니다. 대상 인터프리터에 attached thread state`가 **있어서는 안 됩니다**. 인터프리터 상태는 :c:func:`PyInterpreterState_Clear 호출로 이전에 재설정되었어야 합니다.
