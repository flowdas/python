동기화 프리미티브¶

C-API는 기본적인 뮤추얼 익스클루전 잠금(mutual exclusion lock)을 제공합니다.

type PyMutex¶

상호 배제 잠금장치입니다. PyMutex\는 잠금 해제 상태를 나타내기 위해 0으로 초기화해야 합니다. 예를 들면

PyMutex mutex = {0};

PyMutex`의 인스턴스는 복사하거나 이동해서는 안 됩니다. :c:type:!PyMutex`의 내용과 주소 모두 의미가 있으며, 메모리 내에 고정되고 쓰기 가능한 위치에 유지되어야 합니다.

참고

:c:type:`!PyMutex`는 현재 1바이트를 차지하지만, 크기는 불안정하다고 간주되어야 합니다. 이 크기는 향후 파이썬 릴리스에서 비지원 기간 없이 변경될 수 있습니다.

Added in version 3.13.

void PyMutex_Lock(PyMutex *m)¶
Thread safety: Atomic.

뮤텍스 m 을 잠급니다. 다른 스레드가 이미 록을 잡고 있다면, 호출 스레드는 뮤텍스가 언록될 때까지 차단됩니다. 차단된 동안, 스레드는 존재할 경우 thread state 를 일시적으로 분리합니다.

Added in version 3.13.

void PyMutex_Unlock(PyMutex *m)¶
Thread safety: Atomic.

뮤텍스 m 을 해제합니다. 뮤텍스는 잠겨 있어야 하며, 그렇지 않으면 함수가 치명적인 오류를 발생시킵니다.

Added in version 3.13.

int PyMutex_IsLocked(PyMutex *m)¶
Thread safety: Atomic.

뮤텍스 m 이 현재 잠겨 있으면 0이 아닌 값을 반환하고, 그렇지 않으면 0을 반환합니다.

참고

이 함수는 어설션 및 디버깅 목적으로만 사용되는 것이며, 락 상태는 검사 직후 변경될 수 있으므로 동시성 제어 결정을 내리는 데 사용해서는 안 됩니다.

Added in version 3.14.

Python 임계 영역 API¶

임계 영역 API는 free-threaded CPython용 객체별 록 위에 데드락 방지 계층을 제공합니다. 이는 :term:`global interpreter lock`에 의존하는 것을 대체하는 것이 목적이며, 전역 인터프리터 록이 있는 버전에서는 no-op입니다.

임계 영역은 C-API 확장으로 구현된 사용자 정의 유형에 사용되도록 의도되었습니다. 이들은 일반적으로 :class:`list`와 :class:`dict`와 같은 내장 유형과 함께 사용해서는 안 됩니다. 왜냐하면 이들의 공개 C-API가 이미 내부적으로 임계 영역을 사용하기 때문이며, 주목할 만한 예외는 외부에서 임계 영역을 획득해야 하는 :c:func:`PyDict_Next`입니다.

임계 영역은 활성 임계 영역을 암묵적으로 중지시킴으로써 데드락을 방지하므로, :c:type:`PyMutex`와 같은 전통적인 록이 제공하는 독점 액세스를 제공하지 않습니다. 임계 영역이 시작되면 해당 객체의 개별 객체 록이 획득됩니다. 만약 임계 영역 내부에서 실행된 코드가 C-API 함수를 호출하면 임계 영역을 중지시켜 개별 객체 록을 해제하고 다른 스레드가 같은 객체에 대해 개별 객체 록을 획득할 수 있게 합니다.

Python 객체 대신 PyMutex 포인터를 받아들이는 변형도 사용할 수 있습니다. 이러한 변형을 사용하여 :c:type:`PyObject`가 없는 상황에서 임계 영역을 시작합니다. 예를 들어, :c:type:`PyObject`를 확장하거나 래핑하지 않지만 데드락을 유발할 수 있는 방식으로 C API를 호출해야 하는 C 유형을 다룰 때와 같습니다.

참고

두 객체를 동시에 잠가야 하는 작업은 :c:macro:`Py_BEGIN_CRITICAL_SECTION2`를 사용해야 합니다. 중첩된 임계 영역을 사용하여 한 번에 두 개 이상의 객체를 잠글 수 없습니다. 직사각형의 임계 영역은 외부 임계 영역을 중지시킬 수 있기 때문입니다. 이 API는 한 번에 두 개 이상의 객체를 잠글 수 있는 방법을 제공하지 않습니다.

사용 예:

static PyObject *
set_field(MyObject *self, PyObject *value)
{
   Py_BEGIN_CRITICAL_SECTION(self);
   Py_SETREF(self->field, Py_XNewRef(value));
   Py_END_CRITICAL_SECTION();
   Py_RETURN_NONE;
}

위 예제에서, :c:macro:`Py_SETREF`는 :c:macro:`Py_DECREF`를 호출하며, 이는 객체의 할당 해제 함수를 통해 임의의 코드를 호출할 수 있습니다. 임계 영역 API는 재진입 및 록 순서에 따른 잠재적 데드락을 방지하기 위해, finalizer가 중단되고 :c:func:`PyEval_SaveThread`를 호출하는 경우 런타임이 임계 영역을 일시 중지할 수 있도록 허용합니다.

Py_BEGIN_CRITICAL_SECTION(op)¶
상의 안정 ABI 버전 3.15 이후로.

객체 op 에 대한 개체별 록을 획득하고 임계 영역을 시작합니다.

free-threaded 빌드와 :ref:`Stable ABI <stable-abi>`를 위해 빌드할 때, 이 매크로는 다음과 같이 확장됩니다:

{
    PyCriticalSection _py_cs;
    PyCriticalSection_Begin(&_py_cs, (PyObject*)(op))

기본 빌드에서는 이 매크로가 { 로 확장됩니다.

Added in version 3.13.

Py_BEGIN_CRITICAL_SECTION_MUTEX(m)¶

뮤텍스 m 을 잠그고 임계 영역을 시작합니다.

free-threaded 빌드에서, 이 매크로는 다음으로 확장됩니다:

{
     PyCriticalSection2 _py_cs2;
     PyCriticalSection2_BeginMutex(&_py_cs2, m1, m2)

Py_BEGIN_CRITICAL_SECTION 과 달리 매크로의 인수에 대해 캐스팅이 없다는 점에 유의하십시오. 이는 PyMutex 포인터여야 합니다.

기본 빌드에서는 이 매크로가 { 로 확장됩니다.

Added in version 3.14.

Py_END_CRITICAL_SECTION()¶
상의 안정 ABI 버전 3.15 이후로.

임계 영역을 종료하고 개체별 록을 해제합니다.

free-threaded 빌드와 :ref:`Stable ABI <stable-abi>`를 위해 빌드할 때, 이 매크로는 다음과 같이 확장됩니다:

PyCriticalSection_End(&_py_cs);
}

기본 빌드에서는 이 매크로가 } 로 확장됩니다.

Added in version 3.13.

Py_BEGIN_CRITICAL_SECTION2(a, b)¶
상의 안정 ABI 버전 3.15 이후로.

객체 a 와 b 에 대한 개체별 록을 획득하고 임계 영역을 시작합니다. 록은 잠금 순서 데드락을 방지하기 위해 일관된 순서(가장 낮은 주소 순)로 획득됩니다.

free-threaded 빌드에서, 이 매크로는 다음으로 확장됩니다:

{
    PyCriticalSection2 _py_cs2;
    PyCriticalSection2_Begin(&_py_cs2, (PyObject*)(a), (PyObject*)(b))

기본 빌드에서는 이 매크로가 { 로 확장됩니다.

Added in version 3.13.

Py_BEGIN_CRITICAL_SECTION2_MUTEX(m1, m2)¶

뮤텍스 m1 와 m2 를 잠그고 임계 영역을 시작합니다.

free-threaded 빌드와 :ref:`Stable ABI <stable-abi>`를 위해 빌드할 때, 이 매크로는 다음과 같이 확장됩니다:

{
     PyCriticalSection2 _py_cs2;
     PyCriticalSection2_BeginMutex(&_py_cs2, m1, m2)

Py_BEGIN_CRITICAL_SECTION2 와 달리 매크로의 인수에 대해 캐스팅이 없다는 점에 유의하십시오. 이는 각각 PyMutex 포인터여야 합니다.

기본 빌드에서는 이 매크로가 { 로 확장됩니다.

Added in version 3.14.

Py_END_CRITICAL_SECTION2()¶
상의 안정 ABI 버전 3.15 이후로.

임계 영역을 종료하고 개체별 록을 해제합니다.

free-threaded 빌드와 :ref:`Stable ABI <stable-abi>`를 위해 빌드할 때, 이 매크로는 다음과 같이 확장됩니다:

PyCriticalSection2_End(&_py_cs2);
}

기본 빌드에서는 이 매크로가 } 로 확장됩니다.

Added in version 3.13.

레거시 잠금 API¶

이 API들은 PyMutex 도입으로 Python 3.13부터 사용되지 않습니다.

버전 3.15에서 변경: 이 API들은 이제 PyMutex 를 둘러싼 간단한 래퍼입니다.

type PyThread_type_lock¶

상호 배제 록에 대한 포인터입니다.

type PyLockStatus¶

시간 초과를 포함하여 록을 획득한 결과입니다.

enumerator PY_LOCK_FAILURE¶

록을 획득하는 데 실패했습니다.

enumerator PY_LOCK_ACQUIRED¶

록을 성공적으로 획득했습니다.

enumerator PY_LOCK_INTR¶

록이 신호에 의해 중단되었습니다.

PyThread_type_lock PyThread_allocate_lock(void)¶
상의 안정 ABI.

새로운 록을 할당합니다.

성공 시 이 함수는 록을 반환하고, 실패 시 예외 없이 0 을 반환합니다.

호출하는 쪽은 :term:`첨부된 스레드 상태 <attached thread state>`를 보유할 필요가 없습니다.

버전 3.15에서 변경: 이 함수는 이제 항상 :c:type:`PyMutex`를 사용합니다. 이전 버전에서는 이 운영 체제에서 제공하는 록을 사용했습니다.

void PyThread_free_lock(PyThread_type_lock lock)¶
상의 안정 ABI.

lock 을 파괴합니다. 이 함수를 호출할 때 록은 어떤 스레드에 의해서도 보유되어서는 안 됩니다.

호출하는 쪽은 :term:`첨부된 스레드 상태 <attached thread state>`를 보유할 필요가 없습니다.

PyLockStatus PyThread_acquire_lock_timed(PyThread_type_lock lock, long long microseconds, int intr_flag)¶
상의 안정 ABI.

시간 초과를 사용하여 lock 을 획득합니다.

이것은 microseconds 마이크로초 동안 록을 획득하기 위해 대기합니다. 시간 초과가 지나면, 이 함수는 PY_LOCK_FAILURE 를 반환합니다. microseconds 가 -1 이면, 록이 해제될 때까지 무기한 대기합니다.

intr_flag 가 1 이면 록을 획득하는 과정이 신호에 의해 중단될 수 있으며, 이 경우 이 함수는 PY_LOCK_INTR 을 반환합니다. 중단된 경우, 일반적으로 호출자는 예외를 Python 코드로 전파하기 위해 Py_MakePendingCalls() 를 호출하는 것이 예상됩니다.

록을 성공적으로 획득한 경우, 이 함수는 :c:enumerator:`PY_LOCK_ACQUIRED`를 반환합니다.

호출하는 쪽은 :term:`첨부된 스레드 상태 <attached thread state>`를 보유할 필요가 없습니다.

int PyThread_acquire_lock(PyThread_type_lock lock, int waitflag)¶
상의 안정 ABI.

lock 획득.

waitflag 가 1 이고 다른 스레드가 현재 록을 보유하고 있는 경우, 이 함수는 록을 획득할 수 있을 때까지 대기하며 항상 1 을 반환합니다.

waitflag 가 0 이고 다른 스레드가 록을 보유하고 있는 경우, 이 함수는 대기하지 않고 대신 0 을 반환합니다. 록을 다른 스레드가 보유하고 있지 않은 경우, 이 함수는 록을 획득하고 1 을 반환합니다.

:c:func:`PyThread_acquire_lock_timed`과는 달리, 록을 획득하는 과정은 신호에 의해 중단될 수 없습니다.

호출하는 쪽은 :term:`첨부된 스레드 상태 <attached thread state>`를 보유할 필요가 없습니다.

int PyThread_release_lock(PyThread_type_lock lock)¶
상의 안정 ABI.

lock 해제. lock 이 보유되지 않은 경우, 이 함수는 치명적인 오류를 발생시킵니다.

호출하는 쪽은 :term:`첨부된 스레드 상태 <attached thread state>`를 보유할 필요가 없습니다.
