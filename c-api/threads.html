스레드 상태와 전역 인터프리터 록¶

CPython\의 전역 인터프리터 록 와 같이, 파이썬 인터프리터는 일반적으로 스레드 안전하지 않습니다. 다중 스레드 파이썬 프로그램을 지원하기 위해, 파이썬 객체에 액세스하기 전에 스레드가 보유해야 하는 전역 잠금인 전역 인터프리터 록 또는 :term:`GIL`가 있습니다. 이 잠금이 없으면, 가장 간단한 작업조차도 다중 스레드 프로그램에서 문제를 일으킬 수 있습니다. 예를 들어, 두 스레드가 동시에 같은 객체의 참조 카운트를 증가시킬 경우, 참조 카운트가 두 번이 아닌 한 번만 증가할 수 있습니다.

따라서, GIL을 보유한 스레드만이 파이썬 객체에서 작동하거나 파이썬/C API 함수를 호출할 수 있습니다.

동시성을 에뮬레이션하기 위해 인터프리터는 바이트 코드 명령어들 사이에서 정기적으로 스레드 전환을 시도합니다 ( sys.setswitchinterval()\ 참조). 이것이 순수 파이썬 코드에서도 스레드 안전성을 위해 락이 필요한 이유입니다.

게다가, 전역 인터프리터 록은 파일 읽기 또는 쓰기와 같은 블로킹 I/O 작업 주변에서 해제됩니다. C API에서는 스레드 상태 분리\을 통해 이 작업을 수행합니다.

파이썬 인터프리터는 PyThreadState\이라는 데이터 구조 내에 일부 스레드 지역 정보를 유지하며, 이를 스레드 상태 <thread state>`라고 합니다. 각 스레드는 :c:type:`PyThreadState\를 가리키는 스레드 지역 포인터를 가지며, 이 포인터로 참조된 스레드 상태는 :term:`첨부됨 <attached thread state>`으로 간주됩니다.

스레드는 한 번에 하나의 :term:`첨부된 스레드 상태`만 가질 수 있습니다. 첨부된 스레드 상태는 일반적으로, 무료-쓰레드 빌드의 경우를 제외하고 GIL을 보유하는 것과 유사합니다. GIL이 활성화된 빌드의 경우, 스레드 상태를 첨부하면 GIL을 획득할 때까지 블록됩니다. 그러나 GIL이 비활성화된 빌드에서도 인터프리터가 어떤 스레드가 파이썬 객체에 액세스할 수 있는지 추적해야 하므로, 여전히 첨부된 스레드 상태를 가질 필요가 있습니다.

참고

프리 스레드 빌드에서도 쓰레드를 연결하면 차단될 수 있습니다. GIL이 다시 활성화되거나 가비지 컬렉션 중에 스레드가 일시적으로 중단되는 등의 이유가 있기 때문입니다.

일반적으로 파이썬 C API를 사용할 때, 임베딩 및 메서드 구현 중에도 항상 연결된 쓰레드 상태가 존재하므로 자체적으로 쓰레드 상태를 설정할 필요는 흔치 않습니다. Py_BEGIN_ALLOW_THREADS 블록 내부 또는 새 스레드의 경우에는 쓰레드가 연결된 쓰레드 상태를 갖지 못합니다. 확실하지 않다면, PyThreadState_GetUnchecked() 가 NULL 을 반환하는지 확인하십시오.

만약 쓰레드 상태를 생성해야 하는 경우, PyThreadState_Ensure() 또는 PyThreadState_EnsureFromView() 사용을 권장합니다. 이 함수들은 쓰레드 상태 관리를 대신 수행해 줍니다.

확장 코드에서 쓰레드 상태 연결 해제¶

Most extension code manipulating the thread state has the following simple structure:

로컬 변수에 쓰레드 상태를 저장합니다.
... 블로킹 I/O 작업을 수행합니다 ...
로컬 변수로부터 쓰레드 상태를 복원합니다.

이것은 매우 일반적인 작업이라서 이를 단순화하기 위한 매크로 쌍이 존재합니다:

Py_BEGIN_ALLOW_THREADS
... 블로킹 I/O 작업을 수행합니다 ...
Py_END_ALLOW_THREADS

Py_BEGIN_ALLOW_THREADS 매크로는 새로운 블록을 열고 숨겨진 지역 변수를 선언하며, Py_END_ALLOW_THREADS 매크로가 이 블록을 닫습니다.

위의 블록은 다음 코드로 확장됩니다:

PyThreadState *_save;

_save = PyEval_SaveThread();
... 블로킹 I/O 작업을 수행합니다 ...
PyEval_RestoreThread(_save);

이 함수들이 어떻게 작동하는지 알려 드립니다:

연결된 쓰레드 상태는 인터프리터에 GIL이 유지됨을 의미합니다. 이를 연결 해제하려면 :c:func:`PyEval_SaveThread`가 호출되고 결과가 로컬 변수에 저장됩니다.

쓰레드 상태를 분리함에 따라 GIL이 해제되어, 현재 스레드가 블로킹 I/O 작업을 수행하는 동안 다른 스레드들이 인터프리터에 연결하여 실행할 수 있게 됩니다. I/O 작업이 완료되면, :c:func:`PyEval_RestoreThread`를 호출하여 이전 쓰레드 상태가 다시 연결되며, 이는 GIL을 획득할 때까지 기다립니다.

참고

블로킹 I/O를 수행하는 것이 쓰레드 상태 분리의 가장 흔한 사용 사례이지만, 파이썬 객체나 파이썬 C API에 접근할 필요가 없는 장기 실행 네이티브 코드에 호출하기에도 유용합니다. 예를 들어, 표준 zlib 및 hashlib 모듈은 데이터를 압축하거나 해싱할 때 :term:`쓰레드 상태 <attached thread state>`를 분리합니다.

:term:`free-threaded build`의 경우, 일반적으로 :term:`GIL`은 문제가 되지 않지만, 레이스 조건 없이 파이썬 객체에 대한 일관된 뷰를 얻기 위해 인터프리터가 주기적으로 모든 스레드를 차단해야 하므로 쓰레드 상태 연결 해제가 여전히 필요합니다. 예를 들어, CPython은 현재 가비지 컬렉터를 실행하는 동안 모든 스레드를 짧은 시간 동안 중단시킵니다.

경고

쓰레드 상태 연결 해제는 인터프리터 최종화 중에 예기치 않은 동작을 초래할 수 있습니다. 자세한 내용은 :ref:`cautions-regarding-runtime-finalization`을 참조하십시오.

API¶

다음 매크로들은 보통 후행 세미콜론 없이 사용됩니다. 상세한 사용 예시는 파이썬 소스 배포판에서 찾아보십시오.

참고

이 매크로들은 :term:`free-threaded build`에서도 데드락을 방지하기 위해 여전히 필요합니다.

Py_BEGIN_ALLOW_THREADS¶
상의 안정 ABI.

이 매크로는 { PyThreadState *_save; _save = PyEval_SaveThread(); 로 확장됩니다. 열고 있는 중괄호가 포함되어 있음에 유의하십시오. 뒤따르는 Py_END_ALLOW_THREADS 매크로와 일치해야 합니다. 이 매크로에 대한 자세한 내용은 위를 참조하십시오.

Py_END_ALLOW_THREADS¶
상의 안정 ABI.

이 매크로는 PyEval_RestoreThread(_save); } 로 확장됩니다. 닫는 중괄호가 포함되어 있음에 유의하십시오. 이전 Py_BEGIN_ALLOW_THREADS 매크로와 일치해야 합니다. 이 매크로에 대한 자세한 내용은 위를 참조하십시오.

Py_BLOCK_THREADS¶
상의 안정 ABI.

이 매크로는 PyEval_RestoreThread(_save); 로 확장됩니다: 닫는 중괄호가 없는 :c:macro:`Py_END_ALLOW_THREADS`와 동등합니다.

Py_UNBLOCK_THREADS¶
상의 안정 ABI.

이 매크로는 _save = PyEval_SaveThread(); 로 확장됩니다: 열고 있는 중괄호 및 변수 선언이 없는 :c:macro:`Py_BEGIN_ALLOW_THREADS`와 동등합니다.

외부 스레드에서 C API 사용¶

스레드가 전용 Python API(예를 들어 threading 모듈)를 사용하여 생성될 때, 해당 스레드에는 자동으로 스레드 상태가 연결됩니다. 그러나 네이티브 코드에서 스레드를 생성하는 경우(예: 자체 스레드 관리를 가진 타사 라이브러리에 의한 경우), 연결된 스레드 상태를 가지지 않습니다.

만약 이러한 스레드에서 Python 코드를 호출해야 하는 경우(이는 종종 앞서 언급한 타사 라이브러리에서 제공하는 콜백 API의 일부가 될 것입니다), 새로운 스레드 상태를 생성하고 연결하여 인터프리터에 이러한 스레드를 먼저 등록해야 합니다.

가장 쉬운 방법은 PyThreadState_Ensure() 또는 PyThreadState_EnsureFromView() 를 사용하는 것입니다.

참고

이러한 함수들은 원하는 인터프리터를 가리키는 인수를 필요로 합니다. 이러한 포인터는 스레드를 생성하는 함수에서 PyInterpreterGuard_FromCurrent() (PyThreadState_Ensure 의 경우) 또는 PyInterpreterView_FromCurrent() (PyThreadState_EnsureFromView 의 경우) 호출을 통해 획득할 수 있습니다. 만약 포인터가 없는 경우(예를 들어 주어진 네이티브 스레드 라이브러리가 데이터 인수를 제공하지 않는 경우), PyInterpreterView_FromMain() 을 사용하여 메인 인터프리터에 대한 뷰를 얻을 수 있지만, 이 경우 코드가 서브인터프리터와 호환되지 않게 된다는 점에 유의해야 합니다.

예를 들면

// 이 스레드를 생성한 함수에서 반환되는 PyInterpreterGuard_FromCurrent()의 값입니다.
PyInterpreterGuard *guard = thread_data->guard;

// 인터프리터에 대한 새 스레드 상태를 생성합니다.
PyThreadStateToken *token = PyThreadState_Ensure(guard);
if (token == NULL) {
   PyInterpreterGuard_Close(guard);
   return;
}

// 유효한 스레드 상태가 있음 -- 여기에 Python 작업을 수행합니다.
result = CallSomeFunction();
// 결과를 평가하거나 예외를 처리합니다.

// 스레드 상태를 해제합니다. 이 시점 이후로는 C API 호출이 허용되지 않습니다.
PyThreadState_Release(token);
PyInterpreterGuard_Close(guard);

PyThreadState_Ensure 를 호출하는 것이 항상 새 스레드 상태를 생성하지 않을 수 있으며, PyThreadState_Release 를 호출하는 것도 항상 연결을 해제하지 못할 수 있다는 점을 유념하십시오. 이 함수들은 기존에 연결된 스레드 상태를 재사용하거나, 현재 스레드를 위해 이전에 연결되었던 스레드 상태를 다시 연결할 수도 있습니다.

스레드 상태의 연결/연결 해제¶

PyThreadStateToken *PyThreadState_Ensure(PyInterpreterGuard *guard)¶
상의 안정 ABI 버전 3.15 이후로.

스레드가 guard 에 의해 보호되는 인터프리터에 대한 연결된 스레드 상태를 가지고 있는지 확인하여, 해당 인터프리터를 안전하게 호출할 수 있도록 하십시오.

이 함수는 이미 연결된 스레드 상태를 가진 경우 호출하는 것이 괜찮습니다. 단, 이와 일치하는 PyThreadState_Release() 호출을 이후에 수행할 경우입니다 (이는 이 함수에 대한 “중첩” 호출이 허용됨을 의미합니다).

해당 함수의 효과(있는 경우)는 일치하는 PyThreadState_Release() 호출에 의해 되돌려집니다.

오류가 발생하면, 이 함수는 예외를 설정하지 않은 NULL 을 반환합니다. 이런 경우에는 PyThreadState_Release() 를 호출하지 마십시오.

성공 시, 이 함수는 일치하는 PyThreadState_Release`() 호출에 전달되어야 하는 포인터 값을 반환합니다.

이 함수가 새로운 스레드 상태 <thread state>`를 생성하는 조건은 불안정하고 구현에 따라 달라지는 것으로 간주됩니다. 스레드 상태의 정확한 수명 주기를 제어해야 하는 경우, :c:func:`PyThreadState_New` 사용을 고려하십시오. 다만, 이 함수는 단지 스레드 상태의 수명이 버전 간에 일관되지 않을 수 있다는 이유만으로 사용하는 것을 피해서는 안 됩니다. 이 함수의 변경 사항은 신중하고 하위 호환 가능한 방식으로 이루어질 것입니다. 특히, 스레드 로컬 변수 및 유사한 상태 저장소는 Python 버전을 거쳐 유지될 것입니다.

이 함수가 새로운 스레드 상태를 생성하는지의 정확한 동작은 아래에 설명되어 있지만, 이는 향후 변경될 수 있다는 점을 유의하십시오.

먼저, 이 함수는 연결된 스레드 상태가 존재하는지 확인합니다. 존재한다면, 그 다음으로 해당 스레드 상태의 인터프리터가 guard 에 의해 보호되는 인터프리터와 일치하는지 확인합니다. 만약 그렇다면, 이 함수는 단순히 해당 스레드 상태를 PyThreadState_Ensure 호출에 의해 사용되고 있음을 표시하고 반환합니다.

첨부된 스레드 상태가 없으면, 이 함수는 현재 OS 스레드가 어떤 스레드 상태를 사용했는지 확인합니다. (이는 PyGILState_GetThisThreadState() \에 의해 반환됩니다.) 만약 존재했다면, 이 함수는 해당 스레드 상태의 인터프리터가 guard 와 일치하는지 확인합니다. 일치하면 재첨부되어 사용됨으로 표시됩니다.

그렇지 않다면, 위 두 경우 모두 실패할 때 guard 를 위한 새로운 스레드 상태가 생성됩니다. 그런 다음 첨부되고 PyThreadState_Ensure \에 의해 소유됨으로 표시됩니다.

Added in version 3.15.

PyThreadStateToken *PyThreadState_EnsureFromView(PyInterpreterView *view)¶
상의 안정 ABI 버전 3.15 이후로.

view 에서 참조된 인터프리터에 연결된 스레드 상태를 가져옵니다.

작동 방식과 반환 값은 PyThreadState_Ensure() \와 동일합니다. 추가적으로, 함수가 성공하면 view 에서 참조된 인터프리터는 암시적으로 보호됩니다. 이 보호는 해당 PyThreadState_Release() \ 호출 시 해제됩니다.

Added in version 3.15.

void PyThreadState_Release(PyThreadStateToken *token)¶
상의 안정 ABI 버전 3.15 이후로.

PyThreadState_Ensure() 또는 PyThreadState_EnsureFromView()\ 호출을 되돌립니다.

이것은 성공적인 Ensure 호출마다 정확히 한 번 호출되어야 하며, 이때 token 은 해당 호출의 반환 값으로 설정해야 합니다.

해당 Ensure 호출 이전에 첨부되었던 상태(있는 경우)는 PyThreadState_Release()\가 반환될 때 첨부됩니다.

이 함수가 스레드 상태를 삭제할지 여부는 불안정하며 구현 의존적입니다.

현재 이 함수는 첨부된 스레드 상태의 내부 카운터를 감소시킵니다. 이 카운터가 0 미만으로 내려가면, 이 함수는 치명적인 오류( Py_FatalError()\를 통해)를 발생시킵니다.

첨부된 스레드 상태가 PyThreadState_Ensure\에 의해 소유된 경우, 내부 카운터가 0에 도달하면 첨부된 스레드 상태는 할당 해제 및 삭제됩니다. 그렇지 않으면 카운터가 0에 도달해도 아무 일도 일어나지 않습니다.

Added in version 3.15.

type PyThreadStateToken¶
상의 안정 ABI (불투명한 구조체로) 버전 3.15 이후로.

PyThreadState_Ensure()\ 호출에서 검색된 불투명한 토큰을 해당 PyThreadState_Release()\ 호출에 전달하는 것입니다.

GIL 상태 API¶

The following APIs are generally not compatible with subinterpreters and will hang the process during interpreter finalization (see 인터프리터 파이널라이제이션 관련 주의사항). As such, these APIs were soft deprecated in Python 3.15 in favor of the new APIs.

type PyGILState_STATE¶
상의 안정 ABI.

PyGILState_Ensure()\가 반환하고 PyGILState_Release()\에 전달하는 값의 타입입니다.

enumerator PyGILState_LOCKED¶

PyGILState_Ensure()\가 호출될 때 이미 GIL이 보유되고 있었습니다.

enumerator PyGILState_UNLOCKED¶

PyGILState_Ensure()\가 호출될 때 GIL을 보유하고 있지 않았습니다.

PyGILState_STATE PyGILState_Ensure()¶
상의 안정 ABI.

현재 Python의 상태나 attached thread state\와 관계없이 현재 스레드가 Python C API를 호출할 준비가 되었음을 보장합니다. 이 함수는 각 호출이 PyGILState_Release()\ 호출과 일치하는 한, 스레드에서 원하는 만큼 여러 번 호출될 수 있습니다. 일반적으로 다른 스레드 관련 API는 Release() 이전의 이전 상태로 스레드 상태를 복원하기만 한다면, PyGILState_Ensure()\와 PyGILState_Release()\ 호출 사이에 사용할 수 있습니다. 예를 들어, Py_BEGIN_ALLOW_THREADS\ 및 Py_END_ALLOW_THREADS\ 마크로의 정상적인 사용은 허용됩니다.

반환 값은 PyGILState_Ensure() \가 호출되었을 때의 attached thread state \에 대한 불투명한 “핸들”이며, Python이 동일한 상태로 남겨지도록 하려면 PyGILState_Release() \에 전달해야 합니다. 재귀적 호출은 허용되지만, 이 핸들은 공유 될 수 없습니다. PyGILState_Ensure() \의 각 고유한 호출은 해당 호출의 PyGILState_Release() \를 위한 핸들을 저장해야 합니다.

함수가 반환되면, attached thread state\가 존재하며 스레드는 임의의 Python 코드를 호출할 수 있게 됩니다.

이 함수는 오류를 반환하는 방법이 없습니다. 따라서 오류는 치명적이거나(– 즉, SIGABRT\를 보내 프로세스를 충돌시키거나; Py_FatalError()\)\ 스레드가 영구적으로 차단됩니다 (인터프리터 최종화 중과 같은 경우).

경고

이 함수를 인터프리터가 최종화되는 동안 호출하면, 무한정 스레드를 멈추게 할 수 있으며, 이는 데드락을 유발할 수 있습니다. 더 자세한 내용은 인터프리터 파이널라이제이션 관련 주의사항\을 참조하십시오.

게다가 이 함수는 외부 스레드에서 사용될 때 서브 인터프리터와 일반적으로 작동하지 않습니다. 이는 이 함수가 어떤 인터프리터가 해당 스레드를 생성했는지 알 방법이 없기 때문입니다 (따라서 암묵적으로 메인 인터프리터를 선택합니다).

버전 3.14에서 변경: 인터프리터가 파이널라이즈되는 동안 호출되면, 현재 스레드를 종료하는 대신 대기 상태로 만듭니다.

버전 3.15부터 약하게 폐지 <Soft deprecated>: 대신 PyThreadState_Ensure() 또는 PyThreadState_EnsureFromView() 를 사용하십시오.

fork()에 대한 주의 사항¶

스레드에 관해 알아야 할 또 다른 중요한 사실은 C fork() 호출에 직면했을 때의 동작입니다. 대부분의 시스템에서 :c:func:`fork`가 있으면, 프로세스가 포크된 후에는 포크를 발생시킨 스레드만 존재하게 됩니다. 이는 락을 처리하는 방법과 CPython 런타임의 모든 저장된 상태 모두에 구체적인 영향을 미칩니다.

The fact that only the “current” thread remains means any locks held by other threads will never be released. Python solves this for os.fork() by acquiring the locks it uses internally before the fork, and releasing them afterwards. In addition, it resets any Lock 객체 in the child. When extending or embedding Python, there is no way to inform Python of additional (non-Python) locks that need to be acquired before or reset after a fork. OS facilities such as pthread_atfork() would need to be used to accomplish the same thing. Additionally, when extending or embedding Python, calling fork() directly rather than through os.fork() (and returning to or calling into Python) may result in a deadlock by one of Python’s internal locks being held by a thread that is defunct after the fork. PyOS_AfterFork_Child() tries to reset the necessary locks, but is not always able to.

다른 모든 스레드가 사라진다는 사실은 또한 CPython의 런타임 상태도 os.fork`가 하는 것처럼 적절하게 정리되어야 함을 의미합니다. 이는 현재 인터프리터에 속하는 다른 모든 :c:type:`PyThreadState() 객체와 모든 다른 PyInterpreterState 객체를 파이널라이제이션 함을 의미합니다. 이것과 :ref:`”메인” 인터프리터 <sub-interpreter-support>`의 특수한 특성 때문에, :c:func:`fork`는 CPython 전역 런타임이 원래 초기화된 인터프리터의 “메인” 스레드에서만 호출되어야 합니다. 유일한 예외는 그 후에 :c:func:`exec`가 즉시 호출되는 경우입니다.

호출하는 스레드는 이미 :term:`attached thread state`를 가지고 있어서는 안 됩니다.

참고

런타임이 파이널리징 될 때 스레드에서 이 함수를 호출하면, 스레드가 파이썬에 의해 생성되지 않았더라도 프로그램이 종료될 때까지 해당 스레드가 멈추게 됩니다. 자세한 내용은 인터프리터 파이널라이제이션 관련 주의사항 을 참조하십시오.

버전 3.8에서 변경: PyEval_RestoreThread(), Py_END_ALLOW_THREADS(), 및 :c:func:`PyGILState_Ensure`와 일관되도록 업데이트되었으며, 인터프리터가 파이널라이즈되는 동안 호출되면 현재 스레드를 종료합니다.

버전 3.14에서 변경: 인터프리터가 파이널라이즈되는 동안 호출되면, 현재 스레드를 종료하는 대신 대기 상태로 만듭니다.

:c:func:`PyEval_RestoreThread`는 (스레드가 초기화되지 않았을 때조차) 항상 사용할 수 있는 고수준 함수입니다.

void PyEval_ReleaseThread(PyThreadState *tstate)¶
상의 안정 ABI.

attached thread state 를 분리합니다. NULL 이 아니어야 하는 tstate 인수는 이것이 attached thread state 를 나타내하는지 확인하기 위해서만 사용됩니다—그렇지 않은 경우, 치명적인 오류가 보고됩니다.

:c:func:`PyEval_SaveThread`는 (스레드가 초기화되지 않았을 때조차) 항상 사용할 수 있는 고수준 함수입니다.

비동기 알림¶

메인 인터프리터 스레드로 비동기 알림을 할 수 있는 메커니즘이 제공됩니다. 이 알림은 함수 포인터와 void 포인터 인자 형태로 구성됩니다.

int Py_AddPendingCall(int (*func)(void*), void *arg)¶
상의 안정 ABI.

메인 인터프리터 스레드에서 호출될 함수를 예약합니다. 성공하면 0 이 반환되고 func 는 메인 스레드에서 호출되기 위해 큐에 추가됩니다. 실패 시에는 예외를 설정하지 않고 -1 가 반환됩니다.

성공적으로 큐에 넣으면, func*는 *arg 인자와 함께 결국 메인 인터프리터 스레드에서 호출될 것입니다. 일반적으로 실행되는 Python 코드와 비교할 때 비동기적으로 호출되지만, 다음 두 조건이 모두 충족됩니다:

  • bytecode 경계에서;

  • 메인 스레드가 attached thread state 를 가지고 있는 경우에만 (func 는 따라서 전체 C API를 사용할 수 있습니다).

func 는 성공 시 0 을 반환하거나, 예외를 설정하고 실패 시 -1 을 반환해야 합니다. func 는 다른 비동기 알림을 재귀적으로 수행하기 위해 인터럽트 되지 않지만, thread state 가 분리된 경우 스레드를 전환하기 위해 여전히 인터럽트 될 수 있습니다.

이 함수는 별도의 attached thread state 를 필요로 하지 않습니다. 그러나 서브 인터프리터에서 이 함수를 호출하려면, 호출자가 attached thread state 를 가지고 있어야 합니다. 그렇지 않으면 func 가 잘못된 인터프리터에서 호출되도록 예약될 수 있습니다.

경고

이것은 매우 특별한 경우에만 유용한, 저수준 함수입니다. func*가 가능한 한 빨리 호출된다는 보장은 없습니다. 메인 스레드가 시스템 호출을 실행하느라 바쁘면, 시스템 호출이 반환되기 전까지는 *func*가 호출되지 않습니다. 이 함수는 일반적으로 임의의 C 스레드에서 Python 코드를 호출하는 데 적합하지 **않습니다*. 대신 :c:func:`PyThreadState_EnsureFromView`를 사용하십시오.

Added in version 3.1.

버전 3.9에서 변경: 이 함수가 서브 인터프리터에서 호출되면, func 는 이제 메인 인터프리터에서 호출되는 대신 서브 인터프리터에서 호출되도록 예약됩니다. 각 서브 인터프리터는 이제 자체 예약된 호출 목록을 갖습니다.

버전 3.12에서 변경: 이 함수는 항상 func 를 메인 인터프리터에서 실행하도록 예약합니다.

int Py_MakePendingCalls(void)¶
상의 안정 ABI.

모든 계류 중인 호출을 실행합니다. 이는 보통 인터프리터에 의해 자동으로 실행됩니다.

이 함수는 성공 시 0 을 반환하고, 실패 시 예외와 함께 -1 을 반환합니다.

이것을 메인 인터프리터의 메인 스레드에서 호출하지 않으면, 이 함수는 아무것도 하지 않고 0 을 반환합니다. 호출자는 attached thread state 를 보유하고 있어야 합니다.

Added in version 3.1.

버전 3.12에서 변경: 이 함수는 메인 인터프리터의 대기 중인 호출만 실행합니다.

int PyThreadState_SetAsyncExc(unsigned long id, PyObject *exc)¶
상의 안정 ABI.

스레드에서 비동기적으로 발생할 예외를 예약합니다. 스레드가 이전에 예약된 예외가 있는 경우, 이는 덮어쓰여집니다.

id 인수는 PyThread_get_thread_ident() 에 의해 반환되는 타겟 스레드의 스레드 id입니다. exc 는 발생시킬 예외의 클래스이며, 대기 중인 예외를 지우려면 NULL 을 사용합니다 (있는 경우).

영향을 받은 스레드 상태의 수를 반환합니다. id 가 발견되면 변경 사항이 없더라도 (주어진 exc 가 이미 대기 중이었거나, exc 가 NULL 이지만 예외가 대기 중이지 않은 경우) 일반적으로 1 입니다. 스레드 id를 찾을 수 없으면 0 을 반환합니다. 이 함수는 예외를 발생시키지 않습니다.

단순 오용을 방지하기 위해, 직접 C확장 기능을 작성하여 호출해야 합니다. 이 함수는 attached thread state`와 함께 호출되어야 합니다. 이 함수는 *exc*에 대한 참조를 훔치지 않습니다. 이 함수가 :py:func:`~time.sleep 같은 시스템 호출을 반드시 중단시키는 것은 아닙니다.

버전 3.7에서 변경: id 매개변수의 타입이 :c:expr:`long`에서 :c:expr:`unsigned long`으로 변경되었습니다.

운영 체제 스레드 API¶

PYTHREAD_INVALID_THREAD_ID¶

유효하지 않은 스레드 ID에 대한 센티넬 값입니다.

현재 이것은 (unsigned long)-1 과 동일합니다.

unsigned long PyThread_start_new_thread(void (*func)(void*), void *arg)¶
상의 안정 ABI.

인수 arg 와 함께 함수 func 를 새 스레드로 시작합니다. 결과로 생성된 스레드는 조인할 목적으로 사용되지 않습니다.

func 는 NULL 이 아니어야 하지만, arg 는 NULL 일 수 있습니다.

성공하면 이 함수는 새 스레드의 식별자를 반환합니다. 실패하면 :c:macro:`PYTHREAD_INVALID_THREAD_ID`를 반환합니다.

호출자가 :term:`attached thread state`를 보유할 필요가 없습니다.

unsigned long PyThread_get_thread_ident(void)¶
상의 안정 ABI.

현재 스레드의 식별자를 반환하며, 이 값은 절대 0이 아닙니다.

이 함수는 실패할 수 없으며, 호출자는 :term:`attached thread state`를 보유할 필요가 없습니다.

PyObject *PyThread_GetInfo(void)¶
상의 안정 ABI 버전 3.3 이후로.

현재 스레드에 대한 일반 정보를 구조 시퀀스 객체 형태로 가져옵니다. 이 정보는 Python에서 :py:attr:`sys.thread_info`로 접근할 수 있습니다.

성공하면 스레드 정보에 대한 새로운 강한 참조 를 반환합니다. 실패하면 예외가 설정된 NULL 을 반환합니다.

호출자는 :term:`attached thread state`를 보유해야 합니다.

void PyThread_exit_thread(void)¶
상의 안정 ABI.

현재 스레드를 종료합니다. 이 함수는 일반적으로 안전하지 않은 것으로 간주되며 피해야 합니다. 순수하게 하위 호환성을 위해 남아 있습니다.

이 함수는 전체 호출 스택의 모든 함수가 이를 처리하도록 작성된 경우에만 안전하게 호출할 수 있습니다.

경고

현재 시스템이 POSIX 스레드를 사용하는 경우 (pthreads라고도 알려져 있음), 이것은 pthread_exit(3)`을 호출하며, 이는 일부 libc 구현에서 스택을 풀고 C++ 소멸자를 호출하려고 시도합니다. 하지만 `noexcept 함수에 도달하면 프로세스가 종료될 수 있습니다. macOS와 같은 다른 시스템은 언와인딩을 수행합니다.

Windows에서는 이 함수가 _endthreadex() 를 호출하며, 이는 C++ 소멸자를 호출하지 않고 스레드를 종료시킵니다.

어떤 경우에도 스레드의 스택에 손상될 위험이 있습니다.

버전 3.14부터 폐지됨.
