객체 수명 주기¶

이 섹션은 객체의 수명 주기 동안 한 타입의 슬롯들이 서로 어떻게 관련되어 있는지 설명합니다. 슬롯에 대한 전체 표준 참조를 의도한 것이 아니므로, 특정 슬롯에 대한 자세한 내용은 타입 객체 구조\에서 확인하십시오.

생애 이벤트¶

아래 그림은 객체의 수명 주기 내내 발생할 수 있는 이벤트 순서를 보여줍니다. A*에서 *B*로 향하는 화살표는 이벤트 *A*가 발생한 후에 이벤트 *B*가 발생할 수 있음을 나타내며, 화살표의 라벨은 *B*가 *A 이후에 발생하기 위해 참이어야 하는 조건을 나타냅니다.

설명:

  • 객체를 그 타입을 호출하여 새로 구성할 때:

    1. :c:member:`~PyTypeObject.tp_new`는 새 객체를 생성하는 데 사용됩니다.

    2. :c:member:`~PyTypeObject.tp_alloc`가 :c:member:`~PyTypeObject.tp_new`에 의해 직접 호출되어 새 객체의 메모리를 할당합니다.

    3. tp_init 은 새로 생성된 객체를 초기화합니다. 필요하면 tp_init 을 다시 호출하여 객체를 재초기화할 수 있습니다. 또한, 이 tp_init 호출은 Python 코드가 __new__() 를 호출하여 완전히 건너뛸 수도 있습니다.

  • :c:member:`!tp_init`이 완료된 후, 객체는 사용 준비가 됩니다.

  • 객체에 대한 마지막 참조가 제거된 후 일정 시간이 경과합니다:

    1. 객체가 finalized 로 표시되지 않은 경우, finalized 로 표시하고 해당 tp_finalize 함수를 호출하여 최종화될 수 있습니다. Python은 마지막 참조가 삭제될 때 객체를 최종화하지 않으므로, 항상 tp_finalize 가 호출되도록 보장하려면 PyObject_CallFinalizerFromDealloc() 을 사용하십시오.

    2. 객체가 최종화됨으로 표시된 경우, 가비지 컬렉터에 의해 :c:member:`~PyTypeObject.tp_clear`가 호출되어 객체가 보유한 참조를 정리할 수 있습니다. 객체의 참조 카운트가 0에 도달할 때 호출되지는 않습니다.

    3. :c:member:`~PyTypeObject.tp_dealloc`은 객체를 파괴하는 데 사용됩니다. 코드 중복을 피하기 위해, :c:member:`~PyTypeObject.tp_dealloc`은 일반적으로 객체의 참조를 해제하기 위해 :c:member:`~PyTypeObject.tp_clear`를 호출합니다.

    4. tp_dealloc 이 객체 파괴를 완료하면, 메모리를 해제하기 위해 해당 타입에 적절한 PyObject_Free() 또는 PyObject_GC_Del() 로 설정된 tp_free 를 직접 호출합니다.

  • tp_finalize 함수는 원하는 경우 객체에 대한 참조를 추가하는 것이 허용됩니다. 그렇게 하는 경우, 객체는 resurrected*되어 보류 중인 파괴가 방지됩니다. (오직 :c:member:`!tp_finalize`만 객체를 부활시킬 수 있습니다; :c:member:`~PyTypeObject.tp_clear`와 :c:member:`~PyTypeObject.tp_dealloc`은 :c:member:`!tp_finalize`를 호출하지 않고는 할 수 없습니다.) 객체를 부활시키는 것은 객체의 *finalized 표시가 제거될 수도 있고 그렇지 않을 수도 있습니다. 현재 Python은 가비지 컬렉션을 지원하는(즉, Py_TPFLAGS_HAVE_GC 플래그가 설정된) 부활된 객체에서 finalized 표시를 제거하지 않지만, 가비지 컬렉션을 지원하지 않는 객체에서는 표시를 제거합니다. 이러한 두 가지 동작 모두 향후 변경될 수 있습니다.

  • tp_dealloc 은 객체 파괴를 돕기 위해 해당 코드를 재사용하려면 선택적으로 PyObject_CallFinalizerFromDealloc() 을 통해 tp_finalize 를 호출할 수 있습니다. 이는 파괴 전에 항상 tp_finalize 가 호출되도록 보장하기 때문에 권장됩니다. 예제 코드는 tp_dealloc 문서를 참조하십시오.

  • 만약 객체가 cyclic isolate`의 멤버이고, :c:member:`~PyTypeObject.tp_clear`가 참조 순환을 끊지 못하거나 (어쩌면 :func:`gc.disable`이 호출되었거나, 관련 타입 중 하나에서 :c:macro:`Py_TPFLAGS_HAVE_GC 플래그가 잘못 생략되었을 수도 있습니다), 객체들은 무기한 수거 불가능 상태로 남아 있습니다 (“누수”). :data:`gc.garbage`를 참조하십시오.

객체가 가비지 컬렉션을 지원하는 것으로 표시된 경우 (즉, tp_flags 에 Py_TPFLAGS_HAVE_GC 플래그가 설정된 경우), 다음과 같은 이벤트도 가능합니다:

  • 가비지 컬렉터는 :term:`cyclic isolates <cyclic isolate>`를 식별하기 위해 때때로 :c:member:`~PyTypeObject.tp_traverse`를 호출합니다.

  • 가비지 컬렉터가 cyclic isolate`를 발견하면, 이 그룹의 객체 중 하나를 *finalized*로 표시하고 해당 :c:member:`~PyTypeObject.tp_finalize 함수를 호출하여 최종화합니다. 이는 참조 순환이 존재하지 않거나 모든 객체가 최종화될 때까지 반복됩니다.

  • tp_finalize 는 cyclic isolate 외부에서 참조를 추가하여 객체 부활이 허용됩니다. 새로 추가된 참조로 인해 객체 그룹이 더 이상 순환 격리(cyclic isolate)를 형성하지 않게 됩니다(참조 순환 자체는 여전히 존재할 수 있지만, 만약 존재하더라도 객체들은 더 이상 격리되지 않습니다).

  • 가비지 컬렉터가 cyclic isolate`를 발견하고 그룹 내 모든 객체가 이미 *finalized* 상태로 표시된 경우, 가비지 컬렉터는 각 객체의 :c:member:`~PyTypeObject.tp_clear 함수를 호출하여 그룹 내에서 정리되지 않은 하나 이상의 객체를 정리합니다(가능할 경우 동시성으로 처리). 이 과정은 순환 격리가 여전히 존재하고 모든 객체가 정리되지 않은 상태일 동안 반복됩니다.

순환 격리 파괴(Cyclic Isolate Destruction)¶

아래에 가상의 cyclic isolate`의 수명 주기 단계가 나열됩니다. 이 단계는 각 멤버 객체가 종료되거나 정리된 후에도 계속 존재합니다. 순환 격리가 이 모든 단계를 거친다면 메모리 누수입니다. 모든 객체가 정리되면 사라져야 하며, 경우에 따라 더 일찍 사라질 수도 있습니다. 순환 격리는 참조 순환이 끊어지거나, *finalizer resurrection* 때문에 객체들이 더 이상 격리되어 있지 않게 됨으로써 사라질 수 있습니다 ( :c:member:`~PyTypeObject.tp_finalize 참조).

  1. 도달 가능(Reachable) (아직 순환 격리가 아님): 모든 객체가 정상적으로 도달 가능한 상태에 있습니다. 참조 순환이 존재할 수 있지만, 외부 참조는 객체들이 아직 격리되지 않았다는 것을 의미합니다.

  2. 도달 불가능하지만 일관됨(Unreachable but consistent): 객체들의 순환 그룹 외부의 마지막 참조가 제거되어 객체들이 격리됩니다(따라서 순환 격리가 탄생합니다). 그룹의 객체 중 어떤 것도 아직 finalized 되거나 정리되지 않았습니다. 이 순환 격리는 가비지 컬렉터의 미래 실행이 있을 때까지 이 단계에 남아 있습니다(다음 실행이 모든 객체를 스캔하는 것은 아닐 수 있습니다).

  3. 종료된 것과 종료되지 않은 것의 혼합: 순환 격리 내의 객체들은 하나씩 finalized 됩니다. 이는 순환 격리가 finalized 된 객체와 finalized 되지 않은 객체의 혼합물로 구성되는 기간이 있음을 의미합니다. Finalization 순서는 지정되어 있지 않으므로 임의의 순서로 보일 수 있습니다. Finalized 된 객체는 finalized 되지 않은 객체와 상호 작용할 때 안정적인 방식으로 동작해야 하며, finalized 되지 않은 객체는 참조하는 객체의 임의의 부분 집합이 finalization 되는 것을 견딜 수 있어야 합니다.

  4. 모두 종료됨(All finalized): 순환 격리의 모든 객체가 어떤 것이 정리되기 전에 finalized 됩니다.

  5. 종료된 것과 정리된 것의 혼합: 객체들은 순차적 또는 동시적으로 정리될 수 있습니다(하지만 GIL 을 유지한 상태로); 어느 쪽이든, 일부는 다른 것보다 먼저 종료됩니다. Finalized 된 객체는 참조 객체의 하위 집합이 정리되는 것을 견딜 수 있어야 합니다. PEP 442 는 이 단계를 “순환 쓰레기(cyclic trash)”라고 지칭합니다.

  6. 누수됨(Leaked): 그룹 내 모든 객체가 finalized 되고 정리된 후에도 순환 격리가 여전히 존재한다면, 객체들은 무기한으로 수집 불가능합니다( gc.garbage 참조). 순환 격리가 이 단계에 도달하는 것은 버그입니다—이는 참여 객체들의 tp_clear 메서드가 요구되는 참조 순환을 끊지 못한 것을 의미합니다.

만약 :c:member:`~PyTypeObject.tp_clear`가 존재하지 않았다면, 파이썬은 참조 순환을 안전하게 끊을 방법이 없습니다. 순환 격리 내에서 객체를 단순히 파괴하는 것은 단 하나의 끊어진 포인터를 유발하며, 그 파괴된 객체를 참조하는 객체 자신이 파괴될 때 정의되지 않은 동작(undefined behavior)을 유발합니다. 정리(clearing) 단계는 객체 파괴를 2단계 프로세스로 만듭니다. 먼저 :c:member:`~PyTypeObject.tp_clear`가 호출되어 객체를 부분적으로 파괴하여 서로의 연결 고리를 풀고, 다음으로 :c:member:`~PyTypeObject.tp_dealloc`이 호출되어 파괴를 완료합니다.

클리어링과 달리, 파이널라이제이션은 소멸의 단계가 아닙니다. 파이널라이즈된 객체는 여전히 설계 계약을 이행하는 방식으로 올바르게 작동해야 합니다. 객체의 최종화기는 임의의 파이썬 코드를 실행할 수 있도록 허용되며, 참조를 추가하여 임박한 소멸을 방지하는 것도 허용됩니다. 최종화기는 호출 순서와 관련이 있을 뿐입니다—실행된다면, tp_clear (호출된 경우)로 시작하고 :c:member:`~PyTypeObject.tp_dealloc`로 끝나는 소멸 과정 이전에 실행됩니다.

파이널라이제이션 단계는 순환 격리(cyclic isolate) 내의 객체를 안전하게 회수하는 데 필수는 아니지만, 그 존재 덕분에 객체가 클리어 될 때 제대로 작동하는 타입을 설계하기가 더 쉬워집니다. 객체를 클리어하는 것은 필연적으로 객체를 깨지거나 부분적으로 파괴된 상태로 남길 수 있습니다—따라서 클리어된 객체의 메서드를 호출하거나 속성에 액세스하는 것이 안전하지 않을 수 있습니다. 최종화 기능을 사용하면, 최종화된 객체만이 클리어된 객체와 상호 작용할 수 있으며, 비최종화된 객체는 비클리어(하지만 잠재적으로 최종화된) 객체와만 상호 작용하는 것이 보장됩니다.

가능한 상호 작용을 요약하면 다음과 같습니다:

  • 비최종화된 객체는 비최종화되고 최종화된 객체와 참조를 가질 수 있지만, 클리어된 객체와는 참조를 가질 수 없습니다.

  • 최종화된 객체는 비최종화되고, 최종화되고, 클리어된 객체와 참조를 가질 수 있습니다.

  • 클리어된 객체는 최종화되고 클리어된 객체와 참조를 가질 수 있지만, 비최종화된 객체와는 참조를 가질 수 없습니다.

참조 순환이 없다면, 객체는 마지막 참조가 삭제되는 순간 간단히 파괴될 수 있으며, 사용되지 않는 객체를 안전하게 회수하는 데 최종화 및 클리어 단계는 필수적이지 않습니다. 그러나, 타입을 설계할 때 객체가 순환 격리에 참여했는지 여부와 관계없이 언제나 동일한 일련의 이벤트를 경험하도록 하기 때문에, 파괴되기 전에 tp_finalize 및 :c:member:`~PyTypeObject.tp_clear`를 자동으로 호출하는 것이 여전히 유용할 수 있습니다. 현재 파이썬은 순환 격리를 파괴하는 데 필요한 경우에만 :c:member:`~PyTypeObject.tp_finalize`와 :c:member:`~PyTypeObject.tp_clear`를 호출합니다—이는 미래 버전에서 변경될 수 있습니다.

함수¶

메모리를 할당하고 해제하는 방법은 :ref:`객체 할당 <allocating-objects>`를 참조하십시오.

void PyObject_CallFinalizer(PyObject *op)¶

Finalizes the object as described in tp_finalize. Call this function (or PyObject_CallFinalizerFromDealloc()) instead of calling tp_finalize directly because this function may deduplicate multiple calls to tp_finalize. Currently, calls are only deduplicated if the type supports garbage collection (i.e., the Py_TPFLAGS_HAVE_GC flag is set); this may change in the future.

Added in version 3.4.

int PyObject_CallFinalizerFromDealloc(PyObject *op)¶
상의 안정 ABI 버전 3.15 이후로.

PyObject_CallFinalizer() 와 동일하지만, 객체의 소멸자(tp_dealloc) 시작 부분에서 호출되도록 의도되었습니다. 객체에 대한 참조가 없어야 합니다. 객체의 최종화기가 객체를 부활시킨다면, 이 함수는 -1을 반환하고 더 이상의 소멸은 일어나지 않습니다. 그렇지 않으면, 이 함수는 0을 반환하고 소멸이 정상적으로 계속될 수 있습니다.

Added in version 3.4.
