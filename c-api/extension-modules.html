확장 모듈 정의¶

CPython용 C 확장 인터페이스는 공유 라이브러리(예를 들어, 리눅스에서는 .so 파일, 윈도우에서는 .pyd DLL)이며, 파이썬 프로세스에 로드할 수 있고(예를 들어, 호환 가능한 컴파일러 설정을 사용하여 컴파일됨), export hook 함수(또는 구식 initialization function)\를 내보냅니다.

기본적으로 가져와지려면(즉, importlib.machinery.ExtensionFileLoader`에 의해), 공유 라이브러리는 :py:attr:`sys.path\에 존재해야 하며, importlib.machinery.EXTENSION_SUFFIXES\에 나열된 확장자 이름과 모듈 이름을 따르도록 이름을 지정해야 합니다.

참고

확장 모듈을 빌드, 패키징 및 배포하는 것은 외부 도구를 사용하는 것이 가장 좋으며, 본 문서의 범위가 아닙니다. 적합한 도구 중 하나는 Setuptools이며, 관련 문서는 https://setuptools.pypa.io/en/latest/setuptools.html에서 찾을 수 있습니다.

확장 내보내기 훅¶

Added in version 3.15: PyModExport_<name> 내보내기 훅에 대한 지원은 Python 3.15에 추가되었습니다. 모듈을 정의하는 이전 방식은 여전히 사용할 수 있습니다. 이전 파이썬 버전을 지원하려면 PyInit 함수 섹션이나 이 문서의 이전 버전을 참조하십시오.

내보내기 훅은 다음 서명을 갖는 내보내기 함수여야 합니다:

PySlot *PyModExport_modulename(void)¶

ASCII 전용 이름을 가진 모듈의 경우, export hook 은 PyModExport_{<name>}`으로 이름을 지정해야 하며, 이때 <name> 은 모듈의 이름으로 대체됩니다.

비-ASCII 모듈 이름의 경우, 내보내기 훅은 대신에 PyModExportU_{<name>}`으로 이름을 지정해야 합니다 (U 에 유의하십시오). 이때 <name> 은 파이썬의 punycode 인코딩을 사용하여 하이픈(-)을 언더스코어(_)로 대체하여 인코딩됩니다. 파이썬에서는:

def hook_name(name):
    try:
        suffix = b'_' + name.encode('ascii')
    except UnicodeEncodeError:
        suffix = b'U_' + name.encode('punycode').replace(b'-', b'_')
    return b'PyModExport' + suffix

내보내기 훅은 PySlot 엔트리 배열을 반환하며, 이 배열은 슬롯 ID가 0 인 엔트리로 종료됩니다. 이 슬롯들은 모듈이 어떻게 생성되고 초기화되어야 하는지를 설명합니다.

이 배열은 인터프리터 종료 시까지 유효하고 상수여야 합니다. 일반적으로 static 저장소를 사용해야 합니다. 동적 동작이 필요한 경우에는 Py_mod_create 와 Py_mod_exec 슬롯을 사용하는 것을 선호합니다.

내보내기 훅은 실패를 알리는 예외와 함께 NULL 을 반환할 수 있습니다.

도움 매크로를 사용하여 내보내기 훅 함수를 정의하는 것이 좋습니다:

PyMODEXPORT_FUNC¶
상의 안정 ABI 버전 3.15 이후로.

확장 모듈 내보내기 훅을 선언합니다. 이 매크로는:

  • PySlot* 반환 타입을 지정하고,

  • 플랫폼이 요구하는 모든 특수 링크 선언을 추가하며,

  • C++의 경우, 함수를 extern "C" 로 선언합니다.

예를 들어, spam 이라는 모듈은 다음과 같이 정의됩니다:

PyABIInfo_VAR(abi_info);

static PySlot spam_slots[] = {
    PySlot_STATIC_DATA(Py_mod_abi, &abi_info),
    PySlot_STATIC_DATA(Py_mod_name, "spam"),
    PySlot_FUNC(Py_mod_init, spam_init_function),
    ...
    PySlot_END
};

PyMODEXPORT_FUNC
PyModExport_spam(void)
{
    return spam_slots;
}

내보내기 훅은 일반적으로 모듈의 C 소스에서 정의된 유일한 비-static 항목입니다.

훅은 간결하게 유지하는 것이 좋습니다. 이상적으로는 위 예처럼 한 줄로 구성하는 것이 좋습니다. 만약 이 함수에서 Python C API를 사용해야 한다면, ABI 불일치가 발생할 수 있는 일반적인 경우에 크래시를 일으키기보다는 예외를 발생시키기 위해 먼저 PyABIInfo_Check(&abi_info, "modulename") 을 호출하는 것이 권장됩니다.

참고

여러 내보내기 훅을 정의하여 단일 공유 라이브러리에서 여러 모듈을 내보내는 것이 가능합니다. 하지만, 파이썬의 임포트 메커니즘이 파일 이름에 해당하는 함수만 찾기 때문에, 이들을 임포트하려면 사용자 정의 임포터나 적절하게 이름이 지정된 복사본/링크가 필요합니다. 자세한 내용은 PEP 489 의 Multiple modules in one library 섹션을 참조하십시오.

다단계 초기화¶

확장 모듈을 생성하는 과정은 여러 단계를 따릅니다:

  • 파이썬은 모듈을 생성하는 방법에 대한 정보를 얻기 위해 내보내기 훅을 찾고 호출합니다.

  • 실질적인 코드가 실행되기 전에, 파이썬은 모듈이 어떤 기능을 지원할 수 있는지 확인하고, 환경을 조정하거나, 호환되지 않는 확장의 로드를 거부할 수 있습니다. Py_mod_abi, Py_mod_gil 및 Py_mod_multiple_interpreters 같은 슬롯들이 이 단계에 영향을 미칩니다.

  • 기본적으로 파이썬 자체가 모듈 객체를 생성합니다. 즉, 객체를 생성할 때 이 단계는 :c:data:`Py_mod_create() 슬롯을 사용하여 재정의할 수 있습니다.

  • 파이썬은 __package__ 및 :attr:`~module.__loader__`와 같은 초기 모듈 속성을 설정하고, 모듈 객체를 :py:attr:`sys.modules`에 삽입합니다.

  • 그 후에, 모듈 객체는 확장 전용 방식으로 초기화됩니다. 이는 객체를 생성할 때 이 동작은 :c:data:`Py_mod_exec() 슬롯을 사용하여 지정합니다.

이는 초기화 함수가 완전히 구성된 모듈을 반환하는 레거시(하지만 여전히 지원되는) single-phase initialization 와 구별하기 위해 다단계 초기화 라고 불립니다.

버전 3.5에서 변경: 다단계 초기화(PEP 489)를 지원하도록 추가되었습니다.

여러 모듈 인스턴스¶

기본적으로, 확장 모듈은 싱글톤이 아닙니다. 예를 들어, sys.modules 항목이 제거되고 모듈이 다시 임포트되면, 새로운 모듈 객체가 생성되고, 일반적으로 최신 메서드 및 타입 객체로 채워집니다. 이전 모듈은 일반적인 가비지 컬렉션의 대상이 됩니다. 이는 순수 파이썬 모듈의 동작을 반영합니다.

서브 인터프리터 <sub-interpreter-support>`에서 추가 모듈 인스턴스가 생성되거나(Python 런타임 재초기화 시 :c:func:`Py_Finalize 및 Py_Initialize() 호출 후), 모듈 인스턴스 간에 Python 객체를 공유하는 경우 충돌이나 정의되지 않은 동작을 일으킬 가능성이 높습니다.

이러한 문제를 피하기 위해, 확장 모듈의 각 인스턴스는 격리 되어야 합니다. 한 인스턴스에 대한 변경이 다른 인스턴스에 암묵적으로 영향을 미쳐서는 안 되며, Python 객체에 대한 참조를 포함하여 모듈이 소유하는 모든 상태는 특정 모듈 인스턴스에 국한되어야 합니다. 자세한 내용과 실질적인 가이드는 확장 격리 방법론 를 참조하십시오.

이러한 문제를 피하는 더 간단한 방법은 :ref:`반복 초기화 시 오류 발생 <isolating-extensions-optout>`을 수행하는 것입니다.

모든 모듈은 서브 인터프리터 <sub-interpreter-support>`를 지원해야 하거나, 그렇지 않다면 명시적으로 지원하지 않음을 알리는 것이 기대됩니다. 이는 일반적으로 격리 또는 위에서 설명한 반복 초기화 차단 방식에 의해 달성됩니다. 모듈은 또한 :c:data:`Py_mod_multiple_interpreters 슬롯을 사용하여 메인 인터프리터로 제한될 수 있습니다.

PyInit 함수¶

버전 3.15부터 약하게 폐지 <Soft deprecated>: 이 기능은 새로운 기능이 추가되지는 않을 것이지만, 제거할 계획은 없습니다.

PyModExport_modulename() 대신, 확장 모듈은 다음과 같은 서명을 가진 이전 방식의 :dfn:`초기화 함수`를 정의할 수 있습니다:

PyObject *PyInit_modulename(void)¶

이름은 모듈의 이름으로 <name> 을 대체한 PyInit_{<name>}`이어야 합니다. 비-ASCII 모듈 이름의 경우, 대신 PyInitU_{<name>}` 을 사용하며, 이때 <name> 은 내보내기 훅 의 이름과 동일한 방식으로 인코딩됩니다 (즉, 밑줄이 있는 Punycode 사용).

모듈이 PyInit_<name>`와 :samp:`PyModExport_<name> 둘 다를 내보내는 경우, PyInit_<name> 함수는 무시됩니다.

:c:macro:`PyMODEXPORT_FUNC`와 마찬가지로, 헬퍼 매크로를 사용하여 초기화 함수를 정의하는 것이 권장됩니다:

PyMODINIT_FUNC¶

확장 모듈 초기화 함수를 선언합니다. 이 매크로는:

  • PyObject* 반환 유형을 지정하고,

  • 플랫폼이 요구하는 모든 특수 링크 선언을 추가하며,

  • C++의 경우, 함수를 extern "C" 로 선언합니다.

정상적으로, 초기화 함수(PyInit_modulename)는 비어 있지 않은 m_slots 를 가진 PyModuleDef 인스턴스를 반환합니다. 이는 Python이 다단계 초기화 를 사용할 수 있도록 합니다.

반환되기 전에, PyModuleDef 인스턴스는 다음 함수를 사용하여 초기화되어야 합니다:

PyObject *PyModuleDef_Init(PyModuleDef *def)¶
상의 안정 ABI 버전 3.5 이후로.

모듈 정의가 유형과 참조 횟수를 올바르게 보고하는 올바르게 초기화된 Python 객체임을 보장합니다.

def 를 PyObject* 로 캐스트하거나, 에러가 발생하면 NULL 을 반환합니다.

이 함수 호출은 모듈 초기화 함수에서 :c:type:`PyModuleDef`를 반환하기 전에 필요합니다. 다른 상황에서는 사용해서는 안 됩니다.

Python은 PyModuleDef 구조체가 정적으로 할당된다고 가정합니다. 이 함수는 새로운 참조 또는 빌려온 참조 중 하나를 반환할 수 있으며, 이 참조는 해제해서는 안 됩니다.

Added in version 3.5.

예를 들어, spam 이라는 모듈은 다음과 같이 정의됩니다:

static struct PyModuleDef spam_module = {
    .m_base = PyModuleDef_HEAD_INIT,
    .m_name = "spam",
    ...
};

PyMODINIT_FUNC
PyInit_spam(void)
{
    return PyModuleDef_Init(&spam_module);
}

레거시 단단계 초기화¶

버전 3.15부터 약하게 폐지 <Soft deprecated>: 단단계 초기화는 알려진 단점과 설계 결함이 있는 확장 모듈 초기화의 레거시 메커니즘입니다. 확장 모듈 작성자는 대신 다단계 초기화를 사용할 것이 권장됩니다.

그러나 이를 지원하는 것을 제거할 계획은 없습니다.

단단계 초기화에서는, 이전 방식의 초기화 함수 (PyInit_modulename)가 모듈 객체를 생성, 채우고 반환해야 합니다. 이는 일반적으로 PyModule_Create() 및 :c:func:`PyModule_AddObjectRef`와 같은 함수를 사용하여 수행됩니다.

Single-phase initialization differs from the default in the following ways:

  • 단단계 모듈은, 또는 오히려 포함 하고 있는, “싱글톤”입니다.

    모듈이 처음 초기화될 때, Python은 모듈의 __dict__ 내용(즉, 일반적으로 모듈의 함수와 타입)을 저장합니다.

    후속 임포트의 경우, Python은 초기화 함수를 다시 호출하지 않습니다. 대신, 새 __dict__ 를 가진 새 모듈 객체를 생성하고 저장된 내용을 복사합니다. 예를 들어, 함수 sum 과 예외 클래스 error 를 정의하는 단단계 모듈 _testsinglephase [1] 이 있다고 가정해 보겠습니다.

    >>> import sys
>>> import _testsinglephase as one
>>> del sys.modules['_testsinglephase']
>>> import _testsinglephase as two
>>> one is two
False
>>> one.__dict__ is two.__dict__
False
>>> one.sum is two.sum
True
>>> one.error is two.error
True

    정확한 동작은 CPython 구현 상세로 간주되어야 합니다.

  • PyInit_modulename``이 *spec* 인수를 받지 않는 문제를 우회하기 위해, 임포트 메커니즘의 일부 상태가 저장되어 ``PyInit_modulename 호출 중에 생성되는 첫 번째 적절한 모듈에 적용됩니다. 구체적으로, 서브 모듈이 임포트될 때 이 메커니즘은 부모 패키지 이름이 모듈 이름 앞에 붙여집니다.

    단일 단계인 PyInit_modulename 함수는 가능한 한 빨리 “자신이”의 모듈 객체를, 다른 모듈 객체가 생성되기 전에 생성해야 합니다.

  • 비 ASCII 모듈 이름 (PyInitU_modulename) 은 지원되지 않습니다.

  • 단일 단계 모듈은 :c:func:`PyState_FindModule`와 같은 모듈 검색 함수를 지원합니다.

  • 모듈의 :c:member:`PyModuleDef.m_slots`는 NULL이어야 합니다.

[1]

_testsinglephase 은 CPython의 자체 테스트 스위트에서 사용되는 내부 모듈이며, 사용자 설치에 포함되어 있을 수도 있고 아닐 수도 있습니다.
